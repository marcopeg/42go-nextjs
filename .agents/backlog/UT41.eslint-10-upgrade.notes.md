## Problems Encountered

- None so far.
- Baseline includes known pre-existing warnings:
	- `@typescript-eslint/typescript-estree@8.55.0` warns TypeScript 6 is outside official support range.
	- Next.js build logs one existing NFT tracing warning for `next.config.ts` import trace.
	- Repeated `Cannot access next/headers, falling back to environment matching only` lines during build output.
- Hard blocker during ESLint 10 validation:
	- `yarn lint` and `yarn qa` fail with `TypeError: Error while loading rule 'react/display-name': contextOrFilename.getFilename is not a function`.
	- Failure occurs while linting `eslint.config.mjs` and originates from `eslint-plugin-react` pulled via `eslint-config-next`.
- Follow-up investigation confirmed:
	- The crash also reproduces on real source files and Knex migration files, not only root config files.
	- Latest npm metadata still shows no ESLint 10 peer support for `eslint-plugin-react`, `eslint-plugin-import`, or `eslint-plugin-jsx-a11y`.
	- `typescript-eslint` latest does support ESLint 10 and TypeScript 6, but `eslint-config-next` currently installs older transitive versions.
	- ESLint's official `@eslint/compat` package is specifically designed to wrap older plugin/config rule APIs for ESLint v9/v10.

- Current attempt status:
	- Added `@eslint/compat` and wrapped the imported Next flat configs with `fixupConfigRules()`.
	- `yarn lint` now passes successfully on `eslint@10.2.1`.
	- `yarn qa` also passes successfully on `eslint@10.2.1` with the compat-wrapped config.
	- No source-level lint violations required code fixes once the compat wrapper was in place.
	- Those repo-file changes were then reverted so the branch returns to the ESLint 9 baseline while UT41 stays parked.
	- After the rollback, `/opt/homebrew/bin/npm install && /opt/homebrew/bin/yarn qa` passed and reproduced the pre-existing baseline warnings only.
