# Update Dependencies

**TaskID**: UG00
**Status**: ready

## Elevator's Pitch

Research current dependency versions, identify available updates (patch, minor, major), review changelogs and migration guides for breaking changes, produce an upgrade plan grouped by risk level, and execute the upgrades once the plan is approved.

## Business Gain

Keeps the framework on supported, patched versions. Reduces security exposure from known CVEs in outdated packages. Prevents accumulation of upgrade debt that makes future updates harder. Ensures compatibility with ecosystem tooling and documentation.

## Current State

Last full dependency upgrade was completed in AQE (2026-02-16), which brought the project to Next.js 16.1.6, React 19.2.4, and other major bumps. Current `package.json` pins are a mix of exact versions and caret ranges. No automated dependency monitoring is in place.

Current key versions:

- `next`: 16.1.6
- `react` / `react-dom`: 19.2.4
- `next-auth`: ^4.24.11
- `knex`: ^3.1.0
- `pg`: ^8.16.3
- `zod`: ^4.0.17
- `tailwindcss`: ^4
- `eslint`: ^9.39.2
- `typescript`: ^5

## Desired State

All dependencies updated to their latest stable, compatible versions. A documented upgrade plan that:

1. Lists every dependency with current version, latest available version, and change type (patch/minor/major)
2. Groups updates by risk: safe (patch/minor, no breaking), moderate (minor with behavioral changes), risky (major with breaking changes)
3. Notes breaking changes and required migration steps for each risky update
4. Defines an execution order (safe first, then moderate, then risky one-by-one)
5. Includes a validation checklist after each group

## Definition of Success

- Upgrade plan reviewed and approved by the operator before execution begins
- All dependencies updated to latest compatible versions
- `npm run qa` passes (lint + build)
- No runtime regressions in dev mode
- No new security vulnerabilities introduced
- Any intentionally skipped upgrades are documented with rationale

## Additional Context

Key areas to research during planning:

- **next-auth**: v4 → v5 migration is a major change (Auth.js rewrite). Check if v5 is stable and compatible with the current multi-app auth architecture.
- **knex**: check for major version bumps and PostgreSQL driver compatibility with `pg`.
- **@dnd-kit**: recently had major version jumps (v6/v10). Check stability.
- **zod**: v4 is relatively new. Check for patches and ecosystem compatibility.
- **tailwindcss**: v4 is recent. Check for minor/patch updates.
- **React/Next.js**: check if 16.x has newer patches or if 17.x is on the horizon.
- **OCI SDKs** (`oci-objectstorage`, `oci-secrets`): check for updates and deprecations.
- **Azure SDKs** (`@azure/app-configuration`, `@azure/keyvault-secrets`): check for updates.

Tools to use:

```bash
npm outdated                    # Show current vs latest
npm audit                       # Check for known vulnerabilities
npx npm-check-updates           # Detailed upgrade suggestions
```

## Assumptions

- The project uses `npm` as package manager (not yarn/pnpm).
- Caret ranges in package.json mean some minor/patch updates may already be installed via lockfile without changing package.json.
- The operator wants to review the plan before any upgrades are applied.

## Constraints

- Do not upgrade to versions that break compatibility with the current Next.js 16 / React 19 stack unless the operator explicitly approves.
- **next-auth v4 → v5**: research the migration scope and present findings. The operator will decide after seeing the research whether to include it here or spin off a separate task.
- **OCI/Azure SDKs**: check actual codebase usage during research. If unused, propose removal as part of the upgrade plan. If used, upgrade to latest.
- **Version pinning**: after all upgrades are complete, convert all caret/tilde ranges to exact pins for deterministic builds. Update the `overrides` section accordingly.
- Run `npm run qa` after each upgrade group to catch breakage early.
- Do not modify application code during the research phase — only during approved execution.

## Dos

- Research changelogs and migration guides before upgrading major versions
- Upgrade in small batches, validate after each
- Keep the lockfile in sync (`npm install` after changes)
- Document any workarounds or overrides needed (like the existing `@types/react` override)

## Don'ts

- Don't blindly `npm update` everything at once
- Don't upgrade next-auth to v5 without presenting research and getting explicit approval
- Don't remove OCI/Azure SDKs without verifying they're unused in the codebase
- Don't remove the `overrides` section without verifying type compatibility
- Don't skip `npm run qa` between upgrade batches
