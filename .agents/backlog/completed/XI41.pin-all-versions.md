# Pin All Dependency Versions

**TaskID**: XI41
**Status**: completed
**Order**: ⑥ — dead last, after every other dependency task
**Workflow**: `aexecute` completed — manifest pinning landed on the validated dependency baseline

## Elevator's Pitch

After all dependency updates are complete, convert every `^` and `~` range in package.json to exact version pins for deterministic builds. Update the `overrides` section to match.

## Business Gain

Deterministic builds. No surprise version bumps from semver ranges. Every `npm install` produces identical node_modules.

## Current State

Mix of exact pins (react, react-dom, next) and caret ranges (everything else) in package.json.

## Desired State

- Every dependency in package.json uses an exact version (no `^`, `~`, or `*`)
- `overrides` section matches pinned versions
- `npm run qa` passes

## Definition of Success

- Zero semver ranges in package.json
- `npm run qa` passes

## Additional Context

Run after all other update tasks are complete. This is a pure lockdown step — no version changes, just removing the `^`/`~` prefixes so future `npm install` doesn't silently bump anything.

Current `package.json` has 37 caret-ranged entries across `dependencies` and `devDependencies`. The `overrides` block already uses exact pins (`@types/react`, `@types/react-dom`); it should be kept in sync after pinning.

Execution decision on 2026-04-21: proceed on the current ESLint 9 baseline even though UT41 remains paused. If UT41 eventually lands an ESLint 10 upgrade later, XI41 must be re-verified or re-run afterward.

## Assumptions

- All other dependency update tasks (PE62, DQ53, VD22, KB55, YW51, WP85, QQ22, UT41) are complete before this runs, so the installed versions in `package-lock.json` reflect the intended target versions.
- No semver range is intentionally kept loose — every `^` and `~` is removed without exception, including devDependencies.
- Running `npm install` after editing `package.json` is safe and will only update the lockfile metadata; it will not bump any actual package version because the resolved versions in the existing lockfile already match.
- The `overrides` block contains entries that are already exact; they stay exact and need no changes unless a new override is needed.

## Constraints

- Must be the final dependency task
- All other update tasks must be complete first

## Dos

- Edit `package.json` directly — strip `^` and `~` from every version string in `dependencies`, `devDependencies`, and `overrides`.
- Run `npm install` after editing to regenerate `package-lock.json`.
- Run `yarn qa` to confirm the build passes.
- Verify with `grep` that no `^` or `~` remains in `package.json`.

## Don'ts

- Do NOT change any version numbers — only remove the range prefix.
- Do NOT add or remove packages.
- Do NOT touch files other than `package.json` and `package-lock.json` (regenerated automatically).
- Do NOT pin `package.json` `version` field or `engines` fields (if added in future).
