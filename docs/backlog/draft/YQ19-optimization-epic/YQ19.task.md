# optimization epic

## Elevator's Pitch

Coordinate the performance optimization tasks in the intended execution order and keep the parent epic updated as each subtask is refined, planned, executed, and completed.

## Business Gain

Keeps the performance work coherent, reduces accidental sequencing mistakes, and gives operators one control point for tracking the optimization effort.

## Current State

Five optimization subtasks exist as independent drafts, but their recommended order and cross-task tracking rules are not captured in a parent task.

## Desired State

This epic lists the subtasks in execution order and requires every subtask refinement or execution pass to check and update the epic.

## Definition of Success

Agents can open this epic and immediately see the correct task order, the subtask references, and the rule that the epic must be updated while work progresses.

## Additional Context

- Execution order is based on the performance analysis: quick low-risk wins first, session cleanup before reader split, and the largest reader cache split last.
- This is a coordination task, not a direct implementation task.
- Execution order:
  1. [DV28: optimize: public asset cache headers](../DV28-optimize-public-asset-cache-headers/DV28.task.md)
  2. [AT01: optimize: event partition preparation](../AT01-optimize-event-partition-preparation/AT01.task.md)
  3. [CG67: optimize: api session loading](../CG67-optimize-api-session-loading/CG67.task.md)
  4. [AC03: optimize: root profile guard](../AC03-optimize-root-profile-guard/AC03.task.md)
  5. [IJ10: optimize: reader cache split](../IJ10-optimize-reader-cache-split/IJ10.task.md)
- Whenever any listed subtask is refined, planned, executed, blocked, completed, or reordered, update this epic in the same work session.

## Assumptions

- The subtasks remain scoped to the performance optimizations drafted from the serving analysis.
- Refinement may discover reasons to adjust order, but any change must be recorded in this epic.

## Constraints

- Do not execute implementation work directly from the epic unless the operator explicitly asks for an epic-level execution pass.
- Keep each implementation change inside the relevant subtask.

## Acceptance Criteria

- The epic lists the optimization subtasks in the recommended execution order.
- Each optimization subtask references this epic.
- Each optimization subtask tells agents to follow the epic order when refining or working on that task.
- Each optimization subtask tells agents to update the epic whenever the subtask is refined, planned, executed, blocked, completed, or reordered.

## Dos

- Use this order unless refinement records a better reason to change it: DV28, AT01, CG67, AC03, IJ10.
- Update this epic when a subtask changes status, scope, ordering, or implementation notes.
- Keep cross-task dependencies visible here.

## Don'ts

- Do not start IJ10 before CG67 unless this epic is updated with the reason.
- Do not let subtask plans drift away from the epic order silently.

## Open Questions

- Should the epic move to planned only after all five subtasks are refined, or can it remain as a tracking draft until execution starts?
