# Choose Runtime-Friendly Remote Image Handling

## Elevator's Pitch

The LingoCafe production landing page hit a remote image failure even though the asset was publicly available at https://assets.lingocafe.app/mkt/s2.landing-side-540.webp. The failure came from Next.js image optimization rejecting the remote URL because `images.remotePatterns` is configured through `CONTENT_IMAGE_REMOTE_PATTERNS` in `next.config.ts`, which is evaluated during `next build`. A temporary workaround set the LingoCafe landing image to `unoptimized: true`, but the project needs a deliberate long-term policy for remote content images.

## Business Gain

Deployments should not fail or require confusing build-time public asset configuration when the app only needs to display a stable external image. The final policy should make production behavior predictable, keep Docker images practical to promote, and avoid turning public CDN hostnames into secret-like deployment knobs.

## Current State

The Image ContentBlock renders through `next/image`. Remote image optimization requires `images.remotePatterns` in `next.config.ts`. This repo currently derives those patterns from `CONTENT_IMAGE_REMOTE_PATTERNS`, so the allowlist is frozen into the built standalone app. If the production image was built without `https://assets.lingocafe.app/**`, `/_next/image?url=...` returns an error such as `"url" parameter is not allowed`. The LingoCafe home page now temporarily sets `image.unoptimized: true` for the marketing side image, causing the browser to request the remote URL directly.

## Desired State

The repository has a documented, intentional strategy for remote content images. The strategy should explain when to use optimized Next images, when to render direct remote images with `unoptimized: true`, and how deployment configuration should work for stable public asset hosts.

## Acceptance Criteria

- The final approach prevents the LingoCafe marketing image from failing when `CONTENT_IMAGE_REMOTE_PATTERNS` is absent at runtime.
- The chosen policy is documented near the Image ContentBlock or deployment environment documentation.
- The implementation avoids requiring runtime-only deploy configuration for values that Next.js must know at build time.
- If optimized remote images remain supported, the required build-time behavior is explicit and hard to miss.
- The temporary `unoptimized: true` workaround is either confirmed as the preferred policy for this image class or replaced by a stronger implementation.
