# optimize: reader cache split

## Elevator's Pitch

Split cacheable LingoCafe book/page content from private user progress so static reader data can be cached instead of regenerated for every user request.

## Business Gain

Lets many users read the same books without forcing the server and database to rebuild identical content responses every time.

## Current State

Reader endpoints return no-store responses and combine static book content with profile, progress, and event side effects.

## Desired State

Book metadata, page content, page lists, previous/next links, and cover data are available through cacheable content endpoints. User-specific progress/profile state stays in private no-store endpoints.

## Definition of Success

Opening a reader page can reuse cached book/page content while fetching user progress separately.

## Additional Context

- Performance analysis found LingoCafe reader JSON uses Cache-Control: no-store for all payloads.
- GET /api/lingocafe/books/[bookId]/pages/[pageId] currently reads content, reads profile, saves progress, and records events.

- When refining or working on this task, confirm the current order in YQ19 and keep following that epic unless the epic is updated with a reason.
- Whenever this task is refined, planned, executed, blocked, completed, or reordered, update YQ19 in the same work session.

## Assumptions

- Book/page content changes infrequently compared to progress.
- Per-user progress must remain private and no-store.

## Constraints

- Do not cache user progress, profile, roles, grants, or consent payloads in public caches.
- Keep server-side validation for all private state changes.

## Acceptance Criteria

- Static book/page content endpoint uses public cache headers or Next cache primitives with an explicit revalidation policy.
- Progress/profile endpoint remains no-store.
- Reader UI composes content and progress without user-visible regressions.
- Production build succeeds.

## Dos

- Keep content and private state payloads clearly separated.
- Document cache invalidation behavior for book content updates.

## Don'ts

- Do not make existing private no-store endpoints public by accident.
- Do not write progress or events from a cacheable content GET.

## Open Questions

- What revalidation window is acceptable for book/page content?
- Should cache invalidation be manual, time-based, or tied to book import/update flows?
