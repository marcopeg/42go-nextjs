# optimize: public asset cache headers

## Elevator's Pitch

Add stronger cache headers for version-stable public app themes and icons.

## Business Gain

Reduces repeated requests for static theme and icon assets, improving first-load and repeat-load behavior without increasing server work.

## Current State

Next hashed chunks and fonts cache immutably, but /app-themes/* and /app-icons/* static files currently return public max-age=0.

## Desired State

App theme and icon assets have explicit long-lived cache headers where safe, with a clear cache-busting strategy for changes.

## Definition of Success

Requests to app theme and icon assets return cache headers that allow browser and CDN reuse.

## Additional Context

- Header probe found /app-themes/lingocafe/style.css and /app-icons/lingocafe/favicon-32x32.png use max-age=0.
- Next hashed _next/static assets already return max-age=31536000, immutable.

- When refining or working on this task, confirm the current order in YQ19 and keep following that epic unless the epic is updated with a reason.
- Whenever this task is refined, planned, executed, blocked, completed, or reordered, update YQ19 in the same work session.

## Assumptions

- Theme and icon asset filenames are stable today.
- Long-lived caching needs either filename versioning, query versioning, or conservative max-age.

## Constraints

- Do not cache HTML, private API responses, or auth responses with public headers.
- Avoid cache rules that make theme/icon updates impossible to roll out.

## Acceptance Criteria

- next.config.ts defines targeted cache headers for /app-themes/:path* and /app-icons/:path* or an equivalent production-safe mechanism.
- Cache strategy is documented in code comments or project docs.
- Header checks show stronger cache directives for theme/icon assets.
- Production build succeeds.

## Dos

- Keep cache rules narrow.
- Choose a cache-busting approach before using immutable on stable public paths.

## Don'ts

- Do not set long public caching on dynamic manifest output unless app-specific variation is handled safely.
- Do not cache user-specific endpoints.

## Open Questions

- Should theme/icon URLs include the package version or build ID to support immutable caching?
