# Docker Release Quality and Hardening v2

## Elevator's Pitch

Improve the Docker release workflow after TK89 proved the current image is safe to go live. This task should turn the remaining non-blocking security, quality, and operational hardening items into a reliable repeatable release posture.

## Business Gain

Future releases should be easier to trust: the team can verify the exact published image, keep Docker Hub usage intentional, and avoid slow drift in image quality without blocking day-to-day delivery.

## Current State

- TK89 completed the Docker image security review and removed launch blockers.
- The current image passed a fresh full security check with zero blocking findings.
- The final runtime image has no Trivy image vulnerabilities, no source maps, no env files, no project dumps, no docs/backlog files, no `.agents`, and no baked `contents`.
- `docker-compose.prod.yml` is explicitly local production only and is not the real production deployment model.
- Local dump artifacts remain a deferred backup-system concern and are not a Docker image blocker while they stay ignored and local.
- The remaining work is quality and release hardening: remote digest verification, publish gate design, SBOM/provenance/signing policy, health-check ownership, base-image policy, and optional runtime hardening.

## Desired State

- Docker release quality checks can verify both local images and the exact remote image digest that was published or deployed.
- The publish/deploy safety gate blocks only meaningful release risks: critical/high image vulnerabilities and final-image leaks.
- Docker Hub usage is documented as either private transport or public distribution, with matching SBOM/provenance/signing expectations.
- `latest` tag movement has a written rule.
- Health-check ownership is explicit: deployment platform, Dockerfile, Compose, or a documented combination.
- Base-image update strategy is documented: stay on `node:26-alpine`, pin digests, or rely on scheduled rebuilds and vulnerability scans.
- Host-supported runtime hardening is evaluated without forcing features the current VPS/platform cannot support.

## Definition of Success

- A follow-up quality review decides which hardening items are worth implementing now and which stay documented as policy.
- The release workflow can scan a remote image digest, not just local `42go-next:latest`.
- The security check/reporting flow distinguishes launch blockers from local-only/deferred warnings.
- Relevant Make targets or docs are updated for the chosen publish/deploy policy.
- The task produces clear operator instructions for future releases.

## Additional Context

- This is not a launch blocker. TK89 concluded there is no Docker image security blocker left for the current VPS go-live.
- Do not fold backup/dump relocation into this task unless the operator explicitly expands scope; that belongs to a separate backup-system review.
- Do not switch to distroless, Chainguard, or another major base-image family unless the task explicitly chooses that migration after compatibility testing.
- The current local assessment report showed 6 non-blocking findings: local dumps, local-production Compose warnings, and no final image leaks.

## Assumptions

- The project will keep using Docker Hub for image publication unless a future task changes the registry strategy.
- The current VPS/platform can already run the image successfully.
- The team prefers practical checks that catch real release risk over gates that block on local-only development warnings.

## Constraints

- Do not paste secret values or raw Trivy secret payloads into backlog notes.
- Keep `docker-compose.prod.yml` classified as local production only.
- Avoid blocking publish/deploy on local dump findings while they are Docker-ignored and local-only.
- Do not make the release workflow substantially slower without a clear safety gain.

## Acceptance Criteria

- [ ] Define whether Docker Hub is private transport or public distribution for this project.
- [ ] Add or document a remote-image digest scan flow for the exact pushed image.
- [ ] Decide and implement the publish gate severity policy: block critical/high runtime image vulnerabilities and final-image leaks; treat local-only findings as warnings.
- [ ] Decide whether SBOM generation is required and implement or document the chosen policy.
- [ ] Decide whether provenance/attestation/signing are required and implement or document the chosen policy.
- [ ] Define when the `latest` tag may move.
- [ ] Decide health-check ownership and update Dockerfile, Compose, deployment docs, or platform notes accordingly.
- [ ] Decide base-image policy: floating `node:26-alpine`, digest pinning, or scheduled rebuilds with vulnerability scans.
- [ ] Evaluate optional runtime hardening supported by the deployment platform: read-only filesystem, dropped capabilities, `no-new-privileges`, and writable temp/cache paths.
- [ ] Run `npm run qa` and the Docker security check after any workflow changes.

## Dos

- Keep the task focused on Docker release quality, safety, and performance.
- Prefer small Makefile/script/doc improvements over heavyweight release ceremony.
- Keep security reports sanitized.
- Use TK89 as the baseline for what is already solved.

## Don'ts

- Do not reopen TK89 launch blockers unless new evidence appears.
- Do not treat local-production Compose warnings as real production failures.
- Do not mix backup-system relocation into this task.
- Do not adopt a harder base image without proving Next.js standalone and deployment compatibility.

## Open Questions

- Is Docker Hub only a private transport for this VPS, or should the image be safe for third-party consumption?
- Should published images be signed now, or is SBOM plus digest scan enough for this stage?
- Does the deployment platform already provide health checks and restarts?
- Should the base image be digest-pinned, or should freshness come from scheduled rebuilds?
