# optimize: api session loading

## Elevator's Pitch

Avoid loading the same NextAuth session twice in protected API requests.

## Business Gain

Cuts avoidable auth work from every protected API call, reducing latency and server pressure under concurrent usage.

## Current State

protectRoute() evaluates the policy and loads the session for session-required routes. LingoCafe handlers then call getSessionUserId(), which loads the session again.

## Desired State

Protected handlers can reuse the session/user context resolved by policy evaluation or use a shared per-request helper that memoizes session lookup.

## Definition of Success

A protected LingoCafe API request resolves session identity once per request path.

## Additional Context

- Performance analysis found duplicate session loading in protectRoute() and LingoCafe reader handlers.
- This affects /api/lingocafe/books and reader page endpoints.
- This task is part of [YQ19: optimization epic](../YQ19-optimization-epic/YQ19.task.md), execution step 3 of 5.
- When refining or working on this task, confirm the current order in YQ19 and keep following that epic unless the epic is updated with a reason.
- Whenever this task is refined, planned, executed, blocked, completed, or reordered, update YQ19 in the same work session.

## Assumptions

- NextAuth session lookup is safe to memoize within a single request.
- Handlers still need a trusted user ID after policy passes.

## Constraints

- Do not expose session internals to client payloads.
- Do not weaken protectRoute() error semantics.

## Acceptance Criteria

- Protected route handlers have a supported way to access the already-resolved session user ID.
- LingoCafe reader routes no longer call a second getServerSession() after protectRoute() has already required session.
- 401/403/404 policy behavior remains unchanged.
- Production build succeeds.

## Dos

- Prefer a generic 42go policy/auth helper if it benefits all apps.
- Keep handler APIs readable.

## Don'ts

- Do not cache session data across requests.
- Do not bypass app-scoped user validation.

## Open Questions

- Should protectRoute pass context into handlers, or should getSessionUserId() use request-scoped memoization?
