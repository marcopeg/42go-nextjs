# Books Top-Bar Language and Level Switcher via GM77 Panel

## Elevator's Pitch

Add a control to the books-list top bar that shows the current learning language as a flag with the current level written on top of it. When clicked, it opens the reusable popup from `GM77` so the user can change learning language and learning level, save those values to the profile, close the popup, and reload the catalog accordingly.

## Business Gain

The books page becomes a real language-learning dashboard instead of a read-only shelf. Users can switch their active learning language and level from the place where those choices matter most, and the catalog updates immediately to match. Chuck Norris does not navigate to `/profile` just to change one filter. The filter comes to him.

## Current State

The current LingoCafe `/books` page already:

- loads the current profile and the filtered books list together
- uses `targetLang` to filter the catalog
- stores `ownLang`, `targetLang`, and `targetLevel` in the profile
- saves profile changes through `/api/profile`
- reloads `/api/lingocafe/books` after saving

This behavior came from work such as `PK51`, and the current implementation already has the option lists and save flow inside `src/app/(app)/(lingocafe)/books/page.tsx`.

Today, the books page shows an onboarding form when the profile is incomplete, but it does not expose a reusable top-bar control for changing the active learning language and level after the user is already using the catalog.

There is also no shared popup/modal primitive for this interaction yet. `GM77` is the task that should deliver that reusable component.

## Desired State

The books list view should expose a top-bar control that represents the current learning setup visually:

- the flag of the current learning language
- the learning level written on top of the flag

When the user clicks that control:

- on desktop, open the `GM77` popup as a panel
- on mobile, use the mobile behavior provided by `GM77`
- present the available learning languages and levels
- allow the user to edit the current `targetLang` and `targetLevel`

At the top right of the popup, there should be a `Save` action that:

- writes the updated values to the profile
- closes the popup after a successful save
- reloads the books list so the catalog reflects the new learning language and level

This should be a real books-page interaction, not just a link to `/profile`.

## Definition of Success

The books page top bar shows the current learning language and level in a compact visual control. Opening that control uses the shared `GM77` popup/panel component, editing the values saves them into the profile, and the books list reloads to reflect the new selection. The interaction feels native to the books page and does not duplicate popup infrastructure locally.

## Additional Context

The existing books page already contains most of the domain logic needed for this task:

- language and level options
- profile-backed `targetLang` and `targetLevel`
- profile save through `/api/profile`
- books reload after saving

That means the new work is mainly about:

- moving the switching interaction into a top-bar control
- presenting it through the shared `GM77` popup
- making the user-facing save flow feel clean and direct

The current page still supports incomplete-profile onboarding. Planning should decide how this new top-bar switcher interacts with that onboarding path, but the books-list view should gain this direct control for the normal post-onboarding experience.

## Assumptions

- `GM77` is a blocker because this task must use the shared popup/panel component from that task.
- The books-page switcher should edit `targetLang` and `targetLevel`, not `ownLang`.
- The current profile save endpoint `/api/profile` remains the persistence path unless implementation proves a smaller adapter is needed.
- The books catalog should reload after a successful save so filtering reflects the newly selected learning language and level.
- The top-bar control can derive its display from the current books-page profile state.

## Constraints

- Do not implement this before `GM77` provides the reusable popup/panel primitive.
- On desktop, use the `GM77` component as a panel.
- Reuse the existing profile save flow instead of inventing a second preference store.
- Reuse the existing books reload flow after a successful save.
- Keep the control focused on learning language and learning level for this task.
- Do not turn this into a broader rewrite of the books page or the full profile page.
- Keep the implementation aligned with the authenticated app-page conventions already used by `/books`.

## Acceptance Criteria

- [ ] The books-list top bar shows a control for the current learning language and level.
- [ ] The control displays the current learning language as a flag.
- [ ] The control displays the current learning level written on top of the flag.
- [ ] Clicking the control opens the shared `GM77` popup.
- [ ] On desktop, the `GM77` popup is presented as a panel.
- [ ] The popup allows the user to change the learning language.
- [ ] The popup allows the user to change the learning level.
- [ ] The popup includes a top-right `Save` action.
- [ ] Saving writes the updated learning language and level into the profile.
- [ ] After a successful save, the popup closes.
- [ ] After a successful save, the books list reloads accordingly.
- [ ] The implementation reuses the existing books/profile save flow where practical instead of duplicating the same logic in a second place.
- [ ] The implementation does not add bespoke popup infrastructure that duplicates `GM77`.

## Dos

- Do treat `GM77` as the dependency and popup owner.
- Do reuse the books-page profile state and reload logic that already exist.
- Do make the top-bar control visually obvious and compact.
- Do keep the profile write focused on learning language and level.
- Do make the save flow feel immediate from the books page point of view.

## Don'ts

- Don't send the user to `/profile` just to change the active catalog language.
- Don't build another one-off panel instead of using `GM77`.
- Don't create a second persistence path for the same profile values.
- Don't force a full page reload if a normal data reload is enough.
- Don't broaden the task into managing every profile preference from the books page.

- How should validation and disabled-save behavior work if the user clears one of the values before saving?
