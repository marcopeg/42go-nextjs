# Validate Translation Requests Against Book Page Content

## Elevator's Pitch

Harden the LingoCafe translation backend so every translation request is tied to a specific book page, verified against that page's content, and rejected when it exceeds a strict word-count cap.

## Business Gain

Translation should only operate on text the user is actually reading. Server-side page validation prevents arbitrary freeform translation abuse, reduces provider-cost exposure, and makes the endpoint safer even if a client is modified.

## Current State

The reader client sends `text`, `from`, `to`, `bookId`, `pageId`, and `sentenceId` to `/api/lingocafe/translate`. The backend schema already accepts `bookId` and `pageId`, but they are optional and are only used for event metadata. The backend currently authorizes the user, then calls `translateText(parsed.data)` without loading the page or verifying that the requested text appears in that page. The backend caps `text` at 5000 characters, but there is no hard word-count cap.

## Desired State

The translation endpoint requires `bookId` and `pageId`, loads that exact page on the server, and verifies that the requested translation text exists in the page before checking caches or calling the provider. The page-content check should include all rendered translatable page surfaces: title, summary, and markdown body content. The endpoint also enforces a hard maximum number of words for any translation request and rejects oversized input before cache lookup or provider calls.

## Definition of Success

A modified client cannot use `/api/lingocafe/translate` to translate arbitrary text. Valid reader sentences from the requested page still translate normally. Text that is not found in the requested page, missing page identifiers, unknown pages, or oversized word counts fail with clear client-safe errors.

## Assumptions

- The safety check should use `bookId` and `pageId` from the request body rather than deriving them from session or referrer.
- The source text may come from page title, page summary, or page body content because all three are rendered with sentence wrappers in the reader.
- The word cap value is not finalized yet and should be refined; implementation may start with a named constant or environment-backed default.
- Whitespace and markdown formatting may need normalization so a rendered sentence can match stored markdown content reliably.

## Constraints

- Do not rely on the browser to enforce page membership.
- Do not call Google Translate for text that fails page membership or word-count validation.
- Do not let missing `bookId` or `pageId` fall back to the current optional behavior.
- Do not leak full page contents or sensitive validation details in error responses.
- Do not weaken the existing `feature_flags.translate` authorization check.
- Do not break translation cache behavior for valid sentences.

## Acceptance Criteria

- `/api/lingocafe/translate` requires `bookId` and `pageId` in the payload.
- The backend loads the requested page with `loadBookPage(bookId, pageId)` or equivalent server-side logic.
- Unknown or missing pages return a clear error before translation work begins.
- The backend checks that requested `text` exists in the requested page's title, summary, or body content after documented normalization.
- Text that does not exist in the requested page is rejected before cache lookup and before provider calls.
- The backend enforces a hard maximum word count for `text` before cache lookup and before provider calls.
- Oversized requests return a clear validation error.
- Valid requests for text that appears on the page continue to work through memory, database, and provider translation paths.
- Translation event logging still records `book_id`, `page_id`, and `sentence_id` for accepted requests.
- Tests or smoke checks cover missing page identifiers, unknown page, text not found on page, oversized text, and a valid page sentence.
- `npm run qa` passes after implementation.

## Dos

- Make `bookId` and `pageId` required in the Zod payload schema.
- Keep validation before translation cache/provider work so invalid requests do not create cache entries.
- Normalize text comparison intentionally and document the chosen normalization.
- Use a named constant for the word cap so the limit is easy to find and change.
- Keep successful response shape compatible with the current reader client.

## Don'ts

- Do not translate arbitrary text just because the user has `feature_flags.translate = true`.
- Do not use the existing 5000-character limit as the only hard cap.
- Do not silently truncate user text and translate the truncated version.
- Do not trust `sentenceId` as proof that text belongs to the page.
- Do not move this into client-only validation.

## Open Questions

- What exact word-count cap should be enforced?
- Should the cap be a hardcoded constant, an environment variable with default, or an AppConfig setting?
- Should page membership matching require exact normalized text, or allow sentence-boundary normalization that strips markdown punctuation artifacts?
- Should a text-not-found request return `400 validation`, `403 forbidden`, or another error code?
