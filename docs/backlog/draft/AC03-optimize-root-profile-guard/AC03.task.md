# optimize: root profile guard

## Elevator's Pitch

Remove request-time session and profile loading from the root layout so authenticated app shell entry stays cheap and scalable.

## Business Gain

Reduces server and database pressure on first page loads, refreshes, and deep links for LingoCafe users.

## Current State

The root layout resolves app info and, for LingoCafe's before-profile guard, calls getServerSession() and loadProfile() before rendering the HTML entry.

## Desired State

The root layout stays limited to lightweight app-shell concerns. Profile completion is resolved by client-side cache, a dedicated API, or a signed/versioned cookie that avoids per-request profile DB reads.

## Definition of Success

Authenticated entry to /books no longer performs profile DB loading from src/app/layout.tsx, while incomplete-profile users are still guided through onboarding.

## Additional Context

- Performance analysis found src/app/layout.tsx getInitialProfileComplete() as a per-request server hotspot.
- LingoCafe config enables app.profile.guard.slot = before.
- This task is part of [YQ19: optimization epic](../YQ19-optimization-epic/YQ19.task.md), execution step 4 of 5.
- When refining or working on this task, confirm the current order in YQ19 and keep following that epic unless the epic is updated with a reason.
- Whenever this task is refined, planned, executed, blocked, completed, or reordered, update YQ19 in the same work session.

## Assumptions

- Profile completion can be represented safely outside the root SSR pass.
- Any cookie-based shortcut must be invalidated when profile schema or required consent versions change.

## Constraints

- Do not weaken onboarding enforcement.
- Keep authenticated app pages client-rendered under AppLayout.

## Acceptance Criteria

- src/app/layout.tsx no longer imports or calls getServerSession(), getAuthOptions(), or loadProfile() for initial profile completion.
- LingoCafe onboarding still appears for authenticated users with incomplete profile or missing required consent.
- Production build succeeds.
- A refresh on /books does not issue a profile DB read from root layout code.

## Dos

- Prefer a narrow, measurable change.
- Keep the existing profile guard user experience.

## Don'ts

- Do not move generic app-shell logic into LingoCafe-specific route files unless needed.
- Do not trust client-only state as the only source of required consent truth.

## Open Questions

- Should completion be cached in a signed cookie, or should the client guard call /api/profile after hydration?
