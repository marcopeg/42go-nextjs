# Add AppLayout Profile Guard Before Slot

## Elevator's Pitch

Add a future AppLayout profile guard slot that can synchronously prevent normal app layout rendering and render guard content instead.

## Business Gain

Some authenticated app experiences need to block the entire shell before navigation chrome, page content, or layout effects render. A dedicated before-layout guard gives apps a clean configuration point for these hard stops without forcing route-by-route wrappers.

## Current State

YA00 introduces the first profile guard work around the layout overlay slot for LingoCafe onboarding. The broader guard model identifies a second conceptual slot, app.profile.guard.before, but YA00 intentionally does not implement it.

## Desired State

AppLayout supports a configured profile guard before slot that can run before normal layout rendering and return guard content when the app should not render the shell. The contract should compose with the existing profile guard model and avoid leaking app-specific behavior into non-target apps.

## Definition of Success

The repository has a clear synchronous pre-layout guard contract for app profile guards, documented enough for future app-specific blocking flows, and it coexists with the layout overlay guard from YA00.

## Additional Context

- Follow-up extracted from YA00 refinement.
- YA00 should implement only the layout overlay guard for LingoCafe onboarding.
- The before guard is expected to be synchronous, or at least able to block normal AppLayout rendering before the shell appears.
- The design should fit near app.profile.guard in AppConfig.

## Assumptions

- The before slot is a platform/AppLayout capability, not LingoCafe-specific onboarding work.
- The before slot may need a different contract from layout overlay guards because it prevents layout rendering instead of rendering above it.
- Multiple guard support can stay out of scope unless refinement chooses otherwise.

## Constraints

- Do not duplicate route wrappers across authenticated pages.
- Do not hardcode LingoCafe behavior into AppLayout.
- Do not break existing AppLayout policy handling, menus, mobile nav, toolbar, or footer behavior.
- Preserve multi-app isolation.

## Acceptance Criteria

- The task defines the exact contract for app.profile.guard.before before implementation.
- AppLayout can invoke a configured before guard before rendering the normal shell.
- The before guard can return null to allow normal layout rendering.
- The before guard can return content that replaces normal AppLayout rendering.
- The behavior is app-scoped and inactive for apps without a before guard.
- The implementation is documented in the relevant profile/AppLayout guard docs or task notes.
- npm run qa passes after implementation.

## Dos

- Reuse the guard concepts introduced by YA00 where they fit.
- Keep the contract small and typed.
- Prefer clear AppConfig shape over route-specific special cases.
- Study AppLayout policy behavior before choosing guard ordering.

## Don'ts

- Do not implement this inside YA00.
- Do not make every page import a wrapper manually.
- Do not couple before guards to books, onboarding, or any single LingoCafe route.
- Do not bypass authenticated page policy checks without explicitly documenting ordering.

## Open Questions

- Should before guards run before or after AppLayout policy evaluation?
- Should before guards support lazy loading, or must they be synchronous direct references?
- Should before guards support one guard or multiple ordered guards?
- What fallback UI should render while a before guard is unavailable or fails to load?
