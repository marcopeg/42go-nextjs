# Promote Events to Core App-Scoped Event System

## Elevator's Pitch

Move the existing LingoCafe-only event stream into a reusable 42go core events capability: an app-scoped PostgreSQL event schema, a server-side event writer, a client-side batching collector, core auth lifecycle event recording, and an analytics export path evolved from VI41.

## Business Gain

Every app in the multi-app platform needs product telemetry without rebuilding the same event plumbing. A core event system gives the platform one disciplined way to record signups, logins, logouts, profile changes, reader activity, and future product events while keeping each app's data compartmentalized. PostgreSQL stays the hot ingestion buffer. Local CSV and Parquet archives become the durable analytics source. Chuck Norris does not copy telemetry code between apps. Telemetry copies itself into core.

## Current State

- LingoCafe has a `lingocafe.events` table created by `knex/migrations/20260427150000_lingocafe_schema.js`.
- `lingocafe.events` is append-oriented, partitioned by `created_at`, uses composite primary key `(created_at, id)`, and stores `user_id`, `event_at`, `name`, optional `book_id`, optional `page_id`, `data`, and `meta`.
- `lingocafe.events_prepare_partitions()` creates monthly partitions under the `lingocafe` schema.
- The current writer is `trackReaderEvent()` in `src/app/api/(lingocafe)/lingocafe/_lib/reader.ts`, so non-LingoCafe features cannot use events without depending on LingoCafe internals.
- LingoCafe currently records events such as `book-info`, page events, `profile-update`, and `consent-update`.
- VI41 is in WIP and is designing a download-only LingoCafe export skill that reads `lingocafe.events` through `LC_EVENTS_DATABASE_URL` and writes paired CSV and Parquet batches to a gitignored local archive.
- The existing dump/backup work treats PostgreSQL events as large dynamic data that should not be part of light operational dumps.
- Core authentication currently has NextAuth callbacks in `src/42go/auth/lib/callbacks.ts`, but signup, login, and logout are not recorded into a shared event stream.
- There is no reusable server-side event library under `src/42go/`.
- There is no generic client-side event batching facility or API endpoint for browser-originated events.

## Desired State

- A core `events` PostgreSQL schema exists for platform events.
- Events are app-scoped so each event belongs to a resolved `app_id` and analytics can filter or export by app without ambiguity.
- The task explicitly evaluates and documents the app compartmentalization strategy: one master table partitioned by `(app_id, created_at)` or time with app indexes, partitioning by app and date, or multiple per-app parent tables partitioned by time.
- The chosen approach should optimize for write simplicity, query/export performance, operational maintenance, partition count growth, migration complexity, and app data compartmentalization.
- A likely preferred design is one core parent table such as `events.events` with an `app_id` column and time-based range partitioning by `created_at`, plus app/time indexes and optional future subpartitioning only if scale proves it. The task must still validate this against alternatives before implementation.
- The schema supports generic events with a stable shape: `created_at`, `id`, `app_id`, `user_id` nullable for anonymous events, `session_id` or client identifier where useful, `event_at`, `name`, `source`, `data`, and `meta`.
- The schema preserves ingestion cursor semantics with `(created_at, id)` for export and treats `event_at` as behavioral time.
- Existing LingoCafe event recording is migrated to the core writer while preserving existing event names and useful payload fields.
- A server-only library under `src/42go/events` exposes a small interface to record one event or a batch of events with payload and metadata.
- The server writer resolves or accepts `app_id`, prepares partitions, validates event names and payload boundaries, and inserts through Knex using `getDB()`.
- A protected server API endpoint accepts client-side event batches for the current app, validates them, fills trustworthy server metadata, and rejects oversized or malformed payloads.
- A client-side facility provides a simple `trackEvent(name, data, options)` style interface with lightweight batching, flush-on-interval, flush-on-pagehide/visibilitychange, and no fragile dependency on analytics vendors.
- Core signup, login, and logout events are recorded with app ID, user ID when available, provider, and safe request metadata.
- VI41's export work is evolved from LingoCafe-only into core events export, while preserving the hot-cache PostgreSQL model and paired CSV/Parquet local archive goal.
- Documentation explains how app-scoped event exports are stored locally, likely by app and date or with app fields in Parquet partition paths, without mixing unrelated apps accidentally.

## Definition of Success

The platform has a reusable core events system. LingoCafe no longer owns the only event table or event writer. Any server feature can record an app-scoped event through a small core API, browser code can queue client events safely, auth lifecycle events are captured, and the VI41 export design has a clear path to export core events into app-compartmentalized CSV and Parquet archives.

## Additional Context

- The system is multi-app by request. App resolution starts in `src/proxy.ts` and flows through `getAppID()` / AppConfig.
- The event system must fit the existing PostgreSQL-only and Knex migration model.
- PostgreSQL events are a hot cache for recent writes and operational inspection, not the long-term analytics warehouse.
- Local filesystem analytics storage should remain gitignored and queryable from Python/DuckDB.
- The design should avoid per-feature event table proliferation.
- The design should not require external analytics services.
- Client-originated events are less trustworthy than server-originated events; server should stamp ingestion metadata and app identity.
- Logout is harder than login because browser sign-out currently uses NextAuth client calls from shared UI components. The task must decide whether to use NextAuth events, wrapper endpoints, or explicit client flush/recording around signOut.

## Assumptions

- The initial core schema can be introduced alongside `lingocafe.events` before old LingoCafe rows are migrated or deprecated.
- Historical migration from `lingocafe.events` into `events.events` may be a separate follow-up unless refinement decides it is mandatory.
- The first implementation can support authenticated events first while allowing nullable `user_id` for future anonymous/public telemetry.
- Event payloads should remain JSONB in PostgreSQL and serialized into stable CSV/Parquet fields during export.
- The core writer should be server-only and should not leak database access into client bundles.
- The client batching facility can start small and does not need offline persistence in v1.
- The export env var from VI41 may need to be generalized from `LC_EVENTS_DATABASE_URL` to a core name such as `EVENTS_DATABASE_URL`, with backwards compatibility or migration notes.

## Constraints

- Do not break existing LingoCafe event recording while moving the capability to core.
- Do not make app analytics depend on the LingoCafe schema.
- Do not trust client-provided `app_id`, `user_id`, `created_at`, or server metadata.
- Do not let one app record events for another app through the client batch endpoint.
- Do not add an external analytics vendor or warehouse in this task.
- Do not build dashboards in this task.
- Do not delete, truncate, or clean up existing PostgreSQL event rows.
- Do not make per-app partitioning so clever that migrations and export scripts become brittle.
- Do not add long-running analytics queries to production request paths.
- Keep payload validation and size limits server-side.

## Acceptance Criteria

- A design note or output artifact compares these app-compartmentalization options: single app-scoped events parent table, app/date partitioning, and one parent table per app.
- The chosen schema approach is justified with expected write path, export path, partition maintenance, and app isolation tradeoffs.
- A migration creates the core events schema, parent table, indexes, and partition preparation function.
- The core event table includes `app_id` and supports stable incremental export ordering by `(created_at, id)`.
- A server-only core event library exists under `src/42go/events` or an equivalent core location.
- The server library exposes a small API for recording one event and batching multiple events.
- The server library can resolve the current app ID safely or accept an explicit trusted app ID for cases where request context is unavailable.
- Existing LingoCafe event writes are routed through the core event writer or through a compatibility adapter with no dependency from core to LingoCafe.
- A client batch API endpoint exists and is feature/policy protected as appropriate.
- Client-submitted events are validated, size-limited, stamped with server-side app/user metadata, and inserted as app-scoped events.
- A client-side tracker exists with simple batching and flush behavior.
- Signup, successful login, and logout are recorded as core events.
- Auth events include app ID, user ID when available, provider or auth method when available, and safe metadata only.
- VI41 export design is updated or followed up so core events can be exported to app-compartmentalized CSV and Parquet archives.
- The local archive layout prevents accidental app mixing, either by `app_id` in every row plus query guidance or by app/date partitioned filesystem paths.
- Tests or smoke checks cover server event insertion, client batch validation, and auth event recording where feasible.
- `npm run qa` passes after implementation.

## Dos

- Do inspect VI41 before planning implementation.
- Do inspect the current `lingocafe.events` schema and `trackReaderEvent()` before designing the core API.
- Do use `getDB()` and Knex parameterized inserts.
- Do keep `created_at` as the ingestion and export cursor timestamp.
- Do keep `event_at` as behavioral time.
- Do make app ID mandatory at storage time.
- Do design for cheap app-specific export.
- Do include size limits for event names, payloads, batch counts, and metadata.
- Do keep the client tracker dependency-light.
- Do document how old LingoCafe events should coexist with or migrate to the new core schema.

## Don'ts

- Don't store raw secrets, OAuth tokens, passwords, or full cookies in event metadata.
- Don't rely on browser clocks for ingestion ordering.
- Don't expose a generic unauthenticated write endpoint unless refinement explicitly approves anonymous telemetry.
- Don't make every app own a bespoke event writer.
- Don't duplicate the VI41 export logic in a second incompatible skill.
- Don't let the export cursor use only `event_at`.
- Don't couple core events to LingoCafe fields like `book_id` and `page_id`; keep those in `data` or app-specific compatibility fields only if deliberately chosen.

## Open Questions

- Should the first core schema keep app-specific convenience columns such as `book_id` and `page_id`, or move all app-specific dimensions into `data`?
- Should old `lingocafe.events` rows be migrated into `events.events` as part of this task or handled by a follow-up migration/export bridge?
- Should the physical PostgreSQL partitioning be time-only with app indexes, list-by-app then range-by-time, or separate per-app parent tables?
- What event name convention should core enforce: kebab-case strings, namespaced strings such as `auth.login`, or AppConfig-declared event names?
- Should client event collection be enabled per app through AppConfig features, and what should the feature flag name be?
- How should logout be captured reliably with the current NextAuth client signOut flows?
- Should the core export env var replace `LC_EVENTS_DATABASE_URL` immediately or support both names during transition?
- What retention policy should PostgreSQL hot-cache events eventually use after local archive durability is proven?
