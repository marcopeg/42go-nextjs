# Split AppConfig Into Public and Server-Only Configuration

## Elevator's Pitch

Split the current monolithic AppConfig model into an explicit server-only configuration and a deliberately projected public configuration so OAuth secrets, provider tokens, API keys, database credentials, and future secret-like fields cannot enter client bundles or API responses by accident.

## Business Gain

This removes the architectural root cause behind LU73. The app can keep configuration-driven multi-app behavior while making secret exposure structurally harder. It also gives future app authors a clear place to put public UI config and a separate server-only place for credentials.

## Current State

- `src/AppConfig.ts` defines one runtime `apps` registry that mixes public UI fields, policy fields, auth provider declarations, and OAuth `clientSecret` values.
- `src/42go/config/use-app-config.tsx` imports `apps` directly and returns `apps[appID]` to client code.
- Production/client bundle inspection in LU73 showed the full configuration shape and `clientSecret: process.env.*` references in browser assets, even though non-public env values were not inlined in the tested build.
- The confirmed `/api/todos` raw-config leak was removed, but the broader client-registry exposure remains open.
- `src/app/(public)/layout.tsx` passes raw config into a configurable layout component; the default layout is server-side today, but a future client custom public layout could serialize raw config props.

## Desired State

- Raw server config remains available only to server-only modules.
- Client code consumes a `PublicAppConfig` projection that contains only fields intentionally safe for browser use.
- OAuth `clientSecret`, API keys, tokens, database URLs, passwords, and equivalent credentials are impossible to access through the public config API.
- Public layouts, app layouts, profile blocks, consent, event tracking, policy UI checks, PWA/metadata, and logout logic keep working using the public projection.
- Build/test guardrails fail if raw `@/AppConfig` enters client code or if forbidden secret keys/values appear in `.next/static`.

## Definition of Success

- No client component imports `@/AppConfig` or raw server config helpers.
- `useAppConfig()` returns only `PublicAppConfig` or is replaced by an equivalent public hook/provider.
- Server auth provider construction can still read OAuth `clientSecret` values.
- The browser bundle no longer contains `auth.providers[].config.clientSecret` fields or secret env key names from AppConfig.
- A sentinel build test proves fake secret values and forbidden secret key names are absent from `.next/static`.
- `npm run qa` passes.

## Additional Context

- Related analysis is in `docs/backlog/wip/LU73-audit-appconfig-client-secret-exposure/LU73.output.md` and `LU73.notes.md`.
- The immediate `/api/todos` leak was removed, so this task should focus on the remaining architectural boundary.
- OAuth `clientId` is public, but `clientSecret` is server-only.
- A safe public projection likely includes app name, icons, theme default, public toolbar/footer/meta/pwa/pages/docs metadata as needed, feature flags, app menu, app default page, logout URL, app profile/consent UI schema, and client event-tracking config after review.
- Server-only config should include OAuth provider secrets, translation/API keys, database/runtime secrets, and any value whose key or semantics include secret/token/password/private/credential.

## Assumptions

- The project should preserve the current configuration-driven multi-app architecture.
- Existing app configs can be refactored but should not require app authors to duplicate large config objects manually.
- A projection function or typed split is acceptable if it gives a hard client/server boundary.
- The old raw `TAppConfigItem` type can remain for server code if guarded with `server-only`.

## Constraints

- Do not expose real secrets in logs, tests, snapshots, or backlog artifacts.
- Do not break NextAuth provider setup or app-specific OAuth clients.
- Do not remove needed public config used by AppLayout, ProfilePage, LingoCafe books, PWA metadata, docs, or policy UI checks.
- Prefer explicit typed projections over ad hoc object deletion at call sites.
- Keep backwards compatibility where reasonable, but security boundary wins over convenience.

## Acceptance Criteria

- A server-only raw config module exists and is protected from client imports.
- A public config type/projection exists and contains no secret-bearing fields.
- Client hooks/components use the public config path only.
- Public layout handling cannot pass raw config into a client component.
- Auth provider construction still reads server-only provider secrets correctly per app.
- Static guard or lint/test fails when client code imports raw `@/AppConfig`.
- Bundle sentinel test fails if configured secret values or forbidden secret key names appear in `.next/static`.
- Documentation explains where public config and server-only config belong.
- `npm run qa` passes.

## Dos

- Start from LU73 evidence and verify every current `useAppConfig()` consumer before changing the public shape.
- Use TypeScript to make secret-bearing fields unavailable in client code by construction.
- Add `import "server-only"` to raw config modules or server-only helpers where applicable.
- Preserve current app resolution semantics from proxy/header/app ID matching.
- Add tests or scripts that can run locally and in CI.

## Don'ts

- Do not rely only on developers remembering not to serialize raw config.
- Do not merely rename `useAppConfig()` while it still imports the raw registry.
- Do not put server secrets into `NEXT_PUBLIC_*` variables.
- Do not print or snapshot real `.env` values.
- Do not solve only the removed `/api/todos` endpoint; the client-bundle boundary is the point of this task.

## Open Questions

- Should app configs be physically split into `config.public.ts` and `config.server.ts`, or should a single server-only raw config generate a typed public projection?
- Should `match` rules be considered public, or should the client only receive the resolved app ID and public fields?
- Should feature flags be public for client UI checks, or should some features become server-only?
- Should public config be serialized once in root layout, served from an API endpoint, or generated as a static public registry?
- Should this task also add the optional migration to drop the unused `todos` table, or should that be a separate cleanup task?
