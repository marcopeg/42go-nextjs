# optimize: event partition preparation

## Elevator's Pitch

Stop preparing event partitions on every event insert; prepare them once per process, schedule, or migration window.

## Business Gain

Reduces repeated database overhead from analytics/event traffic as active users increase.

## Current State

recordEvent() and recordEvents() call SELECT events.events_prepare_partitions() before every insert or batch insert.

## Desired State

Partition preparation happens through a low-frequency path while event inserts stay lean.

## Definition of Success

Recording events no longer pays partition-preparation cost on every insert.

## Additional Context

- Performance analysis flagged events.events_prepare_partitions() as repeated tax under reader event load.
- Reader page open and scroll flows record events.
- This task is part of [YQ19: optimization epic](../YQ19-optimization-epic/YQ19.task.md), execution step 2 of 5.
- When refining or working on this task, confirm the current order in YQ19 and keep following that epic unless the epic is updated with a reason.
- Whenever this task is refined, planned, executed, blocked, completed, or reordered, update YQ19 in the same work session.

## Assumptions

- Monthly partitions can be prepared ahead of time safely.
- A fallback path may still be useful when inserts fail due to missing partition.

## Constraints

- Do not break event recording across month boundaries.
- Keep migration and production startup behavior compatible with Docker deployment.

## Acceptance Criteria

- recordEvent() and recordEvents() no longer call partition preparation unconditionally per write.
- A documented partition preparation path exists via migration, startup, scheduled task, or process-level memoization.
- Month-boundary behavior is covered by a test or a clear fallback.
- Production build succeeds.

## Dos

- Prefer batch-friendly behavior.
- Keep event insert path simple and observable.

## Don'ts

- Do not silently drop events if a partition is missing.
- Do not add broad startup work that blocks unrelated app boot for too long.

## Open Questions

- Should partition prep be a Makefile task, app boot hook, cron job, or DB-side scheduled process?
