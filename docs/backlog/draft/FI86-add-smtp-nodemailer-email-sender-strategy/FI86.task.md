# Add SMTP/Nodemailer Email Sender Strategy

## Elevator's Pitch

Add an SMTP delivery strategy to the existing email magic-link/code authentication system so apps can send sign-in emails through any SMTP-compatible service using Nodemailer. This extends the current `console` and `resend` delivery strategies without changing the NextAuth email-token/session flow.

## Business Gain

Gives 42Go apps a portable email-delivery option for operators who already use SMTP providers or company mail infrastructure. Reduces vendor lock-in to Resend while preserving the current strategy-based delivery configuration.

## Current State

- JD93 introduced email magic-link/code auth with delivery strategies.
- Current delivery strategies are `console` and `resend`.
- `nodemailer` is installed because NextAuth v4 EmailProvider imports it at module load time, but the codebase does not currently expose a Nodemailer/SMTP delivery strategy.
- Email auth provider config already supports multiple named delivery strategies in `strategies` and a selected `useStrategy` key.
- Public docs currently explain Resend production setup and console development behavior.
- The new `42go-auth` skill documents sender rules and currently lists SMTP/Nodemailer only as a future-compatible path.

## Desired State

- Email provider config supports an SMTP/Nodemailer delivery strategy, likely `type: "smtp"`.
- Operators can select SMTP by setting `EMAIL_AUTH_SENDER="smtp"` or app-specific equivalents such as `LC_EMAIL_AUTH_SENDER="smtp"`.
- SMTP config supports common host/port/TLS/auth settings through server-only environment variables.
- SMTP strategy sends the same email content as the existing console/Resend strategies: numeric code, magic link, expiry, and safe text/html bodies.
- The existing NextAuth v4 EmailProvider, app-scoped Knex adapter, token hashing, manual-code route, replay prevention, throttle behavior, and session creation flow remain unchanged.
- The implementation documents compatible service examples: Amazon SES SMTP, Mailgun SMTP, Postmark SMTP, SendGrid SMTP, Gmail/Google Workspace SMTP, company SMTP servers, and local development sinks such as Mailpit or MailHog.
- Public docs explain when SMTP/Nodemailer is preferable to Resend: existing SMTP vendor, company mail relay requirements, SES usage, provider portability, local SMTP sink testing, or need for lower-level transport control.
- Public docs explain when Resend remains preferable: simpler SaaS setup, API-key based transactional delivery, less SMTP/TLS configuration, and modern dashboard/deliverability tooling.

## Definition of Success

- A 42Go app can configure and select SMTP as an email auth sender without changing shared login UI or auth token/session code.
- SMTP delivery can be tested locally with a development SMTP sink and configured for a real SMTP provider in production.
- Documentation clearly compares SMTP/Nodemailer and Resend, lists compatible services, and includes production env/config examples.
- Existing console and Resend strategies continue to work.

## Additional Context

- Nodemailer is already in `package.json` for NextAuth EmailProvider import compatibility.
- Official Nodemailer docs should be used during refinement/implementation: https://nodemailer.com/
- Nodemailer supports SMTP transport and can be used with many SMTP-compatible transactional services.
- Amazon SES exposes SMTP credentials separately from normal AWS access keys.
- Gmail/Google Workspace SMTP can require app passwords or OAuth depending on account policy; document cautiously and avoid recommending consumer Gmail as the main production path.
- Local test tools such as Mailpit and MailHog can capture SMTP mail without sending real email.

## Assumptions

- SMTP should be implemented as another delivery strategy under the existing `EmailStrategyConfig` union, not as a separate auth provider.
- SMTP strategy should not alter token generation, storage, hashing, throttle, app scoping, event names, or manual code verification.
- Secrets remain in server environment variables following the current provider config pattern.
- Resend remains the easiest recommended default for LingoCafe unless the operator explicitly chooses SMTP.

## Constraints

- Do not bypass NextAuth email callback/session creation.
- Do not store raw codes or raw tokens.
- Do not expose SMTP passwords to client-side code.
- Do not make SMTP required for local development.
- Do not remove Resend or console strategies.
- Do not clean expired tokens per request as part of this task.
- Run `npm run qa` after code changes.

## Acceptance Criteria

- `EmailStrategyConfig` includes a typed SMTP/Nodemailer strategy variant.
- `sendEmailVerification()` dispatches to SMTP when the selected strategy has `type: "smtp"`.
- SMTP config supports at least `host`, `port`, `secure`, `user`, `password`, `from`, and optional `subject`.
- Missing required SMTP config produces a controlled error that is observable in logs and produces a controlled login failure.
- SMTP sender builds both text and HTML bodies containing the numeric code, magic link, and expiry.
- `.env.example` includes generic SMTP variables and LingoCafe-prefixed examples where useful.
- `contents/default/docs/authentication/README.md` documents SMTP setup and compares it with Resend.
- `docs/articles/EMAIL_AUTH.md` documents the internal strategy shape.
- `.agents/skills/42go-auth/SKILL.md` mentions SMTP/Nodemailer as a supported sender after implementation.
- Local validation uses a development SMTP sink such as Mailpit or MailHog, or documents why it was not run.
- Existing `console` and `resend` strategy behavior is not regressed.

## Dos

- Prefer a narrow sender-only implementation.
- Use Nodemailer SMTP transport instead of inventing raw SMTP code.
- Keep the email body builder shared or consistent across delivery strategies.
- Document provider-specific caveats only where they affect configuration.
- Include both generic and app-prefixed environment variable examples.

## Don'ts

- Do not introduce a full public/server AppConfig split in this task.
- Do not change the login UI unless a small sender-independent copy tweak is required.
- Do not add account-linking behavior or `auth.accounts` rows for email sign-in.
- Do not treat SMTP as more secure by default than Resend; it is more portable, not automatically better.
- Do not recommend storing SMTP credentials in public config.
