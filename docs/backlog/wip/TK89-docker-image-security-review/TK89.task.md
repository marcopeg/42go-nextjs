# Docker Image Security Review


## Elevator's Pitch

Perform a comprehensive security audit of the Docker image to ensure it can safely be published and inspected by third parties without leaking secrets, private content, or unnecessary attack surface.

## Business Gain

Confidence that the production Docker image is safe to distribute publicly. A clear, documented security posture for the image—covering secrets, content, dependencies, runtime hardening, and registry hygiene—so the team can ship without second-guessing what's inside.

## Current State

The project has a Dockerfile with multi-stage builds and a `docker-compose.prod.yml` for production. There is a `check-docker-security.sh` script in `scripts/`. Docker app-resolution issues were addressed by the completed middleware-removal work, and later Docker build performance work validated the production image build path. No formal security audit of the image contents, layers, or runtime configuration has been performed or documented.

Refinement clarified that this review should cover the Dockerfile build process, the resulting image artifact, and the runtime Compose configuration. The review should start with a source-level inspection of the Dockerfile and related production container files, looking for security posture improvements before or alongside final image inspection. The operator specifically suspects issues around the current standalone Dockerfile path, so the review must verify whether the final image contains the right traced runtime files and whether standalone mode is still the correct production runner strategy.

## Desired State

A completed security review covering the Dockerfile, build context, built image artifact, and production Compose runtime posture. The review should compare this repository against current official Next.js and Docker containerization guidance, then produce concrete findings and hardening steps.

The review covers all twelve checklist areas:

1. Build context hygiene (`.dockerignore`, excluded files)
2. Dockerfile copy strategy (multi-stage, minimal final image)
3. Secret handling (no secrets baked into layers)
4. Content exposure (no private content in the image)
5. Dependency exposure and patch state (vulnerability scan, base image policy)
6. Runtime user and file permissions (non-root, read-only where practical)
7. Network and service boundaries (no internal services exposed)
8. Shared image blast radius (multi-app risk assessment)
9. Auth and session material (no credentials in bundles)
10. Analytics and consent boundaries (no premature tracking)
11. Registry and publishing hygiene (tagging, signing, provenance)
12. Final image inspection (filesystem search for secrets and sensitive files)

Each area results in a pass/issues-found/blocked verdict. Issues found produce actionable remediation items in `TK89.output.md`.

## Definition of Success

- Every checklist item has been evaluated with evidence collected.
- A review outcome summary is produced using the provided template (pass / issues found / blocked per area).
- All critical issues (secrets in layers, private content exposure, running as root without justification) are either resolved or tracked as separate tasks.
- A documented rule exists for what content may enter the final image.
- A documented secret injection policy exists.
- Docker Hub publishing posture is evaluated.
- Trivy is used from the local shell for vulnerability scanning.
- `TK89.output.md` contains evidence, verdicts, and an ordered list of actionable hardening steps.

## Additional Context

Key audit commands to use:

```bash
docker history <image-name>
docker run --rm -it --entrypoint sh <image-name>
docker run --rm --entrypoint sh <image-name> -lc "find / -iname '*.env' -o -iname '*secret*' -o -iname '*key*' -o -iname '*dump*' 2>/dev/null"
trivy image <image-name>
```

Refinement v1 operator decisions:

- Scope includes both the Docker build/image artifact and runtime Compose configuration.
- Target registry is Docker Hub.
- Output artifact is `TK89.output.md`.
- The output must include a list of actionable steps.
- Blocking risk is anything that can lead to a security breach once the image is running in production.

Refinement v2 operator decisions:

- Trivy has been installed and should be used from bash.
- Vulnerability release gate is accepted as proposed: critical and high vulnerabilities with a plausible production exploit path block release until fixed or formally accepted; medium vulnerabilities are actionable follow-ups unless they affect auth/session handling, runtime command execution, exposed network services, or known exploited CVEs; low vulnerabilities are documented only.
- Runtime hardening gaps such as read-only filesystem, dropped capabilities, `no-new-privileges`, and tighter bind mounts block only when they create a concrete breach path; otherwise they become ordered hardening actions.
- Default final-image content policy is accepted as proposed after research.
- Dockerfile standalone behavior must be checked carefully because the operator suspects issues with the current standalone version.

Research baseline to consult during execution:

- Docker's Next.js containerization guide: https://docs.docker.com/guides/nextjs/containerize/ — multi-stage builds, standalone output, current secure base image selection, non-root runtime, and minimal final image expectations.
- Next.js `output: "standalone"` docs: https://nextjs.org/docs/app/api-reference/config/next-config-js/output — traced runtime files, `.next/standalone`, manual `public` and `.next/static` copying, narrow tracing includes/excludes.
- Docker build best practices: https://docs.docker.com/build/building/best-practices/ — `.dockerignore`, minimal trusted base images, rebuild/pull cadence, pinned digests tradeoffs, CI image testing, and non-root `USER`.
- Trivy official installation docs: https://trivy.dev/docs/v0.66/getting-started/installation/ — preferred vulnerability scanner, installable via Homebrew on macOS/Linux with `brew install trivy`, or runnable as an official container image.

Final image content policy:

- Allowed: traced `.next/standalone` runtime files, `.next/static`, `public/` assets required at runtime, production runtime dependencies copied by the standalone trace, `package.json` metadata needed by the runtime, app icons/themes under `public/`, and minimal health/runtime support files.
- Not allowed unless explicitly justified: `.env*`, secret material, `.npmrc`, source maps, test files, fixtures, seed data, database dumps/backups, local logs, `.git`, `.agents`, docs/backlog, development caches, editor state, private content, and `contents/` unless the production deployment intentionally mounts or bakes public content.

## Assumptions

- The image is intended for Docker Hub, so the review assumes public-registry inspection risk unless the operator explicitly narrows distribution.
- Secrets are expected to be injected at runtime via environment variables, not baked into the image.
- The multi-app architecture means one image serves multiple sites; this is a deliberate platform decision but needs blast-radius assessment.
- Runtime Compose settings matter because they can change the real production attack surface even when the image artifact is clean.
- `trivy` is available from the local shell.

## Constraints

- The review does not replace application security review, code review for auth/authz, infrastructure security review, or legal review.
- The review focuses on the Docker image artifact, its build pipeline, and production Compose runtime posture.
- Do not make Dockerfile, Compose, or build-pipeline changes during execution unless the task is explicitly expanded from review into remediation.
- If `trivy` unexpectedly fails or is unavailable, document the blocker and command failure in `TK89.output.md` instead of silently substituting a different scanner.
