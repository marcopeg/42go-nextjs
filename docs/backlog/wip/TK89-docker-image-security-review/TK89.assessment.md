## Remaining Findings

The remaining work is not the same kind of problem as the original release blockers. The image itself is clean from Trivy's runtime vulnerability perspective. The remaining items are policy and exposure decisions:

1. Source maps remain in the final image: `54` `.next` maps. This is source disclosure risk, not an active exploit finding.
2. Local dump files remain in the repository tree and contain token-looking material. They are excluded from Docker build context and final image, but they are still a local repository hygiene risk.
3. `docker-compose.prod.yml` is a local-production mimic. Its broad `.env` injection, local DB port exposure, host-gateway, and bind mount are not blockers for the deployed image, but must not be reused as the real production security model.
4. Image-level health check is disabled. This is acceptable if the deployment platform owns health checks, but the choice should be documented.
5. Docker Hub publishing still lacks SBOM, provenance, signing, and a written `latest` tag movement policy.

## Assessment

The remediation already performed fixed the actual runtime blockers: vulnerable Next.js package state, redundant dependency layer, package-manager tooling in the runtime image, and build-context leakage. The current image is suitable for final deployed smoke testing.

The next decisions are about distribution posture. If this image is a private deployment artifact, the remaining actions can be handled as follow-up hardening. If the image is meant to be publicly inspectable or broadly consumed from Docker Hub, source maps and supply-chain metadata become real pre-release work.

The local dump/token issue is separate from Docker publishing. Docker no longer ships it. The repository still carries it. That gets fixed by moving dumps out of the repo or documenting an intentionally private local dump location. Token rotation is only necessary if those dump files left this machine or entered shared infrastructure.
