## Final Image Content Policy

Allowed:

- traced `.next/standalone` runtime files
- `.next/static`
- `public/` assets required at runtime
- production runtime dependencies copied by the standalone trace
- `package.json` metadata needed by the runtime
- app icons/themes under `public/`
- minimal health/runtime support files

Not allowed unless explicitly justified:

- `.env*`
- secret material
- `.npmrc`
- source maps
- test files
- fixtures
- seed data
- database dumps/backups
- local logs
- `.git`
- `.agents`
- docs/backlog
- development caches
- editor state
- private content
- `contents/` unless production intentionally mounts or bakes public content

## Release Gate

Critical and high vulnerabilities with a plausible production exploit path block release until fixed or formally accepted. Medium vulnerabilities become actionable follow-ups unless they affect auth/session handling, runtime command execution, exposed network services, or known exploited CVEs. Low vulnerabilities are documented only.

Runtime hardening gaps block only when they create a concrete breach path; otherwise they become ordered hardening actions.

### Source Inspection

- `.dockerignore` excludes `.git`, `.agents`, `node_modules`, `.next`, docs, `.env*`, `.npmrc`, logs, and `*.pem`.
- `.dockerignore` does not exclude `db-backups`, `knex/dumps`, `*.sql`, `*.dump`, `*.backup`, `*.bak`, `.devcontainer`, `.tmp`, or `.DS_Store` recursively.

## Ordered Actionable Steps

### Assessment Update

- Build context hygiene is now done for Docker publishing purposes: `.dockerignore` excludes the risky dump, backup, local, and Knex paths from the build context.
- This does not clean the repository itself. SQL dump files still exist locally, and Trivy source secret scanning still reports token-like material in `knex/dumps`.
- That remaining issue belongs under secret/repository hygiene, not Docker image build-context hygiene.
- `docker-compose.prod.yml` is local production: it lets engineers run the production build locally and mimic production shape. It is not the real production deployment topology and should not block Docker Hub publishing by itself.
- Dockerfile copy strategy is now addressed: the redundant final `node_modules` copy was removed after the standalone-only trial image booted and passed health plus host-resolution smoke checks.

### Blockers Before Docker Hub Publishing

1. Upgrade vulnerable dependencies, starting with `next` from `16.2.4` to at least `16.2.6`, then rerun `npm run qa`, rebuild the Docker image, and rerun `trivy image`.
2. Move `db-backups` and `knex/dumps` outside the repository or formally accept the local-only repository hygiene risk. The Docker build-context side is covered by `.dockerignore`.
3. Rotate any credentials/tokens that were present in SQL dump files if those files ever left this machine or entered shared build/cache infrastructure.
4. Remove production source maps from the final image or formally accept the source disclosure risk for a public Docker Hub image.

### Hardening Follow-Ups

5. Smoke test auth and one protected route against the lean standalone image.
6. Document `docker-compose.prod.yml` as local production only.
7. For any real Compose production topology, avoid broad `.env` injection, host DB exposure, local host-gateway shortcuts, and writable bind mounts unless explicitly justified.
8. Align the runtime user and group so `nextjs` belongs to `nodejs`, or document the current `nextjs:nogroup` runtime identity as intentional.
9. Decide whether the Dockerfile should have an image-level `HEALTHCHECK` or whether platform/Compose-only health checks are the project policy.
10. Extend the security check beyond `.env` detection to check source maps, dump/backup files, `.npmrc`, docs/backlog, `.agents`, runtime user, final filesystem policy, and Trivy scan status.
11. Add Docker Hub publishing hygiene: SBOM generation, provenance/attestation, image signing, and a policy for when `latest` may be moved.

## Release Recommendation

Blocked.

The current image should not be published to Docker Hub as a public or inspectable production artifact.

Decisive blockers:

- Trivy reports `8` high vulnerabilities in runtime Node packages, including multiple `next@16.2.4` advisories with fixed versions available.
- Trivy repository secret scanning found JWT-like material in local dump files; `.dockerignore` excludes them from Docker build context, but repository hygiene remains unresolved.
- The final image contains source maps, including server route maps, increasing source disclosure risk for a public image.

Shortest path to unblock:

1. Upgrade `next` and vulnerable transitive packages.
2. Resolve or formally accept local dump-file repository hygiene risk.
3. Remove source maps from the final image or accept that risk in writing.
4. Rebuild with no cache and rerun final filesystem inspection plus Trivy.
