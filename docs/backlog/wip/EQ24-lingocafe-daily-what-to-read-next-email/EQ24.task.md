# LingoCafe Daily What To Read Next Email

## Elevator's Pitch

Add a local `42go` CLI automation that emails eligible LingoCafe users a "what to read next" recommendation based on their recent reading activity and current book progress.

## Business Gain

Helps readers return to LingoCafe with a concrete next action: continue the book they are reading or choose a new unread book. This supports retention without sending messages to users who already read recently, while avoiding new app runtime security surface during the pilot.

## Current State

LingoCafe tracks books, pages, and per-user book progress in the `lingocafe` schema. Reader endpoints update progress when users open and save pages. The codebase has app-scoped users with `auth.users.feature_flags`, and LingoCafe email delivery exists for authentication, but there is no periodic reading recommendation email process yet.

The canonical current-reading table for this pilot is `lingocafe.books_progress`. It stores one row per user/book with the latest `page_id`, `progress_bps`, `updated_at`, and optional `completed_at`.

The local `42go` CLI can refresh source data with `42go pull all` and rebuild local analytics with `42go query all`. Refinement verified on 2026-06-18 that the local Parquet structure contains the required raw data:

- `.local/42go-data/auth/users.parquet`: 128 users with `app_id`, `id`, `email`, `feature_flags`, and timestamps.
- `.local/42go-data/lingocafe/books.parquet`: 39 books.
- `.local/42go-data/lingocafe/books_pages.parquet`: 1272 pages with `book_id`, `id`, and `position`.
- `.local/42go-data/lingocafe/books_progress.parquet`: 115 progress rows with `user_id`, `book_id`, `page_id`, `progress_bps`, `updated_at`, and `completed_at`.
- `.local/42go-data/events/events_YYYYMM.parquet`: raw LingoCafe events including `page.open` and `page.scroll`.
- `.local/42go-query/lingocafe-users.parquet` and `.local/42go-query/lingocafe-reads.parquet`: existing aggregate outputs.

The current local data has 115 LingoCafe users, but zero users currently have `feature_flags.lc_daily_email = true`. During refinement, the pilot recipient gate changed from database feature flags to a temporary local whitelist so testing can proceed without changing production user rows.

## Desired State

A local `42go email lingocafe read-tip` CLI command runs periodically outside the Next.js runtime, with Codex/operator automation handling the schedule. The command refreshes data with `42go pull all`, rebuilds analytics with `42go query all`, reads local Parquet files, applies a temporary local recipient whitelist, skips users whose latest `books_progress.updated_at` is within the last 24 hours, and prepares one "what to read next" email for each remaining eligible user.

If a user has an actively reading book that is not complete, the email proposes continuing that book and links to the current page when progress is below 90% complete, otherwise to the next page of the book. If the user has no active reading book, or all reading books are complete, the email suggests picking one unread book using the user's LingoCafe target language and level when possible.

The first version defaults to dry-run/report mode. It sends email only when sender credentials are explicitly configured. It stores a local Parquet delivery log under the ignored `.local/42go-data` tree, so repeated runs can avoid duplicate sends and enforce a cooldown. For now, at least 72 hours must pass between reminder emails to the same user.

When send mode is enabled, the command uses the same Resend HTTP delivery service as LingoCafe magic-link login. It reads `.env`, uses `LC_RESEND_API_KEY`, and follows the existing sender shape from `src/42go/auth/lib/email/senders.ts`. Secrets must not be written to logs, Parquet output, or backlog docs.

The pilot whitelist should live inside the future skill folder as a simple text file. The initial whitelist content should include only `marco.pegoraro@gmail.com` so test sends go only to Marco. The whitelist is temporary and should be removed after the early tests. If the whitelist file contains exactly one line with the exact content `send to all`, the automation should disable the email whitelist and evaluate every LingoCafe user against the normal eligibility rules.

## Definition of Success

Eligible LingoCafe users receive reminder emails only when they appear to need a nudge back into reading: no reading activity in the last 24 hours and no reminder email sent in the last 72 hours. Ineligible users are skipped for a clear reason. The local process can be run periodically by Codex/operator automation without duplicate sends in the cooldown window.

## Additional Context

- For the pilot, process only users whose email appears in the local whitelist file.
- The initial whitelist file should contain `marco.pegoraro@gmail.com`.
- The whitelist is temporary. After a few successful tests, remove the whitelist concept and replace it with the final recipient gate selected for production.
- If the whitelist file contains exactly one line and that line is exactly `send to all`, the automation checks every LingoCafe user and sends only to users who meet all other eligibility rules.
- "User has read in the last 24h" is defined by the user's most recent `books_progress.updated_at`.
- "Actively reading" maps to existing incomplete `lingocafe.books_progress` rows, with the most recently updated row used first.
- `books_progress.progress_bps` is the first implementation's scroll-position source for deciding whether the current page is near its end.
- Unread-book fallback order is: same target language and same level, nearest higher level in the same language, random unread book in the same language, then skip if no unread book exists in that language.
- Links should route back into the authenticated LingoCafe reader UI.
- This first version intentionally avoids a protected Next.js API route and avoids introducing generic 42Go API keys.
- The accepted command path is `42go email lingocafe read-tip`.
- Send mode uses the existing LingoCafe Resend setup with `LC_RESEND_API_KEY` from `.env`.

## Assumptions

- This is a LingoCafe-specific `42go` CLI command, not a generic 42Go notification framework.
- The first version uses a local whitelist as an explicit recipient gate while testing.
- Email content can be plain and transactional for the first release.
- The process should avoid recommending completed books as active continuation targets.
- Local send logs are acceptable for the pilot, as long as the log path is ignored and durable enough for the operator machine running the automation.
- The first email body can be plain text with links.
- The existing codebase uses Resend for LingoCafe magic-link delivery; no separate Reflow sender was found during refinement.

## Constraints

- Do not email users unless their email appears in the local whitelist file, except when the file contains exactly `send to all`.
- Do not email users who read in the last 24 hours.
- Do not expose private progress data in public, cacheable endpoints.
- Avoid duplicate sends when the periodic job retries or runs more than once within the configured cooldown window.
- Enforce at least 72 hours between sent reminder emails to the same user.
- The goal is to remind inactive readers to return to the app, not to email active readers or repeatedly nudge inactive readers too often.
- Do not add a protected app API endpoint or generic API-key system for this first version.
- Do not commit local pulled data, local query outputs, or local delivery logs.
- Do not treat `send to all` as active unless it is the only line in the whitelist file and matches exactly.
- Do not log or persist `LC_RESEND_API_KEY`.
- Run `npm run qa` after code changes.

## Acceptance Criteria

- `42go email lingocafe read-tip` can execute the recommendation email workflow for LingoCafe.
- The automation runs `42go pull all` and `42go query all` before computing recommendations.
- Candidate user selection reads local Parquet files and filters to LingoCafe users whose email appears in the local whitelist file.
- The initial whitelist file is created with `marco.pegoraro@gmail.com` as its only recipient.
- If the whitelist file contains exactly one line with `send to all`, the whitelist gate checks every LingoCafe user and allows only users who meet all other eligibility rules.
- Users whose most recent `books_progress.updated_at` is within the last 24 hours are skipped.
- Users with active, incomplete books receive a continuation recommendation for the incomplete `books_progress` row with the most recent `updated_at`.
- Active-book recommendation links to the current page when the book is below 90% complete.
- Active-book recommendation links to the next page when the current page/book progress is at or above the defined 90% threshold and another page exists.
- Users with no active incomplete book receive a recommendation to pick one unread book, preferring same target language and same target level.
- If no unread same-level book exists, the unread-book fallback chooses the nearest higher level in the same target language.
- If no unread higher-level book exists, the unread-book fallback chooses a random unread book in the same target language.
- If the user has no unread books left in the target language, the user is skipped and no generic library-exhausted email is sent.
- Completed books are not recommended as active continuation targets.
- The automation records a full history of sent emails in a local ignored Parquet file under `.local/42go-data`, alongside the other 42Go Parquet files.
- The sent-email log includes at least user id, recipient email, sent timestamp, recommendation type, reading tip, book id, page id, target link, plain-text email body, sender status/message id, and error if any.
- The sent-email log is used to enforce at least 72 hours between reminder emails to the same user.
- Dry-run mode is the default and reports candidate users, skip reasons, recommendation type, and intended link without sending email.
- Dry-run skip reasons distinguish at least: not whitelisted, read in last 24 hours, emailed in last 72 hours, no eligible recommendation, and sender/configuration issue.
- Dry-run output clearly reports whether the whitelist is in explicit-recipient mode or `send to all` mode.
- Email send mode only runs when sender credentials/configuration are explicitly present.
- Email send mode reads `.env` and uses `LC_RESEND_API_KEY` with the existing Resend HTTP API request shape.
- Email failures are logged without stopping the whole batch.
- Tests or focused verification cover the main decision branches: recent read skip, recent email cooldown skip, active incomplete continuation, near-complete next page, unread-book fallback, whitelist exclusion, and `send to all` mode.
- Production build succeeds.

## Dos

- Keep the first version LingoCafe-scoped.
- Use `42go pull all` and `42go query all` as the data refresh boundary.
- Read local Parquet files rather than calling the application database directly from the automation.
- Keep recommendation selection deterministic enough to test.
- Document how the local CLI command is meant to be invoked by Codex/operator automation.
- Keep the whitelist parser deliberately simple and strict.

## Don'ts

- Do not create a broad marketing email platform in this task.
- Do not send to all LingoCafe users by default.
- Do not keep the whitelist long term after the pilot validates the workflow.
- Do not rely on client-side filtering for eligibility.
- Do not recommend a completed book as the primary action.
- Do not block the full batch because one user email fails.
- Do not introduce generic 42Go API keys as part of this task.
- Do not make a Next.js API route the first implementation path.
