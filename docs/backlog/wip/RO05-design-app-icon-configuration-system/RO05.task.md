# Design App Icon Configuration System

## Elevator's Pitch

Design and implement a shared 42go icon configuration system so every app can configure favicons, home-screen icons, PWA icons, and main app logos without hardcoded one-off behavior.

## Business Gain

Every app in the multi-app platform can look like its own product in browser tabs, installed app launchers, public layout chrome, and protected app navigation. Chuck Norris does not let one favicon rule the whole town.

## Current State

The platform already has partial icon support:

- `src/AppConfig.ts` has a top-level `logo?: string | ComponentType` field.
- `src/42go/layouts/public/AppTitle.tsx` and `src/42go/layouts/app/AppTitle.tsx` render `toolbarConfig?.icon || config.logo`.
- `config.public.pwa.icons.appleTouch180` feeds `src/42go/pwa/HeadTags.tsx`.
- `config.public.pwa.icons.manifest192`, `manifest512`, and `maskable512` feed `src/app/manifest.ts`.

The platform does not appear to have a dedicated favicon configuration field.

Official Next.js App Router guidance prefers metadata file conventions such as `app/favicon.ico`, `app/icon.png`, and `app/apple-icon.png` for static metadata files. That works cleanly for a single static app identity, but this repository resolves app identity dynamically per request through middleware and AppConfig.

`LT47` completed the immediate LingoCafe icon work and deliberately left shared favicon architecture to this task.

Additional repository evidence after `LT47`:

- LingoCafe assets now live under `public/lingocafe` and `public/lingocafe/favicon`.
- `src/config/lingocafe/config.ts` now wires `logo: "/lingocafe/logo.png"` and LingoCafe-specific `public.pwa.icons.appleTouch180`, `manifest192`, and `manifest512`.
- `LT47` left `maskable512` unset because no dedicated safe-area maskable asset was provided.
- `LT47` confirmed the runtime manifest emits LingoCafe icon paths for the LingoCafe host.
- There is still no shared app-specific favicon config field.
- `src/proxy.ts` excludes `/favicon.ico` and `/manifest.webmanifest` from proxy matching, so favicon routing must be designed with request-based app resolution in mind.

## Desired State

42go has a documented, type-safe, config-driven icon model that every app can use.

The refinement direction is:

- add a shared `icons` AppConfig contract for favicon, PWA icon, maskable icon, and main UI icon assets;
- prefer convention over per-app path repetition: icon assets live in a canonical per-app folder under `public/app-icons/<app-id>/`;
- ship a complete boilerplate fallback iconset under `public/app-icons/_default/`;
- define an expected favicon variant set by naming convention, rather than by repetitive config entries;
- allow explicit `icons.*` overrides only when convention-only is insufficient;
- make the new resolver authoritative and update existing app configs, assets, and docs to the new shape instead of preserving legacy configuration;
- document the convention and required/optional variants in shared 42go docs.

The model should cover:

- browser favicon links;
- Apple home-screen icon;
- web app manifest icons;
- maskable manifest icon;
- main app logo used in layouts;
- public toolbar icon override interactions;
- default/fallback icon behavior.

The implementation should respect Next.js App Router metadata behavior while still supporting dynamic app identity.

## Recommended Design

Use a single shared resolver as the source of truth:

```ts
resolveAppIcons(appId, config) => {
  faviconIco,
  favicon16,
  favicon32,
  appleTouch180,
  manifest192,
  manifest512,
  maskable512,
  ui,
}
```

All icon-consuming surfaces should call this resolver or receive its output:

- `src/app/layout.tsx` `generateMetadata()` emits `metadata.icons` from the resolved favicon and Apple icon paths.
- `src/42go/pwa/HeadTags.tsx` emits legacy iOS tags from the same resolved Apple icon path.
- `src/app/manifest.ts` emits manifest icons from the same resolved manifest icon paths.
- public and protected `AppTitle` components use the resolved `ui` icon, while keeping `public.toolbar.icon` as the most local override.

The canonical asset convention is:

```text
public/app-icons/_default/
  favicon.ico
  favicon-16x16.png
  favicon-32x32.png
  apple-touch-icon-180x180.png
  manifest-192x192.png
  manifest-512x512.png
  maskable-512x512.png
  ui.png

public/app-icons/<app-id>/
  ...same optional filenames...
```

The `_default` folder must contain the complete boilerplate iconset. Per-app folders may be partial; missing per-app files automatically fall back asset-by-asset to `_default`.

Explicit config should be supported for apps that cannot or should not use the convention:

```ts
icons: {
  basePath?: "/some/public/path";
  strict?: boolean;
  favicon?: {
    ico?: string;
    png16?: string;
    png32?: string;
  };
  appleTouch180?: string;
  manifest192?: string;
  manifest512?: string;
  maskable512?: string;
  ui?: string | ComponentType<{ className?: string }>;
}
```

Resolution priority should be:

1. local UI override: `config.public.toolbar.icon` for layout title rendering only;
2. explicit `config.icons.*`;
3. convention asset under `config.icons.basePath ?? "/app-icons/<app-id>"`;
4. boilerplate fallback under `/app-icons/_default`.

Maskable icons should not be auto-derived from non-maskable assets. If an app does not provide a dedicated `maskable-512x512.png` or explicit `icons.maskable512`, use the default maskable icon unless strict validation is enabled.

Build-time validation should generate or verify an icon registry before `next build`. The registry should record which convention files exist, so request rendering and client components do not need filesystem checks. Validation rules:

- `_default` iconset is always required and build-failing if incomplete;
- per-app iconsets warn by default when incomplete because fallback is valid;
- `icons.strict: true` fails the build for that app if required app-specific assets are missing;
- `APP_ICONS_STRICT=1` fails the build for all missing per-app assets.

This design intentionally treats bare `/favicon.ico` as a boilerplate fallback only. Per-app browser tabs should use dynamic `metadata.icons` generated from the resolved app config. This avoids pretending a single root favicon can represent multiple URL-resolved apps.

Version 1 should defer `safari-pinned-tab.svg`. It is not needed for the core favicon, Apple touch, manifest, maskable, or UI icon promise.

Existing LingoCafe assets should be migrated into `public/app-icons/lingocafe` during implementation so the first real app proves the convention. `icons.basePath` remains available for future apps that need non-standard public asset placement.

The boilerplate default iconset should be generated from a simple 42go mark and committed as raster assets. Runtime generation is not needed for this task.

Because this project is still early, the implementation should not keep backward-compatible `logo` or `public.pwa.icons` aliases. Existing app configs, existing static assets, and existing documentation should be migrated in the same task.

## Definition of Success

An app author can place favicon assets in one documented per-app convention path, and 42go emits the right metadata, manifest entries, and layout logo behavior for that app without hardcoded per-app conditionals.

The outcome of this task includes the shared icon resolver, metadata integration, manifest integration, UI icon integration, documentation, and a complete committed boilerplate fallback iconset.

## Assumptions

The current `config.public.pwa` shape remains useful for PWA name, display, start URL, scope, and color metadata, but icon paths should move to the new shared icon contract.

The existing `logo` field should be removed or superseded by the new `icons.ui` contract during implementation.

Per-app favicons should not be solved by replacing a global `src/app/favicon.ico`, because that would only represent one app.

The LingoCafe asset layout is useful prior art, and existing LingoCafe icon files should be physically migrated into `public/app-icons/lingocafe` during implementation.

Official Next.js documentation recommends file-based metadata icons where possible, but also supports `metadata.icons` for explicit icon links. In this project, `metadata.icons` is the better fit because app identity is resolved dynamically from the request.

## Constraints

Keep the solution compatible with request-based app resolution.

Do not break existing Quicklist, Notes, default app, or LingoCafe PWA configuration.

Avoid duplicating the same icon paths across unrelated config fields unless the duplication is intentional and documented.

Use official Next.js metadata APIs and conventions where they fit dynamic app identity.

Implementation must include the migration and manual-test artifacts listed in the acceptance criteria.

## Acceptance Criteria

- [ ] Existing icon and PWA behavior is audited across `AppConfig`, `HeadTags`, `manifest.ts`, and app/public layout title components.
- [ ] Official Next.js App Router metadata icon behavior is documented as an implementation constraint.
- [ ] A convention-first favicon contract is specified using a canonical per-app folder path (`public/app-icons/<app-id>/...`).
- [ ] A complete boilerplate fallback iconset is designed and added under `public/app-icons/_default/`.
- [ ] A shared icon resolver returns favicon, Apple touch, manifest, maskable, and UI icon paths for every configured app.
- [ ] App-specific browser favicon links can be emitted from convention-resolved assets without replacing one global favicon for every app.
- [ ] The required vs optional favicon variant matrix is explicitly documented (including naming conventions and file formats).
- [ ] Build-time validation behavior is implemented: missing default icon assets fail the build, and missing app-specific assets fail only when strict mode is enabled.
- [ ] Apple touch icon behavior is driven by the shared resolver.
- [ ] Manifest icon behavior is driven by the shared resolver.
- [ ] Main app logo behavior is documented and, if needed, aligned with the new icon configuration shape.
- [ ] `public.toolbar.icon` remains the highest-priority layout-title override.
- [ ] Existing apps keep equivalent runtime identity after being migrated to the new icon contract.
- [ ] Documentation explains how each app should configure favicon, home-screen icon, PWA icons, and logo.
- [ ] A dedicated favicon documentation page is added to shared 42go docs content, including naming conventions and browser-compatibility variants.
- [ ] Existing app configs and documentation are updated to remove legacy `logo` and `public.pwa.icons` icon-path usage.
- [ ] A `RO05.migration.md` file explains how to move from legacy icon fields and asset locations to `public/app-icons`.
- [ ] A `RO05.test.md` file provides step-by-step manual verification for favicons, PWA icons, manifest output, and UI icons.

## Dos

- Do preserve the multi-app architecture.
- Do keep icon configuration explicit and type-safe.
- Do preserve non-icon PWA metadata such as name, display, start URL, scope, colors, and status bar style.
- Do make the migration path clear for existing app configs.
- Do verify generated head tags and manifest output.
- Do produce explicit favicon naming-convention documentation in shared 42go docs.
- Do prefer build-time checks over request-time checks where possible.

## Don'ts

- Don't hardcode app IDs in shared metadata components.
- Don't replace `src/app/favicon.ico` as the only solution for app-specific favicons.
- Don't break existing non-icon `public.pwa` behavior.
- Don't force every app to provide every icon size if sensible fallbacks are available.
- Don't leave app icon paths split across legacy `logo` and `public.pwa.icons` fields.
- Don't introduce per-request filesystem checks for favicon resolution.
