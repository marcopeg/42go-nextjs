# Promote Events to Core App-Scoped Event System

## Elevator's Pitch

Move the existing LingoCafe-only event stream into a reusable 42go core events capability: an app-scoped PostgreSQL event schema, a server-side event writer, a client-side batching collector, core auth and profile lifecycle event recording, and an analytics export skill evolved from VI41.

## Business Gain

Every app in the multi-app platform needs product telemetry without rebuilding the same event plumbing. A core event system gives the platform one disciplined way to record signups, logins, logouts, profile changes, consent changes, reader activity, and future product events while keeping each app's data compartmentalized. PostgreSQL stays the hot ingestion buffer. Local CSV and Parquet archives become the durable analytics source. Chuck Norris does not copy telemetry code between apps. Telemetry copies itself into core.

## Current State

- LingoCafe has a `lingocafe.events` table created by `knex/migrations/20260427150000_lingocafe_schema.js`.
- `lingocafe.events` is append-oriented, partitioned by `created_at`, uses composite primary key `(created_at, id)`, and stores `user_id`, `event_at`, `name`, optional `book_id`, optional `page_id`, `data`, and `meta`.
- `lingocafe.events_prepare_partitions()` creates monthly partitions under the `lingocafe` schema.
- The current writer is `trackReaderEvent()` in `src/app/api/(lingocafe)/lingocafe/_lib/reader.ts`, so non-LingoCafe features cannot use events without depending on LingoCafe internals.
- LingoCafe currently records events such as `book-info`, page events, `profile-update`, and `consent-update`.
- VI41 completed a download-only LingoCafe export skill that reads `lingocafe.events` through `LC_EVENTS_DATABASE_URL` and writes paired CSV and Parquet batches to a gitignored local archive.
- The existing dump/backup work treats PostgreSQL events as large dynamic data that should not be part of light operational dumps.
- Core authentication currently has NextAuth callbacks in `src/42go/auth/lib/callbacks.ts`, but signup, login, and logout are not recorded into a shared event stream.
- There is no reusable server-side event library under `src/42go/`.
- There is no generic client-side event batching facility or API endpoint for browser-originated events.

## Desired State

- A core `events` PostgreSQL schema exists for platform events.
- Events are app-scoped so each event belongs to a resolved `app_id` and analytics can filter or export by app without ambiguity.
- The chosen physical layout is one core parent table such as `events.events`, partitioned by `created_at` with mandatory `app_id` and app/time indexes.
- The core table does not use app-specific physical partitions in v1. App isolation is logical through `app_id`, not separate parent tables or list partitions.
- The core table keeps indexing lean because PostgreSQL is a hot download buffer, not the live analytics store. Required indexing focuses on the export cursor and app-scoped extraction.
- The schema supports generic events with a stable shape: `created_at`, `id`, `app_id`, `user_id`, `event_at`, `name`, `data`, and `meta`. Optional `source` or `session_id` can be included only if planning proves they are useful.
- App-specific dimensions such as LingoCafe `book_id` and `page_id` move into the JSONB `data` payload instead of becoming core columns.
- The schema preserves ingestion cursor semantics with `(created_at, id)` for export and treats `event_at` as behavioral time.
- Existing LingoCafe event recording is migrated to the core writer. LingoCafe event names should use app-scoped dot names such as `book.info`, `page.open`, `page.scroll`, and `page.translate`; the row's `app_id` provides the app scope. Existing LingoCafe dimensions should be represented in `data`.
- A server-only library under `src/42go/events` exposes a small interface to record one event or a batch of events with payload and metadata.
- The server writer resolves or accepts `app_id`, prepares partitions, validates event names and payload boundaries, and inserts through Knex using `getDB()`.
- Client event collection is enabled per app through AppConfig, not globally.
- A protected server API endpoint accepts client-side event batches for the current app only when the current app enables the feature. The endpoint can require a valid session through config, and that should be the default for most apps.
- Client-submitted rows must be associated with a server-trusted `user_id` when the endpoint requires a session. This is mandatory for abuse investigation.
- A client-side facility provides a simple `trackEvent(name, data, options)` style interface with lightweight batching, configurable batch size/flush interval, frequent default flushing, flush-on-visibility/pagehide behavior, and no fragile dependency on analytics vendors.
- The client receives enough app config to know whether event logging is available for unauthenticated calls. If app code tries to use unavailable logging, the client should emit a console warning and drop or skip the event safely.
- Core user lifecycle events are recorded as `user.signup`, `user.login`, `user.logout`, `user.profile.created`, `user.profile.updated`, `user.consent.created`, and `user.consent.updated`.
- Login and signup events include the auth method in `data`. Logout has no extra data requirement in v1. Profile and consent update events store `prev` and `next` snapshots in `data`.
- Logout tracking is best-effort client-side in v1.
- VI41's export work is superseded by core events export. The existing `lingocafe-events-export` skill should become `42go-events-export`, targeting the new core schema while preserving the paired CSV/Parquet mechanics.
- A new local `42go-events-logging` skill documents how future features should integrate with core event logging. Its `SKILL.md` should stay short and link to load-on-demand subdocuments for details.
- Documentation explains how app-scoped event exports are stored locally, likely by app and date or with app fields in Parquet partition paths, without mixing unrelated apps accidentally.

## Definition of Success

The platform has a reusable core events system. LingoCafe no longer owns the only event table or event writer. Any server feature can record an app-scoped event through a small core API, browser code can queue client events safely when the app enables the feature, user lifecycle events are captured, and VI41's export mechanics are renamed and retargeted to export core events into app-compartmentalized CSV and Parquet archives.

## Additional Context

- The system is multi-app by request. App resolution starts in `src/proxy.ts` and flows through `getAppID()` / AppConfig.
- The event system must fit the existing PostgreSQL-only and Knex migration model.
- PostgreSQL events are a hot cache for recent writes and operational inspection, not the long-term analytics warehouse.
- Local filesystem analytics storage should remain gitignored and queryable from Python/DuckDB.
- The design should avoid per-feature event table proliferation.
- The design should not require external analytics services.
- Client-originated events are less trustworthy than server-originated events; server should stamp ingestion metadata and app identity.
- Logout is harder than login because browser sign-out currently uses NextAuth client calls from shared UI components. V1 accepts best-effort client-side logout event tracking.
- Dot namespace event names are the preferred convention, but enforcement should be limited to reasonable length and character validation. The convention belongs in the `42go-events-logging` skill.

## Assumptions

- The core schema is introduced through a new appended migration so it can be run on the remote database.
- Existing historical `lingocafe.events` rows do not need to be migrated into `events.events`.
- LingoCafe event code support should be removed from the application once current write points use the core APIs.
- The old `lingocafe.events` table creation and partition helper should be removed from current migration files for fresh local rebuilds. Production cleanup is manual and outside this task.
- The first implementation is authenticated-first. Anonymous/public telemetry is only available when an app explicitly enables it.
- Event payloads should remain JSONB in PostgreSQL and serialized into stable CSV/Parquet fields during export.
- The core writer should be server-only and should not leak database access into client bundles.
- The client batching facility can start small and does not need offline persistence in v1.
- The export env var should move from `LC_EVENTS_DATABASE_URL` to the core name `EVENTS_DATABASE_URL` as part of the renamed `42go-events-export` skill. When unset, the exporter should fall back to `DATABASE_URL`.

## Constraints

- Do not break existing LingoCafe event recording while moving the capability to core.
- Do not make app analytics depend on the LingoCafe schema.
- Do not trust client-provided `app_id`, `user_id`, `created_at`, or server metadata.
- Do not let one app record events for another app through the client batch endpoint.
- Do not add an external analytics vendor or warehouse in this task.
- Do not build dashboards in this task.
- Do not delete, truncate, or clean up existing PostgreSQL event rows.
- Do not make per-app partitioning so clever that migrations and export scripts become brittle.
- Do not add long-running analytics queries to production request paths.
- Keep payload validation and size limits server-side.
- Do not migrate old `lingocafe.events` data into the new core table as part of WS58.
- Do not keep LingoCafe-specific event libraries, endpoints, or table support after current write points have moved to core.
- Do not store passwords, OAuth tokens, cookies, secrets, full request headers, or excessive profile data in events.

## Acceptance Criteria

- A design note or output artifact records the chosen physical layout: one `events.events` parent table, monthly range partitions by `created_at`, mandatory `app_id`, and app/time indexes.
- The output artifact explains why app/date subpartitioning and one parent table per app are not used in v1.
- A new appended migration creates the core events schema, parent table, indexes, and partition preparation function.
- The core event table includes `app_id` and supports stable incremental export ordering by `(created_at, id)`.
- Core event table indexes are intentionally limited to hot-buffer extraction needs, especially `(created_at, id)` and app-scoped cursor/download access.
- Current migration files are updated so fresh local rebuilds no longer create `lingocafe.events` or `lingocafe.events_prepare_partitions()`.
- A server-only core event library exists under `src/42go/events` or an equivalent core location.
- The server library exposes a small API for recording one event and batching multiple events.
- The server library can resolve the current app ID safely or accept an explicit trusted app ID for cases where request context is unavailable.
- Existing LingoCafe event writes are routed through the core event writer, and the old LingoCafe event writer/table support is removed from application code.
- LingoCafe events use app-scoped dot namespace names without repeating the app ID.
- LingoCafe `book_id`, `page_id`, and similar dimensions are stored in `data`, not core columns.
- A client batch API endpoint exists and is protected by AppConfig feature/policy controls.
- The client batch endpoint can require a valid session through config, and the default documented posture is session-required.
- Client-submitted events are validated, size-limited, stamped with server-side app/user metadata, and inserted as app-scoped events.
- A client-side tracker exists with simple configurable batching, frequent default flushing, visibility/pagehide flushing, and a console warning when logging is unavailable for the current app/session state.
- `user.signup`, `user.login`, and `user.logout` are recorded as core events.
- Login and signup events include the auth method in `data`.
- `user.profile.created`, `user.profile.updated`, `user.consent.created`, and `user.consent.updated` are recorded when the corresponding `auth.users` JSON fields are first set or changed.
- Profile and consent update events include `prev` and `next` snapshots in `data`.
- Logout recording is best-effort client-side in v1.
- The `lingocafe-events-export` skill is renamed or replaced as `42go-events-export` and targets the new core events structure.
- The export env var becomes a core events env var such as `EVENTS_DATABASE_URL`.
- A new local `42go-events-logging` skill documents the core event logging API, naming convention, AppConfig controls, client tracking rules, payload safety, and examples for future features.
- The local archive layout prevents accidental app mixing, either by `app_id` in every row plus query guidance or by app/date partitioned filesystem paths.
- Tests or smoke checks cover migration partition/index creation, server writer insertion with resolved app ID, client batch rejection/acceptance, LingoCafe compatibility rewrite, auth lifecycle event logging, and export from the new core table.
- `npm run qa` passes after implementation.

## Open Questions

Q: Should the first core schema keep app-specific convenience columns such as `book_id` and `page_id`, or move all app-specific dimensions into `data`?
A: Move app-specific dimensions into `data`.

Q: Should old `lingocafe.events` rows be migrated into `events.events` as part of this task?
A: No. Do not migrate historical rows. Switch current write points to core and remove LingoCafe-specific event code/table support.

Q: Should the physical PostgreSQL partitioning be time-only with app indexes, list-by-app then range-by-time, or separate per-app parent tables?
A: Use one core parent table partitioned by `created_at`, with mandatory `app_id` and app/time indexes.

Q: What event name convention should core use?
A: Prefer dot namespace names such as `user.login`, `book.info`, and `page.open`. Do not prefix app-owned events with the app ID because rows are already scoped by `app_id`. Document this in `42go-events-logging`; enforce only reasonable length and character validation in code.

Q: Should client event collection be enabled per app through AppConfig?
A: Yes. It is strictly per app and protected by config-driven feature/policy controls. Most apps should require a valid session.

Q: How should logout be captured in v1?
A: Best-effort client-side tracking.

Q: Should the core export env var replace `LC_EVENTS_DATABASE_URL`?
A: Yes. The renamed `42go-events-export` skill should target the core schema and use `EVENTS_DATABASE_URL`, with fallback to `DATABASE_URL`.

Q: What retention policy should PostgreSQL hot-cache events eventually use after local archive durability is proven?
A: Out of scope for WS58.
