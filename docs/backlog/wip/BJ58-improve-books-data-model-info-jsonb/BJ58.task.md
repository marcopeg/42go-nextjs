# Improve Books Data Model With Info JSONB

## Elevator's Pitch

Update the LingoCafe books model so it records the source `project`, stores generic book metadata in a flexible `info` JSONB column instead of fixed-purpose columns such as `cover`, and uses that metadata to show optional book metrics on the book info page.

## Business Gain

This gives the content export flow room to ship richer book metadata without a migration for every new field. Chuck Norris does not add columns for every fact. The schema adds itself to his schedule.

## Current State

The `lingocafe.books` table includes a `cover` column.

The `lingocafe.books` table does not yet include a `project` column.

Book cover behavior has also been implemented in the reader UI and API flow, so removing the column may affect catalog and detail rendering.

The book info page already displays book metadata and details, but it does not yet use `info.words_count` or `info.chapters_count` as optional metrics.

The previous LingoCafe books seed file has been removed. There are no book seed rows to migrate or update as part of this task.

## Desired State

The `lingocafe.books` table no longer has the `cover` column.

The `lingocafe.books` table has a `project` field immediately after `id`. It is `text NOT NULL`.

The table has a generic `info` JSONB column that can hold different book-level data produced by the content source or export pipeline.

`lingocafe.books.info` is `jsonb NOT NULL DEFAULT '{}'::jsonb`, matching the style already used by `lingocafe.profiles.data`.

When displaying a book, the book info page reads `info.words_count` and `info.chapters_count` from the book payload and displays those two metrics when available.

If either key is missing, invalid, or not present for a book, the UI skips that metric instead of showing placeholders or broken values.

Book cover URLs are resolved only through the deterministic `/images/lingocafe/<book-id>.jpg` convention plus the existing placeholder fallback. There is no `books.cover` override and no `info.cover` override.

## Definition of Success

The database schema, API mapping, seed/export assumptions, and book info UI all agree on the new books model: `books.project` exists as required text, no persisted `books.cover` column remains, a generic JSONB metadata column exists, and optional word/chapter metrics render only when available.

## Additional Context

The operator asked to remove the column `cover` from the table `books`.

The operator later asked to add a `project` field after `id` in `books`; it should be `text NOT NULL`.

The operator asked to add a generic `info` JSONB field that will contain different data.

The operator asked that when displaying a book, the UI uses `info.words_count` and `info.chapters_count` to display those two metrics in the info page.

The operator clarified that those keys might not be available, and in that case the metrics should be skipped.

Existing cover work (`DG66`) inferred default cover URLs from book IDs and allowed `books.cover` as an override. The operator clarified that override support should be removed: cover rendering should be deterministic only.

The current export contract documentation (`BD89`) describes the current books table. It may need to be updated so future content exports write the new `info` field correctly.

The operator clarified that this task should update the existing LingoCafe schema migration directly, then rebuild and reseed the local development database with `make migrate.rebuild` and `make seed`.

The operator clarified that the old books seed has been removed and will be regenerated after this task. This task should not maintain or recreate book seed data.

During refinement, the operator clarified:

- Keep the database column and client-facing payload language as `info`; do not rename the book metadata object to `data`.
- Make `info` `jsonb NOT NULL DEFAULT '{}'::jsonb`.
- Remove cover override support entirely; use only deterministic cover resolution.
- Add `books.project` immediately after `id` as `text NOT NULL`.
- Update the existing `knex/migrations/20260427150000_lingocafe_schema.js` migration directly instead of creating a new migration file.
- After implementation, rebuild the local development database with `make migrate.rebuild` and run `make seed`.
- Do not preserve a rollback path for the old `books.cover` shape in this task.
- Do not recreate book seeds; they will be regenerated separately after this task is done.

IMPORTANT: operate the change in the current migration. This is a destructive change in the local development database. The development database will be destroyed and rebuilt. Do not create additional migration files to manage a soft migration, and do not use `alter table` for this change.

## Assumptions

The new generic metadata column should be named `info` in the database because the operator used that name.

The new `project` column represents the source project for the book and must be populated by future export/import examples.

The API and client payload should keep the word `info`, so the book info page reads `info.words_count` and `info.chapters_count`.

Existing deterministic cover URL inference from `book.id` is the only cover source after the removed `cover` override column.

The values for `words_count` and `chapters_count` are expected to be numeric counts.

There are no LingoCafe book seeds to update in this task because the operator removed them and will regenerate them later.
