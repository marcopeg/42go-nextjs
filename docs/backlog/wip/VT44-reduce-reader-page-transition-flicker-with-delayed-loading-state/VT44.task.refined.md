# Reduce LingoCafe Route Transition Flicker With Delayed Loading State

## Elevator's Pitch

Reduce flickering across LingoCafe page transitions by delaying route-level `Loading...` states. Wait up to `0.8s` before showing loading, and if loading does appear, keep it visible for at least `0.8s`.

## Business Gain

LingoCafe page transitions feel calmer and more deliberate instead of flashing loading placeholders for fast route changes. Chuck Norris does not blink every time a page changes. The loading card learns patience.

## Current State

Several LingoCafe routes currently switch into visible `Loading...` states immediately when their client-side fetch starts.

The original draft focused on the reader route at `/books/[bookId]/[pageId]`, where `loading` becomes `true` immediately and both reader surfaces show `Loading page...` right away.

Code lookup also shows the same immediate-loading pattern in the other LingoCafe route-level pages that fetch client-side data during navigation:

- `/books`
- `/books/[bookId]`
- `/books/[bookId]/[pageId]`

Today, those pages can visibly flash loading placeholders even when the next request resolves quickly.

## Desired State

When the user is already looking at a visible LingoCafe page and then navigates to another LingoCafe page, the UI should avoid jumping immediately into the loading placeholder.

The intended timing behavior is:

- wait up to `0.8s` before showing the loading phase
- if the request finishes before `0.8s`, do not show the loading phase at all
- if the request takes longer than `0.8s`, show the loading phase
- once the loading phase is shown, keep it visible for at least `0.8s`

This should apply only to in-app LingoCafe page transitions after a page is already visible. It should not apply to first direct load or browser refresh.

The current intended LingoCafe scope is:

- `/books`
- `/books/[bookId]`
- `/books/[bookId]/[pageId]`

The timing values should be easy to change from one place instead of being duplicated across pages.

## Definition of Success

LingoCafe page-to-page navigation no longer flickers into loading placeholders for fast requests. Slow requests still show loading, but only after the configured delay and for the configured minimum visible duration. The first direct load of a route stays unchanged. The timing values are centralized for easy tuning.

## Additional Context

This task started as a reader-only transition polish request and was later expanded to cover LingoCafe route transitions more broadly wherever a route-level `Loading...` state appears during client-side page changes.

The current behavior likely comes from each page setting `loading` to `true` immediately when its fetch starts.

The route-level pages in scope currently use the same general fetch-and-render pattern, so this likely wants a small reusable timing helper rather than one-off ad hoc delays.

## Assumptions

- Only post-render navigation transitions should be smoothed; first direct load and refresh should keep current behavior.
- Keeping the previously visible page on screen during the initial delay window is acceptable and preferred if that avoids flicker.
- The current LingoCafe routes with relevant route-level loading placeholders are `/books`, `/books/[bookId]`, and `/books/[bookId]/[pageId]`.
- The `0.8s` delay and `0.8s` minimum visible duration should remain the default product behavior, but should be centralized for easy tuning.
- Existing error handling should stay functionally the same unless a small adjustment is required to support the timing rule.

## Constraints

- Apply the delayed-loading behavior only to LingoCafe page transitions.
- Do not apply the delay rule to first direct load or browser refresh.
- Do not broadly change loading behavior across the whole app.
- Preserve the existing LingoCafe routes and client-side fetch model.
- Keep the timing values easy to change from one place.
- Preserve current navigation behavior, route correctness, and existing data fetch contracts.
- Preserve current error handling unless a small loading-state adjustment is needed to support the new timing rule.
- Do not regress progress restore or progress tracking behavior in the reader route.

- [ ] The loading state is delayed by up to `0.8s`.
- [ ] If the request resolves before `0.8s`, no loading phase is shown.

- [ ] The rule does not apply to first direct load or browser refresh.

- [ ] The change does not break successful page data loading.

## Dos

- Do scope the change to LingoCafe route transitions only.
- Do centralize the delay and minimum-visible timing values.
- Do preserve the currently visible page during the initial delay window when possible.
- Do verify books list, book detail, and reader routes because all are in scope.

## Don'ts

- Don't delay every loading state in the app.
- Don't apply the rule to first direct load or refresh.
- Don't duplicate the timing constants across multiple pages.
- Don't remove loading entirely for genuinely slow requests.
- Don't let this task spill into reader redesign or unrelated routing changes.
