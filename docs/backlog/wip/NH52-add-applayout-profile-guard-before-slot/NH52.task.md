# Add AppLayout Profile Guard Before Slot

## Elevator's Pitch

Add a `before` profile guard slot to `AppLayout` so configured guards can render instead of the normal app shell until they explicitly release it. Use the existing LingoCafe onboarding guard as the proving case.

## Business Gain

Incomplete-profile onboarding currently appears after the app layout has already started rendering. That creates a visible flicker: layout first, modal second. A before-layout guard removes the flash by letting blocking experiences render first, without route wrappers and without tying onboarding to `/books`.

## Current State

`YA00` completed the first onboarding extraction. LingoCafe now configures `app.profile.guard` with `slot: "layout"` and a lazy `LingocafeOnboardingGuard`.

The guard renders a blocking full-screen modal for authenticated LingoCafe users whose profile is incomplete. It collects own language, reading language, reading level, and configured consent items, saves through the existing `/api/profile` flow, and dispatches `profile:complete` after successful completion.

`/books` no longer owns the inline onboarding form. It renders a plain incomplete-profile note and does not load the catalog while profile data is incomplete.

The current `layout` slot renders over the normal app layout. For LingoCafe onboarding this means the app shell can briefly appear before the modal loader and profile check finish. That flicker is the problem to solve.

## Desired State

`AppLayout` supports an app-configured profile guard `before` slot in addition to the existing `layout` slot.

A configured `before` guard can render before normal AppLayout chrome, navigation, page content, layout effects, footer, or mobile menu appear. While the guard is blocking, the user sees only the guard surface.

The existing LingoCafe onboarding guard is moved to the `before` slot through configuration. The onboarding component should remain basically the same: it is still the existing modal-based full-screen onboarding surface, still uses existing profile and consent APIs, and should not become route-specific.

When onboarding completes and the server reports the profile as complete, the guard releases the app layout. The normal AppLayout should render underneath, then the onboarding modal should be allowed a short close/play-out period, around 300ms, so the transition hides the layout mount instead of flashing. After that, the guard is gone and the resolved app default route/content is visible. For LingoCafe after login, that means `/books` through the existing app default route behavior.

## Definition of Success

Incomplete LingoCafe profiles see only the onboarding component at first render, with no AppLayout flicker. Completed profiles render the normal app layout without an unnecessary blocker. Completing onboarding releases the guard, lets AppLayout render underneath, then closes the modal smoothly.

- `YA00` is completed and should be treated as the current baseline.
- Current implementation lives in `src/app/(app)/(lingocafe)/_components/LingocafeOnboardingGuard.tsx`.
- Current guard runner lives in `src/42go/layouts/app/ProfileLayoutGuard.tsx`.
- Current LingoCafe guard config lives in `src/config/lingocafe/config.ts`.
- Current profile guard typing only allows `slot: "layout"` in `src/42go/components/ProfileBlock/types.ts`.
- Profile persistence and consent evidence should continue through the existing profile system unless refinement chooses a separate, explicit backend change.

## Assumptions

- This task is a platform/AppLayout profile-guard capability, proved with LingoCafe onboarding.
- The `before` guard is route-independent. It must not care whether the current route is `/books`, `/profile`, book detail, or another authenticated app route.
- The existing LingoCafe onboarding fields and consent behavior are already implemented and should not be redesigned here.
- No new onboarding analytics or events are required for this task.
- The current `layout` slot should keep working for future or existing overlay-style guards.

## Constraints

- Do not make `/books` own onboarding again.
- Do not add route-specific onboarding logic.
- Do not redesign the LingoCafe onboarding form, fields, copy, or persistence flow unless required to adapt the guard contract.
- Do not create a second profile persistence or consent evidence path.
- Do not add onboarding analytics/events.
- Preserve the existing `layout` guard slot behavior.
- Preserve AppLayout policy behavior for the normal app shell after the before guard releases.
- Do not reintroduce route-local onboarding forms.
- Do not regress completed-profile access to `/books`, `/profile`, book detail, or reader routes.
- Preserve multi-app isolation and keep non-LingoCafe apps inactive.

## Acceptance Criteria

- Profile guard typing supports a `before` slot alongside the existing `layout` slot.
- AppLayout can invoke a configured `before` guard before rendering normal shell chrome and content.
- A blocking `before` guard can prevent the normal AppLayout shell from rendering.
- A completed/non-blocking `before` guard can release normal AppLayout rendering.
- The `before` guard contract gives the guard or guard runner a way to release the layout after profile completion.
- The release path can keep the modal visible briefly, around 300ms, while AppLayout renders underneath, then allow the modal to close smoothly.
- LingoCafe config uses the `before` slot for the existing onboarding guard.
- The LingoCafe onboarding surface stays modal-based and uses the existing profile/consent save flow.
- An incomplete LingoCafe profile sees only onboarding on initial render, with no visible sidebar/header/content flicker.
- After onboarding save completes and the server returns profile completion, the guard disappears and the normal app route is visible.
- For the default LingoCafe login flow, the visible route after completion is `/books` through existing config behavior.
- Users with completed profiles are not blocked by onboarding and can use normal LingoCafe app routes.
- Existing `layout` slot behavior still works for overlay-style guards.
- Non-LingoCafe apps without configured guards are unaffected.
- Desktop and mobile visual checks verify no initial layout flicker and a smooth completion transition.
- `npm run qa` passes after implementation.

## Dos

- Build on `YA00` instead of reopening solved extraction work.
- Keep the onboarding feature app-specific under the LingoCafe area unless a shared primitive earns its keep.
- Use the existing profile controller, consent component, and profile APIs where possible.
- Keep visual verification tied to the LingoCafe ngrok test flow when browser checks are needed.
- Treat the before slot as route-independent guard infrastructure.
- Keep the existing onboarding modal mostly intact and adapt only what the before-slot lifecycle needs.
- Prefer a small typed guard contract over route wrappers or hardcoded LingoCafe checks.

## Don'ts

- Do not make `/books` own onboarding again.
- Do not ask the before guard to know or choose routes.
- Do not hardcode LingoCafe behavior into shared AppLayout code beyond the existing guard configuration contract.
- Do not silently change profile completeness rules without matching API/server validation.
- Do not add new onboarding analytics in this task.
- Do not turn this into a broader onboarding UX redesign.

## Open Questions
