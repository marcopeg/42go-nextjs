# Show Language-Filtered LingoCafe Books

## Elevator's Pitch

Show the available LingoCafe books on the authenticated `/books` route, using the user's LingoCafe profile language preferences to filter the catalog by `target_lang` when that preference exists.

## Business Gain

This turns the books page from a fake shelf into a useful reading catalog. A learner should land on books they can actually study, not wander through every language like a villain looking for Chuck Norris in the wrong dojo.

## Current State

LingoCafe already has an authenticated `/books` page, but the current route renders a placeholder list instead of loading the real catalog from the LingoCafe data model.

The ongoing LingoCafe data-model work defines `lingocafe.profiles` with `own_lang` and `target_lang`, and `lingocafe.books` with a `lang` field. That task also states that `target_lang` is used to filter the books.

## Desired State

When a user lands on `/books`, the app makes one API call that returns both the current user's LingoCafe profile and the list of books to display.

If the profile has a `target_lang`, the books in the response are filtered to `books.lang = target_lang`.

If the user has no LingoCafe profile, the page shows an onboarding form asking for `own_lang` and `target_lang`. The form should use language option lists from the LingoCafe app configuration instead of hardcoded component-local lists.

If the user has a profile but `target_lang` is missing, the page should still avoid crashing and display available books.

## Definition of Success

The `/books` route displays LingoCafe catalog data, loads the current user's profile in the same request, filters books by `target_lang` when available, and presents onboarding when the user has no profile yet.

## Additional Context

The operator asked: "show list of available books for lingocafe. When landing on the books route, load the user's profile to learn the user's language preferences (own_lang, target_lang) and show the books filtered by target_lang if available."

This work likely belongs after, or alongside, the LingoCafe schema and seed data task because the route depends on `lingocafe.profiles` and `lingocafe.books`.

The first read API should return both `profile` and `books` in one response. This lets the page populate the current catalog and also retain user language information for a later language switch.

The first version should not include a manual language switch. Focus on displaying the books and preparing the response shape for future switching.

## Assumptions

The current user's identity should come from the existing authenticated session.

The books page should remain an authenticated app page under the existing `(app)` route convention.

If `target_lang` is missing on an existing profile, the page should fall back to the full available catalog rather than an empty list.

## Constraints

Follow the existing app-page convention: client component, `AppLayout`, and browser-side fetching with `credentials: "same-origin"`.

Use server-side validation in the API route. Do not trust client-provided user identifiers.

Use the LingoCafe schema and Knex `getDB()` access patterns already established by the data-model work.

Add custom LingoCafe configuration for onboarding language options in `src/config/lingocafe/config.ts`.

The `own_lang` onboarding list should cover a broad set of major European languages.

The `target_lang` onboarding list must be limited to English, Italian, Spanish, German, and Swedish for now.

## Acceptance Criteria

- [ ] The `/books` route loads books from the LingoCafe catalog instead of rendering the placeholder list.
- [ ] One API call returns both the authenticated user's LingoCafe profile and the books list.
- [ ] If the profile has `target_lang`, the displayed catalog is filtered to books where `books.lang` matches `target_lang`.
- [ ] If the profile is missing, `/books` shows an onboarding form asking for `own_lang` and `target_lang`.
- [ ] The onboarding language options come from the LingoCafe app configuration, not from component-local hardcoded arrays.
- [ ] The `own_lang` options cover a broad list of major European languages.
- [ ] The `target_lang` options are English, Italian, Spanish, German, and Swedish.
- [ ] If `target_lang` is not set on an existing profile, the page still displays available books.
- [ ] The first version does not include a manual language switch on the books page.
- [ ] The API response includes enough profile context to support a later language switch without changing the basic response shape.
- [ ] The implementation follows the app-page convention for routes under `(app)`.
- [ ] Server-side access uses the authenticated session and does not accept arbitrary user IDs from the browser.
- [ ] `npm run qa` passes after implementation.

## Dos

- Do reuse the LingoCafe `profiles` and `books` tables from the data-model work.
- Do return `profile` and `books` from one API call for the first page load.
- Do keep the default filter based on `target_lang`, not `own_lang`.
- Do handle users with no profile gracefully.
- Do show onboarding when there is no profile.
- Do put supported onboarding language lists in the LingoCafe app configuration.
- Do keep the UI aligned with the existing app layout and shadcn/Tailwind conventions.
- Do use seeded LingoCafe books and profiles as the first smoke-test path when available.

## Don'ts

- Don't keep the hardcoded placeholder books as the production data source.
- Don't fetch another user's profile based on a client-supplied ID.
- Don't make `own_lang` the catalog filter.
- Don't block the page entirely just because the user has not completed profile setup.
- Don't introduce a separate identity or preference system for LingoCafe.
- Don't add the manual language switch in this first version.
- Don't hardcode onboarding language lists inside the page component.

## Open Questions

Q: Should the language filter be read-only on first implementation, or should the user be able to change the visible language from the books page?
A: Read-only for now. Focus on displaying the books.

Q: Should missing profile state route the user into onboarding, or is showing all books enough for the first version?
A: If there is no profile, show an onboarding form asking for `own_lang` and `target_lang`.

Q: Should the books API return profile data and books together, or should profile and catalog be separate endpoints?
A: One API call should return both `profile` and `books`, so the page can populate the current book list and keep user language info available for a later language switch.

Q: Where should onboarding language option lists live?
A: Both lists should be custom configuration parameters in the LingoCafe configuration file. `own_lang` should use a broad list of major European languages. `target_lang` should use English, Italian, Spanish, German, and Swedish.
