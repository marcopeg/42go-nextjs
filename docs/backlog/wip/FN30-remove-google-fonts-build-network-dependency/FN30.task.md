# Remove Google Fonts Build Network Dependency

## Elevator's Pitch

Investigate why the Next.js production build tries to contact Google Fonts, then make the build reliable in Codex sandbox environments where outbound network access may be limited or unavailable.

## Business Gain

Reliable offline-capable builds reduce broken automation, unblock local and remote Codex execution, and make the project easier to build in constrained CI-style environments. The build should hit like a roundhouse kick: fast, direct, and not asking Google for permission.

## Current State

The build-time Google Fonts access path is confirmed in `src/app/layout.tsx`, which imports `Inter` from `next/font/google` and exposes it as `--font-inter`. `src/app/tailwind.css` maps Tailwind's `font-sans` token to `var(--font-inter)`, so the app shell depends on Next's Google font optimization during production build.

Previous task notes recorded sandboxed builds failing because `next/font` could not fetch Google Fonts, while escalated or online builds passed. The LingoCafe reader is separate from the app shell and currently uses web-safe font stacks in `src/app/(app)/(lingocafe)/books/_components/reader-preferences.ts`.

## Desired State

`npm install` and `npm run build` pass reliably without requiring outbound network access to Google Fonts after dependencies are installed. The app shell uses vendored Inter font files through `next/font/local`, preserving consistent typography without build-time or runtime Google Fonts requests. The LingoCafe reader remains on web-safe font stacks and should not require additional font downloads.

## Definition of Success

- The app is inspected for `next/font/google`, Google Fonts `<link>` tags, CSS `@import` rules, custom document/head/layout usage, global CSS usage, MUI theme usage, metadata configuration, or package-driven font loading.
- The exact file or package causing the Google Fonts access is identified.
- The cause is classified as `next/font/google` via `src/app/layout.tsx`, with Tailwind consumption through `src/app/tailwind.css`.
- The selected fix removes the `next/font/google` build path and replaces it with vendored local Inter font files.
- The plan records rejected alternatives: runtime Google CDN links and OS-dependent system stack as the final typography strategy.
- The recommended fix is implemented with the smallest safe code change.
- `docs/articles/THEMING.md` explains the system-font strategy and why it is compatible with Codex/sandbox builds.
- The LingoCafe reader font options are checked to confirm they remain web-safe and download-free.
- `npm install` and `npm run build` are run and their exact results are reported.
- If build still fails, the next external dependency or build blocker is identified.

## Additional Context

Candidate fixes considered:

- Self-host fonts locally with `next/font/local`.
- Replace `next/font/google` with runtime `<link>` tags.
- Use a system font fallback during CI/Codex builds.
- Vendor the required font files into the repository.

Operator decisions from refinement:

- Use vendored Inter font files through `next/font/local` so typography stays consistent without Google network access.
- Do not add runtime Google CDN links as the selected fix; they may avoid build-time fetching, but they keep a runtime network dependency and do not match the current goal.
- Confirm the LingoCafe reader stays on web-safe fonts only.
- Verification should prove the build no longer pulls network dependencies at build time; `npm install` and `npm run build` in the sandbox are acceptable proof if the source scan shows no build-time font fetch path remains.
- Document the strategy in `docs/articles/THEMING.md`.

The goal is not to redesign typography. The goal is to remove build-time external network dependency while keeping the app visually acceptable.

## Assumptions

- The Google Fonts request comes from project code using Next's `next/font/google` optimization.
- Installed npm dependencies should be the only required external input before `npm run build`.
- Production should preserve Inter typography through local vendored font files.
- The current reader font configuration is already web-safe and should not need font asset changes.
