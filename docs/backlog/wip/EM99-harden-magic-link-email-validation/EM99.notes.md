# Execution Notes — Harden Magic Link Email Validation
**Task**: ./EM99.task.md
**Plan**: ./EM99.plan.md

## Decisions

- Treat the operator's "plan and execute" as plan acceptance plus execution approval.
- V1 keeps validation static and global. No AppConfig policy knobs.
- V1 validates only the app-owned magic-link request/code paths and client UI. It does not modify NextAuth internals.
- Valid accepted email addresses preserve JD93 behavior: verified email-only login can still create app-scoped users.
- User-facing and server API validation errors remain generic.
- Provider alias policy sources:
  - Gmail: Google documents that dots do not matter for consumer Gmail addresses, while Workspace/custom domains can differ: https://support.google.com/mail/answer/7436150
  - Microsoft: Exchange Online documents plus addressing support: https://learn.microsoft.com/en-us/exchange/recipients-in-exchange-online/plus-addressing-in-exchange-online
  - Fastmail: Fastmail documents plus addressing and subdomain addressing for aliases: https://www.fastmail.help/hc/en-us/articles/360060591053
  - Proton: Proton documents plus aliases and hide-my-email aliases: https://proton.me/support/creating-aliases and https://proton.me/support/addresses-and-aliases
  - Yahoo: Yahoo documents disposable email addresses: https://help.yahoo.com/kb/SLN36718.html
  - iCloud: Apple documents iCloud Mail aliases and states aliases cannot create a separate Apple Account: https://support.apple.com/en-ie/guide/icloud/-mm6b1a490a/icloud
- Disposable-domain policy source candidates:
  - `disposable/disposable-email-domains` provides MIT-licensed generated disposable-domain lists and daily update language: https://github.com/disposable/disposable-email-domains
  - `disposable-email-domains/disposable-email-domains` provides a CC0/public-domain style blocklist and maintenance workflow: https://github.com/disposable-email-domains/disposable-email-domains
- V1 will vendor a compact static denylist of common disposable domains plus documented update guidance instead of adding a runtime dependency or network lookup.
- V1 will reject plus aliases globally because plus subaddressing is common across major providers and false positives are acceptable for this auth surface.
- V1 will reject dotted local-parts for consumer Gmail domains only (`gmail.com`, `googlemail.com`) because Google documents dot equivalence there and warns Workspace/custom domains can differ.
- V1 will allow privacy forwarding domains such as `privaterelay.appleid.com`, `mozmail.com`, `duck.com`, Proton/SimpleLogin-style aliases unless they appear in the explicit disposable denylist.
- Added `npm run test:email-validation` using Node's built-in test runner instead of adding Jest/Vitest dependencies.
