# Harden Magic Link Email Validation

## Elevator's Pitch

Strengthen passwordless magic-link sign-in by validating email addresses before any client request is sent and again on the server before any delivery attempt. Validation should be centralized in one shared 42go/core utility that both browser UI and server/API auth code use.

## Business Gain

Reducing invalid, aliased, and disposable email sign-ins improves account quality, protects sender reputation, reduces wasted email delivery attempts, and makes the magic-link authentication flow harder to abuse.

## Current State

- Magic-link/email-only login exists from JD93 and sends codes or magic links through configured sender strategies.
- The login UI can trigger the email auth request from Enter or the primary action.
- Email validation appears local or shallow in some components, and there is no known centralized reusable policy for auth-grade email validation.
- The existing completed task JD93 documents throttling by app ID plus normalized email, so validation and normalization semantics need to be compatible with that flow.

## Desired State

- A single shared email validation utility lives in reusable 42go code and can be imported safely by both client components and server/auth/API modules.
- The utility validates formal email syntax and applies strict auth-grade policy checks for provider alias behavior, disposable/temporary domains, and provider-specific duplicate-account risks where practical.
- The policy is static/global in v1. No AppConfig override is needed.
- The magic-link login UI validates only when the user submits with Enter or clicks the primary magic-link action. It does not validate while typing or on blur in v1.
- Invalid email input blocks the client-side magic-link request without triggering a client exception or network call.
- The app-owned server/API paths for email auth reject invalid email input before contact lookup, token creation, throttling mutation, or sender invocation.
- This task does not hack into NextAuth internals. It should not try to replace or deeply modify the direct NextAuth email sign-in path beyond using the shared normalization/validation where naturally available in existing app-owned code.
- Verified email-only login keeps the current JD93 behavior: valid accepted emails may still create app-scoped users after verification.
- User-facing and server API error messages are generic. They must not expose validation reason details or account existence.

## Definition of Success

- Invalid email input cannot trigger a client-side magic-link API call from Enter or button click.
- Invalid email input cannot cause the server to attempt a contact lookup or email delivery even if the API is called directly.
- Client and server imports use the same static validator.
- Unit tests cover formal syntax, provider alias rules, Gmail-style plus aliases, Gmail dot aliases, common provider behavior, disposable domains, privacy relay allowance, normalization, and expected decisions.
- Route-level tests or equivalent code tests prove invalid inputs are rejected before throttle mutation and sender dispatch.
- Implementation evidence includes `npm run qa` plus targeted auth validation tests.

## Additional Context

- The operator explicitly wants `marco@gmail.com` to be valid and `marco+foobar@gmail.com` to be invalid.
- The operator wants extensive implementation research into major email providers and common temporary/disposable email providers so LingoCafe avoids duplicate accounts for the same underlying mailbox as much as reasonably possible.
- Initial research signals:
  - Google documents that dots do not matter in consumer `gmail.com` addresses, while Google Workspace/custom domains may behave differently: https://support.google.com/mail/answer/7436150
  - Microsoft documents plus addressing in Exchange Online: https://learn.microsoft.com/en-us/exchange/recipients-in-exchange-online/plus-addressing-in-exchange-online
  - Yahoo documents disposable email addresses in Yahoo Mail: https://help.yahoo.com/kb/SLN36718.html
  - Community disposable-domain lists exist and include update workflows, such as `disposable/disposable-email-domains` and `disposable-email-domains/disposable-email-domains` on GitHub.
- Research should favor authoritative provider documentation first, then well-maintained public blocklists where licensing permits vendoring or adaptation.
- The selected disposable-domain list and provider-alias policy must be documented in both public authentication documentation and developer documentation.
- The documentation must state that disposable/alias provider lists need periodic review and updates because providers and disposable domains change.
- Privacy relay and forwarding providers such as Apple private relay, DuckDuckGo Email Protection, Firefox Relay, Proton aliases, and similar services are considered real accounts for v1 and should not be blocked only because they forward mail.

## Assumptions

- This task targets passwordless email/magic-link login first, not every optional email field in the product.
- Strict rejection of plus-address aliases is acceptable even though some real users rely on them.
- A curated denylist is acceptable if it is maintainable, tested, documented, and not treated as perfect protection.
- The validator must not depend on server-only modules if it is imported by client components.
- The current implicit account creation flow from JD93 remains unchanged for valid accepted email addresses.
- Rejecting more aliases is preferred over maximizing acceptance when the provider behavior indicates the address can represent the same underlying mailbox.

## Constraints

- Do not send email, create verification tokens, mutate throttle state, or perform contact/account lookup for invalid email input.
- Do not expose account-existence details through validation or auth errors.
- Do not expose structured validation reason codes in server responses or user-facing UI copy.
- Do not add a heavy runtime dependency unless the planner proves it is worth the bundle/server tradeoff.
- Do not add AppConfig validation-policy knobs in v1.
- Do not modify NextAuth internals or build a parallel email sign-in path.
- Keep the shared utility compatible with Next.js client/server boundaries.
- Use absolute imports and existing project style.

## Acceptance Criteria

- [ ] A shared 42go email validation utility exists and is safe for client and server imports.
- [ ] The validator performs formal syntax validation and returns only a valid normalized email or a generic invalid result to callers that surface errors.
- [ ] Implementation research covers major provider alias behavior and records the chosen rules in developer documentation.
- [ ] Gmail plus aliases such as `marco+foo@gmail.com` are rejected.
- [ ] Gmail dot aliases such as `mar.co@gmail.com` are rejected when the domain is a consumer Gmail domain and the dotted form would risk duplicate accounts.
- [ ] Known provider plus-address aliases are rejected where documented or strongly established, including at least Gmail and Microsoft-hosted consumer/business domains where appropriate.
- [ ] Other provider-specific aliases are rejected when implementation research identifies a reliable address-shape rule with acceptable false positives.
- [ ] Privacy relay and forwarding providers are not blocked only because they forward mail.
- [ ] A researched and maintainable temporary/disposable email domain block policy is implemented or integrated.
- [ ] Public and developer documentation explain the validation policy and periodic blocklist review/update expectation.
- [ ] Magic-link UI prevents Enter and send-button submission when the email is invalid and shows generic controlled validation copy.
- [ ] Magic-link API/server logic rejects invalid emails before contact lookup, verification token creation, throttling mutation, or sender calls.
- [ ] Unit tests cover valid normal emails, invalid syntax, missing TLD/domain errors, plus aliases, Gmail dot aliases, disposable domains, privacy relays, and expected normalization.
- [ ] Route-level tests or equivalent checks cover invalid-email rejection before throttle and sender execution.
- [ ] QA passes with `npm run qa`.

## Dos

- Do centralize validation logic in reusable 42go/core code.
- Do research common providers and disposable email sources before finalizing the deny policy.
- Do keep user-facing errors generic and safe.
- Do preserve existing credentials-only and OAuth login behavior.
- Do preserve JD93 verified email-only account creation for valid accepted addresses.
- Do add targeted tests for the validator and app-owned auth request path.
- Do document the update cadence and maintenance process for the disposable-domain list.

## Don'ts

- Do not implement separate client-only and server-only validators with diverging rules.
- Do not allow invalid emails to reach sender strategy code.
- Do not leak whether an email already belongs to an account.
- Do not silently normalize plus aliases into base addresses; reject them.
- Do not make the blocklist impossible to audit or update.
- Do not show provider-specific validation details to users.
- Do not expose structured validation reason codes through server responses.
- Do not add validation-policy configuration in v1.
- Do not block Apple private relay, DuckDuckGo Email Protection, Firefox Relay, Proton aliases, or similar forwarding accounts only because they forward mail.

## Open Questions

- Exact disposable-domain source, vendoring strategy, and update cadence should be finalized during planning after checking license, package footprint, and maintenance quality.
- Exact provider-specific alias rules should be finalized during implementation research and documented with sources.
