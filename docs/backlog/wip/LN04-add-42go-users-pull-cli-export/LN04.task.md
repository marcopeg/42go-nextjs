# Add 42go Users Pull CLI Export

## Elevator's Pitch

Add a `42go users pull` CLI command that refreshes authentication user data into Parquet files. The command should export `auth.users` rows without password hashes, and export selected `auth.accounts` fields without OAuth/token secrets. It should reuse the existing CLI export patterns where possible and be wired into `42go update` so fresh auth data is pulled before downstream aggregation commands run.

## Business Gain

Operators and analytics jobs can work with user and linked-account data locally without repeatedly querying production tables. Incremental refresh keeps local data current as users and linked accounts are created or edited, while Parquet output keeps the data ready for efficient offline analysis and future aggregation.

## Current State

The Python CLI already contains event export and aggregation commands, including Parquet-oriented event storage and `42go update` orchestration. User/account data currently remains database-only for analytics purposes. Related UI work is refining `/users` visibility into social logins and user activity, but the CLI does not yet provide a local auth data pull.

## Desired State

`42go users pull` downloads `auth.users` into `.local/42go-stats/_data/auth_users.parquet`, excluding password data. It also downloads `auth.accounts` into `.local/42go-stats/_data/auth_accounts.parquet`, exporting only these account fields: `app_id`, `account_id`, `user_id`, `type`, `provider`, `scope`, `token_type`, `expires_at`, `created_at`, and `updated_at`.

Incremental pulls must capture rows created or updated after the last local snapshot. Because create-time cursors cannot detect edits to existing rows, the implementation should use `updated_at` as the high-water mark and include stable identity columns as deterministic tie-breakers. For users, the deterministic ordering should be `(updated_at, created_at, id)`. For accounts, the deterministic ordering should be `(updated_at, created_at, app_id, account_id)`. The command rewrites the Parquet file atomically after merging fetched rows with existing local rows, de-duplicating users by `id` and accounts by `(app_id, account_id, provider)` or the effective account primary key used by the schema. If this upsert-style refresh proves materially too complex during planning or implementation, discuss a full delete-and-redownload strategy before changing the task.

`--reset` rebuilds both Parquet files from the source tables from the beginning. A normal run reads the existing Parquet files and metadata, fetches changed rows, merges them locally, writes atomically, and advances local metadata only after successful smoke validation.

`42go update` runs the users pull before aggregation commands so local user/account data is fresh when aggregations execute. If the users pull fails, `42go update` must fail and stop before aggregations run.

The first implementation should not partition these Parquet files. Keep one file for users and one file for accounts. The task should still document that partitioning can be revisited later, with likely candidates being `app_id`, created month, or both when row volume justifies the extra file layout.

## Definition of Success

The CLI can refresh local authentication datasets into `.local/42go-stats/_data/auth_users.parquet` and `.local/42go-stats/_data/auth_accounts.parquet` without exposing password hashes or OAuth secrets. Repeated pulls update local Parquet rows for both newly created and updated records, `--reset` rebuilds from source, and `42go update` refuses to run downstream aggregations when the auth refresh fails.

Partitioning decision: do not partition in the first implementation. Store stable single-file datasets under `.local/42go-stats/_data/`. Partitioning can be revisited later; partition by `app_id` if most reads become app-scoped, by created month if historical scans and append batches dominate, or by both only when row volume is high enough to avoid tiny-file churn. For account data, avoid partitioning on high-cardinality values such as `account_id`.

## Assumptions

- `created_at` and `updated_at` exist on both `auth.users` and `auth.accounts`.
- The command should connect to the same database source used by the existing CLI event pull machinery unless planning finds a repository-standard data source for auth exports.
- The exported account dataset is for analytics/account-link visibility, not credential restoration; therefore provider tokens and secrets should stay out of Parquet.
- Incremental behavior must detect both new rows and updates to existing rows.
- The local file root `.local/42go-stats/_data/` is the target layout for these two auth Parquet files.

## Constraints

- Do not export password hashes.
- Do not export OAuth tokens or provider credential secrets.
- Do not export account fields outside the approved account field list unless refinement explicitly changes the task.
- Use deterministic SQL ordering for all incremental pulls.
- Do not advance local metadata until merged Parquet output is written atomically and smoke-read successfully.
- Keep the command compatible with the existing installable Python CLI structure.
- Do not partition the first implementation.

## Acceptance Criteria

- `42go users pull` exists in the Python CLI and is discoverable from help output.
- The command exports `auth.users` to `.local/42go-stats/_data/auth_users.parquet`.
- The exported users dataset excludes `password` and any equivalent password-hash field.
- User incremental state captures rows created or updated after the previous local snapshot.
- User incremental ordering is deterministic and uses `updated_at` with `created_at` and `id` as tie-breakers.
- A normal run fetches users whose `updated_at` position is newer than the stored user cursor.
- The command merges fetched users into the existing local users Parquet data, de-duplicating by `id`.
- `--reset` clears or ignores the stored cursor and rebuilds the users export from the source table.
- The command exports `auth.accounts` to `.local/42go-stats/_data/auth_accounts.parquet`.
- The exported accounts dataset excludes OAuth/provider secret fields such as access tokens, refresh tokens, ID tokens, session state, token secrets, and similar credential material.
- The exported accounts dataset includes only `app_id`, `account_id`, `user_id`, `type`, `provider`, `scope`, `token_type`, `expires_at`, `created_at`, and `updated_at`.
- Account incremental state captures rows created or updated after the previous local snapshot.
- Account incremental ordering is deterministic and uses `updated_at` with `created_at`, `app_id`, and `account_id` as tie-breakers.
- A normal run fetches accounts whose `updated_at` position is newer than the stored account cursor.
- The command merges fetched accounts into the existing local accounts Parquet data, de-duplicating by the source account primary key `(app_id, account_id, provider)`.
- `--reset` rebuilds the accounts export from the source table as well as users.
- The command uses atomic writes for rewritten Parquet files.
- The command validates written Parquet files with a smoke read before advancing metadata.
- Cursor or metadata updates happen only after successful write completion and validation for the corresponding dataset.
- The first implementation keeps one Parquet file for users and one Parquet file for accounts, with no partitioning.
- `42go update` invokes the users pull before aggregation commands.
- `42go update` fails and stops if the users pull fails.
- The implementation documents the selected Parquet layout and the no-partitioning decision in code comments, CLI docs, or task notes.
- The implementation includes tests for cursor filtering, update merging, reset behavior, excluded sensitive fields, approved account fields, atomic write validation, and `42go update` command ordering/failure behavior.
- CLI tests pass.
- `npm run qa` passes if repository code outside the Python CLI changes.

## Dos

- Do use `updated_at` for incremental change detection because created-time cursors miss edits.
- Do keep `created_at` in exported data and in deterministic cursor tie-breakers.
- Do merge changed rows into existing local Parquet data instead of blindly appending duplicate records.
- Do keep the auth export files under `.local/42go-stats/_data/`.
- Do smoke-read Parquet output before accepting a run.
- Do treat users pull failure as fatal inside `42go update`.

## Don'ts

- Don't export `auth.users.password`.
- Don't export `auth.accounts.refresh_token`, `access_token`, `id_token`, `session_state`, or any similar provider secret.
- Don't partition auth Parquet files in the first implementation.
- Don't silently continue `42go update` aggregations after auth pull failure.
- Don't switch to full delete-and-redownload without discussing it first if upsert-style refresh becomes difficult.
