### 1. Should this task handle updates to existing users/accounts, or only newly created rows?

The requested cursors are creation-based: users use `(created_at, id)` and accounts use `(created_at, app_id, account_id)`. That is perfect for new rows. It will not detect edits to existing users, such as name/profile changes, email verification changes, or account metadata changes.

### 2. Which `auth.accounts` fields should be exported?

The draft assumes secrets stay out: access tokens, refresh tokens, ID tokens, session state, token secrets, and similar fields. Should the export include every non-secret account column, or only a small analytics shape like `app_id`, `user_id`, `type`, `provider`, `provider_account_id/account_id`, `scope`, `token_type`, `expires_at`, `created_at`, and `updated_at` if present?

Answer:

`app_id`, `account_id`, `user_id`, `type`, `provider`, `scope`, `token_type`, `expires_at`, `created_at`, and `updated_at`

### 3. Should Parquet be partitioned in the first implementation?

Good default options are:

- No partitioning yet: simplest, easiest to append, best for small/medium datasets.
- Partition by `app_id`: best if reads are usually app-scoped.
- Partition by created month: best for long historical scans and append batches.
- Partition by `app_id` and created month: strongest future shape, but can create too many tiny files early.

Answer:

ok, no partitions yet.
