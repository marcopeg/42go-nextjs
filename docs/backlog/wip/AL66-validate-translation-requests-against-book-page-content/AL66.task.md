# Validate Translation Requests Against Book Page Content

## Elevator's Pitch

Harden the LingoCafe translation backend so every uncached translation request is tied to a specific book chapter/page, verified against that content, and rejected unless it is one bounded sentence.

## Business Gain

Translation should only operate on text the user is actually reading. Server-side chapter/page validation prevents arbitrary freeform translation abuse, reduces provider-cost exposure, and keeps the endpoint safe even if a client is modified.

## Current State

The reader client sends `text`, `from`, `to`, `bookId`, `pageId`, and `sentenceId` to `/api/lingocafe/translate`. The backend schema already accepts `bookId` and `pageId`, but they are optional and are only used for event metadata. The backend currently authorizes the user, then calls `translateText(parsed.data)` without loading the page or verifying that the requested text appears in that page. `translateText` checks memory cache, database cache, and then Google Translate in one flow. The route caps `text` at 5000 characters, but there is no `LC_MAX_TRANSLATE_LENGTH` setting, no one-sentence guard, and no backend proof that the text belongs to the requested chapter/page.

## Desired State

Translation is controlled globally by `LC_ENABLE_TRANSLATE=true`; when the env var is missing or any other value, the reader UI and backend endpoint disable translation. The translation endpoint requires the sentence text plus `bookId` and `pageId`. Do not introduce `chapterId` unless a later task changes the reader/page model. The backend uses this order: global env enablement, cheap formal validation, cache-hit check, profile language validation, source-page validation, then upstream provider call only when needed. Cheap formal validation includes required fields, one-sentence shape, and `LC_MAX_TRANSLATE_LENGTH` with a hardcoded default of 500. The one-sentence full-stop guard is intentionally strict for v1: trim and normalize whitespace, reject any `.` that is not the final sentence character, allow zero full stops, allow one terminal `.`, and do not add abbreviation, decimal, or ellipsis exceptions in this task. After a valid memory or database cache hit, the endpoint can return the cached translation without loading the profile or page. If there is no cache hit, `from` must match the user's profile `targetLang`, `to` must match profile `ownLang`, and `from` must match the loaded book language before Google can be called. The backend loads that exact page on the server and verifies that the requested sentence exists in the page before calling the provider. The page-content check should include rendered translatable page surfaces: title, summary, and markdown body content, using intentionally loose normalization.

## Definition of Success

A modified client cannot use `/api/lingocafe/translate` to translate arbitrary uncached text. Valid cached sentences return quickly after cheap formal validation. Valid uncached reader sentences from the requested page still translate normally. Text that is not one sentence, exceeds the configured character limit, is not found in the requested page, or uses missing/unknown page identifiers fails with clear client-safe errors.

## Assumptions

- The safety check should use `bookId` and `pageId` from the request body rather than deriving them from session or referrer.
- The backend should keep `pageId` as the required page identifier; do not add `chapterId` in this task.
- `LC_MAX_TRANSLATE_LENGTH` is the intended environment variable name for the character limit.
- `LC_ENABLE_TRANSLATE` is the global enablement switch and defaults off unless set exactly to `true`.
- A cache hit may short-circuit expensive chapter membership validation, but only after the payload passes basic text shape and length validation.
- A cache hit may also short-circuit profile language validation because cache reads do not spend Google budget.
- On cache misses, `from` should match the user's learning language (`targetLang`) and the loaded book language; `to` should match the user's own language (`ownLang`).
- Single-sentence validation should use a deliberately strict v1 full-stop guard: reject any internal `.`, allow zero full stops, allow one terminal `.`, and do not support abbreviation, decimal, or ellipsis exceptions yet.
- The source text may come from page title, page summary, or page body content because all three are rendered with sentence wrappers in the reader.
- Page-membership matching should be loose enough to prove the text is from the book without needing a perfect rendered-text reconstruction.
- Whitespace, markdown formatting, and simple punctuation differences may need normalization so a rendered sentence can match stored markdown content reliably.

## Constraints

- Do not rely on the browser to enforce page membership.
- Do not call Google Translate for uncached text that fails single-sentence, length, book/chapter, or membership validation.
- Do not let missing `bookId` or `pageId` fall back to the current optional behavior.
- Do not leak full page contents or sensitive validation details in error responses.
- Do not call Google when `LC_ENABLE_TRANSLATE` is not exactly `true`.
- Do not break translation cache behavior for valid sentences.
- Do not silently truncate text to fit the configured maximum.
- Do not add a new `chapterId` payload field unless planning proves the current `pageId` model cannot satisfy the requirement.
- Do not call Google for uncached requests whose `from`/`to` values do not match the reader profile direction.
- Do not call Google for uncached requests whose `from` value does not match the requested book language.

## Acceptance Criteria

- `/api/lingocafe/translate` requires the sentence text, `bookId`, and chapter/page id in the payload.
- The backend and reader payload use `LC_ENABLE_TRANSLATE=true` as the translation enablement switch.
- Missing or non-`true` `LC_ENABLE_TRANSLATE` disables reader translation and makes the translate endpoint return `403`.
- The backend reads `LC_MAX_TRANSLATE_LENGTH` and falls back to 500 when the variable is missing, invalid, or unsafe.
- The backend rejects text longer than the configured max before page/chapter existence checks and before provider calls.
- The backend rejects multi-sentence text or text with full stops in the middle before page/chapter existence checks and before provider calls.
- The backend allows zero full stops and one terminal `.`, with optional simple support for closing quotes/brackets after the terminal full stop.
- The backend does not add abbreviation, decimal, or ellipsis exceptions in this task.
- A valid memory or database translation cache hit can return after cheap formal validation without loading the page and without calling the provider.
- On cache misses, `from` must match the user's profile `targetLang` and `to` must match profile `ownLang` before provider use.
- The backend loads the requested page with `loadBookPage(bookId, pageId)` or equivalent server-side logic.
- On cache misses, `from` must match the loaded book language before provider use.
- Unknown or missing pages return a clear error before translation work begins.
- The backend checks that requested `text` exists in the requested page's title, summary, or body content after documented loose normalization.
- Text that does not exist in the requested page is rejected before provider calls and before cache insertion.
- Oversized requests return a clear validation error.
- Valid requests for text that appears on the page continue to work through memory, database, and provider translation paths.
- Translation event logging still records `book_id`, `page_id`, and `sentence_id` for accepted requests.
- Malformed payloads, oversized text, multi-sentence text, and text not found in the requested page return `400`.
- Unknown book/page identifiers return `404`.
- Disabled translation returns `403`.
- Tests or smoke checks cover max length, multi-sentence rejection, missing identifiers, unknown page, text not found on page, cache hit behavior, and a valid uncached page sentence.
- `npm run qa` passes after implementation.

## Dos

- Make `bookId` and `pageId` required in the Zod payload schema.
- Split cache lookup from provider translation if needed so uncached text can be validated before provider use.
- Keep cheap sentence-shape and length validation close to payload parsing.
- Normalize text comparison intentionally and document the chosen loose normalization.
- Use `LC_MAX_TRANSLATE_LENGTH` with an explicit default of 500.
- Use `LC_ENABLE_TRANSLATE` as the product-level translation switch with an explicit default-off behavior.
- Keep successful response shape compatible with the current reader client.

## Don'ts

- Do not keep `auth.users.feature_flags.translate` in the runtime translation enablement path.
- Do not keep the current 5000-character schema cap as the effective business limit.
- Do not silently truncate user text and translate the truncated version.
- Do not trust `sentenceId` as proof that text belongs to the page.
- Do not move this into client-only validation.
- Do not run page-membership database checks before cheap sentence-shape and length checks.
- Do not call the translation provider before confirming uncached text belongs to the requested chapter/page.
- Do not make page-membership matching so exact that harmless markdown/rendering differences reject normal reader sentences.
- Do not build an abbreviation dictionary or broad natural-language sentence parser in this task.
