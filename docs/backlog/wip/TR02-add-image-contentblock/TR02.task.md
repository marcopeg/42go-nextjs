# Add Image ContentBlock


## Elevator's Pitch

Add an "image" ContentBlock type for composable pages.

## Business Gain

Richer landing pages can be assembled from AppConfig without custom page components. The image block should integrate with the existing composable page ContentBlock system alongside hero, markdown, stack, CTA, and other shared blocks.

## Current State

Composable pages already render server-side ContentBlock items from AppConfig. Existing server blocks include hero, markdown, component, link, pricing, waitlist, feedback, CTA, and stack.

There is no dedicated image ContentBlock for landing-page sections that combine an image with optional markdown side content.

The shared `ScrollAnimation` component exists and currently supports `fade`, `scale`, and `slideUp`.

## Desired State

Add a server-only `image` ContentBlock for landing-page content.

The block renders an image with optional side content. Side content is markdown.

The public configuration API should follow this final shape:

```ts
{
  type: "image";
  image: {
    src: string;
    alt: string;
    width: number;
    height: number;
    sizes?: string;
    align?: "left" | "right" | "top" | "bottom";
  };
  content?: {
    source?: string;
    path?: string;
  };
  imageAnimation?: "none" | "fade" | "scale" | "slideUp";
  contentAnimation?: "none" | "fade" | "scale" | "slideUp";
}
```

Side content should align with the existing `markdown` block naming so implementation can reuse the same model:

```ts
content?: { source: string; path?: never } | { path: string; source?: never };
```

`content.source` is inline markdown. `content.path` is a path to a markdown file to read and render. If `content` is provided, exactly one of `content.source` or `content.path` must be provided.

`imageAnimation` and `contentAnimation` default to `"none"`. Optional animation values are limited to the current `ScrollAnimation` capabilities:

```ts
imageAnimation?: "none" | "fade" | "scale" | "slideUp";
contentAnimation?: "none" | "fade" | "scale" | "slideUp";
```

The block should use Next.js `Image` with explicit dimensions:

```ts
image: {
  src: string;
  alt: string;
  width: number;
  height: number;
  align?: "left" | "right" | "top" | "bottom";
  sizes?: string;
}
```

For `next/image`, `sizes` tells the browser which rendered image width to expect at different viewport sizes so it can choose the right generated source from the responsive `srcset`.

For config string URLs, Next.js does not automatically calculate image dimensions at build time. Automatic dimensions are available when statically importing local images into code, but AppConfig public pages use serializable string paths and URLs. Therefore `width` and `height` are required for this block.

On desktop, `image.align` controls where the image sits relative to the optional content. On mobile, `left` and `right` behave like `top`, so the image stacks above content.

Use the existing `ScrollAnimation` component. Do not extend `ScrollAnimation` in this task. The `image` block should only expose animation values that can be supported by the current component, plus `none` as an opt-out.

Default home page examples should include a new committed demo image under `public/` and an external-image example using a generic remote image allowlist.

Replace the current LingoCafe-specific `LC_ASSETS_BASE_PATH` remote image configuration with a generic 42go boilerplate remote image allowlist. The environment variable should accept a list of URL patterns, and the deployed LingoCafe service must explicitly include `https://assets.lingocafe.app/**` in that list to keep LingoCafe external assets working.

Prefer a generic environment name such as `CONTENT_IMAGE_REMOTE_PATTERNS` or `PUBLIC_CONTENT_IMAGE_REMOTE_PATTERNS`. The final implementation should choose the clearest project-consistent name and document why. The important rule is that the setting is generic to public/content images and is not LingoCafe-specific.

## Definition of Success

Page authors can add image-led landing-page sections from AppConfig without custom components. The block supports optional markdown side content, responsive stacking, server rendering, documented configuration, no-animation defaults, and examples on the default home page.

## Additional Context

Round 1 clarified that this is for landing-page content only, not toolbar actions or authenticated app pages.

Round 2 clarified that animation defaults should be `"none"` and that side content should align with the existing markdown block naming: inline markdown uses `source`, file-backed markdown uses `path`.

Round 3 chose the strict Next.js `Image` API with required `width` and `height`. The `sizes` field must be documented thoroughly and used in the default config examples.

Round 3 chose a generic env-driven remote image allowlist. The current `next.config.ts` supports remote `next/image` URLs only for the configured LingoCafe assets base path, defaulting to `https://assets.lingocafe.app/**`. That LingoCafe-specific path should be replaced by generic boilerplate configuration. CSP already allows HTTPS images through `img-src 'self' data: https:`.

The operator expects to perform the final browser visual test and report feedback after implementation.

## Assumptions

- The block should be registered only in the server ContentBlock renderer.
- The implementation should reuse existing markdown rendering patterns where practical.
- Because AppConfig is developer-authored static configuration, image dimensions are required for the strict Next.js `Image` API.
- New demo images can be committed under `public/` and referenced through stable public paths.

## Constraints

- Keep the task scoped to adding the image block. Do not redesign the ContentBlock system.
- Do not add the image block to the client ContentBlock renderer.
- Do not extend or replace `ScrollAnimation` in this task.
- Preserve existing ContentBlock behavior and existing composable page configs.
- Do not broaden `next/image` remote hosts casually. Remote images require explicit `next.config.ts` allowlisting.
- Remote image allowlisting must be generic to the 42go boilerplate and must not remain hard-wired to LingoCafe.
- Existing LingoCafe remote asset usage must be preserved by documenting that deployments must include `https://assets.lingocafe.app/**` in the new generic allowlist.
- Run lint, build, and type checks through the repository QA path after implementation.

## Acceptance Criteria

- Server ContentBlock supports a typed `image` block.
- The block renders an image with required accessible alt text.
- The block uses Next.js `Image`.
- `image.width` and `image.height` are required.
- `image.sizes` is optional, documented thoroughly, and used in the default config examples.
- The block can render optional markdown side content from either inline `content.source` or file-backed `content.path`, but not both.
- Desktop layout supports image alignments `left`, `right`, `top`, and `bottom`.
- Mobile layout stacks `left` and `right` as image above content.
- Image and content reveal animation are configurable using only current `ScrollAnimation` capabilities.
- Image and content animation default to `"none"`.
- Default app home page includes multiple image block examples, one under the other.
- Default app home page includes a committed local demo image example.
- Default app home page includes an external image example that works through the generic remote image allowlist.
- `next.config.ts` replaces the LingoCafe-specific remote image setting with a generic env-driven list of remote image patterns.
- LingoCafe external assets continue to work when `https://assets.lingocafe.app/**` is included in the generic remote image pattern list.
- Public composable-page documentation documents the image block API and examples.
- The `42go-pages` skill is updated so future agents know how to use the image block.
- Lint, build, and type checks pass.

## Dos

- Reuse existing ContentBlock registration patterns in `src/42go/components/ContentBlock/server.tsx`.
- Reuse existing markdown and scroll animation conventions.
- Prefer a configuration API that is easy to type and hard to misconfigure.
- Keep the desktop and mobile layout behavior explicit in code and docs.
- Explain `sizes` in the public docs and examples.
- Make remote image configuration generic to the boilerplate.

## Don'ts

- Do not add client-side availability for this block.
- Do not introduce new animation primitives.
- Do not accept both `content.source` and `content.path` at the same time.
- Do not silently allow arbitrary remote image hosts without explicit Next.js image configuration.
- Do not leave remote image allowlisting tied to `LC_ASSETS_BASE_PATH`.
- Do not require custom page components for common image + content landing-page sections.
