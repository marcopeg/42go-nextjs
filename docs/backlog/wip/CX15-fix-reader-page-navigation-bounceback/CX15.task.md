# Fix Reader Page Navigation Bounceback

## Elevator's Pitch

Fix the LingoCafe reader bug where advancing from the end of a page with Next or a swipe briefly loads the next page, then snaps back to the page the user was just reading.

## Business Gain

Reader navigation must feel reliable. A page turn that appears to work and then reverses breaks concentration and makes the reader feel possessed. Chuck Norris turns a page once. The page stays turned.

## Current State

Users often, but not always, see this sequence in the LingoCafe reader: they navigate from one page to the next page, the next page appears correctly for a brief moment, then the UI immediately returns to the previous page. Repeating the same page change usually works.

The bug is not tied to a specific book or chapter; it has been observed consistently across many books. It is also not limited to reaching the bottom of the page. It can happen from any scroll position when swiping to the next page, and it can happen when clicking the bottom Next control.

The confirmed failing environment is Chrome on iPhone, including the home-screen installed app/PWA launched through Chrome. Desktop Chrome does not reproduce it, and mobile Safari appears to work.

The strongest current reproduction pattern is the first page turn after the app has been idle in the background for a while, has resumed from the home-screen installed app, or has been unloaded and reopened. The first attempted page change can move from page 1 to page 2 and immediately snap back to page 1. After that first bounceback, subsequent page turns usually work normally: page 1 to page 2 succeeds on the second attempt, then page 2 to page 3 succeeds.

The latest hypothesis is a race between restoring or reopening the last read page and the first explicit user navigation to a new page. This may involve persisted `readingAction` resume hrefs built from `lingocafe.books_progress`, the initial reader page load, `saveBookOpenProgress()`, native history updates, or stale route synchronization after PWA wake/resume.

Current reader code includes client-side route state, native history updates, delayed loading behavior, scroll/progress restoration, and POST-based progress persistence, so this may be a race between navigation state, URL synchronization, scroll restore, progress persistence, or stale in-flight fetch results.

## Desired State

Reader page changes should be stable on the first attempt. When the user clicks Next or swipes to the next page from any scroll position, the reader should load and remain on the intended next page.

The first explicit user page-turn after a Chrome iPhone PWA wake/resume, background idle period, or unload/reopen must be treated as authoritative over any pending resume-last-read or restore-last-state behavior.

While a next-page load is in flight, additional Next or swipe inputs should be ignored so a second gesture cannot collide with the first page turn. The same rule applies to previous-page navigation if the same underlying race affects it.

The fix should preserve current delayed-loading behavior, scroll restoration, progress persistence, browser back/forward support, and table-of-contents navigation.

## Definition of Success

The first Next click or first qualifying next-page swipe consistently lands on the target page and does not bounce back to the source page, especially after Chrome on iPhone resumes the home-screen installed app from idle/background/unloaded state. The fix includes enough instrumentation, reproduction evidence, or tests to prove the stale resume/restore/navigation path has been neutralized instead of hidden.

## Assumptions

- The issue is intermittent and probably timing-sensitive.
- The second navigation attempt working suggests stale route/progress state or an aborted/older async effect may be winning after the first transition.
- The first-page-turn-only pattern suggests the stale actor may come from initial reader restore, PWA resume, or last-read route reconstruction rather than from ordinary steady-state page navigation.
- A robust fix should guard against stale fetch, stale URL-derived route state, and stale progress writes rather than relying on a delay.
- The bug can be reproduced with multiple LingoCafe books through Chrome on iPhone.
- The immediate implementation should treat Chrome on iPhone as the primary manual verification target.

## Constraints

- Keep the authenticated app-page client-side data fetching model.
- Do not remove delayed-loading behavior from `VT44` unless the investigation proves it is the cause and replaces it with equivalent UX.
- Do not regress scroll restoration, reading progress persistence, browser back/forward navigation, table-of-contents navigation, or reader settings interactions.
- Ignore additional page-turn inputs while the target page is loading.
- Do not turn this into a reader redesign or a bookshelf task.
- Run `npm run qa` after code changes during execution.

## Acceptance Criteria

- A reproducible investigation identifies the bounceback source or a defensible race condition in the reader navigation flow.
- Clicking Next at the end of a page loads the target page and keeps the reader there on the first attempt.
- Swiping to the next page from any scroll position loads the target page and keeps the reader there on the first attempt.
- The page does not briefly show the target page and then revert to the previous page during normal next-page navigation.
- The first page turn after Chrome on iPhone resumes or reopens the home-screen installed app does not bounce back to the previous page.
- Resume-last-read or restore-last-state behavior cannot override an explicit user page-turn that happens after wake/resume.
- Additional Next or swipe inputs are ignored while a target page is already loading.
- Previous-page navigation is checked and fixed too if affected by the same race.
- Browser back and forward still update the reader page correctly.
- Table-of-contents navigation still updates the reader page correctly.
- Scroll restoration still lands at the expected position for opened pages.
- Progress persistence still saves the active page and progress without overwriting a newer page with stale data.
- Fast repeated navigation gestures do not allow an older fetch or progress write to win over the latest intended page.
- `npm run qa` passes after implementation.
- Manual verification covers Chrome on iPhone next-page swipe and bottom Next click through the LingoCafe ngrok test URL or an equivalent deployed reader session.
- Manual verification includes leaving the Chrome iPhone installed app idle/backgrounded long enough to trigger the first-page-turn failure mode, then confirming the first page turn after resume stays on the target page.
- Manual verification checks desktop Chrome as a non-regression path even though it does not currently reproduce the bug.
