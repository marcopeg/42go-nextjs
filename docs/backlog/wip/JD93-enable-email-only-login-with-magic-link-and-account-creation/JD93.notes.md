# Execution Notes - Enable Email-Only Login With Magic Link and Account Creation
**Task**: ./JD93.task.md
**Plan**: ./JD93.plan.md

## Decisions

- Used NextAuth v4 EmailProvider with a conditional app-scoped Knex adapter. The adapter is installed only when the resolved app config includes `type: "email"` so apps without email auth keep their existing adapter-free path.
- Added `app_id` to `auth.verification_tokens` and made `(app_id, identifier, token)` the primary key.
- Added `auth.email_auth_throttle` keyed by `(app_id, identifier)` with `meta` reserved for future IP/user-agent expansion.
- Manual code entry posts to `/api/auth/email/verify-code`; the route validates the app-scoped hashed token, then redirects the browser to the normal NextAuth email callback. It does not mint sessions itself.
- `AUTH_SECRET` is passed explicitly to NextAuth and is the only supported auth secret variable.
- Removed the hard-coded `pages.newUser = "/signup"` redirect because `/signup` does not exist and first-time email login must land on the callback URL.

## Problems Encountered

- NextAuth EmailProvider imports `nodemailer` at module load time even when using a custom sender. Added `nodemailer` and `@types/nodemailer`.
- Initial manual-code verification failed because NextAuth was deriving a secret while the manual verifier used the configured auth secret. Passing the secret explicitly fixed token hashing alignment.
- Ngrok requests exposed that callback URLs built from `req.url` could become `https://localhost:3000/...`. The manual-code route now honors forwarded host/proto headers.

## Deviations From Plan

- Resend is implemented through direct HTTP `fetch` instead of adding a Resend SDK dependency.
- The request throttle is enforced by the app-owned `/api/auth/email/request` route used by the UI. Direct calls to NextAuth's email sign-in endpoint still rely on token expiry and one-time consumption; deeper hardening can move throttling closer to token persistence in a follow-up.

## Additional Requests

- After live login testing, changed the credentials-plus-email UI so the first screen shows the email/username field with primary `Continue` for email code/magic-link and secondary `Continue with password`. Email-only hides the password option; credentials-only still uses username plus password fields.
- Renamed email auth event names from underscore form to dash form (`auth.email.code-verified`, `auth.email.login-failed`) to satisfy the core event validator.
- Changed the email primary action default to `Send me a magic link` and made it configurable through `auth.providers[].config.ui.primaryActionLabel`. Changed `Continue with password` to a borderless link-style button.
- Added public authentication documentation at `contents/default/docs/authentication/README.md`, expanded the internal email-auth article, and created the `42go-auth` skill for future auth changes.
- Replaced email token lifetime config from `maxAgeSeconds` to `code.duration` using explicit duration strings such as `"30s"`, `"5m"`, and `"1h"`. Plain numbers now fail app config validation during build/startup.
- Replaced overlapping throttle config (`delaySeconds` plus `progressiveDelaySeconds`) with one ordered `throttle.delay` list. Default is `["30s", "1m", "2m", "3m", "5m", "10m"]`; the last value is reused for later attempts.
- Removed generic email-auth fallback environment wiring from the LingoCafe config. LingoCafe now explicitly selects the Resend sender and requires `LC_RESEND_API_KEY`.
- Removed redundant default `code`, `throttle`, and `ui` blocks from the LingoCafe email provider config. The shared defaults now own code generation, resend throttle sequence, and primary action label.
- Renamed email provider delivery config from `senders` to `strategies` so it lines up with `useStrategy`. LingoCafe now includes both `console` and `resend` strategies and can switch locally with `LC_EMAIL_AUTH_STRATEGY=console`.
- Audited and updated public docs, internal email-auth article, the `42go-auth` skill, and the SMTP follow-up draft to use `strategies`, explain the `type` discriminator, and document local console usage.
- Final auth secret standard is `AUTH_SECRET` only. Removed `NEXTAUTH_SECRET` fallback from NextAuth options, email token hashing, docs, examples, and the local env key.
- Removed static `NEXTAUTH_URL` from active environment examples and local `.env`. The auth route still sets `process.env.NEXTAUTH_URL` internally per request as a NextAuth v4 bridge for multi-app hostnames.

## Known Limitations

- OAuth regression was not live-tested because external Google/GitHub credentials and provider callbacks were not exercised in this run.
- Resend live delivery was not tested because no real API key was used. The sender implementation builds a controlled error when the key is absent.
- The validation matrix is partially complete: console/local request, code screen, app-scoped token storage, manual-code callback, new verified user creation, token one-time consumption, credentials login, throttle, migration status, and QA passed.
