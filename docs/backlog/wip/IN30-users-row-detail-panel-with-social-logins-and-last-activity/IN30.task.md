# Users List Social Logins and Last Activity

## Elevator's Pitch

Improve the authenticated `/users` backoffice feature so the users list shows linked social login providers and each user's latest available activity. Social logins should be represented with provider icons in the user row. Last activity should show the activity date, and hovering or focusing the date should reveal the latest event name. Clicking a user row, or choosing `View user details` from the row menu, opens a read-only detail panel with formatted user information and user actions.

## Business Gain

Backoffice operators can scan the users list and immediately understand how each user signs in and whether the account is active. Social login visibility helps diagnose account-linking issues. Last activity gives operational context without forcing operators to open raw telemetry or inspect the database. The detail panel lets operators inspect a user cleanly without jumping straight into raw edit fields.

## Current State

The `/users` page shows a table loaded from `/api/users`. Rows display user name, email, ID, and join date. The action menu can open an edit panel with raw editable fields and reset actions. `/api/users` lists fields from `auth.users` only and does not include linked accounts or latest event data. There is no read-only user detail panel.

## Desired State

The `/users` list displays each user's login methods and linked social login providers. Linked providers come from `auth.accounts`, support zero, one, or multiple providers, and render as recognizable icons for GitHub, Google, and future configured social providers. Unknown providers use a neutral fallback. If a user has no social provider rows, the UI shows a credentials-login icon when `auth.users.password` is present, otherwise a magic-link/email icon.

The `/users` list also displays `last activity at` for each user using the latest available `events.events` row scoped by `app_id` and `user_id`. Latest activity uses `event_at DESC`. The visible value is the event date/time. Hovering or focusing the value reveals the event `name`. Users with no event history show an empty state such as `Never`.

Clicking a populated user row opens a right-anchored read-only user detail panel. The row's three-dot action menu still works and includes `View user details` as the first item. The detail panel includes an `Edit` action that opens the existing edit panel for the same user. It also includes a red ghost `Delete` placeholder action; after confirmation, it shows `User deletion not yet implemented` and does not delete anything.

The detail panel shows formatted user fields, excluding `app_id` and all sensitive fields. `email_verified` is represented with a verified or unverified icon. When verified, hovering, focusing, or tapping the icon reveals the verification date in readable format. The panel shows user identity, email, username, name, image, profile, consent, feature flags, created date, updated date, social/login method icons, and the latest activity event date and name.

The existing edit action menu, reset actions, policy protection, and browser-side loading model keep working.

## Definition of Success

The `/users` table becomes an operator-friendly account overview: each row includes login-method status and latest activity without exposing OAuth secrets or telemetry payloads. Operators can open a detail panel from the row or row menu, inspect formatted user data, and jump to the edit panel. The implementation includes any database indexes needed to avoid slow joins as the number of users and events grows.

## Additional Context

Relevant schema observations:

- `auth.users` is app-scoped by `app_id` and user `id`.
- `auth.users.password` can be used server-side to infer credentials-login availability, but the password hash must never be returned to the browser.
- `auth.accounts` stores OAuth/social provider links with `app_id`, `user_id`, `provider`, and token columns. It currently has a primary key on `(app_id, account_id, provider)`, which helps provider-account lookup but not list-page joins by `(app_id, user_id)`.
- `events.events` stores app-scoped user events with `app_id`, `user_id`, `event_at`, `name`, `created_at`, and `id`. It is partitioned by `created_at`.
- Existing event indexes are `(app_id, created_at, id)` and `(app_id, name, created_at DESC)`. Neither is ideal for "latest event per user" on `/users`.
- Visual verification should use the default LingoCafe ngrok flow with `john` / `john`.

## Assumptions

- Latest activity uses `event_at DESC` as the user-facing behavioral timestamp.
- Ties may be made deterministic with secondary ordering, but `event_at` is the primary activity timestamp.
- Only provider names and other display-minimal metadata needed for icons should be sent to the browser. OAuth token fields must stay server-only.
- The list response should include the social/login method data and latest activity for all listed users; the page should not perform one detail fetch per row.

## Constraints

- Keep `/users` as a client component wrapped by `AppLayout`.
- Keep `/api/users` protected by `api:users`, authenticated session, `backoffice` role, and `users:list` grant.
- Do not expose `refresh_token`, `access_token`, `id_token`, `session_state`, or other provider secrets.
- Do not expose `auth.users.password` to the client; only expose a boolean such as `hasPassword` when needed to render the credentials-login icon.
- Add indexes only if they support the concrete query path.
- Do not implement real user deletion in this task.

## Acceptance Criteria

- `/api/users` or an appropriate companion endpoint returns linked social login providers by joining `auth.accounts` on `app_id` and `user_id`.
- The list response includes social/login method data and latest activity for every listed user.
- The social login section supports multiple linked providers for one user and an empty state when none are linked.
- The `/users` table renders provider/login-method icons somewhere visible for each user row, with accessible labels or tooltips.
- Provider icon rendering supports at least GitHub and Google, and has a safe fallback for unknown social providers.
- Users without social provider rows render either a credentials-login icon when a password exists or a magic-link/email icon when no password exists.
- OAuth token values from `auth.accounts` such as access token, refresh token, id token, and session state are never returned to the client.
- `auth.users.password` is never returned to the client.
- The data layer fetches the latest event for each relevant user from `events.events`, scoped by `app_id` and `user_id`, returning only one event per user.
- Latest event selection uses `event_at DESC`, with deterministic tie-breakers allowed.
- The users list shows a `last activity at` date/time for each user.
- Hovering or focusing the `last activity at` value reveals the latest event name.
- Users with no linked social accounts or no event history render clear empty states.
- Clicking a populated user row opens a right-side read-only user detail panel.
- The row's three-dot action menu remains usable and does not accidentally trigger row click behavior.
- The row action menu includes `View user details` as the first item.
- The detail panel includes an `Edit` action that opens the existing edit panel for the same user.
- The detail panel includes a red ghost `Delete` placeholder action with confirmation. Confirming shows `User deletion not yet implemented` and does not delete the user.
- The detail panel shows formatted user information without `app_id`, password hashes, OAuth tokens, or provider secret fields.
- Email verification status is represented with an icon; verified users can reveal the verification date by hover, focus, or tap.
- The implementation adds an efficient index for account lookup by app and user if the current schema does not already provide one.
- The implementation adds an efficient index for latest event lookup by app and user if the current schema does not already provide one.
- The users table still loads through browser-side fetch with `credentials: 'same-origin'` and remains wrapped by `AppLayout` policy enforcement.
- Backoffice policy behavior remains unchanged: `page:users`, `api:users`, role `backoffice`, and grant `users:list` continue to protect the feature.
- Visual verification uses the default LingoCafe ngrok flow with `john` / `john`.
- `npm run qa` passes after implementation.

## Dos

- Do keep the query app-scoped. Join accounts on both `app_id` and `user_id`.
- Do aggregate provider links per user instead of duplicating user rows.
- Do use a deterministic latest-event query, such as `DISTINCT ON (user_id)` or a `LATERAL` join sorted by latest activity.
- Do prefer an account index like `auth.accounts (app_id, user_id, provider)`.
- Do prefer an event index like `events.events (app_id, user_id, event_at DESC, created_at DESC, id DESC)` if the query uses `event_at` for latest activity.
- Do keep row click and row menu behavior separate so menu actions do not open the detail panel.
- Do use icons from the existing icon stack where possible instead of emoji for verification and login methods.
- Do verify visually through the default LingoCafe ngrok flow with `john` / `john`.

## Don'ts

- Don't expose OAuth tokens or raw account secret fields to the client.
- Don't expose password hashes to the client.
- Don't compute latest activity in the browser from a full event list.
- Don't introduce an N+1 API pattern for users, accounts, or events.
- Don't sort last activity only by event name or user email. Time decides.
- Don't implement real user deletion in this task.
- Don't remove the existing edit panel or reset actions.
