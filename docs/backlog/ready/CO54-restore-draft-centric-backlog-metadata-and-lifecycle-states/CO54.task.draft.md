# Restore Draft-Centric Backlog Metadata and Lifecycle States

## Elevator's Pitch

Update the repo-local backlog skills so task lifecycle metadata lives only in the original draft task file, then expand the active lifecycle from draft/ready/wip into draft, refining, refined, planned, and wip. The draft file becomes the canonical task record. Other artifacts can do their job without pretending they own the task state.

## Business Gain

Backlog state becomes easier to trust, migrate, and port across repositories. A task has one metadata spine, so agents do not have to reconcile conflicting statuses between draft, refined, plan, notes, and index state. Chuck Norris does not ask three files what state a task is in. One file answers.

## Current State

The current repo-local backlog workflow uses `docs/backlog` and the `.agents/skills/backlog-*` skills. Recent work in `HW55` introduced `.task.draft.md` and `.task.refined.md` artifacts plus Python scaffolding, but lifecycle metadata is now often written to the active task artifact, especially `.task.refined.md`.

Current skill documents and scripts repeatedly describe task frontmatter on draft and refined task files with fields such as `taskId`, `status`, `createdAt`, `updatedAt`, `group`, `reviewedAt`, `plannedAt`, `startedAt`, `completedAt`, and `reviewAfter`. Examples include `backlog-refine`, `backlog-clarify`, `backlog-plan`, `backlog-execute`, `backlog-complete`, `backlog-archive`, `backlog-core`, and `backlog-migrate`.

The current active states are folder/index oriented: `draft`, `ready`, `wip`, and `blocked`, with completed and archived history. `backlog-migrate` currently maps draft folders to `.task.draft.md` and ready/blocked folders to `.task.refined.md`, then sets status on the migrated active file. That is the behavior to change.

The operator sees this as metadata drifting into the refined task file. That makes migration and reuse across other codebases harder because task properties no longer have one source of truth.

## Desired State

Make `<taskid>.task.draft.md` the canonical source of truth for task metadata. The draft file should keep the original user request and all task-level metadata, including status, lifecycle movement dates, reviewed/planned/started/completed timestamps, review-after information, group, and any future task property fields.

Refined task files should contain the refined task content, not lifecycle ownership. Plan, notes, question, and clarify files may keep artifact-local metadata only when useful for that artifact, but they must not duplicate or own task lifecycle status or movement dates. If an implementation needs `taskId` on sidecar files for compatibility, it should be treated as artifact identity, not lifecycle state.

Update the lifecycle states to:

- `draft`: task entered the backlog through the draft workflow and has not started refinement.
- `refining`: refine or clarify has started and at least one question/clarify round is in progress or being folded in.
- `refined`: refinement/clarification is complete enough that the skill proposes planning.
- `planned`: a plan has been created and accepted.
- `wip`: execution has started.

The refine and clarify skills should transition the task from `draft` to `refining` when they begin rounds. They should keep the task in `refining` during rounds. When they determine that refinement is enough and propose planning, they should transition the draft metadata status to `refined`.

The plan skill should transition the draft metadata status to `planned` when the plan is accepted. The execute skill should transition the draft metadata status to `wip` when execution starts. Existing completed and archived behavior can remain out of scope unless it must be touched to avoid breakage. Existing blocked behavior should be preserved or deliberately remapped only after review, because this request does not ask to redesign blocked tasks.

Update `backlog-migrate` so it can analyze existing task folders and migrate task metadata back into `.task.draft.md` when newer metadata currently exists on `.task.refined.md` or legacy active task files. The migration should be portable enough to apply to other codebases using the same backlog skill structure.

## Definition of Success

The backlog system has one authoritative task metadata location: `<taskid>.task.draft.md`. Active lifecycle transitions update that draft metadata instead of refined, plan, notes, or round files. The visible backlog/index and folder movement behavior still works, but status and movement fields no longer drift across artifacts. Running the migration restores existing active tasks to the new metadata model without dropping useful timestamps or user-authored task content.
