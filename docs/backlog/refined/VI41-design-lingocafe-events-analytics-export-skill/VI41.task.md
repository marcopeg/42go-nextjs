# Design LingoCafe Events Analytics Export Skill

## Elevator's Pitch

Design and document a progressive local export workflow for `lingocafe.events` so raw product events can be downloaded from the VPS PostgreSQL database into durable Mac-local filesystem analytics storage and queried from Python without spending production CPU on heavy analytics.

## Business Gain

LingoCafe needs behavioral analytics such as weekly active users, reading-session duration, book engagement, and reader progress patterns. The VPS database should collect live events, then the Mac-local archive should download them before heavy analysis starts swinging. Production CPU belongs to users, not reports.

## Current State

- The repository has `make backup` / `make restore` commands backed by `.agents/skills/42go-backup/scripts/backup.mjs`.
- `make backup mode=full` can dump `lingocafe.events`, while `mode=light` intentionally skips events and other heavy LingoCafe tables.
- The `lingocafe.events` table is an append-oriented PostgreSQL event stream partitioned by `created_at` with a composite primary key `(created_at, id)`.
- `event_at` is stored as the business timestamp for analytics, while `created_at` is the ingestion and partitioning timestamp.
- Event rows include `user_id`, `name`, optional `book_id` and `page_id`, plus JSONB `data` and `meta` payloads.
- There is no local analytics-oriented durable archive or skill that incrementally pulls only new event rows.
- PostgreSQL events should be treated as a transitory hot buffer on the VPS, not as the long-term analytics source of truth.

## Desired State

- A decision is made for the durable local analytics storage formats: immutable CSV batches for human-readable access and immutable Parquet batches for Python/DuckDB analysis.
- A new agent skill exists for downloading LingoCafe events from a database URL supplied through an environment variable.
- The skill can pull only new event rows since the last successful local dump using a durable cursor based on `(created_at, id)`.
- Downloaded data is stored locally in a deterministic gitignored repository folder, such as `.local/lingocafe-analytics`.
- Each export run writes one paired CSV file and one paired Parquet file with the same batch ID.
- The accepted default layout is `.local/lingocafe-analytics/events/csv/`, `.local/lingocafe-analytics/events/parquet/`, `.local/lingocafe-analytics/events/state.json`, and `.local/lingocafe-analytics/events/manifest.jsonl`.
- The first version exports raw `lingocafe.events` only; signup analysis should come from signup events instead of offline `auth.users` snapshots.
- The local filesystem archive, not PostgreSQL, becomes the durable source of truth for historical event analytics.
- The export process is read-only against PostgreSQL and does not drop, delete, truncate, or clean up source event rows.
- The process supports repeated runs without duplicating rows or corrupting previous local data.
- If the script dies before a batch completes, rerunning should overwrite the incomplete batch rather than advancing the cursor or creating duplicate exported records.
- The task produces a `{id}.output.md` artifact documenting the final decision, storage layout, cursor semantics, commands, and example analysis workflow.

## Definition of Success

An operator can run a documented skill command with an env var pointing at a readable PostgreSQL database, download new `lingocafe.events` rows into durable Mac-local CSV and Parquet batches, then use Python tooling on the Mac to compute metrics such as weekly active users and reading-session durations without loading the VPS CPU.

## Additional Context

- This task is about design, decision making, and creating the skill mechanics, not about building a full analytics dashboard.
- The operator specifically mentioned Parquet as a possible storage format, but the task should document the decision instead of assuming blindly.
- The local store should be analysis-friendly for Python. Likely consumers include DuckDB SQL over Parquet, pandas, pyarrow, or notebook scripts.
- Reading-session analytics will likely need timed event sequences such as `page-open` and `page-scroll`, grouped by user, book, and inactivity threshold.
- The skill should be careful with production credentials and should not commit exported event data to git.
- The operator explicitly wants this export task to only download data and not remove source rows from PostgreSQL.

## Assumptions

- The source database connection is provided by `LC_EVENTS_DATABASE_URL`.
- The source database env var name should be documented in the skill README and easy to change through a top-level constant in the export script.
- The local analytics storage root can be configured through an environment variable, with a safe default under a gitignored repository directory such as `.local/lingocafe-analytics`.
- The source table is `lingocafe.events` and the incremental cursor should use `created_at` plus `id`, not only `event_at`.
- The export process can be implemented as a script under an agent skill and documented for operator use.
- Python analysis examples may be lightweight smoke examples rather than production reporting code.
- The first implementation should not include retention, deletion, truncation, or cleanup behavior for PostgreSQL events.

## Constraints

- Do not use the full SQL dump format as the analytics storage format unless the decision record explicitly rejects Parquet or another columnar format for a good reason.
- Do not require write access to the source database.
- Do not mutate `lingocafe.events` during export.
- Do not delete, truncate, or clean up PostgreSQL events in this task.
- Do not trust client timestamps for incremental extraction; use `created_at` and `id` for the extraction cursor.
- Do not store database credentials in generated files.
- Do not commit raw exported event data unless the repository explicitly adds a safe sample fixture.
- Keep the first version focused on event export and local analytics storage, not dashboards, cron automation, or warehouse infrastructure.
- Keep dimension snapshots such as `auth.users`, `lingocafe.books`, and `lingocafe.books_pages` out of the first version unless a later refinement explicitly brings them back.
- Treat local durability as mandatory before any future source retention cleanup.

## Acceptance Criteria

- A `{id}.output.md` artifact documents the chosen local storage format and why it was chosen.
- The output artifact documents the local filesystem layout, cursor file/state format, and recovery behavior after interrupted runs.
- The output artifact documents exact commands and required environment variables for running the event dump.
- A new or updated skill can connect to PostgreSQL using an env var and select from `lingocafe.events` without modifying the database.
- The source database env var is `LC_EVENTS_DATABASE_URL`.
- The skill README documents the env var, default local layout, and where to change the env var constant in the script.
- The first implementation exports raw `lingocafe.events` only.
- Each successful export batch writes CSV output for human-readable inspection and Parquet output for Python/DuckDB analysis.
- Paired CSV and Parquet outputs share the same batch ID.
- The export is incremental and resumes from the last successful cursor using `(created_at, id)` ordering.
- Repeated runs do not duplicate exported events in the intended local query view.
- The cursor advances only after both CSV and Parquet outputs are written and the Parquet output passes a smoke read.
- A failed or interrupted in-progress export can be rerun and overwrites the incomplete batch without advancing the cursor.
- The local store is documented as the durable analytics archive and PostgreSQL is documented as the transitory event buffer.
- The output artifact explicitly states that this skill is download-only and does not delete source rows.
- The implementation includes a small validation or smoke path that proves Python can read the local store and compute at least one metric, such as weekly active users.
- The implementation includes guidance for session-based metrics such as average time spent on a book, including the event ordering and inactivity threshold assumptions.
- Raw exported data is kept out of git through `.gitignore` or an equivalent documented storage path.
- `npm run qa` or the relevant script validation passes after repository code changes, if repository code changes are made.

## Dos

- Do inspect the existing `42go-backup` skill and Makefile backup family before choosing implementation shape.
- Do use server/database timestamps for extraction progress.
- Do make the local data easy to query from Python.
- Do document how to compute WAU and book-session duration from the local store.
- Do keep credentials in env vars only.
- Do make the skill safe to rerun.
- Do keep the default analytics store in a gitignored repository-local folder unless overridden.
- Do design the archive so production analytics run locally on the Mac, not on the VPS.
- Do update the cursor only after the exported batch is durably written and validated.
- Do write both CSV and Parquet outputs from the same extracted batch.
- Do use the same batch ID for paired CSV and Parquet files.
- Do keep the database env var name easy to change through a top-level script constant.
- Do overwrite incomplete batches on rerun before cursor advancement.

## Don'ts

- Don't build a web dashboard in this task.
- Don't dump the entire database for every analytics run.
- Don't store production event exports in the repository.
- Don't rely on `event_at` as the only incremental cursor.
- Don't add warehouse infrastructure before the local filesystem workflow is proven.
- Don't make the skill depend on interactive prompts for normal operation.
- Don't snapshot users, books, or pages in the first version.
- Don't delete source rows.
- Don't add source retention behavior to this task.
