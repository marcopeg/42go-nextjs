# Clarify Round v1 — Design LingoCafe Events Analytics Export Skill

## Questions

### 1. Should the export skill store raw events only, or should it also snapshot dimensions such as `auth.users`, `lingocafe.books`, and `lingocafe.books_pages` so analysis can join human-readable book and user context offline?

### 2. Where should the default local analytics store live: a gitignored repository folder such as `.local/lingocafe-analytics`, a user-home path such as `~/.42go/lingocafe-analytics`, or only an explicit env-var path?

### 3. For the first version, should Parquet be the committed decision unless a blocker appears, with DuckDB as the documented Python query path?

## Answers

### 1.

For now, events only. Signups will be recorded as events.

### 2.

A gitignored repository folder such as `.local/lingocafe-analytics`.

### 3.

Propose alternatives in chat, explain pros and cons, then ask again.

## Resolution Summary

- The first export scope is raw `lingocafe.events` only.
- Offline dimension snapshots are out of scope for the first version.
- Signup analysis should be handled through signup events, not `auth.users` snapshots.
- The default local analytics store should live under a gitignored repository path such as `.local/lingocafe-analytics`.
- The storage/query format decision remains open pending a comparison of alternatives.
