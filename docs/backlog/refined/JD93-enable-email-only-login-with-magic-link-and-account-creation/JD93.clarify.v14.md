# Clarify Round v14 — Enable Email-Only Login With Magic Link and Account Creation

## Questions

### 1. For v1 architecture, should we prefer NextAuth v4 EmailProvider plus a custom Knex adapter, or a custom credentials-style OTP provider that still uses NextAuth sessions?

## Answers

### 1.

Use NextAuth v4 EmailProvider plus a custom app-scoped Knex adapter.

Reason: it is most compliant with the current setup and least disruptive. The repository already uses NextAuth v4, dynamic `AppConfig.auth.providers`, JWT sessions, custom callbacks, and an `auth.verification_tokens` table. The implementation should preserve per-app auth provider configuration exactly as it works today.

## Resolution Summary

- V1 architecture is NextAuth v4 EmailProvider plus a custom Knex adapter.
- The adapter must be app-scoped through the current request AppID.
- JWT sessions and existing NextAuth callbacks remain in use.
- `AppConfig.auth.providers` remains the source of truth so different apps can configure different auth options.
- Custom credentials-style OTP provider and Auth.js migration are rejected for v1 unless implementation research uncovers a blocker.
