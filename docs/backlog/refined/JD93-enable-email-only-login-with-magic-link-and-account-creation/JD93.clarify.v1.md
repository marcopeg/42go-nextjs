# Clarification Q&A v1 - Enable Email-Only Login With Magic Link and Account Creation

This file consolidates refinement clarification rounds v1 through v18.

## Round 1

### Question

Should v1 require manual code entry, or can v1 ship as magic-link-first while we evaluate code entry as a follow-up?

### Answer

By default it should do both. After the user enters the email, the flow should show an "enter the code" screen and also provide a resend-email action.

The auth strategy configuration should remain open for custom behavior later. It should support throttling resend/request attempts, with a default delay of 1 minute. Prefer support for a progressive delay list such as `15s,30s,1m,3m,7m,15m`, where each subsequent resend is delayed accordingly and the UI shows a customizable error message.

## Round 2

### Question

What default manual code format do you want: 6-digit numeric, 8-character alphanumeric, or full verification token?

### Answer

Default: 6 digits.

The code format should be configurable. A possible model is `length=6` plus `dict=0-9`, where `dict` acts like a regexp-style dictionary of available symbols. Naming and strategy can be refined; alternatives such as `alphabet`, `charset`, or presets should be considered.

## Round 3

### Question

Which email sender should be the default target for v1: SMTP/Nodemailer, Resend HTTP API, SendGrid HTTP API, or a custom sender interface?

### Answer

This must be a strategy-based implementation so multiple providers can handle the mechanical sending of the email. Different 42Go apps must be able to configure different sender strategies from the available set.

Pick the easiest and best-documented strategy as the first/default implementation.

## Round 4

### Question

For users created by email-only login, should `username` be null, generated from the email, or generated as a random stable handle?

### Answer

Default username generation should use the email name portion before `@` when available. If that username is already taken for the app, fall back to the full email address, which must already be unique.

Important: when magic link is enabled together with social logins, different auth providers using the same email must resolve to the same app-scoped user.

## Round 5

### Question

Should the AppConfig provider type be named `email`, `passwordless`, or something else?

### Answer

Name: `email`.

## Round 6

### Question

What default token/code lifetime should v1 use: 10 minutes, 15 minutes, 1 hour, or 24 hours?

### Answer

Token expiration should be configurable and default to 5 minutes.

Additional UI clarification: this authentication mode should integrate with the credentials flow because both begin with the same username/email identifier. If only `credentials` is available, the UI works as today with username and password. If only `email` is available, the UI shows the username/email input, then continues into the email-code screen. If both `credentials` and `email` are available, the first step shows one shared username/email input, then lets the user choose between password flow and email flow. The password choice shows the password field; the email choice sends the code and shows the code-entry screen.

## Round 7

### Question

Should successful email sign-ins create rows in `auth.accounts` for provider `email`, or should `auth.users` plus verification tokens be enough?

### Answer

Just the user. `auth.accounts` appears to be mostly for social sign-on.

Advice: keep `auth.accounts` for OAuth/social provider links. Email sign-in should resolve through app-scoped `auth.users.email` and verification tokens, without creating synthetic `auth.accounts` rows for provider `email`.

## Round 8

### Question

Should v1 log audit/events for email requested, email resent, code verified, and email login failed?

### Answer

Yes. Configuration should allow cherry-picking which events to log, with each event exposed as a Boolean key. All event logging keys should default to `true`.

## Round 9

### Question

For local development, should email delivery default to console output, an SMTP sink, provider sandbox mode, or a mocked sender?

### Answer

Use a different approach: create a real `console` email sender strategy and select it by default.

Auth provider config should allow multiple sender strategies as key/value entries, plus a `useStrategy` key defaulting to `console`. The selected strategy property must be configurable via environment variable.

## Round 10

### Question

Should the `console` sender be blocked in production by default, or allowed in production only when explicitly selected?

### Answer

Allowed. If the engineer configures `console` in production by mistake, that is an operator/configuration problem.

## Round 11

### Question

What should the final code-generation config names be: `dict`, `alphabet`, `charset`, or presets like `numeric6`?

### Answer

Use `length` for code size and `mode` for the allowed symbol family. Supported modes should include `digits`, `alphabet`, `alphanumeric`, and `complex`.

Basic `alphabet` and `alphanumeric` should be case-insensitive by default. Add `caseSensitive`, defaulting to `false`, so config can opt into uppercase/lowercase distinction when needed.

Opinion: use `length`, `mode`, and `caseSensitive`. Keep mode values small and explicit. Default config should be equivalent to `{ length: 6, mode: "digits", caseSensitive: false }`.

## Round 12

### Question

Should v1 include both `console` and SMTP/Nodemailer sender strategies, or only `console` plus the strategy interface?

### Answer

V1 should include `console` plus the first suggested external provider.

Chosen first external provider: Resend. Reason: official docs show a free transactional plan, a simple developer-focused HTTP API, and Node/Next.js documentation. `console` remains the default selected strategy; Resend is the first external strategy available for real delivery.

## Round 13

### Question

Should refinement stop here and mark the task refined, or is there another requirement you want to clarify first?

### Answer

Do not stop yet. Keep the task as one task for now, but use the retrospective/failure findings to further disambiguate the task and maximize delivery success rate.

## Round 14

### Question

For v1 architecture, should we prefer NextAuth v4 EmailProvider plus a custom Knex adapter, or a custom credentials-style OTP provider that still uses NextAuth sessions?

### Answer

Use NextAuth v4 EmailProvider plus a custom app-scoped Knex adapter.

Reason: it is most compliant with the current setup and least disruptive. The repository already uses NextAuth v4, dynamic `AppConfig.auth.providers`, JWT sessions, custom callbacks, and an `auth.verification_tokens` table. The implementation should preserve per-app auth provider configuration exactly as it works today.

## Round 15

### Question

For manual code entry, should we implement a dedicated `/api/auth/email/verify-code` route that consumes the same stored verification token and then hands off to NextAuth session creation?

### Answer

Yes, but manual code verification must use the same logic as magic-link verification so there is no duplication after the verification step.

## Round 16

### Question

What keys should throttle email requests and resends: app+email only, app+email+IP, or app+email+IP+user-agent?

### Answer

For v1, throttle by `app+email` only.

Keep the data structure open so future iterations can add IP and possibly user-agent when needed.

## Round 17

### Question

Should email provider config be split into `public` UI metadata and server-only `server` secrets/settings?

### Answer

Use the same simple approach as the current social providers for v1.

The current configuration already puts OAuth provider client IDs/secrets in the `auth.providers` config using environment variables. Email provider config should follow that pattern for now. Do not introduce a full public/server AppConfig split in this task. Iterate and harden later if needed.

## Round 18

### Question

Should the completion gate require a validation matrix covering credentials-only, email-only, credentials+email, new user, existing user, expired/reused code, throttled resend, console sender, and Resend sender?

### Answer

Yes.

Before the task is considered done, the implementation must prove the critical auth cases work: app with only password login, app with only email-code login, app with both password and email-code login, new user created by email, existing user logs in by email, expired/already-used code fails, resend is throttled, console sender works, and Resend sender works.
