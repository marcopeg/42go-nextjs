# LingoCafe Onboarding Follow-up

## Elevator's Pitch

Use this task slot for the next LingoCafe onboarding iteration now that `YA00` moved incomplete-profile onboarding into an AppLayout-level layout guard.

## Business Gain

LingoCafe onboarding should get users from first login to a usable reading catalog with less confusion and less route-specific behavior. The next improvement needs a concrete product outcome before planning starts.

## Current State

`YA00` completed the first onboarding extraction. LingoCafe now configures `app.profile.guard` with `slot: "layout"` and a lazy `LingocafeOnboardingGuard`.

The guard renders a blocking full-screen modal for authenticated LingoCafe users whose profile is incomplete. It collects own language, reading language, reading level, and configured consent items, saves through the existing `/api/profile` flow, and dispatches `profile:complete` after successful completion.

`/books` no longer owns the inline onboarding form. It renders a plain incomplete-profile note and does not load the catalog while profile data is incomplete.

The original `NH52` draft was for the future `app.profile.guard.before` platform slot. The operator repurposed this slot for LingoCafe onboarding, so the before-guard work is not the current scope unless refinement explicitly brings it back.

## Desired State

TBD through refinement. The task should define one concrete LingoCafe onboarding improvement or slice of improvements, such as onboarding UX polish, step structure, post-completion routing, copy, analytics, guard behavior, profile-completion rules, or first-reading guidance.

## Definition of Success

The task is plan-ready when it states exactly what LingoCafe onboarding should change, who it affects, which current guard/profile APIs it should reuse or extend, and how success will be verified on desktop and mobile.

## Additional Context

- `YA00` is completed and should be treated as the current baseline.
- Current implementation lives in `src/app/(app)/(lingocafe)/_components/LingocafeOnboardingGuard.tsx`.
- Current guard runner lives in `src/42go/layouts/app/ProfileLayoutGuard.tsx`.
- Current LingoCafe guard config lives in `src/config/lingocafe/config.ts`.
- Profile persistence and consent evidence should continue through the existing profile system unless refinement chooses a separate, explicit backend change.

## Assumptions

- This task is LingoCafe-specific.
- The existing layout guard is the starting point.
- The task should not revive the future `app.profile.guard.before` work without an explicit reason.
- The first version of onboarding already requires own language, target language, target level, and consent completion.

## Constraints

- Stay scoped to LingoCafe onboarding unless refinement explicitly identifies platform work.
- Reuse the current AppLayout profile layout guard where practical.
- Do not reintroduce route-local onboarding forms.
- Do not create a second profile persistence or consent evidence path.
- Do not regress completed-profile access to `/books`, `/profile`, book detail, or reader routes.
- Preserve multi-app isolation and keep non-LingoCafe apps inactive.

## Acceptance Criteria

- The exact onboarding improvement is selected and documented.
- The affected users and routes are explicit.
- The desired onboarding flow, fields, copy, and completion behavior are concrete enough to plan.
- The task states whether new analytics/events are required or whether existing profile/consent events are enough.
- The task states whether the current full-screen guard stays, is restyled, or is replaced.
- Desktop and mobile verification expectations are listed.
- `npm run qa` passes after implementation.

## Dos

- Build on `YA00` instead of reopening solved extraction work.
- Keep the onboarding feature app-specific under the LingoCafe area unless a shared primitive earns its keep.
- Use the existing profile controller, consent component, and profile APIs where possible.
- Keep visual verification tied to the LingoCafe ngrok test flow when browser checks are needed.

## Don'ts

- Do not make `/books` own onboarding again.
- Do not add a platform before-guard slot unless refinement explicitly changes the scope.
- Do not hardcode LingoCafe behavior into shared AppLayout code beyond the existing guard configuration contract.
- Do not silently change profile completeness rules without matching API/server validation.
