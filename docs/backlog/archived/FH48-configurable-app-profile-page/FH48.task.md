# Configurable App Profile Page

## Elevator's Pitch

Build an app-configurable profile page that works like the current home page configuration model: each app can define a list of platform profile components and app-specific profile components in its AppConfig. The first concrete use case is LingoCafe, where users need to change their native language, learning language, and learning level from the profile page.

## Business Gain

Apps get a reusable profile surface without hardcoding one global profile page for every tenant. LingoCafe gains a direct profile editing path for language preferences, which affects book filtering and the learning experience.

## Current State

The public home page is driven by AppConfig page data through `DynamicPage` and `ContentBlock`. For example, `src/config/lingocafe/home-page.tsx` exports `HomePage`, and `src/config/lingocafe/config.ts` wires it under `public.pages.HomePage`.

The authenticated profile page at `src/app/(app)/profile/page.tsx` is currently hardcoded. It renders account information, demo RBAC/session panels, and a logout action. It does not read profile-page content from AppConfig.

LingoCafe already has profile-related data and persistence through `/api/lingocafe/profile`, and the books page contains language/level selection logic for incomplete profiles. The profile page does not yet expose those selectors as configurable app-specific components.

The current LingoCafe profile API validates and saves `ownLang`, `targetLang`, and `targetLevel` as one complete payload. `saveProfile()` skips no-op writes, upserts `lingocafe.profiles`, and records a `profile-update` event inside the same transaction when values change.

## Desired State

Add a configuration approach for the app profile page that mirrors the home page pattern: an app config can declare ordered profile-page items under `app.profile.items`, and the profile route renders them.

The profile page should use a dedicated profile renderer, not the generic public `ContentBlock` renderer. Profile blocks need a specific contract for user data, validation, save orchestration, and feedback.

The design should support platform components referenced by stable names plus config parameters, such as a future `ChangePassword` component. This is only an example and should not be implemented as part of this draft unless a later plan explicitly chooses it.

The design should also support app-specific components. For the first implementation, LingoCafe should expose one app-specific profile block:

- `LingocafePreferences`

That block should be added to the `lingocafe` app config so users can change native language, learning language, and learning level from `/profile`.

The shared `42go` profile system should include these platform profile blocks:

- `AccountInfo`: shows account basics such as username, email, and signup date.
- `TestRBAC`: contains the current RBAC/session diagnostic panel from the hardcoded profile page.
- `Logout`: renders the logout action inline inside the profile page.

Profile block configuration should follow the public page block pattern as closely as practical: use typed block objects, built-in blocks with string `type` discriminants, and custom/app-specific `component` blocks with direct React component references plus optional props.

For the first implementation, prioritize proving the profile block pattern over completing every final profile feature. Build the minimal LingoCafe profile block needed to test the configuration and rendering model, reference it from the LingoCafe configuration, and keep the design easy to evolve after the pattern is exercised in the app.

The profile page should expose one global `Save preferences` action. When clicked, the page sends a save signal to every rendered profile block. Each block is responsible for its own validation, persistence, internal error state, and success/error result reporting. The page waits for all block results, shows global success only when all blocks validate and persist successfully, and uses block error messages to show global validation or persistence feedback.

If any block reports a validation error, the whole save is blocked until that error is resolved. If validation passes but persistence fails for one or more blocks, successful blocks may stay successful while failing blocks return their own persistence error messages for global feedback.

The `Save preferences` action should be placed in the profile page top bar, where the logout action currently appears. Logout should move into the configurable inline `Logout` profile block.

## Definition of Success

The repository has a clear AppConfig-driven profile-page model and LingoCafe proves it with real editable profile fields. The implementation stays compatible with the authenticated app-page convention: client component, `AppLayout`, browser-side fetching, and policy-aware behavior.

## Additional Context

Study these existing areas before planning implementation:

- `src/app/(public)/page.tsx`
- `src/42go/components/DynamicPage`
- `src/42go/components/ContentBlock`
- `src/config/lingocafe/home-page.tsx`
- `src/config/lingocafe/config.ts`
- `src/app/(app)/profile/page.tsx`
- `src/app/(app)/(lingocafe)/books/page.tsx`
- `src/app/api/(lingocafe)/lingocafe/profile/route.ts`
- `src/app/api/(lingocafe)/lingocafe/_lib/reader.ts`

The profile page likely needs a client-side block/component renderer because authenticated app pages are client-only by convention, while the public `DynamicPage` uses a server renderer for the public home page.

`ContentBlock/client` currently supports only `component` and `link` blocks. The public `DynamicPage` uses the server renderer and server-oriented content block types, so profile-page rendering should not assume that the public dynamic-page renderer can be reused unchanged.

LingoCafe reader language and level options currently live in `src/app/api/(lingocafe)/lingocafe/_lib/reader.ts` through `getReaderLanguages()`, and the books page has client fallback copies of the same option lists. Planning should decide whether to extract those option lists into shared LingoCafe config/types before building profile selectors.

For this task, LingoCafe profile UI code should remain LingoCafe-specific and live under the LingoCafe area of the `app` directory, alongside the existing authenticated LingoCafe routes. Do not move these selectors into shared `42go` components.

This task should also start ProfilePage documentation that explains the new AppConfig profile-page pattern, where profile blocks live, how platform blocks differ from custom/app blocks, and how the save orchestration contract works. The first documentation pass can be concise, but it must give future tasks a clear place to expand.

## Assumptions

The profile configuration should live under `app.profile.items` because `/profile` is an authenticated app route.

The platform component registry and app-specific component registry should be type-safe enough that invalid component names are caught during development where practical.

The LingoCafe preferences block can reuse the existing profile API and option lists or extract shared logic from the books page rather than duplicating behavior blindly.

The first implementation should let each profile block own validation and persistence. For LingoCafe, use one `LingocafePreferences` block that owns the three fields and persists them together through the existing profile payload shape.

Changing the LingoCafe learning language does not need a warning in the first version.

The initial LingoCafe block can be intentionally minimal if it is enough to validate the profile block registration, rendering, and save-signal pattern.

## Constraints

Authenticated app pages must remain client components and use `AppLayout`.

Browser-side data fetching must use `fetch` with `credentials: "same-origin"`.

The global `Save preferences` action must not persist any block when any block currently reports a validation error.

The profile page must await all block persistence results before showing global success.

Do not implement the example `ChangePassword` platform component in this task unless it becomes explicitly required during refinement or planning.

Use absolute imports with the `@/` prefix.

Preserve the existing multi-app behavior: the profile page should render per resolved AppID.

## Acceptance Criteria

- A proposed profile-page AppConfig schema is documented in the task plan before implementation.
- The profile route renders profile-page items from the resolved app config.
- The profile block schema follows the same general pattern as public page blocks.
- The profile renderer supports platform components referenced by stable names plus config parameters.
- The profile renderer supports app-specific components without forcing those components into the shared `42go` platform layer.
- The profile renderer is dedicated to authenticated profile blocks and does not depend on the public dynamic-page server renderer.
- The profile page exposes one `Save preferences` action that broadcasts validation and persistence signals to rendered blocks.
- The `Save preferences` action appears in the profile page top bar, replacing the current top-bar logout action.
- Each profile block can report validation errors to block-local UI and global page feedback.
- If any block has a validation error, the profile page blocks the save and shows validation feedback.
- After successful validation, each block runs its own persistence logic and reports success or failure.
- The profile page waits for all persistence results and shows global success only when all blocks succeed.
- If some blocks persist successfully and others fail, the page shows global feedback using the failing block messages without pretending the whole save succeeded.
- The shared profile registry includes `AccountInfo`, `TestRBAC`, and `Logout` platform blocks.
- `AccountInfo` shows email, username/name, and signup date when available; missing values render gracefully.
- `TestRBAC` contains the current RBAC/session diagnostic behavior from the existing hardcoded profile page.
- `Logout` renders the logout action inline as configurable profile content.
- LingoCafe defines profile page config that includes one minimal `LingocafePreferences` custom/app block using the same component-reference pattern as public page custom component blocks.
- LingoCafe users can update native language, learning language, and target level from `/profile`.
- The profile UI initializes from the existing `GET /api/lingocafe/profile` response and preserves existing values when saving.
- Saving profile settings uses the existing LingoCafe profile API, compatible server-side validated endpoints, or block-level adapters that preserve server-side validation and authenticated-user ownership.
- The previous account/session profile basics are preserved as configurable platform profile blocks.
- ProfilePage documentation is started and describes the profile block AppConfig pattern, block placement, platform versus custom/app blocks, and save orchestration.
- `npm run qa` passes after implementation.

## Dos

- Study the home page configuration and dynamic page rendering before choosing the profile schema.
- Keep platform profile components generic and reusable.
- Keep LingoCafe-specific preferences in an app-specific module.
- Place the LingoCafe preferences component under the LingoCafe area of `src/app/(app)/(lingocafe)/`.
- Reuse existing LingoCafe profile data contracts where possible.
- Make the configuration readable from `src/config/lingocafe/config.ts`.
- Keep validation and persistence outcomes explicit per block.
- Start concise ProfilePage documentation instead of leaving the new pattern implicit in code only.

## Don'ts

- Do not hardcode LingoCafe behavior into the shared profile route.
- Do not implement `ChangePassword`; it is only a schema example.
- Do not server-render app profile data from internal APIs.
- Do not create a one-off profile page that cannot be reused by other apps.
- Do not keep logout as the profile page top-bar action; it belongs in the inline `Logout` block for this task.
- Do not move LingoCafe profile selector components into shared `42go`.
- Do not extract LingoCafe language/level options into shared AppConfig as part of this task.
- Do not overbuild the LingoCafe block before the profile block pattern is validated in the app.
