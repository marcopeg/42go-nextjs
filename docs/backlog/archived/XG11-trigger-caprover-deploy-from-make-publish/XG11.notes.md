# Execution Notes — Add CapRover Deploy Targets
**Task**: ./XG11.task.md
**Plan**: ./XG11.plan.md

## Decisions

- Accepted plan moved `XG11` into WIP.
- The implementation should keep `publish`, `publish.nocache`, and `publish.universal` as registry publish targets.
- The implementation should add `deploy`, `deploy.nocache`, and a separately callable CapRover trigger target.
- `npx --yes caprover deploy --help` confirmed `--imageName`, `--caproverApp`, `--caproverUrl`, and `--appToken`.
- The Makefile uses `npx --yes caprover` by default so CapRover does not need to be committed as a project dependency.
- `deploy.caprover` references `$$CAPROVER_APP_TOKEN` in the recipe so dry-run output does not expose the token value.
- `.env.example` now documents `CAPROVER_URL`, `CAPROVER_APP`, and `CAPROVER_APP_TOKEN`.
- Production deployment docs now describe `make deploy` and `make deploy.nocache`.
- Dry-run validation passed for `deploy.caprover`, `deploy`, `deploy.nocache`, and `publish.universal` using placeholder CapRover values.
- `npm run qa` passed.
- Added `XG11.docs.md` with local instructions for generating and storing the CapRover app token.

## Problems Encountered

- The first sandboxed `npx caprover deploy --help` failed because registry access was blocked. Retried with approved network access and confirmed the CLI flags.

## Deviations From Plan

- The implementation did not add `CAPROVER_MACHINE` fallback support. It uses the token-based non-interactive path with `CAPROVER_URL` and `CAPROVER_APP_TOKEN`, which matches the accepted direction and keeps the command contract tighter.

## Additional Requests

- The operator accepted the plan and requested immediate implementation.
- The operator requested a task-local docs file explaining how to generate the CapRover app token.

## Known Limitations

- Live Docker Hub push and CapRover deployment were not run. First live deploy still requires real `CAPROVER_URL`, `CAPROVER_APP_TOKEN`, Docker Hub push access, and CapRover pull access to the image.
