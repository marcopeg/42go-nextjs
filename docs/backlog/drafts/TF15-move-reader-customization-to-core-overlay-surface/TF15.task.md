# Move Reader Customization to Core Overlay Surface

## Elevator's Pitch

After `GM77` delivers the shared popup / modal / overlay primitive, migrate the LingoCafe reader customization feature to use that core component instead of its current bespoke overlay implementation. The goal is to keep the reader customization UX and settings behavior, while cleaning out custom surface code for opening, closing, overlay handling, sizing, and responsive presentation.

## Business Gain

The reader customization flow stops being a special snowflake and starts behaving like the rest of the system. That lowers maintenance cost, reduces UI drift, and proves the new shared overlay surface can handle a real product feature instead of living as a museum piece. Chuck Norris does not maintain two overlay systems when one can do the job.

## Current State

The current reader customization UI lives in `src/app/(app)/(lingocafe)/books/_components/BookReaderPreferencesPanel.tsx` and is mounted from the book reader surfaces used by the `/books/[bookId]/[pageId]` route.

The current implementation already supports reader preference behavior such as:

- font size controls
- font family selection
- background color selection
- text color selection
- live preview
- reset action

But the surface itself is custom-built. It currently includes bespoke overlay and container behavior such as:

- fixed full-screen overlay markup
- manual backdrop click-to-close handling
- manual Escape-key close handling
- hard-coded right-side panel positioning
- separate mobile and desktop display branching
- custom sizing and stacking behavior

This means the reader customization feature is still carrying its own popup mechanics instead of using a shared core library primitive.

## Desired State

Once `GM77` is implemented, the reader customization feature should be refactored to use that new core overlay surface.

The migrated reader customization flow should:

- keep the existing reader preference capabilities
- use the shared overlay primitive for open/close behavior
- use the shared overlay primitive for backdrop behavior
- use the shared overlay primitive for responsive mobile and desktop presentation
- remove custom surface code that becomes redundant after the migration

The reader customization surface should behave like a real consumer of `GM77`, not like a wrapper that still secretly keeps most of its old bespoke popup implementation.

The migration should also validate that the shared component can support this concrete use case, including:

- reader-specific header content
- scrollable body content
- footer or reset-action placement if needed
- mobile full-screen or near-full-screen behavior
- desktop side-panel behavior

## Definition of Success

The LingoCafe reader customization feature still lets users change font size, colors, and related reading preferences, but the custom popup mechanics are gone. The surface is powered by the shared overlay component introduced by `GM77`, and the remaining reader-specific code is focused on the preference content itself rather than overlay infrastructure.

## Additional Context

This task is intentionally a follow-up to [GM77](./GM77.task.md). `GM77` defines and implements the shared primitive plus its agent skill. This task is the first concrete cleanup and adoption pass for that primitive inside a real feature.

The current reader work already has route and design history through completed tasks such as `ZE11` and `VU18`. Those tasks established the read flow, mobile full-screen behavior, and the current reader surface structure. This follow-up should preserve the behavior that matters to users while replacing the custom overlay plumbing.

The phrase "cleaning it out from any custom code" should be interpreted as removing custom code specifically related to popup / modal / overlay surface mechanics when those mechanics are covered by the shared primitive. Reader-specific preference content, preview rendering, and preference state logic are still expected to remain where they belong unless a smaller shared extraction is obviously justified.

## Assumptions

- `GM77` is a blocker for this task because the migration target does not exist yet.
- The current reader preference content and state model are mostly valid and do not need a feature redesign.
- The main cleanup target is the custom overlay surface implementation, not the reader preference domain logic itself.
- The shared overlay primitive from `GM77` should be able to support both the mobile and desktop reader customization presentations without reintroducing bespoke layout logic here.

## Constraints

- Do not start this task before `GM77` is implemented enough to provide the shared overlay primitive.
- Preserve the current reader customization capabilities such as font size, font family, colors, preview, and reset behavior.
- Preserve the existing reader route and reader preference persistence behavior.
- Remove custom overlay mechanics when the shared component covers them.
- Keep the resulting implementation aligned with the shared primitive instead of forking it locally for reader-specific needs.
- Do not let this task turn into a broad redesign of the reader experience unless a small UX adjustment is required by the new primitive.

## Acceptance Criteria

- [ ] `GM77` is treated as a prerequisite and the migration is implemented on top of the shared overlay primitive it introduces.
- [ ] The reader customization surface uses the shared core overlay component instead of bespoke overlay markup and behavior.
- [ ] The custom reader preferences surface no longer owns its own backdrop implementation when that is provided by the shared primitive.
- [ ] The custom reader preferences surface no longer owns its own Escape-key close implementation when that is provided by the shared primitive.
- [ ] The custom reader preferences surface no longer owns its own fixed-position shell and responsive popup layout logic when that is provided by the shared primitive.
- [ ] Reader customization still supports font size changes.
- [ ] Reader customization still supports font family changes.
- [ ] Reader customization still supports reader color customization such as background and text colors.
- [ ] Reader customization still supports live preview.
- [ ] Reader customization still supports reset behavior.
- [ ] The migrated surface works on mobile using the new shared popup behavior defined by `GM77`.
- [ ] The migrated surface works on desktop using the new shared popup behavior defined by `GM77`.
- [ ] The resulting reader customization implementation contains materially less popup-specific custom code than before the migration.
- [ ] The migration acts as a real proof that the `GM77` primitive is reusable for product features, not just theoretical.

## Dos

- Do treat this as the first real adoption task for `GM77`.
- Do preserve user-facing reader preference behavior while replacing surface mechanics.
- Do remove obsolete overlay code instead of keeping it around as dead weight.
- Do keep the reader-specific content focused on preferences, preview, and state changes.
- Do validate both mobile and desktop behavior after the migration.

## Don'ts

- Don't reimplement the new shared popup logic locally inside the reader feature.
- Don't keep duplicate overlay behavior just because it already exists.
- Don't use this task to redesign the full reader UX from scratch.
- Don't break saved reader preferences or their existing controls.
- Don't treat `GM77` as optional.
