# Collect Profile Update Event

## Elevator's Pitch

Record a `profile-update` event every time the user changes information in their profile.

## Business Gain

Profile changes are important behavior signals. Capturing them in the event stream lets the app understand when users adjust their language-learning setup instead of guessing like a database without Chuck Norris watching it.

## Current State

LingoCafe already has a profile persistence flow through `/api/lingocafe/profile`.

The `lingocafe.events` table already exists as a raw append-only event stream.

Related draft work already expects other reader actions, such as `page-open` and `page-scroll`, to populate the same event stream.

The current profile update route saves profile changes, but the operator has now asked to collect a `profile-update` event whenever the user changes profile information.

## Desired State

Whenever an authenticated user changes LingoCafe profile information, the app records a `profile-update` event.

The event should be tied to the authenticated user, not to any client-supplied user identifier.

The event should only be recorded when the submitted profile values differ from the values already stored for the user's LingoCafe profile.

The event `data` payload should contain a before/after diff of the changed LingoCafe profile values.

## Definition of Success

Every successful LingoCafe profile information change creates one corresponding `profile-update` event in the existing event stream.

Repeated saves that do not change any stored LingoCafe profile values do not create a `profile-update` event.

Failed validation or failed profile writes do not create profile update events.

## Additional Context

The operator asked: "profile update event. Collect the profile-update event every time that the user changes information in their profile."

The operator clarified that this task is strictly for the LingoCafe profile flow, not the generic account `/profile` page.

The operator also clarified that `profile-update` should fire only when there was a real change of values, and the event `data` should contain a before/after diff of the profile data.

## Assumptions

The event name should be exactly `profile-update`.

The event should be collected after a profile change is successfully persisted.

The existing `lingocafe.events` table is the intended destination for this event.

The profile update surface in scope is `/api/lingocafe/profile`.

## Constraints

Use server-side validation and the authenticated session user.

Do not trust client-supplied user IDs.

Do not emit an event for invalid profile submissions.

Do not emit an event if the submitted values match the stored values.

Do not emit an event if the profile save fails.

Limit the event diff to LingoCafe profile data.

## Acceptance Criteria

- [ ] A successful LingoCafe profile information change records a `profile-update` event.
- [ ] A successful LingoCafe profile save with unchanged values does not record a `profile-update` event.
- [ ] The event uses the authenticated user's ID.
- [ ] The event is stored in the existing event stream.
- [ ] The event `data` contains a before/after diff of the changed profile values.
- [ ] Invalid profile update requests do not record a `profile-update` event.
- [ ] Failed profile writes do not record a `profile-update` event.
- [ ] The implementation is scoped to the LingoCafe profile flow.
- [ ] The behavior is covered by an appropriate API-level test or smoke check.
- [ ] `npm run qa` passes after implementation.

## Dos

- Do record the event only after a real LingoCafe profile value change succeeds.
- Do use the existing database access pattern.
- Do keep the event name stable as `profile-update`.
- Do relate the event to the authenticated user.
- Do include a before/after diff of changed profile values in event `data`.
- Do keep the implementation aligned with the existing LingoCafe event model.

## Don'ts

- Don't emit duplicate events for one successful profile change.
- Don't emit an event for a no-op save where submitted values match stored values.
- Don't emit events for rejected validation payloads.
- Don't emit events for failed writes.
- Don't accept a browser-supplied user ID for event ownership.
- Don't invent a separate event storage system.
- Don't extend this task to the generic account profile page.

## Open Questions

Q: Which profile fields should be included in the event payload?
A: The event payload should contain a before/after diff of the changed LingoCafe profile data.

Q: Should repeated saves with unchanged values emit a `profile-update` event?
A: No. Emit `profile-update` only when there was a real change of stored values.

Q: Should the event include before-and-after values, changed field names only, or a minimal marker?
A: The event should include a before/after diff of the profile data.

Q: Should this apply only to the LingoCafe profile or also to any future global account profile page?
A: Strictly LingoCafe only.
