# HeroBlock: Icon Support

## Elevator's Pitch

Implement icon support in the HeroBlock so a page can show an optional icon as part of the hero content.

## Business Gain

Hero sections can communicate page intent faster and with more visual identity, without every app needing a custom hero component.

## Current State

HeroBlock exists, but the operator request implies it does not currently support rendering an icon.

## Desired State

HeroBlock accepts icon configuration and renders the icon consistently with the existing title, subtitle, actions, and alignment behavior.

## Definition of Success

HeroBlock can display an optional icon without breaking existing HeroBlock usages.

## Additional Context

information is missing

## Assumptions

The icon should be optional and should preserve backward compatibility for all existing HeroBlock configurations.

## Constraints

Follow the existing HeroBlock API and styling patterns. Do not force callers to provide an icon.

## Acceptance Criteria

- HeroBlock supports an optional icon.
- Existing HeroBlock usage without an icon continues to render as before.
- Icon placement works with existing HeroBlock layout behavior.
- TypeScript types document and enforce the icon option.

## Dos

- Keep the API configuration-driven.
- Reuse the project’s existing icon strategy if one is already established.
- Keep rendering accessible and visually aligned with the HeroBlock content.

## Don'ts

- Do not make icon support mandatory.
- Do not introduce an unrelated hero redesign.
- Do not break current title, subtitle, actions, or alignment behavior.

## Open Questions

- Should the icon be passed as a component, an icon name, an image URL, or a structured config object?
- Should the icon support custom size, color, or position?
