# ZO85 Questions v2

V1 answers resolved the big direction:

- This is a block-based ProfilePage, not a field/grid form builder.
- `app.profile.items` remains the ordered list of ProfileBlocks.
- `app.profile.schema` is hand-written AJV validation for `auth.users.profile`.
- Consent is separate from profile and is rendered by a core ConsentBlock.
- Required consent participates in profile completeness.
- `/books` onboarding stays custom for now.
- Default UX should include editable account identity and logout.
- Default and Quicklist should exercise core blocks and generic consent (as basic config).

Answer only what you can. Mark `defer` when it should not block planning. The questions below are the ones still holding the plan by the throat.

## 1. Core API Shape

1. Should the core profile API include both server helpers and HTTP endpoints?
   - Example: `@/42go/profile/server` for product endpoints, plus `/api/profile` for client ProfileBlocks.

yes, and I feel we should create a "top level" `@/42go/profile` folder in which to put all the profile-related code (helpers, components).

The `/api/profile` in the app's folder should be a shallow wrapper of the main logic from the 42go/profile folder.

remember that we want to centralize as much core code as possible under @/42go so that it behaves like a portable folders across multiple installations.

2. Should HTTP endpoints be always available for authenticated apps, or enabled by a feature flag such as `api:profile`?

protect them with the feature flag and also require the session.

3. Should client ProfileBlocks use a core hook such as `useProfile()` / `useProfileBlock()` instead of manual `fetch()`?

yes, better abstraction.

4. Should low-level server helpers accept an explicit `userId`, while request helpers derive the user from the session?

yes

5. Should profile helpers always enforce both `id` and `app_id` in database queries?

the userId is already scoped by app_id. so using a specific user id should be enugh.
maybe we should pass the app_id as decorator property so that it is easier to grab relevant configuration.
but we can also choose to let sub-helpers to use the config apis directly.

reason on pros/cons.

but i tend to prefer functional approach in which low level function receive all the needed information to perform their logic. no side-effects, easier to unit-test.

6. Should the core API names stay exactly `profile.load/save/validate/get/set`, or is a module-style API acceptable?
   - Example: `loadProfile()`, `saveProfile()`, `validateProfile()`.

ok, acceptable.

7. Should we add `profile.setMany({...})` for block/page saves, since blocks can own several keys?

yes, remember that any set is subject to global validation.
it could first get existing profile, override (in memory) the requested keys, then apply validation before committing the changes back into the db.

8. Should `profile.save(null)` be rejected, clear the profile back to `null`, or be allowed only through a dedicated clear API?

yes, if we want to nullify the profile, an explicit call/utility/api is preferred.
no need to implement now.

9. Should `profile.set(key, undefined)` delete that key, store `null`, or reject?

set should only change a key value based on the AJV model possibilities.
we should reject "undefined" unless the AJV model explicitly allows it

10. Should profile writes update `auth.users.updated_at`?

yes

11. Should profile writes use read-merge-validate-write or SQL `jsonb_set`?

use "read-merge-validate-write" for now.

12. Do we need optimistic concurrency now to prevent parallel blocks or tabs from overwriting each other?

i don't understand. explain me the usecase and possible problem.

- onSaveError
- requestValidate
- requestSave
