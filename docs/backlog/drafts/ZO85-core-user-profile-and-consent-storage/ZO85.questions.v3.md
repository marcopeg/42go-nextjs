# ZO85 Questions v3

V2 answers resolved the core API direction:

- Put portable core profile code under `src/42go/profile`.
- Expose `/api/profile` as a shallow wrapper over that core logic.
- Protect `/api/profile` with a feature flag such as `api:profile` and session auth.
- Client ProfileBlocks should use a core hook/abstraction instead of manual `fetch()`.
- Low-level helpers should accept explicit inputs such as `userId`, app id, and app config.
- Request-level wrappers can derive session/app context and call low-level helpers.
- Module-style function names are acceptable.
- Add `setMany`.
- Use read-merge-validate-write.
- Update `auth.users.updated_at` on profile writes.
- Do not implement normal profile clearing through `save(null)`.
- Reject `undefined` in `set()` for the first implementation.
- ProfileBlock contract needs managed keys and lifecycle hooks around validation/save.

The questions below are the remaining planning blockers. The profile beast is limping now. Finish it.

## 1. Optimistic Concurrency

Use case: a user opens `/profile` in two tabs. Tab A changes `targetLang` and saves. Tab B still has the old profile snapshot, changes `ownLang`, and saves later. With read-merge-validate-write, Tab B may overwrite Tab A unless the save only patches managed keys or checks a version.

1. Is this risk acceptable for the first implementation if `setMany` writes only the keys submitted by a block/page?

acceptable. last save overrides.

2. Should the profile API return a `profileVersion` or `updatedAt` value and reject stale full-document saves?

no need.

3. Should `profile.save({...})` replace the whole document only for trusted server code, while UI flows use `setMany({...})` to patch managed keys?

replacing the whole document only if it passes validation.

4. Should the first pass skip optimistic concurrency and document the limitation?

yes.

7. Should block custom validation be allowed to inspect the whole profile and consent dataset?

8. Should a block-level `persist()` escape hatch remain available for rare custom behavior?
