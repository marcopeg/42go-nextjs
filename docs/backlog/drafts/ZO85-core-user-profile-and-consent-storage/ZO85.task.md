# Core User Profile and Consent Storage

## Elevator's Pitch

Promote user profile and consent storage into core 42go functionality. Store reusable profile data on `auth.users.profile` and reusable consent evidence on `auth.users.consent`, then migrate LingoCafe away from the app-specific `lingocafe.profiles` table.

## Business Gain

Every app can use the same authenticated-user profile and consent foundation instead of inventing its own custom table and validation flow. LingoCafe becomes the first real migration path, and future apps get the profile hammer without rebuilding the whole dojo.

The value is not a generic field-grid form builder. The value is a reusable block-based profile page where app builders compose rich profile blocks, store their data in a shared place, validate it consistently, and still have enough room to build app-specific UI that looks right.

## Current State

The configurable `/profile` page already exists through `app.profile.items`, and NO51 added LingoCafe-specific legal and consent blocks.

The current ProfilePage is already a block composition system. Apps can add core blocks or custom React profile blocks. A block is a container that owns UI and one or more related data fields.

The current custom LingoCafe blocks still load and persist through `/api/lingocafe/profile` directly. That proves the pattern but keeps too much data and persistence behavior inside LingoCafe.

LingoCafe currently stores language preferences and consent evidence in `lingocafe.profiles`, including a `data` JSONB field. That table is too app-specific for concepts that other apps will likely need: profile data, consent data, profile completeness, and schema validation.

The initial auth migration is `knex/migrations/20240320_auth.js`, and it creates `auth.users` without profile or consent JSONB fields.

The LingoCafe schema migration is `knex/migrations/20260427150000_lingocafe_schema.js`, and it currently creates `lingocafe.profiles`.

The LingoCafe `/books` wizard detects incomplete profile state from the LingoCafe profile table and custom logic.

## Desired State

Add nullable JSONB fields to `auth.users`:

- `profile jsonb default null`
- `consent jsonb default null`

Both fields should default to `null`, not `{}`, so code can tell when a user has never created profile or consent data.

Update the existing local migrations directly because the local database can be destroyed and rebuilt. Also generate a separate `migrate.sql` progressive migration for manual production patching. Do not create a dirty extra Knex migration file for this change.

Keep ProfilePage as block composition, not a form builder. `app.profile.items` remains the ordered list of profile blocks to render. `app.profile.schema` contains the AJV validation schema for profile data. There must be no core `fields`, `sections`, grid DSL, or typed field declaration model in this task.

Each ProfileBlock is a rich mini-app responsible for its own layout and for one or more related data properties. Core blocks should be polished and reusable, while custom blocks can remain full React components referenced from AppConfig.

The normal profile-save path should move toward a central ProfilePage data contract: blocks declare or expose which profile properties they manage where practical, the ProfilePage loads current profile/consent data, asks blocks to validate, gathers updates, runs schema and consent validation, and persists centrally. Blocks should not need to make HTTP requests in the normal path. Exceptions should remain possible through profile page events such as validate, save, load data, or a passed API surface.

Do not add a full ProfilePage override API in this first implementation. Apps that need a fully custom profile experience can implement app-owned UI later and use the same core profile APIs; backend storage, validation, consent evidence, and completeness semantics must remain core.

Create a top-level `src/42go/profile` area for profile-related core code, including server helpers, client hooks, core components, types, and supporting utilities. This folder should be as portable as the rest of `42go`.

Expose a core `/api/profile` route from the app router as a shallow wrapper over `src/42go/profile` logic. The endpoint must be protected by a feature flag such as `api:profile` and must require an authenticated session.

Client ProfileBlocks should use core abstractions such as `useProfile()` or `useProfileBlock()` instead of manual `fetch()` in the normal path.

Add core profile APIs that app-specific components and product endpoints can use:

- `profile.load()` reads the entire profile data from `auth.users.profile`.
- `profile.save({...})` validates and replaces the entire profile JSON.
- `profile.validate({...})` validates profile data against the current app config schema when one exists.
- `profile.get(key1, key2, key3, ...)` reads only selected top-level profile keys and returns a key/value object.
- `profile.set(key, value)` validates the resulting profile document and writes one top-level key.
- `profile.setMany({...})` validates the resulting profile document and writes multiple top-level keys for block/page saves.

Both `profile.save()` and `profile.set()` must run validation before writing.

Do not expose consent as a fully separate public helper family parallel to profile. Consent has dedicated config and evidence logic, but the main application API should flow through the profile surface: profile load/save should include consent, and client store helpers may expose `profile.consent.get()` and `profile.consent.set()` for in-memory reads and writes.

Low-level server helpers should accept explicit inputs such as `userId`, app id, and app configuration rather than hiding request-context lookups. Request-level helpers may derive those inputs from the authenticated session and resolved app. This keeps the core helpers functional, easier to test, and less magical.

Using module-style function names such as `loadProfile()`, `saveProfile()`, `validateProfile()`, `getProfileKeys()`, `setProfileKey()`, or `setProfileKeys()` is acceptable as long as the resulting API clearly covers the drafted `profile.load/save/validate/get/set/setMany` capabilities.

Profile writes should use read-merge-validate-write for now. `profile.save(null)` should not be a normal save path; clearing or nullifying a profile should require a future explicit utility or API. `profile.set(key, undefined)` should be rejected unless a later validation/data contract explicitly supports delete semantics. Profile writes should update `auth.users.updated_at`.

Skip optimistic concurrency for the first implementation. Last save wins. UI flows should prefer managed-key writes such as `setMany()` over full document replacement to reduce accidental overwrites. Full `saveProfile()` replacement remains acceptable when the submitted full document passes validation.

Add optional `app.profile.schema` configuration. It contains an AJV-compatible JSON Schema document used for profile validation. If no schema is configured, no profile validation is applied. Use JSON Schema draft-07 for the first implementation. Run AJV in strict mode by default, but allow an app-level override such as `app.profile.ajv.strict = false` when strict mode blocks a practical app schema.

Introduce profile completeness as a core logic feature. Required profile schema data and required consent data together determine whether a profile is complete. Apps can use profile completeness as a gate to the real product. LingoCafe must use it to block app usage until required profile and consent information exists.

Profile data must be a plain JSON object with camelCase keys. Unknown profile keys are allowed unless the app schema sets `additionalProperties: false`, but documentation should encourage rich, strict app schemas. `app.profile.schema.required` plus required consent items define completeness. AJV validation should run on the client before sending and on the server for authority. Start with basic AJV path errors plus block-local messages; friendly schema-message support can come later.

Add a core consent storage and validation model that complements profile but stores data in `auth.users.consent`. The consent store is dedicated to terms, privacy, marketing permissions, and similar consent or acknowledgement evidence.

Add `app.consent.items[]` configuration. Each item defines one consent element:

```ts
app: {
  consent: {
    items: [
      {
        name: "terms",
        required: true,
        version: "2026-05-01",
        purpose: "Accept Terms and Conditions",
        legalBasis: "contract",
        category: "legal",
        statement: "I accept the Terms and Conditions",
        label: "I accept the Terms and Conditions",
      },
    ],
  },
}
```

`label` can be a string or a React component so apps can render links, dialogs, or app-specific legal copy. `statement` is a required plain string for every consent item. It is the stable evidence text for what the user accepted or acknowledged, while `label` is only presentation.

Add a core `ConsentBlock` / `ProfileConsent` ProfileBlock that renders configured consent items and collects checked values. Required consent items must cause profile-page validation to fail when missing.

When consent is saved, store GDPR-oriented evidence under `auth.users.consent.{name}[]`. Each evidence entry should include the saved value and trace fields similar to the current LingoCafe implementation, such as timestamp, source, method, and statement or document version.

Consent item config stays at `app.consent.items`. ConsentBlock must be explicitly listed in `app.profile.items`; the page should not auto-add it. Each consent item must declare `version`, `purpose`, `legalBasis`, `category`, and `statement`. Documentation must define the supported legal-basis and category values and must state that the app owner is responsible for reviewing final legal wording and legal basis. Saved records use `auth.users.consent.{name}` as an array of evidence entries, not `consent_{name}`. The last array item is the current value. Optional unchecked consent should be stored explicitly as a `value: false` evidence entry on first save. Profile and consent should save in one transaction when submitted from ProfilePage, and profile load should return both profile and consent together.

Consent collection needs configurable evidence fields. `collect` has no default and starts empty. The implementation should define a supported `collect` list such as source, method, IP address, user agent, and other device/context values only if explicitly configured. LingoCafe should explicitly collect `source`, `method`, `ip`, and `ua`. Consent evidence entries should always include `value`, `changedAt`, `version`, and `statement`; `source`, `method`, and other evidence fields are included when configured or derivable.

Consent history should append a new evidence entry only when the saved boolean value changes or the configured `version` or `statement` changes. No-op saves must preserve the existing history and must not mutate `changedAt`. A required consent item is complete only when its latest evidence entry has `value: true` and matches the current configured `version` and `statement`. Optional consent with a stale version or statement should not be treated as an active current opt-in until the user saves that item again.

If no custom profile config is provided, the profile page should be empty rather than silently inventing profile content. Existing apps should receive explicit profile configuration. The default app should include editable account information, logout, generic consent, and `TestRBAC`. Quicklist should be configured with available core blocks and generic consent for terms, privacy, and marketing purposes. Editable account identity is limited to `name` and `image` for now.

The ProfilePage keeps a single global `Save preferences` action. Save should always remain available, but the page must show a dirty-state indicator and warn before leaving with unsaved changes.

The ProfileBlock runtime contract should support at least these concepts: managed profile keys, full profile and consent dataset access for conditional rendering, custom validation-error handling, a validation-request hook, a pre-save hook after successful validation but before persistence, save-success handling, save-error handling, and programmatic validate/save requests where needed. Use React context/reducer plus focused hooks for the first implementation instead of adding a state-management dependency. The store should track profile and consent separately, but expose one combined dataset to blocks. Managed keys should support top-level profile and consent keys only in the first implementation. Prefer explicit managed-key metadata on `app.profile.items` so ownership is visible at the composition point; platform blocks can carry built-in metadata, and reusable custom blocks may export descriptor/factory helpers when that avoids repetition. Blocks should update a central ProfilePage store as values change, and dirty state should be inferred from loaded snapshots versus current store values. Backward compatibility with the current `useProfileBlockHandle({ validate, persist })` contract is not required.

Block custom validation runs before AJV and may inspect the full profile and consent dataset. AJV runs after block validation and must run both client-side and server-side. Blocks should receive AJV errors after save failure so they can show inline cues. A block-level save escape hatch may remain, but the preferred name/concept is `onRequestSave`.

Update LingoCafe to remove `lingocafe.profiles` and use `auth.users.profile` plus `auth.users.consent`.

Move LingoCafe language preferences into `auth.users.profile`, validated by `app.profile.schema`.

Move LingoCafe legal and optional consent choices into `auth.users.consent`, rendered through the core `ProfileConsent` component where practical.

Remove `/api/lingocafe/profile`; LingoCafe profile blocks should move to `/api/profile`. `/api/lingocafe/books` may keep returning its current profile response shape to reduce frontend churn, but it must load profile data through the new core utilities rather than direct LingoCafe profile-table queries.

The ProfilePage/core server path must expose app-specific server hooks so product code can observe validated profile and consent changes. LingoCafe should implement its own hook handler using that core hook system. LingoCafe profile changes should continue to write `profile-update` events in `lingocafe.events`; consent changes should write `consent-update` events. `LingoCafeTermsPrivacy` and `LingoCafeContent` should be deleted once the core ConsentBlock replaces them. `LingocafePreferences` should become a custom block using the new central profile block contract. LingoCafe `ownLang`, `targetLang`, and `targetLevel` should be the exact camelCase keys stored in `auth.users.profile`. LingoCafe language/level option lists means the `own`, `target`, and `levels` arrays currently returned by `getReaderLanguages()`. Move those options into an explicit LingoCafe-owned AppConfig extension keyed by `ownLang`, `targetLang`, and `targetLevel`, rather than a generic `app.profile.options` bucket.

Update the `/books` wizard so it opens when the profile is incomplete according to the LingoCafe `app.profile.schema`, rather than relying on custom `lingocafe.profiles` completion checks.

The LingoCafe wizard should be able to render `ProfileConsent` so onboarding can collect profile fields and consent fields in one validated flow.

For this task, `/books` onboarding remains a custom LingoCafe feature. A future task may promote onboarding to a core feature, but this task should not turn onboarding into another generic builder.

This draft may be planned as one implementation task with a precise multi-phase plan. The expected phases are: core profile storage/API, ProfileBlock runtime/ProfilePage store, core consent system, LingoCafe migration, default/quicklist configuration plus docs/skills, and manual production patch. The planner should still call out phase boundaries, verification gates, and rollback risks clearly. If planning reveals the one-shot path is too broad to execute safely, the planner may recommend splitting before execution.

## Definition of Success

The repository has one reusable authenticated-user profile and consent storage model in `auth.users`. Profile pages remain block-composed and app-customizable, not field-config builders. LingoCafe no longer depends on `lingocafe.profiles`, and both `/profile` and `/books` use the same validation rules and consent component model. Production can be patched with `migrate.sql` without adding a one-off Knex migration.

## Additional Context

This task follows NO51, which intentionally stored consent evidence in `lingocafe.profiles.data`. That was useful for proving the LingoCafe flow, but it now looks like hyper customization.

The current LingoCafe consent evidence pattern stores readable objects with `value`, `date`, `version`, `source`, and `method`. This task should preserve that evidence quality while moving storage to the core `auth.users.consent` field.

The existing profile block system lives under `src/42go/components/ProfileBlock` and is configured from `app.profile.items`.

The core profile APIs should be usable by custom profile components and by custom product endpoints such as LingoCafe `/books` and `/api/lingocafe/*`.

The operator clarified during refinement that the overall goal is a core feature that lets custom apps describe, fill, and update a user profile. The design must balance an easy setup/declaration path with enough customization for apps to make the profile experience beautiful.

V1 refinement answers clarified that the ProfilePage must stay block-based. Blocks are responsible for UI and layout. AJV validates the profile data. Consent is separate storage and logic because legal evidence has special requirements. Required consent contributes to profile completeness.

An extended refinement questionnaire for this large task is stored in `ZO85.questions.v1.md`.

A second refinement round based on answered V1 questions is stored in `ZO85.questions.v2.md`.

A third refinement round based on answered V2 questions is stored in `ZO85.questions.v3.md`.

A fourth refinement round based on answered V3 questions is stored in `ZO85.questions.v4.md`.

V4 refinement answers clarified that a one-shot phased plan is still preferred over pre-splitting the draft, React context/reducer should be used before adding a state-management library, profile data uses camelCase keys, AJV should use draft-07 strict mode with app override support, consent stores history arrays with the last entry as current state, consent evidence uses `changedAt`, LingoCafe records `consent-update` events for consent changes, and LingoCafe option lists refer to language and reading-level choices.

The reason for a plain consent `statement` string is evidence traceability. A React label can include links, dialogs, or rich UI, but it is not a stable serializable record of what the user agreed to. The implementation should preserve a plain statement in config for every consent item, so consent evidence can reference text/version without trying to serialize rendered React.

Final review changed that recommendation from conditional to required: every consent item should have a plain `statement`, even when `label` is a string, because it gives consent evidence a stable contract and avoids special cases.

Planning guidance clarified that the LingoCafe profile option config should use the same keys as the persisted profile object: `ownLang`, `targetLang`, and `targetLevel`.

## Assumptions

Profile schema validation is app-scoped and resolved from the current request's app configuration.

Consent item validation is separate from profile schema validation, but the profile page save orchestration should report both types of failures coherently.

Profile data includes app-specific settings, preferences, and onboarding answers. Consent does not belong in `profile`; it belongs in `consent`.

Top-level identity data such as `name`, `image`, and maybe `username` remains separate from `auth.users.profile` and should be edited by core ProfileBlocks.

AJV is the desired validation engine for JSON Schema documents. Planning should verify whether it is already installed or needs to be added.

The progressive production patch should be a SQL artifact named `migrate.sql` unless planning chooses a more precise repository location.

Existing local seed data can be adjusted to populate `auth.users.profile` and `auth.users.consent` instead of `lingocafe.profiles`.

React components and functions are acceptable in app profile config, consistent with the existing component-reference pattern.

The implementation should favor functional low-level helpers that receive the data they need directly. Request-aware wrappers can resolve session and app context, then call the low-level helpers.

Migration can start production users with no profile and no consent. Existing `lingocafe.profiles` data does not need to be copied. Non-consent keys in `lingocafe.profiles.data` can be ignored.

The implementation is expected to proceed through a multi-phase plan inside ZO85 unless planning proves the blast radius is too high.
