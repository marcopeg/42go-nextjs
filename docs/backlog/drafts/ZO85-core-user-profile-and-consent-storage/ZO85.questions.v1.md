# ZO85 Questions v1

These questions are intentionally broad. `ZO85` is a core platform feature, a data migration, a profile UI system, and a LingoCafe migration wearing one jacket. Answer inline where possible. Mark anything as `defer` only when it should not block planning.

## 1. Product Goal and Scope

1. What is the primary user story for this task: app builders declaring profile fields, end users filling a profile, product endpoints reading profile data, or LingoCafe migration?

All you listed is part of the story. I'd say the goal is "collect and store profile and consent data".
The items you listed are different aspects of the same problem.

2. Is the first implementation allowed to ship as a reusable core API plus one LingoCafe-specific UI, or must it include a generic declarative profile form builder immediately?

The ProfilePage is a generic form builder in which an App config adds blocks (it works like this already, it needs polishing).

We need one new core component for the ConsentBlock that should dynamically render and collect the consent information

3. Should `ZO85` remain one large implementation task, or should it be split into core storage/API, declarative profile UI, consent, and LingoCafe migration tasks?

I want to attempt one shot because most of it is already implemented.
But you should be extra careful in the planning so to structure a great list of phases and tasks so that you don't get lost during implementation.

4. What does "easy to setup and declare" mean concretely: a JSON/TS config only, a small number of built-in field types, or no custom React for common cases?

It should be easy to declare in an app's config.
NOTE: this is not a "form builder". We are not going to provide fields and grids. The configuration will list "blocks". A block is a container that knows how to deal (ui / data) with one or more fields.
The configuration already allows to add a custom block as a fully fledged react component.

You must craft and refine a few core containers (already defined in `src/42go/components/ProfileBlock`).
You must polish the `42go-profile-block` and `42go-consent` skills so that an agent can help composing the configuration and crafting custom blocks.

5. What does "customizable to make it look beautiful" require concretely: custom labels, custom field components, custom section layouts, app-specific wrappers, theme tokens, or full render overrides?

The theme and the tokens already exist and this part should leverage it.
In a second phase we will progress and polish the UI of the core blocks, and probably encode the UI guidelines for the custom blocks in the `42go-profile-block` skill.
Yes, the ProfilePage and the ConsentBlock should allow a form of full override if an App wants to completely do over the entire page taking the full responsibility of it (well, maybe still using the core apis).

6. Which apps besides LingoCafe should the first design keep in mind, even if they are not implemented yet?

Quicklist and default don't really have custom configuration so you should configure them with the available core blocks and a generic consent for terms, privacy and marketing purposes.

7. What is the minimum feature set that would make this core profile system worth using for a new app?

if nothign is configured, it should show a block to change user's name, image, and the logout block

8. What must not be solved in the first implementation, even if it is tempting?

don't do a composable form with typed field. that's out of scope.
this is a block based page composition, one block after the other.
a block is responsible for one or more fields and their layout.

## 2. Domain Boundaries

1. What belongs in `auth.users.profile` versus top-level identity columns like `name`, `email`, and `image`?

It can be any info that an app needs.
For example in lingocafe we will keep the user's own language and the learning language.

2. Should user-editable account identity fields ever be represented in `app.profile.schema`, or are they always separate from app profile data?

separate. the profile schema is only for the profile data.
the name, image and maybe the username will be edited by custom core ProfileBlocks.

3. Is `auth.users.profile` app-specific because `auth.users` is already scoped by `app_id`, or do we need any extra app namespace inside the JSON document?

`auth.users.profile` means: field "profile" or table "users" in schema "auth".
(same for `auth.users.consent`)

4. Should profile data represent long-lived app preferences only, or can it include onboarding-only answers?

both.

5. Should app settings, preferences, and profile be separate concepts, or is "profile" the platform bucket for all per-user app JSON?

for now all goes into profile esxcept consent information that have their own explicit field.

6. Should consent be strictly separate from profile in all APIs and UI, or can profile form sections include consent components?

one of the ProfileBlock will be the ConsentBlock that will display the consent infor as configured.
It's like a sub-system but all is editable via the ProfilePage.
Then the backend should be smart enough to store the consent info in their own field.

The separation is needed because behind a single cosent checkbox, there will be a variety of info to collect to be gdpr compliant (and this list might as well be dynamic and configurable) so the logic can be more complex for this specific information.

7. Should required consent affect "profile completeness", or should profile completeness and consent completeness be two different states?

yes it does. if a consent required info is not provided, the profile is not complete.

i want to introdcue the concept of "profile completeness" as a logic feature flag.
For instance, in Lingocafe i want to use it as a gate to the real app.
A user can not use the app without completing the required profile info first.

8. Do we need a core vocabulary for `profile`, `consent`, `preferences`, `settings`, and `onboarding`, or is `profile` enough for now?

profile includes also preferences and settings.
consent is separated because is a legal specific concept.
i think it is a good idea to document this.

## 3. AppConfig Shape

1. Should the schema live at `app.profile.schema`, or should `app.profile` become a richer object with `schema`, `fields`, `sections`, and `items`?

- `app.profile.schema` contains the data validation schema (AJV)
- `app.profile.items` contains the list of blocks to render

There are no fields or sections. Just a list of blocks.
Each block can be a rich multi-field mini-app.

2. Should consent live at `app.consent.items`, as drafted, or under `app.profile.consent` to keep all profile-page concerns together?

Keep it separated to stress that is it an important aspect of designing an app.

3. Should `app.profile.items` remain the profile page composition model while a new `app.profile.fields` declares editable data fields?

THERE ARE NO "FIELDS".
The profile page is a composition of a list of rich ProfileBlock items or custom items.
This is not a form builder.

4. Should core profile fields be declared separately from the JSON Schema, or should the JSON Schema be the only source of field definitions?

There are not fields.

5. If both schema and field declarations exist, which one is authoritative for validation?

There are not fields.

6. Should field declarations generate an AJV schema automatically for common cases?

There are not fields.

7. Should AJV schema be hand-written for full control, or should most apps use a simpler declarative field DSL?

Hand written. Make sure the `42go-profile-block` skills guides the agent in writing a working ajv schema

8. Should the config support sections/groups such as `Language preferences`, `Notifications`, and `Legal`?

This is ProfileBlock responsibility.

9. Should field declarations include layout hints such as grid width, order, icon, help text, placeholder, and empty-state copy?

No, just one block after the other.

10. Should a field declaration support a custom React component override for only that field?

There are not fields.

11. Should a section declaration support a custom React component override for the whole section?

There are not fields or sections.

12. Should custom components receive the profile API through props/context, or continue to own their own fetch/persist logic with `useProfileBlockHandle()`?

a custom block should interact via a standard api interface.
ideally a block should expose the list of profile properties that are managed by the block iteself so that the ProfilePage is aware of what info to pass and what to expect when it needs to retrieve updated values.

if this is not possible, we should pass the full apis to the block as props so that the block can perform their own get/set of the values in memory.

but the validation / persistance should be fully managed by the ProfilePage.
Ideally a block doesn't perform any http request.

Of course, the has to be possible for a block to make exception and load or persist stuff hooking into some form of ProfilePage events like "requestValidate", "requestSave", "requestLoadData".

13. Should profile config support conditional visibility, such as showing a field only when another value is selected?

Up to the block's logic.
A block should be able to check on any current value of the overall profile dataset via api/events so that it can implement these kind of logic.

14. Should profile config support read-only computed fields?

Up to the block's logic.

15. Should profile config support app-specific validation messages per field?

This is managed by AJV model.

16. Should profile config support `defaultValue`, or should missing values stay missing until explicitly saved?

This is managed by AJV model.

17. Should profile config support option providers for select fields, such as static arrays, functions, or API endpoints?

Up to the block's logic.

18. Should field names be limited to top-level keys, or should config support nested paths?

This is managed by AJV model and block's logic.

19. Should app profile config be fully serializable, or are React components/functions acceptable because config already uses component references?

React components/functions acceptable
