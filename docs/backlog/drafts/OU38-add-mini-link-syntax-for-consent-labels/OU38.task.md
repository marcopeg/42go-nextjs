# Add Mini Link Syntax for Consent Labels

## Elevator's Pitch

Extend the core consent label rendering so a string label can mark one segment as a link using a tiny wildcard-style marker, with companion config for the destination URL and link target. This should stay intentionally narrow and behave like a super-mini markdown feature with just one supported mechanic.

## Business Gain

Apps can keep consent labels simple and configuration-driven while still linking legal text such as Terms or Privacy pages without needing a custom React component for every consent item.

## Current State

Consent items already support `label` as either a plain string or a React component. When `label` is a string, the core renderer displays it as raw text with no inline link syntax or structured link metadata.

## Desired State

If a consent item's `label` is a string and the item also provides link metadata, a marked substring inside the label should render as an anchor tag. The configuration should also support a target choice equivalent to HTML anchor target behavior, with same-tab or new-tab/new-window options, and the default should open in a new tab/window.

## Definition of Success

An app can configure a consent item using a plain string label plus one link configuration object or property, mark the link segment inside the string with the agreed wildcard marker, and the consent UI renders that segment as a clickable link while leaving the rest of the string as plain text.

## Additional Context

Relevant core files already exist:

- `src/42go/profile/types.ts` defines `TConsentItem`, where `label` currently supports string-like React content or a component.
- `src/42go/profile/ProfileConsent.tsx` renders consent labels and is the likely place to interpret the mini syntax.
- `src/42go/components/ProfileBlock/blocks/Consent.tsx` hosts the profile block that uses `ProfileConsent`.
- Current app configs in `src/config/default/config.ts`, `src/config/quicklist/config.ts`, and `src/config/lingocafe/config.ts` use plain string consent labels today.

The requested behavior is intentionally narrow:

- only one mini-markup mechanic
- only for string labels
- only for rendering a single link segment
- not a general markdown parser

## Assumptions

- React-component labels should keep working unchanged.
- The link target should default to opening in a new tab or new window.
- The consent evidence `statement` should remain plain persisted text and should not become HTML.

## Constraints

- Keep the feature in the core consent system, not in app-specific consent rendering.
- Do not introduce full markdown support for consent labels.
- Do not require every app to replace string labels with custom React components just to add one link.
- Preserve backwards compatibility for existing consent configurations that use plain string labels with no link metadata.

## Acceptance Criteria

- [ ] The core consent config supports one new link-related property or object for string labels.
- [ ] A string label can mark one segment with a simple wildcard-style delimiter and that segment renders as a link.
- [ ] The link target supports same-screen or new-screen behavior.
- [ ] The default link target opens in a new tab or new window.
- [ ] Consent labels without link metadata keep rendering as plain text.
- [ ] React-component labels keep working unchanged.
- [ ] The feature is implemented as a tiny custom mechanic, not as general markdown parsing.

## Dos

- Do choose a clear property name for the link config, even if the initial idea mentioned both `label link` and `label URL`.
- Do keep the parsing rules explicit and very small.
- Do keep the persisted evidence statement plain and stable.

## Don'ts

- Don't introduce a general markdown engine for consent labels.
- Don't store rendered HTML in consent evidence.
- Don't break existing consent items that already use string labels or component labels.

## Open Questions

- What is the exact delimiter token for the wildcard-style link marker inside the string label?
- What is the final config shape and name: a single `labelLink` object, a `labelUrl` plus `labelTarget`, or another small structure?
- Should the mini syntax support exactly one link segment or multiple link segments in the same label string?
