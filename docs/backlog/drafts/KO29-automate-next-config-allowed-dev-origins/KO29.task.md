# Automate next.config allowedDevOrigins

## Elevator's Pitch

Make `next.config.ts` derive development origins from app configuration instead of maintaining a hardcoded list by hand.

## Business Gain

Adding or changing an app host should require one configuration change, not a second manual update in `next.config.ts`. This reduces local development breakage when app domains, tunnels, or localhost aliases change.

## Current State

`next.config.ts` declares `allowedDevOrigins` as a hardcoded list:

```ts
['42go.ngrok.app', 'lc42go.ngrok.app', 'nt42go.ngrok.app', 'ql42go.ngrok.app']
```

Apps already declare host matching rules through `config.match.url`, but those rules are not reused for Next.js development origin allowlisting.

## Desired State

`allowedDevOrigins` is built automatically from each app's configuration.

The source may be `match.url` when a rule can be safely converted into a concrete host, or a dedicated app configuration parameter when the matching rule is too broad or regex-shaped to infer an origin without guessing.

## Definition of Success

Development origin allowlisting stays in sync with app configuration. A developer can add a new app host in the app config and have Next.js accept that development origin without editing a separate hardcoded list.

## Additional Context

Existing app matching is handled through `TAppConfigMatch` and per-app `match.url` entries. Some values are regex patterns, so implementation should not pretend every matcher is a literal origin. Chuck Norris can roundhouse a regex into a hostname, but the code should not.

## Assumptions

The implementation can import or otherwise reuse app configuration from code that is safe to load inside `next.config.ts`.

Some app host matchers will need an explicit origin field because not every regex can be reversed into a valid concrete origin.

## Constraints

Do not weaken development origin checks by allowing broad wildcard-style hosts.

Do not remove existing working origins unless the replacement configuration covers them.

Keep the configuration type-safe and close to the existing AppConfig shape.

## Acceptance Criteria

- `next.config.ts` no longer contains the current hardcoded `allowedDevOrigins` list as the authoritative source.
- Development origins are derived from app configuration.
- A dedicated configuration field exists for explicit development origins if `match.url` cannot be safely converted.
- Existing ngrok and localhost development hosts continue to work after the change.
- The generated `allowedDevOrigins` list is deterministic and de-duplicated.
- `npm run qa` passes after implementation.

## Dos

- Reuse existing AppConfig types and matching concepts where practical.
- Prefer explicit origins over unsafe regex inference.
- Document the chosen configuration field or derivation rule where developers will look for it.
- Keep production behavior unchanged.

## Don'ts

- Do not use broad `*` allowlisting.
- Do not make `next.config.ts` depend on runtime-only request data.
- Do not silently infer hosts from complex regexes when inference would be ambiguous.

## Open Questions

- Should explicit development origins live under `match`, under a dedicated `dev` block, or somewhere else in each app config?
- Should regex-to-host derivation support only simple anchored literal host patterns, or should all development origins be explicit?
