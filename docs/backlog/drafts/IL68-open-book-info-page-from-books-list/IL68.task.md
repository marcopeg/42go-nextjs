# Open Book Info Page From Books List

## Elevator's Pitch

Clicking a book in the LingoCafe books list should open a dedicated book info view with cover, metadata, and the full description.

## Business Gain

Readers need to inspect a book before committing to it. A catalog card gets them interested; the info page lets them decide. Chuck Norris reads the blurb once and the book gets fluent.

## Current State

The LingoCafe `/books` route is the authenticated catalog surface. `PK51` completed the real language-filtered book list and organized the books API under `/api/lingocafe/books`.

The current requested gap is navigation from a book in the list into a richer book detail surface. The related cover task `DG66` is still a hard prerequisite because this task depends on its cover URL and placeholder behavior.

## Desired State

From the books list, clicking a book opens the book info page for that selected book.

On desktop, the book info view uses a dedicated route at `/books/[bookId]`. The route should support refresh, sharing, and normal browser back behavior. The top bar should include a back button that returns to `/books`.

On mobile, clicking a book should still navigate to `/books/[bookId]` so page refresh lands back on the same selected book. The mobile rendering should use a full-screen popup-style base component above everything else, similar to the list experience in QuickList. The popup should include a top-bar back button that returns to `/books` and reveals the books list again.

The info view should show the book cover, title, author, tags, and full description. Opening the book info view should track a `book-info` event in the existing LingoCafe events stream.

## Definition of Success

Users can select a book from the LingoCafe catalog and inspect a complete book info surface at `/books/[bookId]`. Desktop shows the detail as main page navigation, while mobile shows the same route through a full-screen popup-style presentation that survives refresh and keeps back navigation predictable. Each successful book info view is recorded as a `book-info` LingoCafe event.

## Additional Context

The operator asked: "clicking on a book should show the info page. From the books list, clicking on a book should land on the book info page with cover, meta information and full description. UI: desktop: it's a main page navigation; mobile: should be a full-screen popup that go above everything else (like the list in quicklist); there should be the back button in the top bar to go back to the books list."

This should build on the catalog data returned by the `/books` work and the cover conventions from the cover task.

`PK51` completed the catalog work. Its execution notes established `/api/lingocafe/books` and `/api/lingocafe/profile` as the LingoCafe API paths, `api:lingocafe` as the API policy feature, and `src/app/(app)/(lingocafe)/books/page.tsx` as the frontend route location.

The operator clarified during refinement that desktop should use a dedicated route such as `/books/[bookId]`. Mobile should also update the URL to that route, so a refresh lands on the same selected book, but the mobile UI should keep the full-screen popup as the base detail component.

For the first implementation, book info metadata is limited to title, author, tags, description, and cover.

This task depends on `DG66: Display Book Covers`. `DG66` must be completed first so cover URL resolution, placeholder behavior, and the book-cover client rendering path are already established before the book info page is implemented.

The operator later added that opening the book info page should track the `book-info` event in the LingoCafe events system.

## Assumptions

The selected book data should come from the LingoCafe catalog API or a book-specific API endpoint, not from hardcoded UI-only fixtures.

The book model has, or will have, fields for title, author, tags, description, and cover. If one of those fields is missing, the implementation should expose the gap explicitly rather than inventing content.

The mobile popup-style route should preserve or quickly restore the books list context behind it so going back feels immediate.

## Constraints

Follow the existing app-page convention for authenticated routes: client component, `AppLayout`, and browser-side data fetching with `credentials: "same-origin"`.

Use server-side validation in any API route that returns book detail data.

Keep book-detail API work under the LingoCafe API namespace and policy shape established by `PK51`, rather than reviving the removed `/api/books` route.

Track `book-info` in `lingocafe.events` when a user opens a book info view. The event should use the authenticated user, the selected `book_id`, and the existing raw event stream rather than a separate analytics table.

Do not break the language-filtered books list flow.

Do not make mobile detail content a small modal. It should be full-screen and above the rest of the UI.

Keep the back button in the top bar for both desktop and mobile detail views.

Use `/books/[bookId]` as the canonical detail URL on both desktop and mobile.

Keep the first metadata scope limited to title, author, tags, description, and cover.

Implement this only after `DG66` is complete, so cover URL resolution and placeholder behavior are already available.

## Dos

- Do reuse the existing LingoCafe books route patterns.
- Do build on the completed `PK51` route and API structure.
- Do use `/books/[bookId]` so detail navigation can be shared, refreshed, and backed out of predictably.
- Do model the mobile overlay after the existing QuickList full-screen interaction pattern where practical.
- Do keep layout stable while cover and detail data load.
- Do keep the detail surface accessible with semantic buttons and focus handling.
- Do use the cover behavior established by `DG66`.
- Do treat `DG66` as a prerequisite, not just a related task.
- Do write the `book-info` event through the existing LingoCafe events table.

## Don'ts

- Don't turn the desktop view into a transient modal if it should behave like main navigation.
- Don't hide the back action in browser chrome only.
- Don't duplicate book metadata formatting in unrelated places if a small shared formatter or component already exists.
- Don't rely on client-supplied book data without validating the selected book server-side.
- Don't block or regress onboarding and empty-state behavior on `/books`.
- Don't expand the first metadata scope beyond title, author, tags, description, and cover unless the operator asks for it.
- Don't create a separate analytics storage path for `book-info`.
