# LingoCafe Data Model and Knex Migrations

## Elevator's Pitch

Discuss and design the data model for the LingoCafe reading app, then implement the proper Knex migrations so the database structure matches the agreed domain model.

## Business Gain

This gives LingoCafe a real persistence foundation instead of a fake reading list. A reading app without a data model is like Chuck Norris without a beard: technically possible, but suspicious.

## Current State

LingoCafe already exists as an app configuration and currently exposes a basic authenticated `/books` page with placeholder content, but there is no implemented LingoCafe-specific database schema yet for the reading domain.

## Desired State

The project has a clear LingoCafe domain model that covers the core reading-app entities and relationships, the design is documented well enough to guide implementation decisions, and the required Knex migrations exist so the schema can be created through the standard migration flow.

## Definition of Success

The LingoCafe reading domain has an explicit, reviewable data model, the chosen tables and constraints match the app's intended behavior, and the database schema can be created through Knex migrations without bypassing the repository's PostgreSQL-first patterns.

## Additional Context

The operator asked to discuss and design the data model for the LingoCafe reading app and to implement the proper Knex migrations.

The request uses the term `lingocafe datamodel`, which suggests the work should start from domain modeling instead of jumping straight into table creation.

There is already completed work for creating the LingoCafe app configuration in [HU36](../../completed/HU36-create-lingocafe-app-config/HU36.task.md).

The current LingoCafe surface appears to be a reading-oriented app, with `/books` used as the authenticated landing page.

The exact product scope of the reading app is still partially open. It is not yet explicit whether the first version needs only books and reading progress, or also lessons, vocabulary extraction, highlights, annotations, imports, shared libraries, or subscription features.

## Assumptions

LingoCafe should follow the repository's PostgreSQL-only and Knex-based database patterns.

The data model should likely support app isolation and authenticated users in a way that fits the existing multi-app architecture.

The first pass should focus on the minimum durable domain entities for a reading app, then leave optional or higher-risk features explicit rather than smuggling them into the schema.

If linguistic features such as saved words, sentence breakdowns, or reading annotations are important, they should be represented intentionally instead of being hidden inside JSON blobs unless there is a strong reason to do so.

## Constraints

Use Knex migrations under `knex/` and keep the implementation aligned with the repository's PostgreSQL-first approach. Look at existing migrations to take inspiration.

### PG Schema

### Tables

#### profiles

Per-user profile information that is relative to the LingoCafe reader app.

- `user_id` foreign key to `auth.users`
- `own_lang` mother language
- `target_lang` learning goal, used to filter the books

#### books

This is the inventory of existing books.
This is the dataset used to populate the `/books/:id?` view for listing the available readings, and displaying a particula book's details before starting on the first chapter (the description).

- `id` unique identifier (eg. `crusoe-sv-a2`) this value is set from the LingoCafe "books" repo project
- `lang` language in which the book is written (eg. en, it, de)
- `level` complexity level of the language (eg. a2, b1, b2)
- `title`
- `description`
- `tags`
- `cover`

#### books_pages

Detail table (`books` is master) that contains the content of a book.
This is the content that is sent out when reading a chapter.

- `id` string, ref to the original content from the source repository
- `book_id` FK books.id
- `position` integer, progressive number that regulates the ordering of the content item
- `kind` text, eg. part, chapter, other - future proof label on the data point
- `prefix` text, it's like "part 1" or "kapitel 1" or "del 1" - set by the source repo at export time
- `title` text
- `summary` text
- `content` text

PK: id, book_id

NOTE: the original books are organized in sections, in which each section has a title+summary. We can export the sections as standalone chapters without content so that they look like normal pages. If we proceed with this, we need to think how to display "Del 1" or "Kapitel 1", maybe an `is_section` flag in the chapter, or a plain `prefix` field in which to hardcode "Del 1 | Kapitel 1" at export time.

#### books_progress

Stores the user's reading progress for each opened book

- `user_id` FK auth.users.id
- `book_id` FK books.id
- `page_id` part of a composite FK with `book_id` towards books_pages(book_id, id)
- `progress` int (0-10000), it's the reading progress within the chapter, obtain and reflected to the scroll position of the text in the viewport
- `created_at` the record is first created at the "start reading" from the book's details page
- `updated_at` updated at change chapter or scroll event
- `completed_at` used for "mark as completed" feature

NOTE: unique(user_id, book_id)

NOTE ON EXPORT BOOKS:
The Management app is responsible for exporting a full and authoritative snapshot of each book toward the Reader database.

For a given `book_id`, the export flow should:

- upsert the `books` record
- upsert the current set of `books_pages` records for that book
- delete any existing `books_pages` row for that book whose `id` is not part of the newly exported page-id list

The deletion step is intentional. In this model, if a page disappears from the exported set, that means the old reading position is no longer valid for that book structure.

Because of that, `books_progress` should depend on `books_pages` through the `(book_id, page_id)` relation so that deleting a page also deletes any linked progress row automatically.

This keeps the database consistent during re-export and treats page-id changes as a structural content change rather than as a minor content edit.

#### analytics

This table collects users actions during a first period of time, before introducing more complex tools.

- `etag` timestamp
- `year` _etag breakdown, used for fast aggregations_
- `month` _etag breakdown, used for fast aggregations_
- `day` _etag breakdown, used for fast aggregations_
- `hour` _etag breakdown, used for fast aggregations_
- `user_id`
- `action_id` string, reference an application level library (eg. chapter, scroll, search)
- `action_data` jsonb, generic payload for the action

## Acceptance Criteria

- [ ] The LingoCafe reading domain is described in a concrete data model with the main entities and relationships called out.
- [ ] The design makes clear which parts are core MVP scope versus deferred or optional scope.
- [ ] The schema design aligns with existing project conventions for PostgreSQL and Knex.
- [ ] Proper Knex migrations are added for the agreed LingoCafe tables, keys, constraints, and indexes.
- [ ] The migration naming and structure are clear enough for future agents and humans to extend without reverse-engineering intent.
- [ ] Any important open tradeoffs or unresolved modeling decisions are documented instead of buried.

## Dos

- Do inspect the current LingoCafe routes, config, and any existing placeholder UI before finalizing the schema.
- Do model the reading domain from actual product behavior: books, reading state, user progress, and related entities as needed.
- Do use PostgreSQL features deliberately where they simplify correctness.
- Do document tradeoffs between a narrow MVP schema and a richer long-term content model.
- Do add indexes and uniqueness constraints where the domain clearly requires them.

## Don'ts

- Don't jump straight to migrations without first making the domain model explicit.
- Don't hide core relational structure inside generic JSON columns just because it's faster today.
- Don't create tables for speculative features unless the need is concrete enough to justify the maintenance cost.
- Don't bypass existing auth/user tables with a parallel identity system.
- Don't treat placeholder UI data as proof of the final product shape.
