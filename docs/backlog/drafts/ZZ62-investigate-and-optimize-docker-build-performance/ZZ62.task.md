# Investigate and Optimize Docker Build Performance

## Elevator's Pitch

The maker builder currently builds the solution as a Docker image, and each build run takes around five to ten minutes on a MacBook M2. This task investigates where the time is spent, identifies the most expensive build steps, and defines practical optimizations such as better Docker layer caching and CI-focused cache strategy.

## Business Gain

Faster Docker builds reduce developer wait time, shorten CI feedback loops, and lower compute costs. Faster iteration means more features shipped and fewer context switches.

## Current State

Docker image builds are perceived as too slow, roughly five to ten minutes per run on a MacBook M2. Root causes are not yet documented step-by-step, and no validated optimization strategy is recorded.

## Desired State

A clear build-time profile exists for local and CI build paths, with the slowest stages identified and ranked. A concrete optimization plan is defined and validated, including cache-layer improvements (especially in CI) and recommended Dockerfile or pipeline changes.

## Definition of Success

- Build stages and timings are measured and documented.
- Top bottlenecks are identified with evidence.
- At least one cache strategy for CI is proposed and validated.
- Recommended changes produce measurable build-time improvement over baseline.

## Additional Context

The current local execution environment is a MacBook with Apple M2 processor. The task should consider differences between local builds and CI runner architecture where relevant.

## Assumptions

- Docker build time can be reduced through caching and build-step optimization.
- Current build process likely includes one or more invalidated layers causing repeated heavy work.
- CI environment supports a cache mechanism that can be leveraged.

## Constraints

- Keep the existing functional behavior of the built image.
- Do not reduce security posture just to gain speed.
- Any optimization must be reproducible and suitable for CI pipelines.

## Acceptance Criteria

- A baseline build-time report is documented.
- Slowest Docker build steps are explicitly listed with timing data.
- Candidate optimizations are documented with expected impact.
- A CI cache-layer approach is documented and tested or clearly scoped for implementation.
- Final recommendation includes next implementation steps.

## Dos

- Measure first, then optimize.
- Compare no-cache vs cached builds.
- Separate local-machine effects from CI-runner effects.
- Keep findings evidence-based.

## Don'ts

- Do not rely on guesswork without timing data.
- Do not optimize by removing required build or security checks.
- Do not propose changes that only work on one machine setup unless clearly marked.

## Open Questions

- Which exact Dockerfile stages are dominating the build time?
- How much of the delay comes from dependency install, app build, or image export?
- What CI provider and cache backend are currently used?
- Should multi-arch builds be part of this investigation scope?
