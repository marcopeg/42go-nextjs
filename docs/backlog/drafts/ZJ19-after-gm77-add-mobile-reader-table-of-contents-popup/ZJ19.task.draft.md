# After GM77, Add Mobile Reader Table of Contents Popup

## Elevator's Pitch

After `GM77` introduces the core reusable popup/modal component, use it to add a real Table of Contents popup to the top bar of the mobile reading experience in LingoCafe.

## Business Gain

The mobile reader gets a real navigation affordance instead of a placeholder. Readers can jump through the book structure from the top bar without custom one-off UI, and the new core overlay primitive from `GM77` gets a concrete production use case. Chuck Norris does not scroll blind through chapters. Chapters line up and salute.

## Current State

`VU18` explicitly kept the mobile reader top-right controls as placeholders for first release. Its notes say the mobile bottom toolbar was removed and controls such as `Innehåll` were moved into the top-right menu placeholder.

The current mobile reading experience already has a top bar and a placeholder entry for `Innehåll`, but it does not open a real Table of Contents popup yet.

The reader payload already contains ordered page summaries, and the desktop reader surface already uses that data to render a contents list in its side panel.

There is not yet a shared core popup/modal component for this flow. `GM77` is the task that should produce that reusable primitive.

## Desired State

Once `GM77` is completed, the mobile reading top bar should expose a real Table of Contents popup built on top of that core popup/modal component.

The popup should:

- be opened from the mobile reader top bar
- use the shared core popup/modal primitive from `GM77`
- present the book contents in a mobile-friendly overlay surface
- reuse the ordered page summary data already available in the reader flow
- highlight the current page clearly
- fit the mobile reading experience instead of behaving like a desktop side panel squeezed into a phone

The task is specifically about the mobile reading experience. The goal is to replace the current placeholder `Innehåll` behavior with a real reusable popup-driven interaction.

Whether the popup should open from the existing `...` menu path or from a more direct dedicated top-bar affordance can be finalized during refinement, but the final implementation should still live in the mobile top bar experience and use the `GM77` primitive.

## Definition of Success

After `GM77`, the mobile reader exposes a working Table of Contents popup through the top bar using the shared popup/modal component instead of bespoke overlay code or a dead placeholder. The popup feels native to the mobile reader, shows the current page state clearly, and proves the new shared popup can power real reader UX.

## Additional Context

`VU18` already established two important pieces:

- mobile secondary controls belong in the top-right menu or top-bar experience
- the reader payload already includes the ordered page summaries needed for contents navigation

The desktop reader already has a contents list in its left panel, so this task should not invent a second source of truth for book contents. It should reuse the same underlying reader data and adapt the presentation for mobile through the shared popup primitive from `GM77`.

This task is intentionally sequenced after `GM77` so the implementation does not slip back into handcrafted popup logic.

## Assumptions

- `GM77` is the blocking dependency because this task must use the shared popup/modal component created there.
- The existing mobile reader top bar is the right place to trigger this interaction.
- The current reader API payload already has enough page-summary data for a first useful Table of Contents popup.
- The desktop contents list can inform the data model and selection behavior, but the mobile UI should be designed specifically for mobile.

## Constraints

- Do not implement this before `GM77` is completed enough to provide the shared popup/modal primitive.
- Use the shared popup/modal component from `GM77` instead of building another custom overlay.
- Keep the scope focused on the mobile reading experience.
- Reuse the existing reader payload and contents data where possible.
- Do not turn this task into a broader rewrite of the desktop reader side panel.
- Preserve the existing stable reader route and current reading experience outside the new contents popup.

## Acceptance Criteria

- [ ] The task is implemented only after `GM77` provides the shared popup/modal primitive.
- [ ] The mobile reader top bar exposes a real Table of Contents popup.
- [ ] The popup uses the shared popup/modal component from `GM77`.
- [ ] The popup replaces the current placeholder `Innehåll` behavior in the mobile reading experience.
- [ ] The popup reuses the existing ordered page-summary data from the reader flow.
- [ ] The current page is visually highlighted in the popup.
- [ ] The popup is designed for mobile and does not feel like a desktop side panel transplanted into a phone.
- [ ] The implementation does not add bespoke overlay/backdrop/escape/animation logic that duplicates the shared primitive from `GM77`.
- [ ] The existing mobile reader route and reading behavior continue to work after the change.
- [ ] At least one concrete interaction path in the mobile top bar opens the popup predictably.

## Dos

- Do treat `GM77` as the dependency and source of the popup/modal primitive.
- Do reuse the reader's existing contents data.
- Do keep the interaction anchored to the mobile top bar experience.
- Do make the current page obvious inside the Table of Contents.
- Do use this task as a real proving ground for the shared popup in a reader workflow.

## Don'ts

- Don't build another custom popup just because it feels faster.
- Don't fork a separate contents data source for mobile.
- Don't quietly expand the task into desktop reader redesign work.
- Don't leave the `Innehåll` affordance as a placeholder after this task is complete.
- Don't regress the current mobile reading flow to squeeze in the new popup.

## Open Questions

- Should the Table of Contents open from the existing `...` menu flow or from a dedicated visible top-bar button?
- Should tapping a contents row immediately navigate to that page and close the popup, or should the popup support a different confirmation pattern?
- Should the first version include only contents, or also leave room for the old placeholder ideas such as bookmarks?
- How much of the desktop contents visual language should carry over to mobile?
