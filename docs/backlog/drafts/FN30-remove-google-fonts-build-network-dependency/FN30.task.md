# Remove Google Fonts Build Network Dependency

## Elevator's Pitch

Investigate why the Next.js production build tries to contact Google Fonts, then make the build reliable in Codex sandbox environments where outbound network access may be limited or unavailable.

## Business Gain

Reliable offline-capable builds reduce broken automation, unblock local and remote Codex execution, and make the project easier to build in constrained CI-style environments. The build should hit like a roundhouse kick: fast, direct, and not asking Google for permission.

## Current State

The current failure appears related to Google Fonts fetching during `npm run build`. This is a Next.js app, and the build must work inside Codex sandbox environments after dependencies are installed. The exact source of the Google Fonts access has not yet been confirmed.

## Desired State

`npm install` and `npm run build` pass reliably without requiring outbound network access to Google Fonts after dependencies are installed. The app keeps visually acceptable typography locally and in production.

## Definition of Success

- The app is inspected for `next/font/google`, Google Fonts `<link>` tags, CSS `@import` rules, custom document/head/layout usage, global CSS usage, MUI theme usage, metadata configuration, or package-driven font loading.
- The exact file or package causing the Google Fonts access is identified.
- The cause is classified as `next/font/google`, CSS import, framework optimization, or another dependency path.
- Two or three safe fixes are documented with pros and cons.
- The recommended fix is implemented with the smallest safe code change.
- Documentation explains the font strategy and why it is compatible with Codex/sandbox builds.
- `npm install` and `npm run build` are run and their exact results are reported.
- If build still fails, the next external dependency or build blocker is identified.

## Additional Context

Candidate fixes to evaluate:

- Self-host fonts locally with `next/font/local`.
- Replace `next/font/google` with runtime `<link>` tags.
- Use a system font fallback during CI/Codex builds.
- Vendor the required font files into the repository.

The goal is not to redesign typography. The goal is to remove build-time external network dependency while keeping the app visually acceptable.

## Assumptions

- The Google Fonts request likely comes from project code or a framework-level font optimization triggered by project code.
- Installed npm dependencies should be the only required external input before `npm run build`.
- Production should remain visually acceptable even if the chosen fix avoids Google-hosted fonts at build time.

## Constraints

- The build must work in local and remote Codex sandboxes with limited or no outbound network access.
- Do not depend on external network access after dependencies are installed.
- Keep the change small and safe.
- Preserve existing Next.js App Router conventions and repository coding standards.
- Run `npm install` and `npm run build` for verification.

## Acceptance Criteria

- [ ] Google Fonts usage is fully inspected across app code, layout/head files, CSS, and relevant configuration.
- [ ] The build-time Google Fonts access path is explained with file names.
- [ ] The selected fix removes build-time Google Fonts access.
- [ ] The app has an acceptable local and production font fallback or local hosted font.
- [ ] Documentation records why fonts are handled this way for Codex/sandbox compatibility.
- [ ] `npm install` passes, or its failure is recorded with exact cause.
- [ ] `npm run build` passes, or its failure is recorded with exact cause and next blocker.

## Dos

- Search for `next/font/google`, `fonts.googleapis.com`, `fonts.gstatic.com`, and CSS `@import` font rules.
- Inspect custom document/head/layout files, global CSS, theme config, metadata config, and any UI framework theme setup.
- Prefer local or system-font solutions that keep builds deterministic.
- Document the selected tradeoff clearly.
- Report exact verification results.

## Don'ts

- Do not leave `next/font/google` in the build path if it forces build-time network access.
- Do not make the build depend on runtime-only external font fetches unless explicitly justified as a temporary compromise.
- Do not widen the task into unrelated visual redesign.
- Do not change dependency versions unless the investigation proves it is necessary.

## Open Questions

- Which exact file or package currently triggers the Google Fonts request?
- Which font family is currently intended for the app?
- Is preserving the exact same font required, or is a high-quality system stack acceptable?
- Should this project vendor font files in the repo, or prefer system fonts to keep the asset footprint minimal?
