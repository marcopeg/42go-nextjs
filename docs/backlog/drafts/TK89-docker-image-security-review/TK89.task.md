# Docker Image Security Review


## Elevator's Pitch

Perform a comprehensive security audit of the Docker image to ensure it can safely be published and inspected by third parties without leaking secrets, private content, or unnecessary attack surface.

## Business Gain

Confidence that the production Docker image is safe to distribute publicly. A clear, documented security posture for the image—covering secrets, content, dependencies, runtime hardening, and registry hygiene—so the team can ship without second-guessing what's inside.

## Current State

The project has a Dockerfile with multi-stage builds and a `docker-compose.prod.yml` for production. There is a `check-docker-security.sh` script in `scripts/`. There is an existing task (AEM) addressing Docker build issues with middleware. No formal security audit of the image contents, layers, or runtime configuration has been performed or documented.

## Desired State

A completed security review covering all twelve checklist areas:

1. Build context hygiene (`.dockerignore`, excluded files)
2. Dockerfile copy strategy (multi-stage, minimal final image)
3. Secret handling (no secrets baked into layers)
4. Content exposure (no private content in the image)
5. Dependency exposure and patch state (vulnerability scan, base image policy)
6. Runtime user and file permissions (non-root, read-only where practical)
7. Network and service boundaries (no internal services exposed)
8. Shared image blast radius (multi-app risk assessment)
9. Auth and session material (no credentials in bundles)
10. Analytics and consent boundaries (no premature tracking)
11. Registry and publishing hygiene (tagging, signing, provenance)
12. Final image inspection (filesystem search for secrets and sensitive files)

Each area results in a pass/issues-found/blocked verdict. Issues found produce actionable remediation items.

## Definition of Success

- Every checklist item has been evaluated with evidence collected.
- A review outcome summary is produced using the provided template (pass / issues found / blocked per area).
- All critical issues (secrets in layers, private content exposure, running as root without justification) are either resolved or tracked as separate tasks.
- A documented rule exists for what content may enter the final image.
- A documented secret injection policy exists.
- A decision is recorded on whether the image should be public or private.

## Additional Context

The operator provided a detailed review document with twelve checklist sections, suggested audit commands, a review outcome template, and a release gate definition. The full document should be used as the working checklist during execution.

Key audit commands to use:

```bash
docker history <image-name>
docker run --rm -it --entrypoint sh <image-name>
docker run --rm --entrypoint sh <image-name> -lc "find / -iname '*.env' -o -iname '*secret*' -o -iname '*key*' -o -iname '*dump*' 2>/dev/null"
trivy image <image-name>
```

Existing related files:

- `Dockerfile` — the build definition
- `docker-compose.prod.yml` — production compose config
- `.dockerignore` — build context exclusions
- `scripts/check-docker-security.sh` — existing security check script

## Assumptions

- The image may be published to a public registry; the review assumes worst-case (anyone can pull and inspect).
- Secrets are expected to be injected at runtime via environment variables, not baked into the image.
- The multi-app architecture means one image serves multiple sites; this is a deliberate platform decision but needs blast-radius assessment.

## Constraints

- The review does not replace application security review, code review for auth/authz, infrastructure security review, or legal review.
- The review focuses strictly on the Docker image artifact and its build pipeline.

## Acceptance Criteria

- [ ] `.dockerignore` reviewed and confirmed to exclude `.env*`, secrets, backups, dumps, logs, editor folders, temp exports
- [ ] Dockerfile copy strategy reviewed; final stage contains only runtime necessities
- [ ] No secrets present in any image layer (verified by inspection)
- [ ] No private or unintended content in the final image filesystem
- [ ] Vulnerability scan executed on the built image
- [ ] Container confirmed to run as non-root (or root usage justified)
- [ ] Shared image blast radius documented and accepted
- [ ] Registry and tagging policy documented
- [ ] Final image filesystem inspected for `.env`, `key`, `secret`, `token`, `backup`, `dump`
- [ ] Review outcome summary produced with per-area verdicts
- [ ] Release gate criteria evaluated (pass or documented exceptions)

## Dos

- Use the twelve-section checklist from the operator's review document as the working guide
- Collect evidence for each checklist item (file contents, command outputs, scan results)
- Produce actionable remediation items for any issues found
- File separate tasks for remediation work that falls outside the scope of a review

## Don'ts

- Do not conflate this with application security review or code review
- Do not make changes to the Dockerfile or build pipeline during the review—only document findings
- Do not assume the build is correct without inspecting the final artifact directly

## Open Questions

- Is `trivy` available in the local environment, or should an alternative scanner be used?
- Should the review also cover CI pipeline configuration, or only the local build?
- What is the target registry (GHCR, Docker Hub, private)?
- Is the `scripts/check-docker-security.sh` script current and covering all areas, or is it outdated?
