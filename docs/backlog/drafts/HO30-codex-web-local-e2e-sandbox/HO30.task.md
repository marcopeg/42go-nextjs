# Codex Web Local E2E Sandbox

## Elevator's Pitch

Make the project runnable inside a Codex Web sandbox with no external Postgres dependency, so delegated Codex tasks can run real end-to-end tests, log in with seeded users, navigate the app, and capture browser screenshots.

## Business Gain

Delegated implementation work becomes more autonomous and more trustworthy. Codex Web can validate behavior before handing work back, instead of stopping at static checks because the database is unreachable. That means fewer blind changes, fewer manual smoke-test loops, and faster iteration on authenticated multi-app flows.

## Current State

The project is PostgreSQL-only. Runtime DB access goes through Knex and `pg`, with configuration built from `DATABASE_URL`. The Knex migration stack uses PostgreSQL-specific features including schemas, `jsonb`, arrays, `uuid-ossp`, PL/pgSQL, and partitioned tables.

Local development currently expects a Postgres server. That works on a developer machine with Docker or a local Postgres service, but it is a weak fit for Codex Web delegation when the sandbox cannot depend on an external database server.

SQLite is not a clean fit. It would require either a parallel schema or substantial migration/query compatibility work. That is a bar fight without a prize.

The likely file-backed Postgres option is PGlite: PostgreSQL compiled to WASM, able to run in Node and persist to the filesystem. PGlite Socket is especially relevant because it exposes a Postgres wire-compatible server that existing `pg` and Knex code can connect to through a normal `DATABASE_URL`.

## Desired State

The repository provides a documented local e2e test harness that can run inside Codex Web with a file-backed or in-memory Postgres-compatible database. The harness should boot the database, run the existing Knex migrations and seeds, start the Next.js app, execute e2e tests, and support browser automation against the resulting app.

The first implementation path should spike PGlite Socket because it preserves the current Knex/`pg` shape. If the stock socket CLI cannot load required extensions or migration features, the next option is a project-local PGlite launcher that instantiates PGlite with needed extensions such as `uuid_ossp`, starts a socket server, and runs the e2e command with the generated `DATABASE_URL`.

## Definition of Success

Codex Web can clone or open the repository, run one documented command, and get a working app backed by an isolated local database. The command should prove the system by running migrations, seeding known credentials users, logging in through the UI, navigating at least one authenticated flow, and producing at least one screenshot artifact from the running app.

## Additional Context

Investigation notes:

- PGlite is the closest match to "file system Postgres database."
- PGlite can run in Node and persist to the filesystem.
- PGlite Socket gives `pg` clients a server interface, which matches the current Knex setup better than switching drivers.
- The current `knexfile.js` rejects non-Postgres connection strings, which is compatible with a socket-based PGlite approach because it can still provide a `postgres://...` URL.
- Current migrations use `CREATE EXTENSION IF NOT EXISTS "uuid-ossp"`, `CREATE SCHEMA`, `jsonb`, `text[]`, PL/pgSQL functions, and range partitions. Any candidate database must be tested against the full migration chain, not only against simple queries.
- PGlite is expected to be good enough for autonomous e2e validation, but real Postgres should remain the production and concurrency truth.

Useful references:

- PGlite overview: https://pglite.dev/docs/about
- PGlite filesystem support: https://pglite.dev/docs/filesystems
- PGlite ORM and Knex support: https://pglite.dev/docs/orm-support
- PGlite Socket: https://pglite.dev/docs/pglite-socket
- PGlite extensions: https://pglite.dev/extensions/

## Assumptions

Codex Web can run Node commands, install project dependencies, bind a localhost port, and execute a browser automation tool against localhost.

The initial e2e scope can use seeded credentials-provider users. OAuth provider flows are not required for the first pass because they depend on external provider credentials and network behavior.

The app should continue to use PostgreSQL semantics. This task is about local autonomy for testing, not replacing production Postgres.

## Constraints

Do not introduce SQLite as the primary path unless the PGlite spike proves impossible. The existing migrations and runtime queries are Postgres-native.

Do not rewrite the ORM layer. The current ORM/data access tool is Knex with `pg`, and the preferred solution should preserve that contract.

Do not require Docker or an externally managed Postgres server for the Codex Web e2e command.

Do not weaken production database behavior to satisfy the test harness.

Keep the setup deterministic. Test database state should be disposable or resettable between runs.

## Acceptance Criteria

- A documented command exists for Codex Web/local sandbox e2e execution.
- The command starts an isolated Postgres-compatible database without requiring external Postgres.
- Existing Knex migrations run successfully against the sandbox database.
- Existing seeds, including credential test users, run successfully against the sandbox database.
- The Next.js app can start against the sandbox `DATABASE_URL`.
- An e2e smoke test logs in with a seeded credentials user.
- The e2e smoke test navigates at least one authenticated route.
- The e2e smoke test captures at least one screenshot artifact.
- Failure output is useful enough for a delegated Codex task to diagnose migration, app boot, login, or navigation failures.
- Documentation explains when to use the sandbox database versus real Postgres.

## Dos

- Start with a PGlite Socket spike.
- Verify the full migration chain before building broader e2e flow.
- Check `uuid-ossp` support explicitly.
- Check PL/pgSQL function creation and partitioned table DDL explicitly.
- Prefer a project-local launcher if extension loading or lifecycle control needs more grip than the CLI provides.
- Keep `DATABASE_URL` as the runtime interface for the app and Knex.
- Add the smallest e2e smoke flow that proves login, app navigation, and screenshots.
- Make cleanup/reset behavior obvious.

## Don'ts

- Do not convert the application to SQLite.
- Do not create a second migration system just for tests.
- Do not skip migrations by loading hand-written schema snapshots unless explicitly chosen as a later optimization.
- Do not rely on an already-running Postgres server.
- Do not require manual database setup for Codex Web.
- Do not make OAuth login part of the first autonomous e2e gate.

## Open Questions

- Can stock PGlite Socket load `uuid-ossp` and satisfy the current migrations as-is?
- Does PGlite support the current partitioned `lingocafe.events` migration well enough for e2e use?
- Should the sandbox database default to in-memory for speed or filesystem persistence for easier post-failure inspection?
- Which e2e runner should be standardized for Codex Web: Playwright, browser-use workflow, or a thin project script around one of them?
- Should `npm run qa` eventually include this e2e smoke gate, or should it remain a separate heavier command?
