# Reduce Reader Scroll Progress Save Delay From 10 Seconds

## Elevator's Pitch

Reduce the delay before LingoCafe saves the reader scroll position so progress is persisted faster than the current 10-second wait, with the likely target being 5 seconds or possibly 3 seconds.

## Business Gain

Saved reading position feels more trustworthy because the app remembers where the reader stopped with less lag. Chuck Norris does not reread the same paragraph because a timer was taking a nap.

## Current State

The LingoCafe reader route at `src/app/(app)/(lingocafe)/books/[bookId]/[pageId]/page.tsx` currently batches scroll persistence with a `setTimeout(..., 10000)`. Each new scroll event resets that timer, so the app waits up to 10 seconds after scrolling settles before posting the latest `progress_bps` value to the existing page API. The route still performs a final keepalive save on teardown.

## Desired State

Reader progress should be saved sooner after scrolling activity stops. The current 10-second delay should be reduced to a shorter value, likely 5 seconds or possibly 3 seconds, while preserving the existing low-volume event model, the final teardown save, and the current `progress_bps` persistence path.

## Definition of Success

A reader who scrolls and then pauses has their latest reading position persisted noticeably sooner than today, without turning raw scroll movement into a flood of `page-scroll` writes or breaking resume behavior.

## Additional Context

Earlier reader work established the current behavior. `ZE11` introduced scroll progress reporting with a 10-second collection window and a final keepalive save, and `XI31` later refined resume behavior around saved progress. The current code path still lives in the reader page component and posts to the existing `/api/lingocafe/books/[bookId]/pages/[pageId]` endpoint using `progress_bps`.

## Assumptions

- The existing authenticated page API for `page-scroll` events and `books_progress` updates should remain the persistence path.
- The final target delay is not decided yet and should stay open between 5 seconds and 3 seconds until refinement.
- The final keepalive save on unmount/navigation should remain in place unless implementation proves it redundant.
- Resume behavior from saved `progress_bps` should continue to work exactly as it does today.

## Constraints

- Keep the task scoped to reader scroll-save timing, not a broader reader redesign.
- Do not remove validation or clamping for `progress_bps`.
- Do not change the existing event type or persistence endpoint just to shorten the delay.
- Avoid increasing write volume so much that the reader starts emitting near-per-scroll network requests.
- Preserve existing resume and restore behavior for `/books/[bookId]/[pageId]`.

## Acceptance Criteria

- [ ] The delay before persisting the last observed reader scroll position is reduced from the current 10 seconds.
- [ ] The final target delay is finalized during refinement or implementation between the currently suggested values of 5 seconds and 3 seconds.
- [ ] Reader scroll persistence still posts the last observed `progress_bps` value through the existing authenticated API path.
- [ ] Continuous scrolling still avoids flooding the app with raw per-scroll writes.
- [ ] The final keepalive save on teardown/unmount still preserves the latest known position when applicable.
- [ ] Existing resume behavior continues to reopen the reader near the latest saved position.
- [ ] The change does not break mobile or desktop reader scroll tracking.

## Dos

- Do reuse the existing reader progress persistence flow.
- Do keep the change focused on the timing threshold.
- Do verify both desktop and mobile reader surfaces because both use the same reader progress mechanism.
- Do preserve the low-volume intent of the `page-scroll` event model.

## Don'ts

- Don't turn scroll tracking into a network request on every browser scroll event.
- Don't change the `progress_bps` contract or API shape for this task.
- Don't remove the final teardown save without proving the replacement is equally safe.
- Don't broaden the task into unrelated reader navigation, loading, or layout changes.

## Open Questions

- Should the new delay be `5s` or `3s`?
- Should the shorter delay apply only to the reader route for now, or is there any other progress-saving reader surface that should stay aligned?
