# Book Page Read UI

## Elevator's Pitch

Implement the LingoCafe route and UI for reading one page from a book, using a URL that can survive full page reloads and restore the reader's vertical position.

## Business Gain

Readers need the "Read now" and "Continue reading" actions to land on a real reading surface. A saved position should reopen where the reader left off. Chuck Norris does not scroll back to his place. The page scrolls to him.

## Current State

`IL68` defines the book details route and base URL shape for book-specific navigation.

`PR58` defines the book details primary action and says resume links should carry `progress_bps` as a URL parameter from `books_progress`. It explicitly does not create the read page UI.

The requested gap is the page reading route and presentation layer.

## Desired State

The app supports a route for reading a specific page from a specific book. The route should extend the base URL shape from `IL68` by adding the page ID, so full page reloads reopen the same book page.

When the URL includes the progress indicator from `PR58`, the read UI should restore the reader's initial vertical scroll position using the `progress_bps` value. The value is a basis-point percentage from `0` to `10000`.

Opening a page should record a `page-open` event. This can be populated server-side by the page content API when it returns the requested page.

Scrolling the page should record `page-scroll` events from the client. Scroll events should be collected over a 10-second window and only the last scroll position in that window should be sent, so continuous scrolling does not flood the event stream.

On desktop, the read page should feel similar to the book detail surface and include a back link to the book detail.

On mobile, the read page should render as a full-screen popup above the book details popup. Going back from the read page should slide this top popup away and reveal the book details underneath.

## Definition of Success

Users can open a stable read-page URL for a book page, refresh it without losing the selected page, and optionally resume at the vertical position encoded in the URL. Desktop reads as an internal page with a back link to book details. Mobile layers the read popup above book details so back navigation reveals the details popup underneath.

The read flow also records `page-open` and low-volume `page-scroll` activity into the LingoCafe event stream, with scroll reporting reduced to the last position observed within each 10-second collection window.

## Assumptions

The read URL should build on `/books/[bookId]` and add a page identifier beneath it.

The `progress_bps` URL value should be treated as the requested initial scroll position, clamped to the valid `0` to `10000` range.

The page content should come from the LingoCafe book page data, not from the book details payload alone.

## Constraints

This task is blocked by `PR58`.

Follow the authenticated app-page convention: client component, `AppLayout`, and browser-side data fetching with `credentials: "same-origin"`.

Use server-side validation for any API route that returns book page content.

The read route must include both the book ID and page ID so refresh and direct navigation are stable.

Do not trust client-supplied progress values without validation and clamping.

Do not make mobile read UI a small modal. It should be a full-screen popup above the book details popup.

Populate `page-open` from the server-side page content API.

Populate `page-scroll` from the client and debounce or batch it so only the last scroll position observed during a 10-second window is sent.

- Don't ignore the `progress_bps` resume parameter.
- Don't emit a `page-scroll` event for every raw browser scroll event.
- Don't use a small dialog for the mobile read UI.
- Don't collapse the mobile flow directly back to the books list when backing out of the read popup.
