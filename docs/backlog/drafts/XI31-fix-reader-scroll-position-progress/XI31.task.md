# Fix Reader Scroll Position Progress

## Elevator's Pitch

Fix LingoCafe reader progress so a book page stores progress from the actual scroll position, records scrolling events, and restores the saved scroll position when the reader reopens the last read page on both mobile and desktop.

## Business Gain

Readers should return to the exact place where they stopped reading. If desktop forgets the scroll position while mobile remembers it, the app feels unreliable. Chuck Norris does not reread the same paragraph twice. The paragraph rereads itself.

## Current State

The LingoCafe book page reader already has completed work for `/books/[bookId]/[pageId]`, `page-open`, `page-scroll`, and `lingocafe.books_progress` persistence.

The operator noticed that scroll-position progress seems to work locally in the mobile version, but not in the desktop version.

The current reader implementation has separate desktop and mobile scroll refs. The desktop surface uses a fixed full-screen layout with its own `overflow-y-auto` scroll container, while mobile uses a separate full-screen overlay scroll container.

The read page currently restores only from the `progress_bps` URL parameter. The page data payload does not include the saved `books_progress.progress_bps` value for the current user.

The page-content API currently records `page-open` and immediately saves `progress_bps = 0` for the current book and page. That may overwrite an existing non-zero saved progress before the client has a chance to restore or report the saved scroll position.

The operator clarified that `page-open` should always create the `page-open` event, but progress persistence should be conditional. If no previous `books_progress` record exists for that book, opening the page may create progress for the opened page with `progress_bps = 0`. If a progress record exists for the same book and same page, opening the page again should not update progress. If the progress record exists for the same book but points to a different page, opening the new page may update progress to that new page with `progress_bps = 0`.

## Desired State

When reading a book page, the reader uses the current vertical scroll position to calculate progress.

Scrolling writes `page-scroll` events and updates saved progress for the authenticated user, current book, and current page.

When reopening the last read page, the reader restores the saved scroll position on both mobile and desktop, even when the saved position comes from persisted progress rather than only from a `progress_bps` URL parameter.

Opening a page should not destroy a valid non-zero saved progress value before restore can happen.

Opening a different page in the same book should update the saved page pointer to the newly opened page with `progress_bps = 0`.

## Definition of Success

The LingoCafe reader consistently persists and restores scroll-position progress on both mobile and desktop reader layouts.

Leaving a page after scrolling and then reopening the last read page returns the reader to the saved vertical position instead of the top of the page.

`page-scroll` events continue to be written through the existing reader event model without flooding the event stream.

## Additional Context

The operator asked: "fix scroll position progress. When reading a book's page, we should use the scroll position to store the progress and write scrolling events. When re-opening the last read page, the scroll should restore to the saved position. I noticed that this seems to be working locally when using the mobile version, but not in the desktop. It should work in both mobile and desktop version."

`ZE11` implemented the read page UI and explicitly included restoring vertical scroll from `progress_bps`, recording `page-scroll` events, and updating `lingocafe.books_progress`.

`ZE11` execution notes mention a mobile read overlay above the book detail overlay and a 10-second client scroll collection window with a keepalive final save for teardown.

Source inspection during refinement found the current reader page in `src/app/(app)/(lingocafe)/books/[bookId]/[pageId]/page.tsx`, the reader surfaces in `src/app/(app)/(lingocafe)/books/_components/BookReaderSurfaces.tsx`, and the page API in `src/app/api/(lingocafe)/lingocafe/books/[bookId]/pages/[pageId]/route.ts`.

`PR58` established that "Continue reading" links include `progress_bps` as a URL parameter when the book detail API has saved progress.

`OG12` established that `books_progress.progress_bps` is the scroll position within the chapter, represented as an integer from `0` to `10000`.

## Assumptions

The intended progress value remains `progress_bps`, a basis-point vertical scroll percentage from `0` to `10000`.

The existing `/api/lingocafe/books/[bookId]/pages/[pageId]` API remains the intended endpoint for reading page data and recording scroll progress unless implementation discovers a cleaner existing endpoint.

The bug is likely caused by one or more of these implementation details: desktop and mobile use different scroll containers, restore timing depends on the currently active viewport, saved progress is only passed by URL, or `page-open` resets saved progress to zero.

## Constraints

Keep the existing read URL shape: `/books/[bookId]/[pageId]`.

Do not introduce a new progress storage model if `lingocafe.books_progress` already supports the required behavior.

Do not trust client-supplied user IDs.

Clamp scroll progress to the valid `0` to `10000` range.

Preserve the existing low-volume scroll event behavior instead of sending one event for every browser scroll callback.

Do not let `page-open` erase a valid non-zero saved scroll position when reopening a page.

Allow `page-open` to create progress when no record exists for the book, or move progress to a newly opened page with `progress_bps = 0` when the saved record points to a different page.

## Acceptance Criteria

- [ ] Desktop reader scroll position is used to calculate `progress_bps`.
- [ ] Mobile reader scroll position is used to calculate `progress_bps`.
- [ ] Scrolling a book page records `page-scroll` events through the existing LingoCafe event stream.
- [ ] Scrolling a book page updates `lingocafe.books_progress` for the authenticated user, current book, and current page.
- [ ] Reopening the last read page restores the saved scroll position on desktop.
- [ ] Reopening the last read page restores the saved scroll position on mobile.
- [ ] Restore works when progress comes from saved `books_progress`, not only from an explicit URL query parameter.
- [ ] Restore runs after the relevant reader content and scroll container are available.
- [ ] Opening a page does not overwrite existing non-zero progress before the reader can restore it.
- [ ] Opening a page still records a `page-open` event.
- [ ] Opening a page creates `books_progress` at `progress_bps = 0` when no progress record exists for the authenticated user and book.
- [ ] Opening a page does not update `books_progress` when the existing record already points to the same book and same page.
- [ ] Opening a different page in the same book updates `books_progress.page_id` to the newly opened page and sets `progress_bps = 0`.
- [ ] If both URL progress and persisted progress are available, the implementation uses the documented precedence.
- [ ] Continuous scrolling remains throttled or batched so the app does not flood `page-scroll` events.
- [ ] Invalid progress values are safely ignored or clamped.
- [ ] The behavior is covered by a focused manual smoke check for both desktop and mobile layouts.
- [ ] `npm run qa` passes after implementation.

## Dos

- Do inspect the reader scroll container separately for desktop and mobile.
- Do use the actual scrollable element when calculating progress.
- Do restore scroll position against the same element used for progress calculation.
- Do keep `page-scroll` event payloads aligned with the existing reader event model.
- Do use authenticated server-side ownership for progress persistence.
- Do verify the desktop reader specifically, because that is the reported failure mode.
- Do check whether `page-open` should update only the page pointer or also reset `progress_bps`.
- Do preserve the `page-open` event even when progress persistence is skipped for an already-saved page.

## Don'ts

- Don't rely only on `window.scrollY` if the reader content scrolls inside a nested container.
- Don't make desktop and mobile progress semantics diverge.
- Don't write scroll events on every raw scroll event.
- Don't reset saved progress to the top when reopening a page that has saved progress.
- Don't depend only on the book detail "Continue reading" link to provide resume progress if the read page can load the saved progress directly.
- Don't skip the `page-open` event just because the progress row does not need to change.
- Don't solve unrelated reader layout or book detail design issues in this task.

## Open Questions

- Is the desktop failure caused by progress calculation, event submission, saved progress retrieval, scroll restoration, or a mismatch between scroll containers?
- Should the reader prefer an explicit `progress_bps` query parameter over saved progress when both are present?
- Q: Should `page-open` continue to write `progress_bps = 0`, or should it avoid overwriting non-zero saved progress when the user is reopening the same page?
  A: `page-open` should always create the event. It should create progress at `0` only when no progress record exists for that authenticated user and book. It should not update progress when the existing record points to the same book and page. It may update progress to the newly opened page with `progress_bps = 0` when the existing record points to a different page in the same book.
- Should direct navigation to `/books/[bookId]/[pageId]` without `progress_bps` restore saved progress for that same page when it exists?
- Q: Is a focused manual smoke check enough, or should implementation add an automated test?
  A: The operator will manually test this behavior.
