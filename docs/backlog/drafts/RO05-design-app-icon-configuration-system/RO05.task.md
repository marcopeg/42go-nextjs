# Design App Icon Configuration System

## Elevator's Pitch

Design and implement a shared 42go icon configuration system so every app can configure favicons, home-screen icons, PWA icons, and main app logos without hardcoded one-off behavior.

## Business Gain

Every app in the multi-app platform can look like its own product in browser tabs, installed app launchers, public layout chrome, and protected app navigation. Chuck Norris does not let one favicon rule the whole town.

## Current State

The platform already has partial icon support:

- `src/AppConfig.ts` has a top-level `logo?: string | ComponentType` field.
- `src/42go/layouts/public/AppTitle.tsx` and `src/42go/layouts/app/AppTitle.tsx` render `toolbarConfig?.icon || config.logo`.
- `config.public.pwa.icons.appleTouch180` feeds `src/42go/pwa/HeadTags.tsx`.
- `config.public.pwa.icons.manifest192`, `manifest512`, and `maskable512` feed `src/app/manifest.ts`.

The platform does not appear to have a dedicated favicon configuration field.

Official Next.js App Router guidance prefers metadata file conventions such as `app/favicon.ico`, `app/icon.png`, and `app/apple-icon.png` for static metadata files. That works cleanly for a single static app identity, but this repository resolves app identity dynamically per request through middleware and AppConfig.

`LT47` captures the immediate LingoCafe icon work and should not design the whole shared icon system.

## Desired State

42go has a documented, type-safe, config-driven icon model that every app can use.

The model should cover, or deliberately decide not to cover:

- browser favicon links;
- Apple home-screen icon;
- web app manifest icons;
- maskable manifest icon;
- main app logo used in layouts;
- public toolbar icon override interactions;
- default/fallback icon behavior.

The implementation should respect Next.js App Router metadata behavior while still supporting dynamic app identity.

## Definition of Success

An app author can configure app icons in one clear AppConfig shape, and 42go emits the right metadata, manifest entries, and layout logo behavior for that app without hardcoded per-app conditionals.

## Additional Context

This task was created as a follow-up from `LT47`.

The immediate LingoCafe task should wire existing supported fields and generate missing practical PWA sizes, but favicon-system design belongs here.

## Assumptions

The current `config.public.pwa` shape is the existing authority for PWA metadata unless this task deliberately supersedes or extends it.

The existing `logo` field is useful but may need clearer documentation or integration with a broader icon config.

Per-app favicons should not be solved by replacing a global `src/app/favicon.ico`, because that would only represent one app.

## Constraints

Keep the solution compatible with request-based app resolution.

Do not break existing Quicklist, Notes, default app, or LingoCafe PWA configuration.

Avoid duplicating the same icon paths across unrelated config fields unless the duplication is intentional and documented.

Use official Next.js metadata APIs and conventions where they fit dynamic app identity.

Run `npm run qa` after implementation.

## Acceptance Criteria

- [ ] Existing icon and PWA behavior is audited across `AppConfig`, `HeadTags`, `manifest.ts`, and app/public layout title components.
- [ ] Official Next.js App Router metadata icon behavior is documented as an implementation constraint.
- [ ] A shared AppConfig icon shape is proposed and implemented, or the existing `public.pwa.icons` plus `logo` fields are explicitly extended and documented.
- [ ] App-specific browser favicon links can be configured without replacing one global favicon for every app.
- [ ] Apple touch icon behavior remains config-driven.
- [ ] Manifest icon behavior remains config-driven.
- [ ] Main app logo behavior is documented and, if needed, aligned with the new icon configuration shape.
- [ ] Existing apps keep their current behavior unless intentionally migrated.
- [ ] Documentation explains how each app should configure favicon, home-screen icon, PWA icons, and logo.
- [ ] `npm run qa` passes after implementation.

## Dos

- Do preserve the multi-app architecture.
- Do keep icon configuration explicit and type-safe.
- Do reuse existing PWA and logo support where it still makes sense.
- Do make the migration path clear for existing app configs.
- Do verify generated head tags and manifest output.

## Don'ts

- Don't hardcode app IDs in shared metadata components.
- Don't replace `src/app/favicon.ico` as the only solution for app-specific favicons.
- Don't break existing `public.pwa` behavior.
- Don't force every app to provide every icon size if sensible fallbacks are available.

## Open Questions

- Should the final shape be `public.icons`, an extension of `public.pwa.icons`, or a normalized top-level `icons` field?
- Should 42go support multiple favicon sizes and MIME types, or only a single favicon path plus PWA icons?
- Should maskable icons require an explicit asset to avoid unsafe cropping?
- Should missing recommended sizes warn during development?
