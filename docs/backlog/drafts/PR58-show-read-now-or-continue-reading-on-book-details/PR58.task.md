# Show Read Now or Continue Reading on Book Details

## Elevator's Pitch

The LingoCafe book details page should show the correct primary reading action for the authenticated user: "Read now" for a new book, or "Continue reading" when saved progress exists.

## Business Gain

Readers should not have to guess whether they are starting fresh or resuming. The book details page should send them to the right page immediately. Chuck Norris does not lose his place. The chapter remembers him.

## Current State

`IL68` defines the dedicated `/books/[bookId]` book info page with metadata, cover, and description.

The LingoCafe data model already includes `lingocafe.books_pages` and `lingocafe.books_progress`. `books_progress` stores one progress row per user and book, including the current `page_id` and `progress_bps`.

The requested gap is the book details page primary action. The details API must return enough reading-state data for the UI to choose the correct button label and link.

## Desired State

The book details API call also fetches the authenticated user's `books_progress` record for the selected book, if one exists.

If no progress exists, the details page shows a "Read now" button that links to the first available page from `books_pages` for that book.

If progress exists, the details page shows a "Continue reading" button that links to the saved progress page and includes `progress_bps` as a URL parameter so the future read page can lock the initial scroll position when opened.

This task is only about displaying the correct button with the correct link. It must not create the read page UI.

## Definition of Success

The book details page uses API-provided reading state to render a single primary action that accurately starts a new book or resumes an existing one, with links pointing to the correct page and preserving saved scroll progress when available.

## Assumptions

The first available page means the lowest-position page in `lingocafe.books_pages` for the selected book.

The read URL shape can be decided during implementation, but it must include enough path information to identify the book and page, plus a query parameter containing `progress_bps` for resume links.

The authenticated user is the only user whose progress should be checked.

## Constraints

This task must be implemented after `IL68`.

Follow the existing authenticated app-page convention: client component, `AppLayout`, and browser-side data fetching with `credentials: "same-origin"`.

Use server-side validation in the details API and do not accept arbitrary user IDs from the browser.

Keep the book-detail API work under the LingoCafe API namespace and policy shape established by completed catalog work.

Do not create the read page UI or reading experience in this task.

Do not create or update `books_progress` as part of rendering the button. This task only reads progress and builds the action link.

## Acceptance Criteria

- [ ] The book details API fetches the authenticated user's `books_progress` row for the selected book when one exists.
- [ ] The book details API returns enough reading-action data for the client to decide between "Read now" and "Continue reading".
- [ ] When no progress row exists, the book details page renders a "Read now" primary button.
- [ ] The "Read now" button links to the first available page from `lingocafe.books_pages` for the selected book.
- [ ] When a progress row exists, the book details page renders a "Continue reading" primary button.
- [ ] The "Continue reading" button links to the saved progress page from `books_progress.page_id`.
- [ ] The "Continue reading" link includes `progress_bps` as a URL parameter.
- [ ] The UI renders only the correct button for the current state, not both.
- [ ] The implementation handles books with no available pages without producing a broken link.
- [ ] The implementation does not create the read page UI.
- [ ] The implementation depends on `IL68` being completed first.
- [ ] `npm run qa` passes after implementation.

## Dos

- Do extend the book details API response rather than making the client guess progress state.
- Do use the authenticated session user when querying `books_progress`.
- Do select the first readable page from `books_pages` when no progress exists.
- Do include `progress_bps` in resume links so the future read page can restore initial scroll position.
- Do keep the UI change scoped to the book details page primary action.
- Do keep the task compatible with the details page route and API from `IL68`.

## Don'ts

- Don't build the read page UI in this task.
- Don't create the reading scroll-lock behavior in this task.
- Don't create progress rows just because the user views the details page.
- Don't trust a user ID passed from the browser.
- Don't show "Continue reading" when the saved progress row is missing or invalid for the selected book.
- Don't break the metadata, cover, or description behavior from `IL68`.
