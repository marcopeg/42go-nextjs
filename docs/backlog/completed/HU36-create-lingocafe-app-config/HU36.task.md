# Create LingoCafe App Config

## Elevator's Pitch

Create a new app configuration named `LingoCafe` with both password and social login enabled, plus a minimal home page that simply asks the user to log in.

## Business Gain

This creates a new branded app entry for LingoCafe so the product can ship as its own tenant on the shared platform, with the right hostname, branding, and authentication options already aligned to the use case.

## Current State

The project already supports multiple app configurations, password login, and social login per app, but there is no LingoCafe configuration yet.

## Desired State

There is a `LingoCafe` app configuration wired into the existing multi-app system, it resolves on the requested production hostname, exposes credentials plus Google authentication, and serves a minimal public home page with a basic hero section whose main call to action pushes the user into the login flow.

## Definition of Success

LingoCafe resolves as its own app on the requested domain, shows a basic hero section, supports password and Google login, and gives the user a direct path from the hero action into authentication without introducing a custom auth architecture.

## Additional Context

The request is specifically to create a new app config named `LingoCafe`.

The home page should stay basic. It should simply ask the user to log in instead of trying to act like a finished marketing site.

The requested hostname is `lingocafe.app`.

The requested authentication mix is Google plus password login.

The home page should include a basic hero section.

The hero action should send the user to `/login` so they can continue with either Google or password login through the existing entry point.

## Assumptions

LingoCafe should follow the same architectural patterns used by the existing app configurations in `src/AppConfig.ts` and `src/config/`.

Password login means the existing credentials provider.

Social login means Google for this app.

The minimal home page can be implemented through the existing public page and app configuration mechanisms rather than introducing a custom standalone architecture.

## Constraints

Use the current multi-app configuration system and keep the implementation configuration-driven.

Reuse the existing authentication stack and only enable providers that are already supported by the project.

Keep the landing page intentionally simple and login-oriented.

Do not require a separate codebase, custom auth stack, or non-standard routing model for this app.

Prefer implementation through existing page configuration, hero/action patterns, and login routes unless there is a concrete limitation that requires a different approach.

## Acceptance Criteria

- [ ] A new app configuration named `LingoCafe` exists in the project.
- [ ] LingoCafe is wired into the existing app resolution flow for `lingocafe.app`.
- [ ] LingoCafe enables password login through the credentials provider.
- [ ] LingoCafe enables Google login.
- [ ] LingoCafe has a basic public home page with a hero section whose main purpose is to ask the user to log in.
- [ ] The hero action routes to `/login`.
- [ ] The implementation follows existing app config and authentication patterns already used in the repository.

## Dos

- Do reuse established examples for app configuration, public page setup, and auth provider wiring.
- Do keep the homepage copy and layout minimal.
- Do align the solution with the existing dynamic social login support already implemented across app configs.
- Do use `/login` as the stable login entry point for this app.

## Don'ts

- Don't invent a brand-new authentication flow.
- Don't turn the home page into a full marketing site unless requirements change.
- Don't hardcode assumptions about unsupported social providers.
- Don't force a brittle direct OAuth jump from the hero.

## Open Questions

Should LingoCafe have custom branding assets or theme colors beyond the default basics?
