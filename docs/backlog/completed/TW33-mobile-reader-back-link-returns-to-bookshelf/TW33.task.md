# Reader Back Links Return To Bookshelf

## Elevator's Pitch

When reading a LingoCafe book, the visible reader back links should return directly to the bookshelf index at `/books`.

## Business Gain

Readers need a fast escape hatch back to the book catalog. If they are done reading or want another book, the app should send them to the shelf in one move. Chuck Norris does not tap back twice. The navigation retreats first.

## Current State

The LingoCafe reader route exists at `/books/[bookId]/[pageId]`.

Completed reader work established mobile read pages as full-screen reader surfaces layered above the book details view, with back navigation that reveals the book details view underneath.

Current source inspection shows `src/app/(app)/(lingocafe)/books/[bookId]/[pageId]/page.tsx` computes a single `backHref` as `/books/${bookId}` when a `bookId` exists, then passes that same value to `AppLayout`, `BookReaderMobileSurface`, and `BookReaderDesktopSurface`.

Current source inspection also shows `BookReaderMobileSurface` in `src/app/(app)/(lingocafe)/books/_components/BookReaderSurfaces.tsx` renders the visible mobile header back link using that shared `backHref`, with `aria-label="Back to book details"`.

The operator first wanted the mobile reader back link to go to the bookshelf index `/books`, not to the book detail route.

The operator then expanded the same rule to desktop as well.

## Desired State

On mobile, the reader's visible header back link returns to `/books`.

The mobile link should behave as a bookshelf/back-to-index action, not as a step back to `/books/[bookId]`.

The mobile back link's accessible label should also match the new target, such as "Back to books" or equivalent, instead of "Back to book details".

On desktop, reader back navigation should also return to `/books`.

The AppLayout reader back button should also return to `/books`.

Back link accessible labels should match the bookshelf target instead of saying "Back to book details".

## Definition of Success

From a reading page such as `/books/<bookId>/<pageId>`, activating the visible reader back link navigates to `/books` on mobile and desktop. The back links are labeled for the bookshelf target. The bookshelf loads normally, and the change does not regress reader loading, scroll progress tracking, or previous/next page controls.

## Additional Context

The operator asked: "when reading a book from mobile, the link back should go to the bookshelf, so to the index: /books."

The operator confirmed before planning that only the mobile reader header back icon should go to `/books`; the desktop/AppLayout back button should keep going to `/books/[bookId]`.

The operator later changed that instruction and asked to apply the same `/books` rule to desktop as well.

`ZE11` created the read page route and mobile full-screen reader layer.

`VU18` redesigned the reader and kept the full-screen mobile read view behavior from the earlier implementation.

`IL68` and `DJ89` define the book details route and its own back-to-books behavior. This task is specifically about the reader surface on mobile.

Relevant implementation entry points:

- `src/app/(app)/(lingocafe)/books/[bookId]/[pageId]/page.tsx`
- `src/app/(app)/(lingocafe)/books/_components/BookReaderSurfaces.tsx`

## Assumptions

The "link back" refers to the visible back control in the mobile reader UI, not the browser's hardware/history back gesture.

The intended target is the canonical bookshelf route `/books`.

The implementation should be route-local to the LingoCafe books reader components if possible.

The AppLayout `backBtn`, desktop reader side-panel back link, and mobile reader header back link should all use `/books`.

## Constraints

Follow the authenticated app-page convention for routes under `(app)`: client component, `AppLayout`, and browser-side data fetching with `credentials: "same-origin"`.

Preserve the read route shape `/books/[bookId]/[pageId]`.

Preserve reader data loading, `page-open` tracking, `page-scroll` tracking, saved progress updates, and previous/next page navigation.

Do not redesign the reader UI as part of this task.

Do not change the canonical book detail route `/books/[bookId]`.

Do not override browser history or hardware back behavior in this task. Change the visible reader back link targets only.

Do not change the canonical book detail route, but reader back links should no longer target it.

Run `npm run qa` after implementation.
