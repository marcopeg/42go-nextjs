# Enable Email-Only Login With Magic Link and Account Creation

## Elevator's Pitch

Enable passwordless email login so a user can enter only an email address, receive a temporary sign-in message, and authenticate either by clicking the magic link or by entering the delivered code. If the email does not belong to an existing user for the current app, the verified flow should create the account implicitly.

## Business Gain

Reduces signup friction for LingoCafe and other 42Go apps. Supports users who do not want OAuth or passwords. Keeps authentication configurable per app through AppConfig instead of hard-coding one app's login strategy.

## Current State

- The project uses `next-auth@4.24.14` with JWT sessions, dynamic providers from `AppConfig.auth.providers`, and custom Knex-backed `auth.users` / `auth.accounts` handling for credentials, Google, and GitHub.
- Current provider types are `credentials`, `github`, and `google` in `src/42go/auth/lib/providers/types.ts`.
- The login page renders explicit strategy components from `src/42go/auth/components/login-strategies` based on the configured provider types.
- The database already has `auth.verification_tokens`, but there is no NextAuth adapter configured in `getAuthOptions()`.
- The existing `auth.verification_tokens` table is not app-scoped today. This task may alter the table and must make verification tokens app-scoped before email auth can be considered secure.
- Existing credentials and OAuth flows already perform app-scoped user resolution and account linking in custom provider/callback code. Adding a NextAuth adapter changes a critical auth boundary and must be proven before UI or sender work is treated as stable.
- Official NextAuth v4 documentation says the Email provider sends magic links, creates the user after verified first use, signs existing users in by email, and requires a database adapter for email sign-in. It also supports custom `sendVerificationRequest`, `generateVerificationToken`, and `normalizeIdentifier` hooks.
- Official Auth.js provider docs also document HTTP email providers such as Resend and SendGrid in the newer Auth.js ecosystem, but v1 stays on NextAuth v4.

## Desired State

- A new app-configurable email/passwordless provider can be enabled per app, alongside or instead of credentials/OAuth providers.
- The AppConfig provider type is `email`.
- A user can request sign-in with only an email address.
- The email contains a sign-in link that carries the verification token through the standard NextAuth callback flow.
- By default, the flow supports both clicking the magic link and entering the delivered temporary code.
- Manual code entry uses a dedicated verification route but must converge with the magic-link verification logic after the token/code is verified.
- The default manual code is a 6-digit numeric code.
- Token/code expiration is configurable and defaults to 5 minutes.
- The code generation strategy uses `length`, `mode`, and `caseSensitive`.
- Supported code generation `mode` values are `digits`, `alphabet`, `alphanumeric`, and `complex`.
- Default code generation is `{ length: 6, mode: "digits", caseSensitive: false }`.
- After email submission, the login UI shows an "enter the code" screen with a resend-email action.
- The same delivered secret can be used through both the magic-link URL and the code-entry form when the selected implementation supports it without fighting NextAuth semantics.
- Verified first-time email login creates an `auth.users` row scoped to the current `app_id`; existing email users are signed into the matching app-scoped account.
- Verification tokens are scoped to the resolved AppID. A token issued for one app must not be valid for another app, even when the normalized email address and raw token/code match.
- New email-only users get a default username from the email local-part before `@` when available, falling back to the full email address if the local-part username is already taken.
- Magic-link/email-code login and social login providers must resolve the same verified email to the same app-scoped user.
- Email sign-ins do not create `auth.accounts` rows by default; `auth.accounts` remains focused on OAuth/social provider links.
- Email delivery goes through a strategy-based sender abstraction so different 42Go apps can configure different email-sending providers from the available set.
- Email provider config supports multiple named sender strategies as key/value entries and a selected strategy key, tentatively `useStrategy`.
- The selected sender strategy defaults to `console` and must be configurable through environment variables.
- `console` is a first-class sender strategy that writes email contents, magic links, and codes to logs/output for development and test usage.
- `console` sender is allowed in production when selected by configuration; this is considered operator responsibility and must be documented.
- Resend is the first external sender strategy for v1 because it has official developer-focused docs, Node/Next.js examples, and an official free transactional plan.
- SMTP/Nodemailer can remain a later sender strategy unless planning decides to include it in v1 as well.
- Auth strategy configuration remains open for future custom behavior, including resend throttling policy and user-facing throttle messages.
- Code-generation config must be explicit and easy to validate: `length`, `mode`, and `caseSensitive`.
- The email login UI shares the initial username/email identifier step with the credentials flow.
- If only `credentials` is configured, login works as it does today: username/email plus password.
- If only `email` is configured, login asks for the username/email identifier, sends the email, then shows the code-entry screen.
- If both `credentials` and `email` are configured, login first asks for the shared username/email identifier, then lets the user choose whether to continue with password or email code/magic link. The selected choice controls whether the next screen asks for password or code.
- Email auth emits audit/events for email requested, email resent, code verified, and email login failed by default.
- Event logging is configurable per event with Boolean keys, all defaulting to `true`.
- V1 architecture uses NextAuth v4 EmailProvider plus a custom app-scoped Knex adapter.
- JWT sessions and existing NextAuth callbacks remain in use.
- `AppConfig.auth.providers` remains the source of truth so each app can configure its own available auth providers.
- V1 follows the existing social-provider configuration pattern under `auth.providers`; it does not introduce a full public/server AppConfig split.

## Definition of Success

- A refined plan implements the selected NextAuth v4 EmailProvider plus custom Knex adapter path.
- The implementation does not fork the authentication model or bypass NextAuth session/callback handling.
- Manual code verification and magic-link verification share post-verification behavior.
- The resulting login flow works for new users, existing users, expired/invalid tokens, and app-scoped duplicate emails.
- Refinement must reduce known delivery risks without splitting the task yet.
- Completion requires a validation matrix across the critical auth-provider combinations and failure cases.
- Planning must explicitly sequence technical risk before product/UI polish. The task remains one task, but the plan must contain gated macro steps with stop conditions.

## Additional Context

- NextAuth v4 EmailProvider imports from `next-auth/providers/email` and internally builds `/api/auth/callback/email?callbackUrl=...&token=...&email=...` links.
- The callback consumes verification tokens through adapter methods and hashes stored tokens before lookup.
- NextAuth v4 asserts that email login has an adapter with at least `createVerificationToken`, `useVerificationToken`, and `getUserByEmail`; real callback handling also needs user create/update behavior against the current schema.
- The custom adapter must be designed around app scope from the start. Planner should not assume request-level AppID is available in every adapter call without a proof step.
- Potential sender options to compare during refinement: `console`, Resend HTTP API, SMTP via Nodemailer-compatible service, SendGrid HTTP API, or a larger Auth.js v5 migration.
- Reference docs: NextAuth v4 Email Provider https://next-auth.js.org/providers/email ; Auth.js Resend Provider https://authjs.dev/getting-started/providers/resend ; Auth.js Nodemailer Provider https://authjs.dev/getting-started/providers/nodemailer
- Resend pricing docs currently describe a free transactional tier with 3,000 emails/month and a 100 emails/day cap: https://resend.com/docs/knowledge-base/what-is-resend-pricing
- Resend official docs include Node.js and Next.js quickstarts: https://resend.com/docs

## Assumptions

- Email address remains the primary contact identifier for this task.
- The flow should be app-scoped through the resolved AppID, matching existing credentials and OAuth behavior.
- Email is unique per app and can be used as the durable account-linking key across email-only and social providers.
- Email-provider identity is represented by `auth.users.email` plus verification-token proof, not by a synthetic provider row in `auth.accounts`.
- JWT sessions remain the session strategy unless refinement explicitly chooses otherwise.
- The selected architecture is the least-disruptive path because it preserves NextAuth v4, JWT sessions, callbacks, and app-configured provider lists.
- Implicit account creation is allowed only after the email token/link is verified.
- The task can add provider config types, a custom Knex adapter, and login UI components, but should not require a broad auth-library migration.
- The task may include a migration that adds `app_id` and supporting indexes/constraints to `auth.verification_tokens`, or an equivalent schema design that makes wrong-app token reuse impossible.

## Constraints

- Stay compliant with the current authentication library instead of building a parallel auth system.
- Before implementing user-facing login changes, prove that NextAuth v4 EmailProvider can run with a request/app-scoped Knex adapter in this repository without breaking credentials, GitHub, Google, JWT session stamping, or RBAC loading.
- Do not duplicate user creation, token consumption, or session creation logic between magic-link and manual-code paths.
- Preserve multi-app provider selection through `AppConfig.auth.providers`.
- Do not expose whether an email already has an account before verification.
- Use server-side validation and normalized email handling.
- Normalize email in one shared server utility used by token creation, token consumption, user lookup, throttling, event payloads, and sender dispatch.
- Follow the existing social-provider pattern for config/secrets: values are sourced from environment variables in `auth.providers`.
- Do not introduce a full public/server AppConfig split in v1.
- Consider rate limiting, token expiry, replay prevention, and event logging before implementation.
- Email request and resend flows must be throttled. The default throttle is 1 minute unless app/provider config overrides it.
- V1 throttle identity is app ID plus normalized email only.
- V1 does not store IP or user-agent for throttling, but the throttle data structure should be extensible so IP and user-agent can be added later if needed.
- Token/code expiration defaults to 5 minutes unless app/provider config overrides it.
- Prefer a configurable progressive delay list such as `15s,30s,1m,3m,7m,15m`, where each subsequent resend attempt follows the next delay.
- Throttle responses should use customizable, user-safe messaging and must not reveal whether the email is registered.
- Sender configuration must be app-scoped. One 42Go app can use a different sender strategy than another.
- Sender secrets/settings follow the current OAuth-provider config style and should be sourced from environment variables.
- Sender strategy selection defaults to `console` unless app/provider config or environment overrides it.
- The implementation must not block `console` only because the runtime is production, but docs should warn about the risk.
- Event logging config must allow each supported email-auth event to be enabled or disabled independently.
- Sender implementation must not be coupled to UI state. The server flow owns request validation, throttle checks, token generation, event logging, and delivery result handling.
- Resend work must not start until the console sender can complete a real login flow through the selected adapter/token model.
- Run `npm run qa` after code changes in the implementation task.

## Acceptance Criteria

- Implementation uses NextAuth v4 EmailProvider plus a custom app-scoped Knex adapter.
- Custom credentials-style OTP provider and Auth.js v5/provider migration are not part of v1 unless implementation research uncovers a blocker.
- Implementation includes an adapter proof before broad login UI changes: email request creates the expected verification token, callback consumes it once, a verified new user is created or reused, and the JWT session has the correct `appId`.
- Implementation updates `auth.verification_tokens` or equivalent persistence so tokens are app-scoped and indexed for safe lookup/consumption.
- Implementation preserves per-app provider configuration through `AppConfig.auth.providers`.
- Implementation adds a typed `email` provider option to `AppConfig.auth.providers`.
- Implementation provides a shared identifier-first login UI that supports credentials-only, email-only, and credentials-plus-email configurations.
- Credentials-only configuration preserves today's username/email plus password behavior.
- Email-only configuration asks for username/email, sends the email, and displays the code-entry screen.
- Credentials-plus-email configuration asks for username/email first, then offers password login or email login as explicit continuation choices.
- After email submission, implementation displays a code-entry screen with a resend-email action.
- Magic-link click signs in and redirects to the configured callback URL/default app page.
- Manual code-entry signs in by default through the same verified email-login flow, unless refinement proves this cannot be done safely with the selected NextAuth-compatible design.
- Manual code-entry uses a dedicated verification route that consumes the same stored verification token model as the magic-link flow.
- After token/code verification succeeds, manual code-entry and magic-link verification converge on shared user/session creation logic.
- Manual code-entry implementation must document exactly how it enters the NextAuth callback/session path after verification. If the design cannot converge safely, implementation must stop and revise the plan before coding around NextAuth.
- Resend throttling supports a default 1-minute delay and an optional progressive delay sequence such as `15s,30s,1m,3m,7m,15m`.
- Throttle state is keyed by app ID plus normalized email for v1.
- Throttle implementation keeps schema/config extension points for future IP and user-agent fields without collecting them in v1.
- Throttled resend attempts display a configurable, user-safe message.
- Manual code generation defaults to 6 numeric digits.
- Manual code generation supports configurable `length`, `mode`, and `caseSensitive`.
- Manual code generation supports `digits`, `alphabet`, `alphanumeric`, and `complex` modes.
- `caseSensitive` defaults to `false`; default code config is `{ length: 6, mode: "digits", caseSensitive: false }`.
- Token/code expiration supports provider/app configuration and defaults to 5 minutes.
- Email sending is implemented behind a strategy interface with at least one production-ready default strategy.
- Email provider config supports multiple named sender strategies plus a selected strategy key such as `useStrategy`.
- The selected sender strategy defaults to `console` and can be overridden through environment configuration.
- `console` sender strategy prints enough information to complete local login manually, including code and magic link.
- `console` sender works in any environment when selected; production use is allowed but documented as risky.
- Resend sender strategy is implemented as the first external provider strategy.
- Resend configuration supports API-key based delivery through server-only environment variables.
- SMTP/Nodemailer is left as a future sender strategy unless explicitly pulled into v1 during planning.
- App configs can select among available sender strategies without changing shared login code.
- New verified emails create app-scoped users; existing app-scoped emails reuse the existing user.
- Email-only implicit account creation assigns `username` as email local-part when available, else full email.
- If the same verified email signs in through magic link/email code and a social provider, both flows resolve to one app-scoped `auth.users` row.
- Email sign-ins do not create `auth.accounts` rows unless a future refinement explicitly changes that model.
- Audit/event logging covers email requested, email resent, code verified, and email login failed.
- Audit/event logging can be disabled per event through Boolean config keys and defaults to enabled for all supported email-auth events.
- Expired, missing, reused, malformed, and wrong-app tokens fail without creating a session.
- Email delivery failures produce a controlled login error and are observable in logs.
- Docs and `.env.example` are updated for the chosen email service configuration.
- Docs mention that email provider config follows the existing social-provider pattern and that a future public/server config split may harden this boundary.
- Completion evidence covers credentials-only, email-only, credentials-plus-email, new user by email, existing user by email, expired/reused code failure, throttled resend, console sender, and Resend sender.

## Dos

- Use official NextAuth/Auth.js documentation during refinement.
- Start planning with an adapter and token-schema proof, not with the login UI.
- Prefer extending the existing provider builder and login-strategy pattern.
- Implement a Knex adapter that maps to the existing `auth` schema and scopes user/token operations by AppID.
- Keep provider configuration app-scoped and type-safe.
- Preserve auth event logging for login/signup where practical.

## Don'ts

- Do not implement passwordless login by storing raw one-time codes in plaintext.
- Do not send a different response that reveals whether an email is registered.
- Do not hard-code LingoCafe-only behavior into shared auth primitives unless the scope is explicitly narrowed.
- Do not add Resend before the console sender proves the full auth flow locally.
- Do not redesign the login UI before the adapter/token/session path is proven.
- Do not bypass NextAuth cookies, CSRF, callbacks, or session creation.
- Do not pick an external email service without documenting local/dev and production behavior.

## Planner Macro Steps

The task must stay intact, but the implementation plan should be staged with explicit gates:

1. Adapter and schema proof: design the app-scoped Knex adapter, add or revise `auth.verification_tokens` app scoping, and prove token create/use/user-create/user-reuse behavior through NextAuth v4 EmailProvider.
2. Session and account-link proof: prove new and existing verified email users receive JWT sessions with the correct `appId`, roles/grants still load, and OAuth by the same verified email still resolves to the same app-scoped user.
3. Console sender MVP: implement typed `email` provider config, code/token generation defaults, console sender, and magic-link login with controlled errors.
4. Manual code route: add the dedicated code-entry verification route only after the magic-link path works, and make the route converge with the same post-verification/session behavior.
5. Throttle and audit hardening: add app+normalized-email throttle state, progressive delay configuration, safe user-facing messages, and configurable events.
6. Login UI state machine: implement the shared identifier-first flow for credentials-only, email-only, and credentials-plus-email configurations without changing credentials-only behavior.
7. Resend sender and docs: add Resend after the local console flow passes, update `.env.example`, document production risks, and validate delivery failure behavior.
8. Matrix verification: complete QA and auth matrix evidence before completion.

## Planning Stop Conditions

- Stop planning if the adapter cannot reliably resolve AppID during token creation, token consumption, user lookup, and user creation.
- Stop planning if wrong-app token replay cannot be prevented at the database or lookup-contract level.
- Stop planning if manual code entry requires bypassing NextAuth cookies, CSRF, callbacks, or session creation.
- Stop planning if credentials-only or existing OAuth login semantics would regress.
- Stop planning if the plan cannot produce deterministic local proof with the console sender before external delivery work.

## Post-Mortem Risk Notes

- The main failure mode is treating this as a login form feature. It is an auth-boundary migration with UI attached.
- The second failure mode is mixing adapter discovery, token semantics, Resend delivery, throttling, and UI redesign in one ungated implementation pass.
- The third failure mode is assuming the current `auth.verification_tokens` table is ready because it exists. It is not ready until app scope and replay behavior are explicit.
- The fourth failure mode is letting manual code entry become a parallel auth system. Manual code entry must be a front door into the same verified email auth result, not a side door through homemade sessions.
