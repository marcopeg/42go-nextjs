# Restore Draft-Centric Backlog Metadata and Lifecycle States

## Elevator's Pitch

Update the repo-local backlog skills so every task has one canonical task file, `<taskid>.task.md`, that owns the original request, task content, custom task frontmatter, and lifecycle metadata. Replace the current draft/refined metadata split with a status-driven workflow: `draft`, `refining`, `refined`, `planned`, `wip`, and manually assigned `blocked`, with `backlog-doctor` keeping physical folders and indexes honest.

## Business Gain

Backlog state becomes easier to trust, migrate, inspect, and port across repositories. A task has one metadata spine, so agents do not have to reconcile conflicting statuses between draft, refined, plan, notes, questions, and index state. Chuck Norris does not ask three files what state a task is in. One file answers.

The on-disk backlog also becomes more human approachable: task folders live under the folder matching their current status, and each status folder has a local index for quick inspection.

## Current State

The current repo-local backlog workflow uses `docs/backlog` and the `.agents/skills/backlog-*` skills. Recent work in `HW55` introduced `.task.draft.md` and `.task.refined.md` artifacts plus Python scaffolding, but lifecycle metadata is now often written to the active task artifact, especially `.task.refined.md`.

Current skill documents and scripts repeatedly describe task frontmatter on draft and refined task files with fields such as `taskId`, `status`, `createdAt`, `updatedAt`, `group`, `reviewedAt`, `plannedAt`, `startedAt`, `completedAt`, and `reviewAfter`. Examples include `backlog-refine`, `backlog-clarify`, `backlog-plan`, `backlog-execute`, `backlog-complete`, `backlog-archive`, `backlog-core`, and `backlog-migrate`.

The current active states are folder/index oriented: `draft`, `ready`, `wip`, and `blocked`, with completed and archived history. `backlog-migrate` currently maps draft folders to `.task.draft.md` and ready/blocked folders to `.task.refined.md`, then sets status on the migrated active file.

The operator sees this as metadata drifting into the refined task file. That makes migration and reuse across other codebases harder because task properties no longer have one source of truth.

The current visible backlog also does not fully reflect the requested lifecycle. A task can be refined while still living under `drafts`, and `ready` does not match the new requested `refined` or `planned` statuses.

## Desired State

Make `<taskid>.task.md` the canonical source of truth for task metadata and primary task content. Do not keep the `.task.draft.md` suffix as the long-term main task filename.

The canonical `<taskid>.task.md` file should contain:

- the original user request or drafted request
- the current task definition
- all task-level frontmatter
- current status
- lifecycle movement dates
- group and custom task attributes supplied by the user
- any other metadata that describes the task itself

Non-main artifacts should not own frontmatter. This includes refined task artifacts, plan files, notes files, question files, and clarify files. If the implementation keeps those artifacts, they should be content-only unless a narrowly justified compatibility step is needed during migration.

Update the lifecycle statuses to:

- `draft`: task entered the backlog and has not started refinement.
- `refining`: refine or clarify has started and one or more rounds are in progress or being folded in.
- `refined`: refinement/clarification is complete enough that the skill proposes planning.
- `planned`: a plan has been created and accepted.
- `wip`: execution has started.
- `blocked`: a manually attributable status for a task that cannot currently proceed. There is no dedicated blocked skill required by this task.

Every status should have a matching physical subfolder under the backlog root. The other backlog skills should move the whole task folder into the appropriate status subfolder whenever they change the task status.

Each status subfolder should have a local index named `BACKLOG_<STATUS>.md`, for example `BACKLOG_DRAFT.md`, `BACKLOG_REFINING.md`, `BACKLOG_REFINED.md`, `BACKLOG_PLANNED.md`, `BACKLOG_WIP.md`, and `BACKLOG_BLOCKED.md`. The root `BACKLOG.md` should still provide the high-level active backlog view.

Add a new repo-local `backlog-doctor` skill. When executed, it should analyze each task's canonical `<taskid>.task.md` status, move the task folder to the correct status subfolder, and rebuild the status-local indexes and root backlog index. It should be able to verify and repair drift between status frontmatter, folder location, and index links.

Update `backlog-migrate` so it can analyze existing task folders, merge metadata currently written on `.task.refined.md` or legacy task files back into the canonical `<taskid>.task.md`, and remove lifecycle frontmatter from non-main artifacts. Because current skills write task metadata to refined files, the migration conflict rule is `refined -> task`: refined-file metadata should be reflected into the canonical task file.

Keep the implementation repo-local only under `.agents/skills/backlog-*`. Do not update `/Users/marcopeg/.agents/skills/backlog-*`; the operator will copy changes manually.

## Definition of Success

The backlog system has one authoritative task metadata location: `<taskid>.task.md`. Active lifecycle transitions update that file instead of refined, plan, notes, question, or clarify artifacts. The visible backlog/index and folder movement behavior still works, but status and movement fields no longer drift across artifacts.

Running migration restores existing active tasks to the new metadata model without dropping useful timestamps, custom frontmatter, or user-authored task content.

Running `backlog-doctor` can verify and repair the on-disk backlog so every task folder lives under the subfolder matching its canonical status and every status-local index reflects the folders under that status.
