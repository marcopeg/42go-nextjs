# Refinement Questions v1 - Extract LingoCafe Profile Onboarding Overlay

Answer inline under each question, then return to chat and say `next round`.

## Questions

### 1. Where should the onboarding guard live?

Pick the preferred ownership model:

- A new LingoCafe authenticated route layout/wrapper under `src/app/(app)/(lingocafe)/`.
- A shared `AppLayout` extension driven by LingoCafe config.
- A route-local wrapper imported by each LingoCafe page for now.

Answer:

The best place would be some form of injection into the AppLayout so that a guard like that can take over and render a popup on top of the app layout. it could actually be a couple of "core places" where to place the guard:

- `app.profile.guard.before` should be able to PREVENT rendering the AppLayout and render the guard instead if the profile is incomplete
- `app.profile.guard.layout` shoulr be able to render content OVER the app layout like a popup or full-screen surface so to use also fade-in/out or slide-in/out effects

### 2. Should `/profile` itself be guarded by the onboarding overlay?

If `/profile` is guarded, users see the same overlay there. If `/profile` is exempt, it remains the canonical escape hatch for full profile editing while other LingoCafe routes are blocked.

Answer:

the guard should behave in relation to the AppLayout, regardless of the route.

### 3. What should the onboarding save metadata be?

The current books-owned flow likely uses a books-specific source. Should this task standardize on `source: "lingocafe-onboarding"` and `method: "profile-onboarding-submit"` for the overlay save, or use different exact values?

Answer:

I say the guard is just a component that should somehow complete the profile and be able to finally tell to the guard-manager: "I'm done, remove me".

In a way, the guard component could have a contract as simple as:

- return `null` to say "there is nothing to do"
- return `stuff to render` to actually render something.

the trick is that the guard before the layout should be synchronous so to prevent rendering the layout.
the guard on top of the layout can be independent and asynchronous

### 4. Should the overlay offer a `/profile` link?

The overlay is blocking until completion. Should it include a secondary link/action to `/profile`, or should the embedded onboarding form be the only completion path?

Answer:

No, so far, i want a fully LingoCafe custom surface that we will customize into a multi-step onboarding.
Maybe a small logout button at the very bottom.

### 5. How should route content behave while the profile check is loading?

Should LingoCafe routes hide their main content until the profile completeness check resolves, show the normal route loading state, or allow content to render behind the blocking overlay once incompleteness is known?

Answer:

/books should just show "Profile incomplete, can't show books now" until the profile becomes complete.

NOTE: the "profile becomes complete" must be some kind of client-only event system of sort.

### 6. Which non-books route should prove cross-route behavior?

Name the second authenticated LingoCafe route or route shell scenario to validate besides `/books`. If there is no suitable existing route, say whether a shell-level test/demo is acceptable.

Answer:

the guard should not be route depenendent. it's a route-agnostic behavior.
