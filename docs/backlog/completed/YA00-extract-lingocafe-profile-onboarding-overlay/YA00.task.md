# Extract LingoCafe Profile Onboarding Overlay

## Elevator's Pitch

Separate the current `/books` incomplete-profile form into a LingoCafe onboarding feature that can appear on any authenticated LingoCafe route as a full-screen modal overlay when the user profile is incomplete.

## Business Gain

Onboarding should be a product-level LingoCafe concern, not a books-page implementation detail. Users who land on any protected LingoCafe route should be guided to complete their required profile before the app tries to show route-specific content.

## Current State

The `/books` page owns profile onboarding inline. It calls `/api/lingocafe/books`, receives profile data, and when `profile.isComplete` is false it renders a local form with language selects, consent checkboxes, and a `/api/profile` save. The API already returns an empty `books` array when the profile is incomplete, but the page still owns the onboarding UI and save/reload flow.

## Desired State

Create an AppLayout-level profile layout guard system that can be configured per app. The first concrete guard is a LingoCafe profile onboarding guard that detects incomplete profile state for authenticated LingoCafe users and opens a full-screen takeover popup on both mobile and desktop.

The broader guard model has two conceptual slots under app profile configuration:

- `app.profile.guard.before`: a synchronous guard hook that can prevent rendering the normal `AppLayout` and render guard content instead.
- `app.profile.guard.layout`: a layout-level guard hook that can render content over the normal `AppLayout`, such as a modal, full-screen surface, fade-in, or slide-in overlay. This hook can be asynchronous/client-driven.

For this task, implement only the `layout` slot. The `before` slot should be captured as a separate future draft task, not implemented here. The LingoCafe onboarding UI should use the layout-level guard surface. The guard component owns the incomplete-profile completion flow and signals completion back to the guard manager so the guard can be removed without depending on `/books`.

The configuration should stay close to the existing profile component model, but avoid forcing the onboarding bundle into every app route if possible. Prefer a guard configuration shape that can reference a lazily loaded guard component and a slot, such as a single `app.profile.guard` entry with `slot: "layout"` and a component loader. Planning should inspect the existing AppConfig/client boundary and choose the smallest implementation that preserves lazy loading or code splitting. If framework constraints make direct component references unavoidable, the plan should document the tradeoff before implementation.

The onboarding content can reuse the existing form behavior for the first version: own language, target language, target level, consent, and save through `/api/profile`. The existing profile and consent APIs remain responsible for persistence and event logging. The overlay should close only when the profile becomes complete. The guard should dispatch a `profile:complete` browser event after successful completion so the guard manager can re-evaluate profile completeness and remove or unload the component. `/books` should also be able to listen for that event and reload or continue. The `/books` route should stop hosting the form; when profile data is incomplete it should not load or display books and should show only a small note such as `Profile incomplete, can't show books now` until completion.

## Definition of Success

The incomplete-profile experience is reusable across LingoCafe routes, the books page is no longer the onboarding owner, and incomplete profiles do not trigger book catalog rendering. The app behaves the same on mobile and desktop because the onboarding overlay takes over the full viewport.

## Additional Context

- Current inline form lives in `src/app/(app)/(lingocafe)/books/page.tsx`.
- Current profile save endpoint is `src/app/api/profile/route.ts`.
- Current books API profile gate is in `src/app/api/(lingocafe)/lingocafe/_lib/reader.ts`; `loadReaderData()` already returns no books when profile is incomplete.
- The shared modal component lives in `src/42go/components/modal/Modal.tsx` and supports `size="full"`.
- `AppLayout` currently has no profile guard slots; this task should add the smallest useful layout guard-manager contract around the existing app layout.
- LingoCafe already defines `app.profile.schema`, `app.profile.items`, and `app.consent.items` in `src/config/lingocafe/config.ts`; the profile guard configuration should fit near that model.
- The backend profile completeness check already exists in the profile system through `isProfileComplete`; planning should inspect whether the frontend can reuse the same logic directly or needs a shared client-safe equivalent.

## Assumptions

- Onboarding should apply only to authenticated LingoCafe app routes, not every app in the multi-app shell.
- The full-screen overlay should be blocking while the profile is incomplete.
- The existing form fields and consent behavior are enough for the first onboarding feature.
- The guard should behave in relation to `AppLayout`, regardless of route; `/profile` should not receive special route-based exemption unless implementation discovers a hard blocker.
- The `/books` fallback note can be intentionally simple because the overlay is the primary completion path.
- The LingoCafe onboarding surface is intentionally custom and can later become a multi-step onboarding flow.
- The onboarding overlay should not include a `/profile` link in the first version; a small logout action at the bottom is acceptable.
- Guard completion should dispatch a `profile:complete` browser event so the guard manager and route content can react when the profile becomes complete.
- Profile completeness should be evaluated through the same required-field/profile-schema logic used by the existing profile system, not through a new LingoCafe-only definition.

## Constraints

- Do not duplicate the profile form logic in multiple routes.
- Do not make `/books` responsible for app-wide onboarding.
- Do not load or render the books catalog while the profile is incomplete.
- Do not apply LingoCafe-specific onboarding to non-LingoCafe apps.
- Do not bypass existing `/api/profile` validation, consent evidence, or profile lifecycle hooks.
- Do not use a bespoke fixed overlay when the shared modal system can support the behavior.
- Do not make the guard route-dependent; it belongs to the app layout/profile guard model.
- Do not add a `/profile` escape link to the onboarding overlay in this task.
- Do not implement `app.profile.guard.before` in this task.
- Do not add a separate profile or consent event logging path; use existing profile and consent APIs.

## Acceptance Criteria

- `AppLayout` supports an app-configured profile layout guard that can render over the normal layout.
- A follow-up draft task exists for the future synchronous `app.profile.guard.before` slot.
- The initial profile guard configuration is app-scoped and does not activate for non-LingoCafe apps.
- The guard configuration supports, or explicitly plans around, lazy loading/code splitting so LingoCafe onboarding is not unnecessarily bundled into unrelated app surfaces.
- A LingoCafe onboarding component or feature module exists outside `src/app/(app)/(lingocafe)/books/page.tsx`.
- The onboarding feature detects authenticated LingoCafe users with incomplete profile data.
- The onboarding feature evaluates completeness through shared profile-schema completeness logic or a client-safe equivalent of it.
- An incomplete profile opens a full-screen blocking overlay on desktop.
- An incomplete profile opens a full-screen blocking overlay on mobile.
- The overlay contains the current required onboarding form fields: own language, target language, target level, and configured consent items.
- Saving onboarding data posts through the existing profile/consent API path and relies on that path for persistence and event logging.
- The onboarding component can tell the guard manager that completion is done so the overlay disappears without route reload as the only mechanism.
- Successful onboarding dispatches a `profile:complete` browser event.
- After a successful save that completes the profile, the overlay disappears and route content can proceed.
- `/books` no longer renders the inline onboarding form.
- `/books` does not render or load the book catalog when profile data is incomplete.
- `/books` shows a simple incomplete-profile note when needed, with copy close to `Profile incomplete, can't show books now`.
- Profile completion emits `profile:complete` so `/books` can reload or continue once the profile becomes complete.
- The implementation validates that the guard is route-agnostic and tied to the app layout, not to `/books`.
- `npm run qa` passes after implementation.

## Dos

- Prefer reusing profile helpers such as `useProfileController`, `ProfileConsent`, and existing LingoCafe profile options.
- Keep onboarding app-specific under the LingoCafe route/component area unless refinement chooses a config-driven platform abstraction.
- Use the shared `Modal` component or extend it if full viewport behavior needs a small capability addition.
- Keep the `/books` empty/incomplete state plain and route-local.
- Preserve profile lifecycle event behavior and consent evidence metadata.
- Prefer fitting guard configuration under `app.profile.guard` beside the existing app profile schema/items.
- Keep the guard manager small; the LingoCafe guard component can own its own async profile check and onboarding UI.
- Prefer a single guard entry with an explicit `slot: "layout"` for this task; multiple guard arrays and `slot: "before"` can wait until there is a real need.
- Prefer lazy component loading for the LingoCafe onboarding guard if it fits the current AppConfig/client architecture cleanly.

## Don'ts

- Do not create a second profile persistence path.
- Do not hide validation errors silently.
- Do not leave the user able to dismiss the blocking overlay while the profile remains incomplete unless refinement explicitly asks for that.
- Do not fetch heavy book catalog data just to discover onboarding is required.
- Do not make onboarding depend on the books API if a profile-specific load is cleaner.
- Do not turn this into route-by-route wrapper work.
- Do not redirect users to `/profile` as the completion path for this first onboarding overlay.
- Do not implement multiple guard orchestration before the first single layout guard works.

## Open Questions

- Q: Where should the onboarding guard live?
  A: In or around `AppLayout` through app-configured profile guard slots, not in `/books` and not as a route-local wrapper for every page.
- Q: Should `/profile` itself be guarded by the onboarding overlay?
  A: The guard should behave in relation to `AppLayout`, regardless of route.
- Q: Should the overlay offer a `/profile` link?
  A: No. The first version should use a fully custom LingoCafe onboarding surface. A small logout action at the bottom is acceptable.
- Q: How should `/books` behave while profile is incomplete?
  A: `/books` should show a simple incomplete-profile note and not show books until a client-only completion event or state signal indicates that the profile is complete.
- Q: Which route should prove cross-route behavior?
  A: The behavior should be route-agnostic and tied to the app layout rather than validated as a route-specific feature.
- Q: Should this task implement both guard slots now?
  A: No. Implement only the `layout` slot for LingoCafe onboarding. Create a separate draft task for the future `guard.before` slot.
- Q: What guard configuration shape should be used?
  A: Prefer an app profile guard config with an explicit `slot: "layout"` and a lazily loadable component reference/loader, so onboarding code is loaded only when needed where practical. Planning must verify the best code-splitting shape for the current AppConfig boundary.
- Q: Should guard slots support one guard or multiple guards?
  A: Start with a single guard entry. Multiple ordered guards can be future work.
- Q: How should LingoCafe guard load profile completeness?
  A: Use the existing concept of profile completeness based on required profile schema fields. Provide or reuse frontend and backend utilities so the logic is defined in one place where practical.
- Q: What exact save metadata should onboarding use?
  A: The onboarding component should use existing profile and consent APIs. Those APIs remain responsible for storing data and logging their respective events.
- Q: What should the client completion signal look like?
  A: Dispatch `profile:complete`; the guard can re-evaluate profile completeness and remove or unload the component.
