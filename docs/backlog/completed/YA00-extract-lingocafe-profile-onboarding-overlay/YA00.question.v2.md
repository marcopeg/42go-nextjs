# Refinement Questions v2 - Extract LingoCafe Profile Onboarding Overlay

Answer inline under each question, then return to chat and say `next round`.

## Questions

### 1. Should this task implement both guard slots now?

The v1 answer defined two slots: `app.profile.guard.before` and `app.profile.guard.layout`. Should `YA00` implement both in the core `AppLayout` contract now, even if LingoCafe only uses `layout`, or should this task implement only the `layout` slot and leave `before` as a documented future extension?

Answer:

only the one "layout" so to be able to use it in LingoCafe's onboarding experience.
but generate the new draft task to implement the "guard.before" in the future.

### 2. What shape should guard configuration use?

Should `app.profile.guard` reference direct React components, similar to profile `type: "component"` blocks, or should it use named registry entries? For example:

```ts
guard: {
  layout: { component: LingocafeOnboardingGuard }
}
```

Answer:

what is the best approach at component loading level?
so that the onboarding gets loaded only when needed, instead of being a hard-reference in the config and therefore bloating the production bundle?

### 3. Should guard slots support one guard or multiple guards?

For this task, should `before` and `layout` each accept a single guard component, or an ordered list of guards where the first non-null render wins?

Answer:

it should be either. the config could also be `app.profile.guard.component = <div>...</div>` and `app.profile.guard.slot = "before|layout"`

### 4. How should the LingoCafe guard load profile completeness?

Should the guard use the generic `/api/profile` endpoint and `useProfileController`, add a lighter profile-status API, or reuse an existing LingoCafe endpoint? The task already says not to depend on the books API unless there is a strong reason.

Answer:

there is already the concept of profile completeness based on the profile schema required fields.
it should use the same.
there should be a frontend and backend utility like "isProfileComplete()" that does the logic in one place.

### 5. What exact save metadata should onboarding use?

The v1 answer described the component/guard contract but did not settle event/consent metadata. Should the LingoCafe onboarding save use `source: "lingocafe-onboarding"` and `method: "profile-onboarding-submit"`, or another exact pair?

Answer:

at the end of the onboardin expediente the component effectively should use profile and consent apis to store data.
it's those apis responsibility to log their respective events.

### 6. What should the client completion signal look like?

Should the guard manager expose an explicit callback/context such as `completeGuard()`, should the guard dispatch a browser event such as `profile:complete`, or should completion be handled only by shared React state inside the guard manager?

Answer:

i'd say dispatch a `profile:complete` is ok, so that the gate can re-evaluate the profile completenes and remove or unload the component
