# Document and Configure Lingocafe iOS Home Screen Launch Behavior

## Elevator's Pitch

When Lingocafe is installed on iOS Home Screen, users can log in, kill the app with a full swipe-up, reopen it, and end up on the public home page even though their Google session is still valid. Update the Lingocafe PWA launch configuration and document the relevant AppConfig `public.pwa` options so future apps do not repeat the mismatch.

## Business Gain

Improve trust and retention on mobile by making Lingocafe feel like a real installed app, with predictable reopen behavior.

Improve maintainability by documenting PWA launch configuration where app builders can find it, instead of leaving `startUrl` behavior tribal knowledge.

## Current State

In Lingocafe installed as iOS Home Screen app, after login and force-close/reopen, user lands on home page and appears logged out.

Operator confirmed that with Google login the user is still logged in. This means the problem is not Google session loss; the user is being routed to a public page despite having a valid session.

Quicklist app installed on Home Screen does not appear to have this behavior.

Repository evidence found during refinement:

- `src/app/manifest.ts` defaults manifest `start_url` to `/` when `config.public.pwa.startUrl` is not set.
- `src/config/quicklist/config.ts` defines `public.pwa.startUrl: "/quicklists"`.
- `src/config/lingocafe/config.ts` currently has `app.default.page: "/books"` but no `public.pwa` block, so its manifest start URL currently falls back to `/`.
- HU36 notes say LingoCafe was later aligned to land on `/books`.
- ACZ defines `config.public.pwa` as the authoritative source for iOS/PWA install metadata and explicitly includes `startUrl`.
- `contents/default/docs/multiple-apps/README.md` documents AppConfig basics but does not document `public.pwa`.
- `contents/default/docs` currently has no dedicated PWA page.

## Desired State

On iOS Home Screen relaunch, logged-in Lingocafe users should land in the Lingocafe app flow, not the public home page.

Lingocafe should declare the PWA configuration needed for installed-app launch behavior, including a `startUrl` that targets the app entry point.

The expected in-app destination for Lingocafe is `/books`.

Default documentation should explain the relevant `public.pwa` options, especially `startUrl`, `scope`, `display`, name fields, theme/background colors, status bar style, and icon requirements.

The documentation target is `contents/default/docs/pwa/README.md`, linked from `contents/default/docs/SIDEBAR.md`.

## Definition of Success

- Lingocafe's installed-app launch route is explicit and configuration-driven.
- Lingocafe reopen flow sends a valid Google-authenticated user to `/books`.
- The docs explain how `public.pwa.startUrl` controls installed-app launch behavior.
- The docs make clear that icons are required for full iOS/PWA polish, but Lingocafe custom icon work is out of scope for this task.
- Quicklist behavior remains correct and is used as comparison baseline.

## Additional Context

Operator hypothesis:
- issue may be tied to start point currently aiming to home page;
- possible solutions include:
  - auto-redirect from home when user is already authenticated;
  - app-specific custom launch path for Lingocafe such as `/books`.

The root cause is treated as launch routing/configuration, not Google session persistence, because the operator confirmed Google remains logged in after relaunch.

Operator does not have Lingocafe icon assets yet. Custom Lingocafe icon design should be handled by a future task.

Operator chose a config-only runtime fix for WP59. Do not add an authenticated `/` redirect in this task.

## Assumptions

- iOS Home Screen launch path and browser launch path may differ in practice, but manifest `start_url` is expected to shape installed-app launch.
- Existing Google auth/session persistence is working in the reported scenario.
- Lingocafe and Quicklist differ in config: Quicklist has `public.pwa.startUrl`, Lingocafe does not.
- The default fallback icon can be used or documented for now if Lingocafe-specific icons are not available.
