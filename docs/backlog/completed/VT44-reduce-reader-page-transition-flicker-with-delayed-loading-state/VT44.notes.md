# Execution Notes — Reduce LingoCafe Route Transition Flicker With Delayed Loading State
**Task**: ./VT44.task.refined.md
**Plan**: ./VT44.plan.md

## Decisions

- Keep the delayed-loading helper scoped to LingoCafe route pages for now.
- Centralize the timing values in one helper file so the product timings stay easy to tune.
- Add `useLingocafeRouteLoading` under the LingoCafe books components and export the shared delay and minimum-visible constants from there.
- Keep previously visible content onscreen during delayed loading by driving each page from `showLoading` instead of swapping immediately to the loading card.
- Suppress route-level error panels while delayed loading is active, then show the error normally once the delayed-loading window ends.
- Keep the core policy hook generic and localize policy-identity stabilization to the three LingoCafe books pages with module-level `Policy` constants.
- Treat the broken save flow as a client-side interaction bug, not a profile persistence bug, because authenticated API checks proved that `/api/profile` and `/api/lingocafe/books` still work for the affected LingoCafe user data.
- Make the reader route's top-level modal overlay non-interactive and transparent so a leaked dialog overlay cannot block later taps on `/profile` or `/books`.
- Record the final resolution plainly: the struggle was a misleading chain of UI symptoms around loading, header flicker, and dead save buttons, but the effective solution was to keep the delayed-loading helper local, avoid shared policy churn, and neutralize the reader route overlay that could block later interactions.
- Replace reader page-turn route pushes with state-driven reader navigation for normal previous/next clicks, swipe gestures, and table-of-contents page picks. The reader now keeps the mounted page visible, shallow-updates the URL through native history, and lets the existing delayed-loading helper decide whether to reveal the loading scaffold.
- Stabilize `/profile` with a module-level policy constant so save/dirty render changes do not hand `ProtectComponent` a fresh policy object and briefly re-enter the policy loading scaffold.
- Keep the reader top-right controls icon-only while preserving accessible labels through `aria-label`.
- Disable Radix open autofocus for the full-screen reader shell so it does not visually focus the `Aa` preferences trigger on page load.
- Keep existing reader content mounted during desktop page turns even if the delayed-loading timer expires; the URL now updates only after the target page fetch succeeds.
- Reduce the delayed-loading no-loader grace window from `800ms` to `400ms`.
- Reduce the scroll progress idle save debounce from `10000ms` to `4000ms`.

## Problems Encountered

- The documented task transition helper is still absent in this repository snapshot, so the task folder was moved to `wip` manually.
- LingoCafe has no persistent route-group layout above the books pages, so the implementation can only preserve already-visible content within mounted page components, not across fresh route mounts, without a larger refactor.
- The first draft of the helper tripped lint rules by using impure time reads during render and synchronous state updates inside effects; it was rewritten so timer callbacks own those transitions instead.
- Header flicker was still visible after the delayed-loading change because `ProtectComponent` briefly returned to its loading scaffold when inline `policy={{ ... }}` objects were recreated, which blanked mobile fixed surfaces and made their back buttons disappear.
- A broad attempted fix inside `useEvaluatePolicy` introduced a regression in profile flows, so it was rolled back in favor of a books-only stabilization change.
- Authenticated direct checks against the running app showed that Marco's LingoCafe profile row still exists, `/api/profile` GET works, `/api/profile` POST works, and `/api/lingocafe/books` returns a complete profile plus books; this shifted the diagnosis away from persistence and toward a client blocker layer.
- The final symptom stack was deceptive: save looked broken from both `/profile` and `/books`, but the data layer stayed healthy throughout. The real struggle was separating “dead UI” from “failed persistence” and not letting earlier header/loading hypotheses keep dragging the diagnosis off course.
- Operator reported that a top-bar flicker toward `Reading` was still visible after the first delayed-loading implementation. The remaining cause was that reader page turns still changed route first and loaded data second; this could remount or re-evaluate chrome before the next page data was available.

## Deviations From Plan

- `/books` now consumes the shared helper, but without a persistent LingoCafe layout it cannot preserve prior route content across a fresh remount. That broader cross-route smoothing would require a larger shell/layout refactor than VT44 planned.
- Step 7 is only partially complete in this session: `npm run qa` passed, but targeted manual transition checks were not run.
- The first attempt to solve header flicker changed shared policy behavior globally; that deviated too far from VT44 scope and was replaced with stable page-level policy constants in the affected LingoCafe routes only.
- The later save regression investigation also ruled out the backend: the actual route APIs were healthy under an authenticated LingoCafe session, so the fix stayed on the reader modal surface instead of the profile routes.
- Operator later confirmed that the issue now appears solved after the overlay fix, which supports the client-blocker diagnosis over any server-side save failure theory.
- Follow-up work expanded the reader fix from delayed loading to state-driven page turns. This is still scoped to the reader route and does not introduce a persistent LingoCafe route-group layout.
