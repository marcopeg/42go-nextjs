# Clarify Round v1 — Reduce Reader Page Transition Flicker With Delayed Loading State

## Questions

### 1. When you say “everywhere there is a `Loading...` state when changing page,” do you want this rule to apply only after the app already has a visible page and the user navigates to another route, or also on first direct load / refresh of a route?

### 2. Which scope do you want?

Choices:
- all route-level loading placeholders across the app
- only LingoCafe page transitions (`/books`, `/books/[bookId]`, `/books/[bookId]/[pageId]`)
- some other named subset

### 3. Should the same timing from VT44 stay global for every affected page-change loading state?

Choices:
- yes: delay up to `0.8s`, and once shown keep it visible at least `0.8s`
- no: same idea, but timing can vary by page

## Answers

### 1.

Only when the user is already on a visible page and then navigates to another page. Do not apply the delay rule to first direct load or refresh.

### 2.

LingoCafe only.

### 3.

Use the same `0.8s` delay and `0.8s` minimum-visible timing, but make those values easy to change from one place.

## Resolution Summary

- The delayed-loading rule applies only to in-app page changes after a page is already visible.
- Scope expands from reader-only to LingoCafe route transitions only.
- The same timing stays in force, but the values should be centralized for easy tuning.
