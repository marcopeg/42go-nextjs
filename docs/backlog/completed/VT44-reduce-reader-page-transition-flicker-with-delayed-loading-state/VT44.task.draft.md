# Reduce Reader Page Transition Flicker With Delayed Loading State

## Elevator's Pitch

Reduce flickering when moving across reader pages by delaying the `Loading page...` phase. Wait up to `0.8s` before showing loading, and if loading does appear, keep it visible for at least `0.8s`.

## Business Gain

The reader feels calmer and more deliberate instead of flashing a loading card every time the user moves to the next or previous page. Chuck Norris does not blink at every page turn. The page turn blinks at him.

## Current State

The LingoCafe reader route at `/books/[bookId]/[pageId]` currently flips into a loading state immediately when the page fetch starts.

In `src/app/(app)/(lingocafe)/books/[bookId]/[pageId]/page.tsx`, the reader page fetch sets `loading` to `true` right away when `apiHref` changes.

In `src/app/(app)/(lingocafe)/books/_components/BookReaderSurfaces.tsx`, both reader surfaces show `Loading page...` as soon as `loading` becomes true.

This means that moving across pages can produce a visible flicker even when the next page request resolves quickly.

## Desired State

When the user moves across reader pages, the UI should avoid jumping immediately into the loading placeholder.

The intended timing behavior is:

- wait up to `0.8s` before showing the loading phase
- if the request finishes before `0.8s`, do not show the loading phase at all
- if the request takes longer than `0.8s`, show the loading phase
- once the loading phase is shown, keep it visible for at least `0.8s`

This should apply only when moving across pages inside the reader flow.

The goal is to smooth page-to-page transitions such as:

- previous page navigation
- next page navigation
- swipe-driven page changes
- other internal reader page transitions that reuse the same route pattern

The goal is not to globally delay all loading states everywhere in the app.

## Definition of Success

Page-to-page navigation inside the reader no longer flickers into a loading card for fast requests. Slow requests still show a loading state, but only after the `0.8s` delay and for a minimum visible duration of `0.8s`. The behavior feels smoother while preserving correct page loading and error handling.

## Additional Context

This is specifically about the reader transition experience, not about the first-ever loading of the entire app or unrelated routes.

The current behavior likely comes from treating every `apiHref` change the same way, with an immediate `setLoading(true)` and immediate rendering of `ReaderState`.

The change should focus on transition UX when moving across pages, which is where the flicker is currently noticeable.

## Assumptions

- The problem is the immediate transition into the loading placeholder during reader page-to-page navigation.
- Keeping the old page visible during the initial delay window is acceptable and preferred if that avoids flicker.
- The `0.8s` delay and `0.8s` minimum visible duration should be treated as explicit product behavior for this task.
- This task should preserve the existing reader route, fetch contract, and progress tracking behavior.

## Constraints

- Apply the delayed-loading behavior only to reader page-to-page transitions.
- Do not broadly change loading behavior across the whole app.
- Keep the existing `/books/[bookId]/[pageId]` route and page-fetch model.
- Preserve current previous/next navigation behavior and swipe navigation behavior.
- Preserve existing error handling unless a small loading-state adjustment is needed to support the new timing rule.
- Do not regress progress restore, scroll tracking, or page navigation correctness.

## Acceptance Criteria

- [ ] Reader page-to-page transitions no longer show `Loading page...` immediately.
- [ ] The loading state is delayed by up to `0.8s` when moving across reader pages.
- [ ] If the page request resolves before `0.8s`, no loading phase is shown.
- [ ] If the page request takes longer than `0.8s`, the loading phase is shown.
- [ ] Once shown, the loading phase remains visible for at least `0.8s`.
- [ ] The delayed-loading behavior applies only when moving across pages in the reader flow.
- [ ] The change does not break previous-page navigation.
- [ ] The change does not break next-page navigation.
- [ ] The change does not break swipe-driven page changes.
- [ ] The change does not break successful page data loading.
- [ ] The change does not break reader error-state handling.
- [ ] The change does not regress progress restore or progress tracking behavior.

## Dos

- Do scope the change tightly to reader page transitions.
- Do encode the `0.8s` delay and `0.8s` minimum visible loading duration explicitly.
- Do preserve the existing navigation behavior while smoothing the visual transition.
- Do verify both desktop and mobile reader surfaces because both consume the same loading state.

## Don'ts

- Don't delay every loading state in the app.
- Don't keep the loading state flashing for very fast requests.
- Don't remove the loading state entirely for genuinely slow requests.
- Don't let this task spill into reader redesign or routing changes.
- Don't regress previous/next or swipe navigation while chasing smoother visuals.

## Open Questions

- Should the delayed-loading rule apply only to in-reader navigation after the first page is already visible, or also to the very first load of the reader route?
- When a slow request ends in error after the loading placeholder has appeared, should the error replace loading immediately or still respect the minimum visible duration?
