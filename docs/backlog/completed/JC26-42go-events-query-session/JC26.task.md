# 42go Events Query Session

## Elevator's Pitch

Add a local `42go events query session` analytics command that clusters raw event rows into user sessions and stores app-scoped reusable session caches.

## Business Gain

The operator can inspect real usage sessions from the local event archive without hammering production PostgreSQL. Sessions make event streams readable: one user, one time cluster, one clean chunk. Chuck Norris does not count events one by one when a session can roundhouse-kick them into shape.

## Current State

- `42go events pull` exports core `events.events` rows into local monthly CSV and Parquet files under `.local/42go-events/events/`.
- `42go events query stats` reads local Parquet files with DuckDB and prints high-level archive statistics.
- `42go events query users growth` is a cached analytics command with app-scoped aggregate state under `.local/42go-stats/{app-id}/`.
- Raw event rows include `created_at`, `id`, `app_id`, `user_id`, `event_at`, `name`, `data`, and `meta`; `data` and `meta` are JSON strings in local files.
- Events without `user_id` are not expected in normal data and should be ignored by the session query if they appear.
- No command currently produces isolated chronological sessions from the event stream.

## Desired State

- A discoverable CLI command exists at `42go events query session`.
- The command reads local Parquet files from the existing `.local/42go-events` archive layout and never queries production PostgreSQL.
- The command segregates cache/results by app the same way as `42go events query users`.
- Within each app, the command clusters events into sessions by `user_id` using `event_at` ordering.
- Sessions are not split by device/browser.
- Sessions include every event name for the same user.
- Each session belongs to exactly one user and never mixes events from multiple users.
- A new session starts when the gap between two consecutive events for the same app/user is greater than 1 hour.
- Each session item contains: `id`, `start_at`, `end_at`, `duration`, `user_id`, and `events`.
- `id` is the id of the first event in the session.
- `start_at` is the timestamp of the first event in the session.
- `end_at` is the timestamp of the last event in the session.
- `duration` is derived from `end_at - start_at` and stored as seconds in data/cache/JSON output.
- Console output formats duration as human-readable text.
- `events` contains the ordered list of raw events that belong to the session.
- The command stores local processed-session cache data as flat app-scoped Parquet tables under `.local/42go-stats/{app-id}/`, separate from raw exporter state.
- Cache state is independent by `app_id`; sessions for different apps are never mixed.
- Within an app cache, sessions are grouped by `user_id` plus the time clustering rule.
- All event names for a user participate in session clustering.
- Normal runs process newly available raw events without rebuilding all sessions when possible.
- Normal runs may edit or replace the last cached session for an app/user when new events are logically part of it, because a late run can extend the open tail session.
- `--reset` removes the session cache and reprocesses all available local Parquet events.
- `--user-id xxx` filters visible terminal or JSON output by user without corrupting or narrowing the underlying app-scoped cache.
- `--app-id xxx` filters visible output by app. The command may still refresh all app caches by default, but output must only show the requested app when this flag is present.
- `--user-id`, `--app-id`, and `--limit` control visible output only; they do not narrow cache processing.
- Normal processing computes all available events into local aggregate cache files.
- `--reset` forces a full reflow from all available events into a fresh local aggregate cache.
- Human-readable console output is the default and shows session summaries, not full event arrays.
- JSON output includes full `events[]` arrays for each returned session.
- Local cache storage format is an implementation choice, but should follow the best fit from existing event query commands.
- Visible output is limited to the last 20 sessions by default.
- A `--limit <number>` flag can request an arbitrary number of visible sessions, for example `--limit 100`.

## Definition of Success

- A first run builds deterministic app/user session caches from all available local Parquet data.
- A later run with no new Parquet data returns from cache.
- A later run after `42go events pull` processes new events and correctly extends the previous last session when the next event arrives within 1 hour.
- A later run after `42go events pull` creates a new session when the next event arrives more than 1 hour after the previous event for the same app/user.
- `--reset` deletes/rebuilds the session cache from scratch.
- Output filters by `--user-id` and `--app-id` affect visible results, not cache isolation.
- `--limit` also affects visible results only.
- Output returns the most recent sessions first or otherwise clearly documents ordering, with only the last 20 visible by default unless `--limit` is provided.
- Session data remains app-scoped and user-scoped, and raw event archive files remain read-only.

## Additional Context

- Use `event_at` for session clustering. `created_at` and `id` may be used only as stable processing/cursor metadata.
- The session gap threshold is exactly 1 hour: if the gap is greater than 1 hour, start a new session; if the gap is equal to or less than 1 hour, keep the event in the current session.
- Cache shape is an implementation choice. Prefer the best fit after inspecting existing event query commands and their aggregate cache formats.
- Because every session includes an `events[]` list, JSON output is the full-fidelity format. Text output summarizes sessions and includes event counts while JSON returns full event arrays.
- Use the existing `users_growth` command as the cache behavior reference: source file fingerprints, schema version, app-scoped storage, safe app-id path validation, `--reset`, and deterministic tests.
- The command should update `.agents/skills/42go-cli/SKILL.md` and the events export skill docs so agents can discover the new query.

## Assumptions

- This task belongs to the existing pipx-installable Python CLI under `cli/`.
- The command path should be singular `42go events query session`, matching the operator request.
- Events are segregated by `app_id` following the `query users` app-cache pattern, then grouped by `user_id`; events without `user_id` are ignored because they are not expected in normal data.
- Device/browser metadata does not affect session grouping.
- Every event name for the same user is included in clustering for now.
- Event ordering for clustering is by `event_at`, with a deterministic tie-breaker such as `created_at` then `id`.
- The visible `--app-id` flag should coexist with existing `--app` behavior from `users growth`; implementation should either use `--app-id` exactly or document an intentional alias if both are supported.

## Constraints

- Do not query production PostgreSQL for this metric command.
- Do not mutate raw CSV or Parquet archive files.
- Do not overwrite `events/state.json`, `manifest.jsonl`, or other exporter-owned state.
- Do not commit `.local` cache data.
- Keep analytics state separate from exporter state.
- Keep app session caches isolated by `app_id`.
- Sanitize `app_id` before using it in cache paths.
- Keep command help complete and agent-navigable.
- Preserve complete raw event data inside each session item, including event id, timestamps, name, data, and meta where available.
- Do not hide the command behind Make only.

## Acceptance Criteria

- A CLI command is added at `42go events query session` and appears in `42go --help` / nested `--help` output.
- The command accepts `--reset`.
- The command accepts `--user-id` and filters visible output to sessions for that user.
- The command accepts `--app-id` and filters visible output to sessions for that app.
- The command accepts `--limit` and defaults to showing the last 20 sessions when no limit is passed.
- `--user-id`, `--app-id`, and `--limit` filter/limit output after cache processing; cache processing still covers all available app/user events.
- The command reads local Parquet files from the existing `.local/42go-events/events/parquet/` archive layout.
- The command stores local session cache data under app-scoped flat files like `.local/42go-stats/{app-id}/query_session_sessions.parquet` and `.local/42go-stats/{app-id}/query_session_events.parquet`.
- Cache files and state are isolated by `app_id`.
- The command segregates by app, then clusters events by user using `event_at`, with a deterministic tie-breaker for equal timestamps.
- Sessions are not split by device/browser.
- Sessions are user-scoped and include all event names for that user.
- Events with missing `user_id` are excluded from session clustering.
- A gap greater than 1 hour starts a new session.
- A gap equal to or less than 1 hour keeps the event in the current session.
- Each output session item includes `id`, `start_at`, `end_at`, `duration`, `user_id`, and `events`.
- Machine-readable session data stores `duration` as seconds; human-readable console output displays a formatted duration.
- Console output shows a summary for each returned session and does not print full event arrays by default.
- JSON output includes full event arrays for returned sessions.
- Local aggregate cache files are Parquet tables stored flat under `.local/42go-stats/{app-id}/`.
- Visible output is limited to the last 20 sessions by default, while `--limit <number>` can request an arbitrary count such as 100.
- The session `id` equals the first event id in that session.
- The `events` array is ordered chronologically and contains the raw events in that session.
- Normal runs with unchanged source files read from cache.
- Normal runs with newly pulled events update caches and can extend or replace the last session for an app/user when new events fall within the 1-hour threshold.
- `--reset` removes the existing local session cache and recomputes from all available Parquet files.
- `--reset` is the only flag that forces a full cache reflow from all available events.
- Focused tests cover first run, cached no-new-data run, reset rebuild, session split after more than 1 hour, session continuation at 1 hour or less, last-session extension after incremental input, `--user-id` output filtering, and `--app-id` output filtering.
- CLI docs or skill docs define the session threshold, ordering rule, cache path, reset behavior, and output fields.

## Dos

- Use DuckDB over local Parquet for reading raw events.
- Use `event_at` for session time.
- Make cache writes idempotent.
- Store metadata such as processed raw file fingerprints, metric schema version, source event range, and last processed event key where useful.
- Recompute a small tail window or the last session per app/user so incremental processing handles events that belong to the previous final session.
- Prefer structured data handling over ad hoc string parsing.
- Keep JSON output available for full `events[]`; keep console output compact with summaries.
- Choose the local cache storage format pragmatically from the existing event-query command patterns.
- Keep raw export archives read-only.
- Update CLI help and agent skill docs.

## Don'ts

- Do not run heavy analytics against production PostgreSQL.
- Do not couple session cache state to the exporter cursor as if it were the same contract.
- Do not mix multiple app IDs into one cache file unless the file lives under an explicit app-scoped directory and the schema still records app_id.
- Do not drop event payloads from the `events[]` array.
- Do not treat `--user-id` as a cache partition unless implementation explicitly proves it will not duplicate or fragment app session state.
- Do not let `--user-id`, `--app-id`, or `--limit` skip aggregate cache processing for other available data.
- Do not silently use a different inactivity threshold than 1 hour.
