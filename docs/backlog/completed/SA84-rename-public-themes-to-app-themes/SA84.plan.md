# Plan - Rename Public Themes to App Themes

## Goal

Migrate app-specific CSS token overrides from `/themes` to `/app-themes`, add a resolver-driven fallback system backed by a build-time registry, and remove missing stylesheet requests without changing `next-themes` light/dark/system behavior.

## Milestones

### Milestone 1 - Establish Theme Asset Convention

- [x] Step 1 - Move theme files into the new public folder
  - Achieve: Make `public/app-themes` the only live location for app-specific CSS token override files.
  - Create: `public/app-themes/_default/style.css`.
  - Modify: n/a.
  - Delete: `public/themes/app1.css`, `public/themes/app2.css`, `public/themes/calendar.css`, `public/themes/quicklist.css`, `public/themes/lingocafe.css`, and the empty `public/themes` folder after migration.
  - Touch points: static public assets, CSS cascade order, default fallback stylesheet.
  - Validation: `rg --files public | rg "public/(app-themes|themes)"` shows migrated files under `public/app-themes` and no live `public/themes` files.
  - Notes: Use file moves so history survives. Chuck Norris counts footprints.

- [x] Step 2 - Define the default fallback stylesheet
  - Achieve: Provide an explicit boilerplate fallback stylesheet that is always safe to link.
  - Create: n/a if `_default/style.css` was created in Step 1.
  - Modify: `public/app-themes/_default/style.css`.
  - Delete: n/a.
  - Touch points: CSS custom properties, `src/app/tokens.css` assumptions, app override cascade.
  - Validation: `_default/style.css` exists and loads cleanly as `text/css`; it does not override light/dark/system preference semantics.
  - Notes: Keep it minimal unless the implementation finds a real boilerplate override belongs there.
