# Rename Public Themes to App Themes

## Elevator's Pitch

Rename the app-specific theme override folder from `public/themes` to `public/app-themes` and resolve custom app CSS with the same convention-first, default-fallback approach designed for `public/app-icons`.

## Business Gain

App-specific visual identity becomes predictable across the multi-app platform. Every app can have its own CSS token overrides without producing missing stylesheet noise. Chuck Norris does not let a 404 stylesheet throw the first punch.

## Current State

App-specific CSS token overrides currently live under `public/themes/{app-id}.css`.

Existing theme files include:

- `public/themes/app1.css`
- `public/themes/app2.css`
- `public/themes/calendar.css`
- `public/themes/quicklist.css`
- `public/themes/lingocafe.css`

`src/42go/config/InjectAppID.tsx` links `/themes/${id}.css` for every non-default app ID. The current implementation does not check whether that stylesheet exists before linking it.

`src/42go/config/InjectAppName.tsx` appears to duplicate the same theme injection behavior and should be audited, updated, or removed if obsolete.

Documentation and comments reference the old folder:

- `src/app/tokens.css` says token overrides live in `public/themes/{config}.css`.
- `contents/default/docs/multiple-apps/README.md` documents `@/public/themes/{app-id}.css`.
- Historical backlog notes mention `public/themes`, especially `AAF`.

The current behavior can produce missing CSS requests or strict MIME warnings when an app has no matching theme file.

## Desired State

Theme overrides live under `public/app-themes`.

The canonical convention is:

```text
public/app-themes/_default/style.css
public/app-themes/<app-id>/style.css
```

The `_default/style.css` file is the boilerplate fallback theme override. It may be minimal, but it must exist so fallback behavior is explicit.

Each app may provide `public/app-themes/<app-id>/style.css`. If the app-specific file exists, the root layout links `_default/style.css` first and the app file second so app overrides win. If the app-specific file is missing, the root layout links only `_default/style.css`.

A shared resolver should own this behavior:

```ts
resolveAppThemeStylesheet(appId) => {
  defaultHref: "/app-themes/_default/style.css",
  appHref?: "/app-themes/<app-id>/style.css",
}
```

The resolver should use a generated or static build-time registry of existing files, mirroring the `public/app-icons` direction. Rendering should not do request-time filesystem checks.

No backward-compatible `/themes` path, alias, symlink, or legacy config should remain. This is an early project; the task should migrate the app, not drag old baggage behind it.

## Definition of Success

Every configured app either loads its own CSS override from `public/app-themes/<app-id>/style.css` or cleanly falls back to `public/app-themes/_default/style.css` without requesting a missing stylesheet.

Existing app theme files, documentation, comments, and injection logic are migrated from `/themes` to `/app-themes`.

## Additional Context

This task follows the same architectural direction as `RO05`: convention-first public assets, shared resolver, complete default fallback, no request-time filesystem checks, and no legacy compatibility layer.

`RO05` has already implemented the useful precedent for this task: generated public asset registry, resolver entry points, build integration, and static fallback assets. `SA84` should mirror that shape for CSS theme files instead of inventing a second asset-resolution style.

The existing `theme.default` AppConfig field still controls light/dark/system behavior through `next-themes`. This task is only about app-specific CSS token override files.

## Assumptions

The current files under `public/themes` are still the intended app-specific CSS overrides.

The base tokens in `src/app/tokens.css` remain the global default token set. `public/app-themes/_default/style.css` can be a minimal explicit fallback layer unless implementation decides it should hold boilerplate brand overrides.

The app ID resolved from request headers, hostname, subdomain, or environment remains the key for app-specific theme lookup.

## Constraints

Do not preserve backward-compatible `/themes` behavior.

Do not add per-request filesystem checks to decide whether a CSS file exists.

Do not change `next-themes` light/dark/system preference semantics.

Update existing app assets, code references, comments, and user-facing docs in the same implementation.

Create and maintain `SA84.migration.md` and `SA84.test.md` as sibling task artifacts.

Never modify `CHANGELOG.md`.

## Acceptance Criteria

- [ ] `public/app-themes/_default/style.css` exists as the explicit fallback stylesheet.
- [ ] Existing app override files are moved from `public/themes/*.css` to `public/app-themes/<app-id>/style.css`.
- [ ] The old `public/themes` app override folder is removed after migration.
- [ ] A shared app-theme stylesheet resolver returns the default stylesheet plus an app-specific stylesheet only when the app-specific file exists in the build-time registry.
- [ ] `InjectAppID` or its replacement links `/app-themes/_default/style.css` for resolved apps and only links `/app-themes/<app-id>/style.css` when that file exists.
- [ ] Missing app-specific theme files do not produce browser 404, MIME, or console stylesheet warnings.
- [ ] `InjectAppName.tsx` is audited and either updated to the new resolver or removed if obsolete.
- [ ] `src/app/tokens.css` comments refer to `public/app-themes`.
- [ ] `contents/default/docs/multiple-apps/README.md` documents `public/app-themes/{app-id}/style.css`.
- [ ] Any other live documentation references discovered by `rg "public/themes|/themes/"` are updated.
- [ ] Existing app configs and app material are checked for theme path assumptions and updated if needed.
- [ ] `SA84.migration.md` explains how to move projects from `public/themes` to `public/app-themes`.
- [ ] `SA84.test.md` gives step-by-step manual verification for default fallback, app-specific overrides, and missing-file behavior.
- [ ] `npm run qa` passes.

## Dos

- Do keep the implementation aligned with request-based app resolution.
- Do use `public/app-themes` as the canonical folder name.
- Do use a resolver instead of hardcoding stylesheet links directly in JSX.
- Do load the default stylesheet before the app-specific stylesheet.
- Do update docs and comments in the same task.
- Do treat missing app-specific CSS as normal fallback behavior.

## Don'ts

- Don't keep `/themes` as a compatibility path.
- Don't add symlinks from `public/themes` to `public/app-themes`.
- Don't link app-specific CSS when the file is not known to exist.
- Don't perform request-time filesystem checks.
- Don't change light/dark/system theme preference behavior.
- Don't bury migration or manual test instructions only in execution notes.
