# Plan — Show Read Now or Continue Reading on Book Details

## Goal

Extend the existing LingoCafe book detail API and detail page so the primary action is derived from the authenticated user's reading state: "Read now" links to the first book page, while "Continue reading" links to the saved progress page with `progress_bps` in the URL. Do not build the read page UI.

## Milestones

### Milestone 1 — Settle the Action Contract

- [x] Step 1 — Define the response shape
  - Achieve: Define a small, explicit reading-action payload that can represent start, resume, and unavailable states without making the client infer database state.
  - Create: Nothing expected.
  - Modify: `src/app/(app)/(lingocafe)/books/_components/book-types.ts`.
  - Delete: Nothing expected.
  - Touch points: Add a type such as `ReaderBookAction` or `ReaderBookReadingAction` and attach it to `ReaderBookInfo`; include fields like `kind`, `label`, `href`, `bookId`, `pageId`, and `progressBps`.
  - Validation: TypeScript should force the detail page and content component to handle the action shape explicitly.
  - Notes: Use `progress_bps` as the query parameter name because the task asks for that URL parameter directly. Use a read-route href that extends `/books/[bookId]` with the page id, aligning with `ZE11`; do not create the route here.

- [x] Step 2 — Preserve the existing detail payload
  - Achieve: Keep the current book metadata, cover, tags, and description response compatible while adding the action payload.
  - Create: Nothing expected.
  - Modify: `src/app/api/(lingocafe)/lingocafe/_lib/reader.ts`, `src/app/api/(lingocafe)/lingocafe/books/[bookId]/route.ts`.
  - Delete: Nothing expected.
  - Touch points: `loadBookInfo`, `mapBookInfo`, `BookInfoResponse`, and the existing `{ book }` JSON response.
  - Validation: Existing `/api/lingocafe/books/[bookId]` consumers should still receive `book` with the previous fields plus the new action field.
  - Notes: Keep `book-info` event tracking unchanged.
