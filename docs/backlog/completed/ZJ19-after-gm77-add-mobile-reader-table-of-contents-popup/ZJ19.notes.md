# Execution Notes — After GM77, Add Mobile Reader Table of Contents Popup
**Task**: ./ZJ19.task.refined.md
**Plan**: ./ZJ19.plan.md

## Decisions

- Move the reader TOC into its own feature component instead of expanding `BookReaderSurfaces.tsx` further.
- Replace the desktop left contents rail with top-bar navigation plus a `Modal`-driven TOC so mobile and desktop share one surface model.
- Keep the TOC list backed by the existing `bookPage.pages` payload and reuse the old current-page marker pattern.
- Revert the shared `Modal` back to the original GM77 portal/content wiring after the temporary `DialogContent` refactor did not solve the reader bug.
- Keep only one active reader overlay mounted at a time (`preferences` or `toc`) so the page does not hold multiple live dialog roots during debugging.
- Treat the remaining issue as iOS/WebKit-specific until proved otherwise because desktop and desktop touch simulation both passed while real iPhone Chrome failed.
- Replace the reader page's bespoke fixed overlay shell with an always-open shared `Modal` panel. The reader should be an overlay only through the same core primitive as its child panels.
- Restore independent reader child overlay state after moving the reader itself into `Modal`; the core modal stack now owns layering instead of the feature forcing a single active child slot.
- Put the TOC trigger at the far right of the reader header and show the settings trigger label on desktop while keeping the compact Aa icon on mobile.
- Add `/demo-modal` examples for modal-in-modal and stacked panel flows so future modal work can visually exercise nested surfaces.
- Updated the `42go-ui-modal` skill so future agents know how to use nested `Modal` stacks, full-screen overlay route shells, and the `/demo-modal` nested demos.

## Implementation Attempts And Diagnoses

1. Initial TOC implementation
   - Change: Added `BookReaderTableOfContents`, added a new `isTableOfContentsOpen` state, removed the old desktop left rail, added a new TOC trigger in the reader headers, and rendered the TOC modal from the reader surfaces.
   - Why it seemed reasonable: The TOC should behave like the already-existing reader preferences modal and reuse the shared `Modal`.
   - What was wrong: The reader page renders both mobile and desktop surfaces at the same time, so a portal-based overlay is risky when mounted inside those split surfaces.

2. Diagnosis: duplicate TOC modal mounts from hidden surfaces
   - Change: Added breakpoint gating to the TOC component, mirroring the existing preferences panel behavior, so only one TOC modal would render per viewport.
   - Why it seemed reasonable: Hidden parents do not stop Radix portals from rendering, so duplicate modal instances were a credible cause.
   - Outcome: Plausible and likely part of the problem, but not sufficient. The real device still failed.

3. Mobile header trigger placement fix
   - Change: Moved the TOC trigger to the right side of the mobile header and tightened the hit target to an icon-sized button.
   - Why it seemed reasonable: The screenshot showed the TOC icon on the left, and the user explicitly wanted it on the right.
   - Outcome: Cosmetic improvement only. It did not explain the invisible blocker.

4. Page-level TOC singleton
   - Change: Moved `BookReaderTableOfContents` out of `BookReaderSurfaces.tsx` and mounted it once from the reader page.
   - Why it seemed reasonable: It removes split-surface portal duplication for the TOC completely.
   - Outcome: Good structural cleanup, but the user then reported that the existing font customization modal also failed, which proved the bug was not TOC-specific.

5. Incorrect diagnosis: “two modal triggers on one page”
   - Change: Concluded that multiple dialog roots on the reader page were the problem and started reshaping the page around that assumption.
   - Why it seemed reasonable: Both TOC and preferences were modal consumers, and the page had become more complex after the TOC work.
   - Why it was incomplete: The operator correctly pointed out that `/demo-modal` hosts multiple modals successfully. Multiple triggers alone are not the bug.

6. Temporary core modal refactor to `DialogContent`
   - Change: Reworked `src/42go/components/modal/Modal.tsx` to use the shared `DialogContent` wrapper instead of the original `DialogPortal` + `DialogOverlay` + `DialogPrimitive.Content` composition.
   - Why it seemed reasonable: The user explicitly told the agent to stop freestyling the modal and respect the shared modal path.
   - Why it was later reverted: This was a speculative core refactor, not a reader-specific fix, and it did not resolve the issue. GM77 had already browser-verified the original core modal path.

7. Page-level preferences singleton
   - Change: Moved `BookReaderPreferencesPanel` out of `BookReaderSurfaces.tsx` and mounted it once from the reader page, matching the TOC singleton approach.
   - Why it seemed reasonable: The preferences modal shared the same split-surface risk pattern as the TOC. Keeping modal consumers out of the duplicated reader surfaces is cleaner.
   - Outcome: Structural simplification kept because it reduces moving parts and removes another split-surface portal consumer.

8. Single active overlay slot
   - Change: Replaced separate reader overlay booleans with one page-level `activeOverlay: "preferences" | "toc" | null` so only one modal root is mounted at a time.
   - Why it seemed reasonable: If iOS WebKit was sensitive to multiple live dialog roots on a fixed fullscreen page, this would eliminate that class of issue entirely.
   - Outcome: Kept as a cleanup because it simplifies state and guarantees one active reader overlay at a time, but it was not accepted as the full explanation once the operator cited `/demo-modal`.

9. WebKit/Radix diagnosis from external clue
   - Change: Reviewed a Radix issue about overlay/content sibling structure and `RemoveScroll` tree behavior.
   - Why it seemed reasonable: The bug only reproduced on real iPhone Chrome/WebKit, not desktop or desktop touch simulation.
   - Outcome: Led to the next modal-core change.

10. Overlay/content tree fix in core `Modal`
   - Change: Nested `DialogPrimitive.Content` inside `DialogOverlay` and changed the mobile content shape from `fixed inset-0 h-[100dvh]` to a `relative min-h-full w-full` layer inside the fixed overlay, while keeping desktop `md:fixed` positioning.
   - Why it seemed reasonable: The demo page proved “multiple modals” was not enough to explain the bug; the reader page is special because it sits inside a fixed fullscreen shell, and iOS WebKit is known to be sensitive to full-screen/fixed/scroll-lock interactions.
   - Current status: This is the strongest remaining diagnosis, but it still requires live iPhone verification.

11. Reader shell migrated to core `Modal`
   - Change: Wrapped the reading route content in an always-open `Modal presentation="panel" anchor="right" size="full"` and converted the mobile/desktop reader surfaces from custom `fixed inset-0` overlays into normal flex children that fill the modal body.
   - Why it is stronger: It removes the architectural mismatch the operator called out. The reader is no longer a separate full-screen overlay with other modal portals layered above it; it is now the parent modal layer.
   - Current status: Implemented. Needs QA and real iPhone confirmation.

12. Core modal stack levels for nested panels
   - Change: Added a small stack context inside `Modal` so nested modals/panels receive increasing overlay/content z-index levels. Reader preferences and TOC state are independent again and are rendered as children of the reader modal.
   - Why it is stronger: Multiple triggers are not the bug. Unmanaged overlay ownership is. The core primitive now handles stacked modal levels while the reader feature keeps only product state.
   - Current status: Implemented. Needs QA and real iPhone confirmation.
