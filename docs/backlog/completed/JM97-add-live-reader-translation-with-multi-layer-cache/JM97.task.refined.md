# Add Live Reader Translation With Multi-Layer Cache

## Elevator's Pitch

Add live sentence translation to the LingoCafe reader. When the user taps or clicks a sentence in a book page, the sentence should highlight and a contextual popover should show the translation into the user's own language. The first translation may hit Google Translate through an API-key-based request, but repeated translations should use database, server-memory, and client-local caches so the same sentence can later resolve quickly, including across devices after persistence.

## Business Gain

This turns the reader into an actual learning tool instead of just a content viewer. Users can inspect meaning in context without leaving the page, breaking reading flow, or copying text into another app. Chuck Norris does not open a second app to translate one sentence. The sentence translates itself before it hits the mat.

## Current State

The LingoCafe reader already renders book-page markdown through `src/app/(app)/(lingocafe)/books/_components/BookPageReader.tsx` and reader surfaces through `src/app/(app)/(lingocafe)/books/_components/BookReaderSurfaces.tsx`.

The reader content path is route-local and intentionally narrow: `ReactMarkdown` allows headings, paragraphs, `em`, and `strong`, and skips HTML. The current rendered markdown does not expose sentence-level wrappers or a pre-tokenized sentence model.

The reader surface already contains browser text-selection awareness through `hasActiveTextSelection()`, but there is no sentence-level click/tap interaction, no translation endpoint, no translation cache, no user-level `feature_flags` storage, and no feature gate dedicated to translation.

LingoCafe profile data is now stored in `auth.users.profile` through the shared `/api/profile` path. The current LingoCafe profile schema in `src/config/lingocafe/profile-options.ts` requires `ownLang`, `targetLang`, and `targetLevel`, and sets `additionalProperties: false`. The LingoCafe app currently enables `page:books`, `api:lingocafe`, and `api:profile`.

The existing auth migration `knex/migrations/20240320_auth.js` creates `auth.users` with `profile` and `consent` JSONB columns, but it does not yet include a `feature_flags` JSONB column.

The current LingoCafe seed data gives Jane Doe (`jane.doe@example.com`) a complete LingoCafe profile and consent. John Doe exists as a LingoCafe test user but does not currently receive the complete profile in `knex/seeds/20260427151000.lingocafe.data.js`.

## Desired State

Inside the LingoCafe reader, clicking or tapping a sentence should identify the full sentence from stop to stop, highlight that sentence, and open a translation popover anchored near it.

The popover should choose above or below placement based on available viewport space and content height so that the original sentence and the translation remain visible together. A second click anywhere outside the active sentence/popover should close the popover and clear the highlight.

Translation should use Google Cloud Translation Basic v2 REST with an API key provided by environment configuration. The Google API key must stay on the server and must not be exposed to the browser bundle. The provider integration should be modular and interchangeable, using an abstraction or factory-style boundary so a future provider swap does not rewrite the reader, API route, or cache orchestration.

The feature must be available only when the current authenticated user's `auth.users.feature_flags` JSONB contains `translate: true`. Missing `feature_flags`, a missing `translate` key, or any value other than boolean `true` means translation is unavailable. Both frontend and backend must enforce that rule.

`translate` must not live in `auth.users.profile`, must not be part of the LingoCafe profile schema, and must not be editable in the profile UI. Normal `/api/profile` saves must not overwrite or mutate `auth.users.feature_flags`.

The local clean-rebuild path should update the existing auth migration in place to add `feature_flags jsonb default null` to `auth.users`. The implementation should also provide a task-folder manual production SQL patch named `migrate.sql`, following the prior ZO85 pattern, so the operator can alter the deployed `auth.users` table manually.

Translation cache behavior must be explicit across layers and must use this lookup order:

- client local-storage cache first, with the same hash basis, last-used tracking, and a 1 MB cap
- API server in-memory cache second, with hash keys, last-used tracking, environment-configured maximum size in MB, defaulting to 20 MB
- persistent database cache third, in the `lingocafe` schema, keyed by a stable hash of source text, source language, and target language
- Google Cloud Translation Basic v2 REST last

After a sentence has been translated once and stored in the persistent cache, later requests for the same sentence should resolve from cache when possible, including on another device after the database cache has been populated.

## Definition of Success

A LingoCafe user with `auth.users.feature_flags.translate = true` can tap or click a sentence in the reader, see the full sentence highlighted, get a well-placed translation popover in their own language, close it with a second outside click, and benefit from cache hits that make repeated translations substantially faster.

Unauthorized users never get the feature, even if they try to call the API directly. Cache behavior is observable enough during development or verification to distinguish client, server-memory, database, and upstream translation paths.

## Additional Context

The source language for a translation request is expected to be the book language from the reader payload, while the target language is expected to be the user's `ownLang` profile value.

The task introduces a new LingoCafe persistence concern because sentence translations should survive across sessions and devices. This likely requires a Knex migration for a `lingocafe` translation-cache table.

The task also changes core auth-user storage by adding `auth.users.feature_flags`. Local development may update the existing auth migration directly because this repository already treats local database rebuilds as destructive during this project phase. Production must use a separate manual SQL patch stored in the task folder.

The current reader API and event helpers live under `src/app/api/(lingocafe)/lingocafe/_lib/reader.ts`; prior reader work kept page loading, progress, and event writes route-local.

The operator explicitly wants the remote translation token to be an API key, not OAuth user tokens or a service-account session flow. The selected provider for this task is Google Cloud Translation Basic v2 REST, but the implementation should hide the concrete provider behind a small server-side abstraction.

The operator selected client local-storage, then API server memory, then database, then Google as the required lookup order.

Sentence detection should favor wrapping/tokenizing sentences before or during markdown rendering so each translatable sentence is represented by a click/tap target such as a `span`. The active sentence must be visibly highlighted after tap/click so the user can clearly see what is being translated.

The LingoCafe seeded test user that already has a complete profile should be given `feature_flags: { translate: true }` for local verification. In the current seed files this is Jane Doe.

The persistent translation cache should store at least `hash`, `from`, `to`, `text`, and `translation`. `hash` is the primary key. `text` should retain the original source sentence used for translation/debugging, while the hash is based on the normalized inputs.

## Assumptions

- The source language for a translation request is the current book/page content language.
- The target language is the user's `ownLang` value unless the operator corrects that.
- The sentence boundary logic should be deterministic enough to treat "stop to stop" as punctuation-delimited sentence extraction rather than arbitrary text selection length.
- Sentence tokenization should prefer sentence wrapper elements in the rendered reader content over raw DOM text extraction when practical.
- The translation popover can be implemented with existing UI primitives or a route-local anchored overlay, as long as placement keeps both source and translated text visible.
- The backend owns the Google API call so the API key remains server-side.
- The database cache is the cross-device durability layer, while server-memory and local-storage caches are performance layers on top of it.
- The translation hash should be based on aggressively normalized inputs to reduce duplicate translations for equivalent meaning.
- User-level translation availability is controlled by `auth.users.feature_flags.translate`, not by profile data.

## Constraints

- Keep the feature limited to LingoCafe reader pages, not the entire app or all markdown surfaces.
- Enforce authorization in both frontend and backend using `auth.users.feature_flags.translate === true`.
- Use app-level feature/policy gating where appropriate for rollout control in addition to the per-user `feature_flags.translate` check.
- Do not add `translate` to `auth.users.profile` or the LingoCafe profile schema.
- Do not expose `translate` as a user-editable profile UI setting.
- Do not let profile saves overwrite or clear `auth.users.feature_flags`.
- Update `knex/migrations/20240320_auth.js` in place for the local clean-rebuild schema path instead of creating a new dirty Knex migration only for `auth.users.feature_flags`.
- Create a task-folder manual production patch named `migrate.sql` that adds `auth.users.feature_flags` for deployed databases.
- Seed the existing LingoCafe test user with a complete profile, currently Jane Doe, with `feature_flags.translate = true` for local verification.
- Use Google translation via API-key configuration from environment variables.
- Keep Google-specific translation logic behind a small server-side provider abstraction so a future provider can replace it without touching the cache and reader interaction layers.
- Do not send repeated requests upstream when any valid cache layer can satisfy the request.
- Check caches in this order: client local-storage, API server memory, database, Google.
- Persist translation cache rows in the `lingocafe` schema and key them by a stable hash of source text, source language, and target language.
- Store at least `hash`, `from`, `to`, `text`, and `translation` in the persistent translation cache.
- Cap the server in-memory cache by maximum size in MB from env, defaulting to 20 MB, and evict older entries to make room for new ones.
- Cap the client local-storage cache to 1 MB with comparable eviction-by-last-used behavior.
- Normalize cache-hash inputs at least by trimming, collapsing or removing insignificant whitespace, lowercasing text where acceptable, applying Unicode normalization, and normalizing language codes.
- Keep the second-click-to-close interaction simple and reliable across desktop and mobile.
- Do not weaken existing authenticated app-page conventions, profile validation rules, or restricted markdown rendering.

## Acceptance Criteria

- [ ] A LingoCafe reader user can click or tap a sentence and the full sentence becomes highlighted.
- [ ] The reader identifies sentences from stop to stop instead of translating only the exact character subset of an arbitrary text selection.
- [ ] A translation popover appears near the highlighted sentence and shows the translated text in the user's own language.
- [ ] The popover chooses above or below placement based on available space so both the source sentence and the popover remain visible together.
- [ ] A second click or tap outside the active sentence/popover closes the popover and clears the active highlight.
- [ ] The feature is visible and usable in the frontend only when `auth.users.feature_flags.translate === true` for the current user.
- [ ] Missing `feature_flags`, a missing `translate` key, or a non-true `translate` value disables the feature.
- [ ] The translation backend rejects requests when the current user is not allowed to translate, even if the client attempts to call it directly.
- [ ] `translate` is not stored in `auth.users.profile` and is not part of the LingoCafe profile schema.
- [ ] `translate` is not user-editable from the profile UI.
- [ ] Profile saves do not overwrite, reset, or otherwise mutate `auth.users.feature_flags`.
- [ ] `knex/migrations/20240320_auth.js` adds `auth.users.feature_flags jsonb default null` for local rebuilds.
- [ ] A task-folder manual production SQL patch named `migrate.sql` exists for adding `auth.users.feature_flags` to deployed databases.
- [ ] The LingoCafe seeded test user with a complete profile, currently Jane Doe, receives `feature_flags.translate = true` for local verification.
- [ ] Translation requests use Google Cloud Translation Basic v2 REST through backend API-key-based configuration.
- [ ] Google provider-specific code is isolated behind a small server-side abstraction or factory-style boundary.
- [ ] The Google API key is never exposed to the browser bundle.
- [ ] Cache lookup order is client local-storage, API server memory, database, then Google.
- [ ] The persistent translation cache lives in the `lingocafe` schema and is keyed by hash(`text`, `fromLang`, `toLang`).
- [ ] The persistent translation cache stores at least `hash`, `from`, `to`, original source `text`, and `translation`.
- [ ] Translation hash inputs use documented normalization rules for text and language codes.
- [ ] When the database cache misses, the system can store the fresh translation back into the persistent cache for future cross-device reuse.
- [ ] The backend maintains an in-memory translation cache with hash keys, last-used timestamps, env-configured max size in MB, and eviction of older entries when the limit is reached.
- [ ] The default server-memory cache limit is 20 MB when no env override is provided.
- [ ] The client maintains a local-storage translation cache with the same hash key basis, last-used tracking, and eviction behavior capped to 1 MB.
- [ ] Once a sentence has been translated and persisted, re-translating the same sentence on another device can resolve without another upstream Google call when the persistent cache is available.
- [ ] The implementation includes enough observability or diagnostics to distinguish cache hits from upstream translation misses during development or verification.

## Dos

- Do reuse existing LingoCafe reader surfaces and profile infrastructure where practical.
- Do model the translation cache key explicitly from normalized text, source language, and target language.
- Do make the popover feel native to reading flow on both desktop and mobile.
- Do keep cache eviction deterministic and inspectable.
- Do keep translation authorization in `auth.users.feature_flags`.
- Do preserve the strict LingoCafe profile schema without adding a user-editable translation field.
- Do keep Google Cloud Translation Basic v2 behind a provider boundary.
- Do preserve room in planning for migrations, API surface, and smoke-test coverage.
- Do keep the reader markdown rendering surface narrow unless planning explicitly proves sentence wrapping requires a small controlled extension.

## Don'ts

- Don't put the Google API key in client-side code.
- Don't translate arbitrary freeform selections if the requirement is sentence-level translation.
- Don't bypass backend authorization checks just because the UI is hidden.
- Don't put translation authorization into `auth.users.profile`.
- Don't make translation authorization user-editable.
- Don't let `/api/profile` mutate `auth.users.feature_flags`.
- Don't implement only one cache layer and call the job done.
- Don't make the popover obscure the translated sentence or the original sentence when placement can avoid it.
- Don't broaden this task into notes, vocabulary saving, highlights, or other reader annotations unless planning explicitly adds them later.
- Don't rely on browser-only state for behavior that must work across devices.
