# Stats Users Growth

## Elevator's Pitch

Add a local `42go` analytics command that computes user growth statistics over time from the local event Parquet archive and keeps reusable partial aggregates under `.local`.

## Business Gain

The operator can inspect LingoCafe user growth, subscribed audience growth, active usage, and inactivity trends without running heavy analytics against production PostgreSQL. Chuck Norris does not ask the production database for a trend line twice.

## Current State

- `42go events pull` exports core `events.events` rows into local monthly CSV and Parquet files under `.local/42go-events/events/`.
- `42go events query stats` reads local Parquet files with DuckDB and prints high-level archive statistics.
- Raw event rows include `created_at`, `id`, `app_id`, `user_id`, `event_at`, `name`, `data`, and `meta`; `data` and `meta` are JSON strings in local files.
- Known LingoCafe reader event names include `page.open`, `page.scroll`, `page.translate`, `read.settings.opened`, and `read.settings.changed`.
- Core lifecycle and consent/profile events include `user.signup`, `user.login`, `user.profile.created`, `user.profile.updated`, `user.consent.created`, and `user.consent.updated`.
- No stored user-growth aggregate currently exists; prior tasks deferred incremental aggregate generation.
- Current local Parquet sampling after the latest production pull found one app, `lingocafe`, with 1,352 events, 25 distinct users, and event coverage from 2026-05-11 16:47:03.807 Europe/Rome through 2026-06-10 10:15:27.807 Europe/Rome.
- Current top reading-related events include `page.translate` (496 events), `page.scroll` (439), and `page.open` (208). Settings events exist but should not count as reading activity.
- LingoCafe promotional email consent is configured as the `mkt` consent item in `src/config/lingocafe/config.ts`; consent save events are emitted from `src/42go/profile/server.ts`.

## Desired State

- A discoverable `42go` CLI command exists at `42go events query users growth`.
- The command updates its internal local aggregate cache and returns human-readable terminal output with the data.
- The command may also support a machine-readable output option such as `--format json`, but human-readable output is the required default.
- The command computes user counts over time for all requested granularities: day, week, month, and year.
- The command computes subscribed users over time, where subscribed means the latest known `mkt` consent value for the user is `true` at the data point.
- The command computes weekly active users over time, where active means users with reading events within the trailing 7-day window ending at each data point.
- The command computes monthly active users over time, where active means users with reading events within the trailing 30-day window ending at each data point.
- The command computes inactive users over time as known users at the bucket boundary minus monthly active users at that same boundary.
- The command uses `Europe/Rome` for day, week, month, and year bucket boundaries.
- Week aggregation uses ISO weeks starting Monday, with the bucket label as the ISO week start date.
- The command maintains partial aggregate data locally under `.local/42go-stats/{app-id}/users/growth/`, separate from raw exporter state and raw Parquet files.
- Aggregate state and files are isolated per `app_id`.
- By default, the command computes and updates aggregate data for all app IDs present in the local Parquet archive.
- The command supports `--app` as an output filter accepting a comma-separated list such as `--app lingocafe,app1,default`; this filter affects terminal output and any explicit output selection, not whether all app aggregates can be refreshed.
- On normal runs, the command reads existing aggregate state and only processes newly available raw Parquet rows when possible.
- When no new raw rows exist, the command returns data from the existing aggregate without recomputing all history.
- When run with a `--reset` flag, the command removes the local user-growth aggregate and recomputes it from all available local Parquet data.
- The aggregate output is deterministic, documented, and suitable for future JSON export, charts, or dashboard surfaces.

## Definition of Success

- A future implementer can run the command repeatedly without duplicating metrics or corrupting local state.
- A first run builds the aggregate from all available local Parquet data.
- A later run after `42go events pull` processes only new data or touched periods needed for correctness.
- A later run with no new Parquet data returns quickly from the existing aggregate.
- A run with `--reset` deletes/rebuilds the user-growth aggregate from scratch.
- Results include day, week, month, and year buckets for each requested metric.
- Results are app-scoped, and terminal output can be filtered with `--app`.
- Promotional subscription counts use the latest event-level `next.mkt` choice available at each data point, including updates that switch a user from true to false.
- Reading activity counts only `page.open`, `page.scroll`, and `page.translate` unless implementation sampling proves another event should be included.
- The raw `.local/42go-events/events/` archive remains read-only from this analytics command.

## Additional Context

- Use `event_at` for behavioral analytics buckets and rolling active-user windows. Use `created_at` and `id` only as archive/export cursor metadata where useful.
- The local aggregate root is `.local/42go-stats/{app-id}/users/growth/`.
- Recommended aggregate shape is one or more Parquet files for metric data plus a small JSON metadata/state file. JSON is acceptable for state; Parquet is preferred for data that may later feed charts or other analytics.
- DuckDB is already the preferred local query engine for Parquet analytics.
- The command should update `.agents/skills/42go-cli/SKILL.md` because command discoverability is part of the CLI contract.
- Consent payloads store historical choices under `data.next.mkt` as an array. The implementation must read the latest entry by `changedAt` from the latest relevant consent event at or before each data point.
- Consent events are `user.consent.created` and `user.consent.updated`; both must be considered.

## Assumptions

- This task belongs to the existing pipx-installable Python CLI under `cli/`.
- The source of truth for raw analytics is local Parquet, not production PostgreSQL.
- Reading events are `page.open`, `page.scroll`, and `page.translate`; settings-only events such as `read.settings.opened` and `read.settings.changed` do not count as reading activity.
- Promotional email subscription is inferred from `user.consent.created` and `user.consent.updated` events by reading the latest `data.next.mkt[]` entry for each user.
- Known users over time can be inferred from user-scoped events such as `user.signup` and first-seen `user_id` timestamps if no full user dimension table is present in Parquet.
- Incremental aggregation may need to recompute a small trailing window or touched buckets so rolling weekly/monthly active counts stay correct.
- Output is human-readable by default.
- Optional JSON output is useful and acceptable if it does not delay the required command behavior.
- The command should inspect all app IDs in the local event archive and maintain isolated aggregate state per app.

## Constraints

- Do not query production PostgreSQL for this metric command.
- Do not mutate raw CSV or Parquet archive files.
- Do not overwrite `events/state.json`, `manifest.jsonl`, or other exporter-owned state.
- Do not commit `.local` aggregate data.
- Do not parse JSON payloads with fragile string splitting; use DuckDB JSON functions or structured parsing.
- Keep analytics state separate from exporter state.
- Keep command help complete and agent-navigable.
- Use `.local/42go-stats/{app-id}/users/growth/` for local partial aggregate storage.
- Support a `--reset` flag that removes and recomputes the aggregate.
- Keep app aggregate state isolated by `app_id`.
- `--app` filters output but must not collapse or mix app-scoped aggregate storage.
- Use `Europe/Rome` for bucket boundaries.

## Acceptance Criteria

- A CLI command is added at `42go events query users growth` and appears in `42go --help` / nested `--help` output.
- The command accepts a `--reset` flag.
- The command accepts `--app` with a comma-separated list of app IDs for output filtering.
- The command reads local Parquet files from the existing `.local/42go-events` archive layout.
- The command stores its partial aggregate dataset under `.local/42go-stats/{app-id}/users/growth/` in a documented shape.
- The command computes or refreshes app-scoped aggregates for every app ID present in local Parquet, while `--app` controls which apps are shown in the terminal output.
- The command computes total users over time by day, week, month, and year.
- The command computes subscribed users over time by day, week, month, and year.
- The command computes weekly active users over time by day, week, month, and year.
- The command computes monthly active users over time by day, week, month, and year.
- The command computes inactive users over time by day, week, month, and year.
- Subscribed users are computed from `user.consent.created` and `user.consent.updated` events using the latest `data.next.mkt[]` entry by `changedAt` for each user at each data point.
- Reading activity is computed from `page.open`, `page.scroll`, and `page.translate`.
- Weekly active users use a trailing 7-day window ending at each data point.
- Monthly active users use a trailing 30-day window ending at each data point.
- Inactive users equal known users minus monthly active users at each data point.
- Day/week/month/year buckets use `Europe/Rome`; week buckets are ISO weeks starting Monday.
- Normal runs process newly available raw rows or touched periods without full recomputation when possible.
- Normal runs with no new raw rows read from the existing aggregate.
- `--reset` removes the existing local aggregate and recomputes from all available Parquet files.
- Metric definitions, bucket boundaries, timezone assumptions, and event-name filters are documented in the CLI help or skill docs.
- Focused tests or smoke checks cover first run, no-new-data run, incremental run, and reset rebuild behavior.
- `.agents/skills/42go-cli/SKILL.md` is updated with the new command.

## Dos

- Use DuckDB over local Parquet for metric computation.
- Use `event_at` for analytics time buckets.
- Design aggregate writes to be idempotent.
- Store aggregate metadata such as processed raw file fingerprints, max processed event key, covered date range, and metric schema version.
- Use a schema/version strategy so future metric changes can force rebuild cleanly.
- Prefer recomputing touched buckets or a bounded trailing window when rolling active metrics require it.
- Document the exact definition of subscribed, weekly active, monthly active, and inactive users.
- Isolate aggregate storage and metric results by `app_id`.
- Parse consent JSON structurally and choose the latest `mkt` evidence entry by `changedAt`.
- Keep human-readable output as the default.
- Keep the raw export archive read-only.
- Update the `42go-cli` skill when adding the command.

## Don'ts

- Do not run heavy analytics against production PostgreSQL.
- Do not couple aggregate state to the exporter cursor as if it were the same contract.
- Do not hide the command behind Make only.
- Do not implement this as a one-off script outside the CLI.
- Do not leave unclear whether week buckets use ISO weeks, calendar weeks, or trailing seven-day windows.
- Do not treat consent as subscribed without verifying the exact promotional email consent event or payload.
- Do not leave aggregate files in a repo-tracked path.
- Do not mix app IDs in one aggregate file without an explicit app-scoping strategy.
- Do not count settings-only events as reading activity.
