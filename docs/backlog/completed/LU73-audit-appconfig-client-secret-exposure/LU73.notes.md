# Execution Notes - Audit AppConfig Client Secret Exposure
**Task**: ./LU73.task.md
**Plan**: ./LU73.plan.md

## Decisions

- Keep execution analysis-only, per refinement answers. No production code changes.
- Use command-scoped sentinel values for build/runtime proof. Do not modify `.env`.
- Redact real environment-derived values and avoid copying raw secrets into the task artifacts.
- Classify `/api/todos` raw config serialization as a confirmed exposure path after policy success, even though unauthenticated runtime checks returned 401.
- Classify the client bundle AppConfig registry as a fragile boundary because sentinel values were not inlined, but secret-bearing `process.env.*` references and `auth.providers` structure are present in browser assets.

## Problems Encountered

- First production build failed because the sandbox could not fetch Google Fonts through `next/font`. The build was rerun with approved escalation and passed.
- Starting the built Next server inside the default sandbox failed with `listen EPERM`; it was rerun with approved escalation on `127.0.0.1:45673`.

## Deviations From Plan

- No authenticated `/api/todos` request was completed with a user satisfying role `foo` and grant `users:list`. Source analysis proves the handler returns raw config after the policy gate; unauthenticated runtime proof confirmed the gate returns 401 before handler execution.

## Additional Requests

- Local runtime proof found that `admin/admin` can pass `/api/todos` policy and receive secret-bearing AppConfig fields; real values were not copied into task artifacts.
- After that proof, the operator requested a static-only pass for similar code patterns outside the known leaking endpoint.
- Static-only pass found no second direct API route returning raw AppConfig.
- Static-only pass found a broader client-bundle pattern: `src/42go/config/use-app-config.tsx` imports `@/AppConfig` and is consumed by multiple client components and protected app pages.
- Static-only pass also flagged `src/app/(public)/layout.tsx` passing raw `config` to a configurable `LayoutComponent`; current default layout is server-side, but a future client custom public layout would serialize raw config across the server/client boundary.
- Removed the stale todos page/API and their feature flags after operator agreed to delete the unused surface.
- `npm run qa` passed after deleting the stale todos page/API; the build route list no longer includes `/todos` or `/api/todos`.
- Migration follow-up: deleting `knex/migrations/20250703150513_create_todos_table.js` can break future Knex migrations in any database that already has that filename recorded in `knex_migrations`.
- Verified Knex 3.2.9 validates completed migrations against files on disk and throws `The migration directory is corrupt, the following files are missing: ...` when an applied migration file is absent.
- Recommended migration-safe cleanup is to keep the original todos migration file and add a new migration that drops the unused `todos` table with `dropTableIfExists("todos")`.
- Manual production cleanup is possible only if the production `knex_migrations` row is also handled consistently, but that is brittle migration-table surgery and not the recommended path.

## Known Limitations

- Runtime checks used sentinel secrets and public/unauthenticated endpoints. Real `.env` values were not printed or copied into artifacts.
