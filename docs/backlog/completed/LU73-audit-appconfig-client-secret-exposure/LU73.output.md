# LU73 Output - Audit AppConfig Client Secret Exposure

## Executive Summary

Conclusion: confirmed exposure path, plus a fragile client boundary.

The current system does not have an enforceable server/client AppConfig boundary. `src/42go/config/use-app-config.tsx` is a client hook that imports the full `apps` registry from `src/AppConfig.ts`, and production client assets contain the full app configuration shape, including `auth.providers[].config.clientSecret` properties as `process.env.*` references. The sentinel secret values were not inlined into the browser bundle, but the secret-bearing structure and environment variable names are shipped to the client.

There is also a direct server-side API exposure path: `src/app/api/todos/route.ts` returns `Response.json({ config, todos })`. If that policy gate passes for a user, the server serializes the raw resolved AppConfig. Because AppConfig contains OAuth `clientSecret` values sourced from `process.env`, this route can expose actual runtime secrets. Chuck Norris does not call that a boundary. He calls it a door with a polite sign.

## Scope

- Complete repository analysis for AppConfig and config-derived client exposure paths.
- Analysis-only execution. No production code changes.
- Real secrets were not printed. Sentinel values were used for build/runtime proof.

## Method

1. Source traced AppConfig definitions, app registry imports, server helpers, client hooks, API routes, auth provider assembly, rendered head injection, manifest, and PWA tags.
2. Classified AppConfig fields into public, server-only, and mixed-risk categories.
3. Ran a production build with command-scoped sentinel values for OAuth secrets and adjacent sensitive env vars.
4. Searched emitted browser and server build artifacts for sentinel values and secret-bearing env keys.
5. Started the built app locally with the same sentinel values and checked public HTML, login HTML, manifest, health/test endpoints, NextAuth providers/session, and `/api/todos`.

The first build attempt failed because sandboxed network access could not fetch Google Fonts. The build was rerun with escalation and passed.

## AppConfig Field Classification

Clearly public or intended client-visible:

- `name`
- `icons`
- `theme.default`
- `public.toolbar`
- `public.meta`
- `public.pwa`
- `public.pages`
- `public.docs.source` and cache metadata, if paths are intended to be public
- `public.footer`
- `app.default.page`
- `app.menu`
- `features`
- `match` only when exposing routing rules is acceptable

Mixed-risk:

- `app.profile`: UI/schema information is often client-visible, but schema details can reveal internal account/profile structure.
- `app.consent`: consent labels and versions are client-visible by design.
- `app.events`: client collection settings are consumed by client code. They should not carry secrets.
- `auth.logout.url`: client-visible and used by client logout components.

Server-only:

- `auth.providers[].config.clientSecret`
- OAuth refresh/access tokens stored through auth callbacks
- Database URLs and pool settings
- Translation/provider API keys such as `LC_GOOGLE_TRANSLATE_API_KEY`
- NextAuth secret
- Any field semantically equivalent to `secret`, `token`, `key`, `password`, or `credential`, unless explicitly marked as a public identifier such as OAuth `clientId`

Important distinction:

- OAuth `clientId` can be public.
- OAuth `clientSecret` must be server-only.

## Source Data Flow

Server resolution:

1. `src/proxy.ts` resolves an app ID and sets the request header used downstream.
2. `src/42go/config/app-config.ts:23-96` resolves app ID from headers, environment, matchers, or fallback.
3. `src/42go/config/app-config.ts:98-102` returns the raw app config from `apps`.
4. Server consumers use raw config for metadata, PWA, public layouts, docs, policy, auth, profile config, events, and APIs.

Client path:

1. `src/42go/config/InjectAppID.tsx:17-20` injects only `window.__APP_ID__`.
2. `src/42go/config/use-app-config.tsx:1-2` imports the runtime `apps` registry in a hook consumed by client components.
3. `src/42go/config/use-app-config.tsx:17-19` returns `apps[appID]` to client code.
4. Client components then use this config for layout, menu, logout URL, profile blocks, consent, events, policy evaluation, and LingoCafe pages.

This means the client does not receive a deliberate public projection. It imports the same source registry that contains server-only fields. The current protection depends on Next/webpack/turbopack not inlining non-public env values into browser assets.

## Protection Analysis

What currently helps:

- Next.js does not inline non-public env values into browser bundles in the tested build.
- Root layout injects only `window.__APP_ID__`, not raw config.
- Metadata, manifest, PWA head tags, profile API, events API, and LingoCafe reader code mostly project selected config fields instead of returning raw AppConfig.
- NextAuth public providers/session endpoints do not expose OAuth client secrets in the checked runtime.
- `/api/todos` is gated by feature, session, role, and grant checks.

What does not protect secrets:

- TypeScript interfaces do not enforce a public/server config split.
- `useAppConfig()` imports the full `apps` registry into client code.
- There is no `PublicAppConfig` projection or redaction function.
- There is no lint/static guard preventing client modules from importing `@/AppConfig`.
- There is no bundle regression test that fails if secret sentinels appear in browser assets.
- There is no API response guard preventing `Response.json({ config })`.
- `/api/todos` returns raw AppConfig after its policy gate.

Security classification:

- Confirmed exposure path: `src/app/api/todos/route.ts` can serialize server-side secrets after policy success.
- Fragile client boundary: client bundle includes secret-bearing AppConfig shape and env-key references, but not sentinel values in this build.

## Findings

### Finding 1 - Raw AppConfig API Response Can Expose OAuth Secrets

Severity: high.

`src/app/api/todos/route.ts:8-15` returns the raw resolved `config`. Because config includes OAuth provider secrets, any request that passes the route policy can receive those secrets in JSON.

Evidence:

- Raw config returned at `src/app/api/todos/route.ts:15`.
- OAuth secrets configured in `src/config/default/config.ts:311-319`, `src/config/app1/config.ts:84-85`, `src/config/app2/config.ts:53-54`, `src/config/quicklist/config.ts:20-21`, and `src/config/lingocafe/config.ts:59-60`.
- Unauthenticated runtime check returned `401`, but the handler remains dangerous after successful authorization.

### Finding 2 - Client Hook Bundles Full AppConfig Registry

Severity: high for architecture, medium for direct current leak.

`src/42go/config/use-app-config.tsx` imports and returns the full `apps` registry in client-consumed code. Production browser chunk `.next/static/chunks/0-h3ujaqq2fis.js` contains the AppConfig registry and `clientSecret: process.env.*` references.

Evidence:

- Runtime import at `src/42go/config/use-app-config.tsx:1-2`.
- Client return of `apps[appID]` at `src/42go/config/use-app-config.tsx:17-19`.
- Bundle inspection found `clientSecret: t.default.env.GITHUB_CLIENT_SECRET`, `clientSecret: t.default.env.GOOGLE_CLIENT_SECRET`, and equivalent app-specific secret references.
- Sentinel values were not inlined, so this is not a direct value leak in the tested build.

### Finding 3 - No Enforced Public AppConfig Boundary

Severity: medium-high.

The project relies on developer discipline and framework env behavior rather than an explicit redaction boundary. A future literal secret or `NEXT_PUBLIC_*` secret in AppConfig would be bundled because client code imports the complete registry.

Evidence:

- `TAppConfigItem` mixes public, client-consumed, and server-only concerns in one type.
- There is no public projection function.
- There is no test or lint rule preventing client imports of `@/AppConfig`.

## Follow-Up Recommendations

1. Create a `PublicAppConfig` type and projection function.
   - Include only public fields needed by client components.
   - Exclude `auth.providers[].config.clientSecret` and any server-only keys by construction.

2. Replace `useAppConfig()` so it consumes only a public projection.
   - Prefer a server-produced serialized public config or an app-ID-indexed generated public registry.
   - Keep `window.__APP_ID__` if useful, but do not use it to index the raw server registry on the client.

3. Move OAuth provider secrets out of client-importable app config modules.
   - Options: server-only provider registry, provider factory keyed by app ID, or server-only `auth` config split.
   - Add `import "server-only"` to modules that must never be imported by client code.

4. Remove or rewrite `/api/todos` raw config response.
   - Return only a minimal public diagnostic config if the endpoint is still needed.
   - Add tests proving secret-bearing fields are not serialized.

5. Add a bundle sentinel regression test.
   - Build with fake sentinel secrets.
   - Fail if sentinels or forbidden env key names appear in `.next/static`.

6. Add static import guards.
   - Prevent client files and client dependency chains from importing `@/AppConfig`.
   - Flag `Response.json({ config })` and raw `getAppConfig()` API responses.

7. Document AppConfig public/server field policy.
   - Mark which fields are intentionally public.
   - Mark which fields are server-only.
   - State that secrets must never live in modules imported by client components.

### Static Conclusion

Current confirmed direct response leak remains `/api/todos`.

The broader repeated pattern is raw AppConfig reuse across client and server code. The dangerous calls are not only HTTP endpoints; they are also imports and component calls that place `@/AppConfig` in the browser dependency graph. The fix should remove client access to the raw registry, not just patch `/api/todos`.
