# Audit AppConfig Client Secret Exposure

## Elevator's Pitch

Run a focused 42GO core security analysis that proves how app configuration moves from server resolution into browser code, whether OAuth client secrets or any other server-only values can reach the client, and what guardrails currently exist or must be added. The outcome is evidence, not vibes. Chuck Norris reads the bundle before the bundle reads him.

## Business Gain

Prevent accidental exposure of OAuth client secrets, API keys, or server-only configuration through the shared multi-app configuration system. This protects every current and future app that stores provider credentials or operational settings in AppConfig.

## Current State

App resolution starts in `src/proxy.ts` and server helpers in `src/42go/config/app-config.ts`. The root layout injects only `window.__APP_ID__` through `src/42go/config/InjectAppID.tsx`, but the client hook `src/42go/config/use-app-config.tsx` imports `apps` directly from `src/AppConfig.ts` and returns the selected config. Several app config files place OAuth provider `clientSecret` values under `auth.providers`, including `src/config/default/config.ts`, `src/config/app1/config.ts`, `src/config/app2/config.ts`, `src/config/quicklist/config.ts`, and `src/config/lingocafe/config.ts`. There is no confirmed, documented allowlist or redaction boundary yet proving that these values cannot be bundled, serialized, inspected, or otherwise exposed client-side.

## Desired State

The repository contains a task-local analysis artifact at `LU73.output.md` that maps every path by which AppConfig reaches client code, classifies which AppConfig fields are public versus server-only, verifies browser runtime output and built assets for secret leakage, and states concrete remediation tasks if any leak or weak boundary is found.

This task is analysis-only. It must not implement a hotfix even if confirmed exposure is found. If exposure or weak protection is found, the output must document the threat and recommend follow-up implementation work.

## Definition of Success

- The task folder contains `LU73.output.md` with source-level traces, evidence, findings, and conclusions.
- `LU73.output.md` contains or links to evidence proving what was inspected, for example bundle grep results, rendered HTML snippets, client runtime observations, API response checks, and source path references.
- The analysis explicitly answers: what currently prevents secrets from reaching the client, what does not prevent it, and whether the current boundary is enforceable by type, runtime sanitizer, build behavior, tests, or convention only.
- If risk is found, the analysis highlights the threat and proposes the smallest safe remediation path and separate follow-up implementation tasks instead of changing behavior inside this analysis task.

## Additional Context

- Primary files to inspect: `src/AppConfig.ts`, `src/42go/config/app-config.ts`, `src/42go/config/use-app-config.tsx`, `src/42go/config/InjectAppID.tsx`, `src/app/layout.tsx`, `src/app/(public)/login/page.tsx`, `src/42go/auth/lib/providers/types.ts`, and `src/42go/auth/lib/providers/get-providers.ts`.
- Relevant config files to inspect: `src/config/default/config.ts`, `src/config/app1/config.ts`, `src/config/app2/config.ts`, `src/config/quicklist/config.ts`, `src/config/lingocafe/config.ts`, plus any future configs containing `secret`, `token`, `key`, or provider credentials.
- The suspicious pattern found while drafting is a client component hook importing the complete `apps` registry while auth provider config types include `clientSecret`.
- Do not rely only on Next.js environment variable behavior. Verify the emitted browser assets and runtime state because Chuck Norris does not trust a locked door until he has kicked it once.
- The audit scope is the complete codebase, not only 42GO core. If a vulnerability exists, it does not matter which module introduced it.
- The audit may read `.env` if needed, but must never modify it and must treat it as sensitive because it can contain database URLs and credentials.

## Assumptions

- OAuth `clientSecret` and any API keys, tokens, database URLs, private webhook secrets, or server-only provider settings must never be available to browser JavaScript, rendered HTML, public JSON endpoints, source maps, or client bundles.
- Public app configuration may include branding, icons, public pages, public metadata, menu labels, feature flags, docs settings that are safe to disclose, and non-secret OAuth provider identifiers where needed.
- The correct long-term shape may require a public AppConfig projection or explicit server/client split, but this task should prove the need before designing the full fix.

## Constraints

- Do not rotate or print real secret values. Use sentinel test values or redacted matches when collecting evidence.
- Do not commit `.env` or any generated artifact containing real secrets.
- Do not make behavioral code changes in this analysis task unless the operator explicitly upgrades the task from analysis to implementation.
- Run a production build or equivalent build inspection with sentinel secret values so emitted browser assets can be checked for leaks.
- Do not rely on source-level inspection alone. Also assess whether secrets can leak through API calls, rendered HTML, injected globals such as `window.__APP_ID__`, hydration payloads, and browser bundles.
- Use `npm run qa` only if code changes are introduced; otherwise evidence commands, build inspection, and runtime checks are enough for this analysis deliverable.

## Acceptance Criteria

- `LU73.output.md` documents the AppConfig data flow from request matching to server helpers, layouts, injected globals, client hooks, auth providers, public layouts, app layouts, and API usage.
- The analysis identifies every client import path that can pull `src/AppConfig.ts` or per-app config modules into browser code.
- The analysis identifies every server-only field currently present or reasonably likely to be added to AppConfig, including `auth.providers[].config.clientSecret`, API keys, webhook secrets, database URLs, provider tokens, private credentials, and any key matching or semantically equivalent to `secret`, `token`, `key`, `password`, or `credential`.
- Evidence includes source references with file paths and line numbers for the current data flow and secret-bearing fields.
- Evidence includes a build or bundle inspection strategy that can be reproduced, with redacted/sentinel search strings and clear pass/fail notes.
- Evidence includes browser/runtime inspection for every relevant surface, including rendered HTML, browser JavaScript bundles, source maps if present, `window` globals, React or Next hydration payloads, NextAuth endpoints, public app routes, protected app routes after login where relevant, and API responses that expose config-derived data.
- The conclusion states one of: no exposure found with enforceable protections, no exposure found but only by fragile build/runtime behavior, or confirmed exposure with exact affected paths.
- If protections are missing or fragile, the task output recommends concrete follow-up work such as `PublicAppConfig` typing, server-side redaction/projection, lint/static checks, bundle regression tests, or moving auth provider secrets outside client-importable config modules.

## Dos

- Do trace imports with `rg` and dependency reasoning before drawing conclusions.
- Do inspect both source-level data flow and built browser artifacts.
- Do use sentinel secret values where practical so evidence can prove whether a value appears in emitted client output without exposing a real secret.
- Do distinguish `clientId` from `clientSecret`; OAuth client IDs may be public, client secrets are not.
- Do inspect the complete codebase for AppConfig and config-derived leak paths.
- Do write the final evidence and conclusion in `LU73.output.md` inside the task folder.

## Don'ts

- Don't assume `process.env` behavior alone is a sufficient protection.
- Don't paste real credentials into task notes, logs, screenshots, or command output.
- Don't fix the architecture inside the analysis unless explicitly asked; capture remediation as follow-up scope.
- Don't treat TypeScript interfaces as security boundaries unless there is an enforced public/server split.
- Don't modify `.env`.

## Open Questions

- Which AppConfig sections are intentionally public, and which must be marked server-only by type?
- Does the build output prove secret-bearing values are absent from browser assets, or are protections only accidental because of current environment variable behavior?
- Should follow-up implementation tasks split AppConfig into server-only config plus a public projection, move auth credentials to a server-only registry, and add regression checks?
