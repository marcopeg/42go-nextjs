# Add Makefile Database Backup and Restore Commands

## Elevator's Pitch

Add `make backup` and `make restore` commands that export and import data-only PostgreSQL dumps for the configured `DATABASE_URL`, storing dumps under `knex/dumps` with UTC timestamped mode-specific filenames.

## Business Gain

Developers and operators need a repeatable way to preserve and move application data without dumping schema objects that are already owned by Knex migrations. A reliable restore path makes local rebuilds, production incident handling, and data migration rehearsals safer.

## Current State

The Makefile already owns database commands such as `migrate`, `seed`, and rebuild helpers, but there is no documented `make backup` or `make restore` flow. Knex migrations create the database structure across public, auth, quicklist, notes, and lingocafe schemas. Some LingoCafe tables contain heavy or disposable data such as events, translation cache, books, and book pages.

There is no repo-local `42go-backup` skill yet. The operator wants this task to introduce that skill and place the reusable backup/restore script there so it can be invoked both by Codex as a skill and by Makefile commands.

## Desired State

`make backup mode=full` and `make backup mode=light` dump only table data from the connected database into `knex/dumps/{timestamp}.dump.{mode}.sql`, where `{timestamp}` uses UTC. The generated SQL dump is self-contained for data restoration: it includes the dependency-safe restore instructions needed to clear target data and insert data in the correct order after migrations have built the schema.

`make restore from=<path>` does not need a separate `mode` parameter. It executes the selected dump file against the connected database. The same dump file should also be usable manually in a database management tool after the operator runs migrations.

Restore truncates target tables first in dependency-safe order, then imports data in dependency-safe order. Restore must succeed without disabling data integrity permanently and must respect foreign-key dependencies by writing referenced parent rows before child rows.

The reusable implementation lives in a new `.agents/skills/42go-backup/` skill. The public human command surface is the Makefile. The skill also documents how Codex can operate the same backup and restore flow directly.

## Definition of Success

The repository has Makefile-backed backup and restore commands that are deterministic, mode-aware, documented enough for developers to use, and verified against a migrated empty PostgreSQL database. The shared behavior is implemented through a new `42go-backup` skill with a reusable script instead of fragile Makefile-only logic.

## Additional Context

Mode rules from the operator: `full` includes every schema and table. `light` skips `lingocafe.events*`, `lingocafe.translation_cache`, `lingocafe.books`, and `lingocafe.books_pages`. The dump format must be SQL, not a custom pg_dump archive, and must contain data only. The table write order matters because the restore target already has constraints created by migrations.

The operator answered the first refinement round with these decisions:

- Restore should truncate first and must be careful with table order.
- Dumps should be data only.
- Migration metadata tables should not be included in the data dump; migrations are run manually before restore.
- Restore should use `make restore from=...`.
- Dump filename timestamps should use UTC.
- A repo-local helper script is acceptable, but it should live under a new `42go-backup` skill and work both when invoked as a skill and through Makefile.

The operator answered the second refinement round with these decisions:

- `make restore from=...` should just execute the instructions in the dump file.
- The dump-generation side owns producing a good SQL file, including restore ordering.
- The dump can also be restored manually in a database management tool.
- Migrations must be run manually before restore, not before backup.
- The script is not meant as a separate public CLI surface; public usage is Makefile plus Codex operation through the `42go-backup` skill.

## Assumptions

- `DATABASE_URL` remains the source of truth for the connected PostgreSQL database, loaded through the existing Makefile `.env` include pattern.
- The implementation may use PostgreSQL-native tools such as `pg_dump` and `psql` as long as the resulting dump is data-only SQL.
- The restore target has already run `make migrate` or an equivalent Knex migration flow before restore.
- Backup reads the current connected database as-is and does not run migrations automatically.
- The new `42go-backup` skill may include scripts under `.agents/skills/42go-backup/scripts/`.

## Constraints

- Do not dump schema definitions, migrations metadata as schema DDL, ownership statements, or privilege statements.
- Do not include `knex_migrations` or `knex_migrations_lock` rows in the dump unless a later refinement answer reverses this.
- Do not rely on seed files for restore correctness.
- Do not make `light` mode include `lingocafe.events` partitions, `lingocafe.translation_cache`, `lingocafe.books`, or `lingocafe.books_pages`.
- Do not require SSR, app runtime, or Next.js server startup for backup or restore.
- Generated SQL dumps must contain the dependency-safe restore instructions needed to truncate target tables and import data correctly.
- Restore must truncate before import, and truncation/import order must be dependency-safe.
- Restore must be designed for a migrated database whose data can be replaced, not for merging into arbitrary existing data.

## Acceptance Criteria

- A new `.agents/skills/42go-backup/SKILL.md` exists and documents how Codex should run backup and restore for this repository.
- The `42go-backup` skill includes reusable implementation code for generating and executing dumps, callable from Makefile and operable by Codex through the skill.
- `make backup mode=full` creates `knex/dumps/{utc-timestamp}.dump.full.sql` using only data from the configured database.
- `make backup mode=light` creates `knex/dumps/{utc-timestamp}.dump.light.sql` and excludes `lingocafe.events*`, `lingocafe.translation_cache`, `lingocafe.books`, and `lingocafe.books_pages`.
- Dumps exclude `knex_migrations` and `knex_migrations_lock` rows because migrations are expected to run manually before restore.
- The generated dump SQL includes the dependency-safe truncate/import instructions required for restore.
- `make restore from=knex/dumps/<file>.dump.<mode>.sql` executes the selected dump file against a database that already has schema from migrations.
- The same dump SQL can be run manually in a database management tool after migrations have been run.
- Restore does not require `mode=<full|light>` because the dump file contains the restore instructions.
- Restore truncates target tables first in dependency-safe order before importing dump contents.
- The dump SQL is ordered so referenced parent rows are inserted before child rows where constraints require it.
- The restore process preserves constraint integrity after completion and fails loudly on SQL errors.
- The implementation creates `knex/dumps` if missing and keeps generated dump files out of source control if needed.
- The final task includes validation notes showing backup plus restore were tested against a migrated empty database.

## Dos

- Use the existing Makefile style and `DATABASE_URL` configuration pattern.
- Prefer deriving or explicitly maintaining a dependency-safe table order over relying on arbitrary database catalog order.
- Account for current foreign-key relationships such as `auth.users` before auth child tables, quicklist projects before tasks/invites/collabs, `lingocafe.books` before pages/progress, and books pages before progress.
- Keep backup modes exactly `full` and `light` unless refinement changes the API.
- Use `make restore from=...` for restore invocation.
- Keep `make restore` as an executor for the selected dump file; put mode-specific behavior in dump generation.
- Fail with a clear message when required parameters are missing or invalid.

## Don'ts

- Do not run `npm dev`.
- Do not dump schema DDL.
- Do not silently ignore missing dump files.
- Do not create restore behavior that only works because constraints are disabled forever.
- Do not treat LingoCafe event partitions as regular data in light mode.
- Do not put the reusable backup logic only in the Makefile if a script would be clearer.
- Do not make the helper script a separate documented public CLI unless implementation needs a narrow internal command for Makefile and skill usage.
