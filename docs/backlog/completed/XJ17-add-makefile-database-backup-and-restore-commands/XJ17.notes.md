# Execution Notes — Add Makefile Database Backup and Restore Commands
**Task**: ./XJ17.task.refined.md
**Plan**: ./XJ17.plan.md

## Decisions

- Started from an accepted plan because the operator explicitly asked to plan and execute in one step.
- Restore behavior is implemented as execution of the dump file. The dump-generation path owns mode-specific exclusions, dependency-safe truncation, and dependency-safe inserts.
- Light mode excludes the specified LingoCafe heavy tables and recursively excludes dependent selected tables that would not restore without those parents. That means `lingocafe.books_progress` is excluded in light mode because it depends on `lingocafe.books` and `lingocafe.books_pages`.
- The helper loads `.env` itself with quiet dotenv output because Makefile variables from `-include .env` are not guaranteed to be exported into recipe processes.
- The generated dump uses SQL `INSERT` statements instead of `COPY FROM stdin` so the file remains easier to run through database management tools.
- Restore uses `psql --set ON_ERROR_STOP=1` so Makefile restore fails on the first SQL error.
- Restore now accepts either a readable explicit path or a bare filename that resolves under `knex/dumps/`.
- Both backup modes now exclude dynamic `notes.notes_*` bucket tables. Those tables are created at runtime by notes functions, not by migrations, so a data-only dump cannot restore them into a migration-only database.
- Restore now also strips `notes.notes_*` truncate entries and data sections from older dumps before executing them, so existing dumps created before the exclusion can still restore.
- Full dumps now include `lingocafe.events_YYYYMM` partition creation statements for months present in dumped event rows.
- Restore injects those same event partition creation statements for older dumps before executing `psql`.

## Problems Encountered

- Initial backup validation failed inside the sandbox with `EPERM` when connecting to PostgreSQL. Re-ran the database-connected checks with approved escalation.
- The first failed backup attempt printed an empty error because the thrown error had no `message`; the script now prints stack/message/string fallback.
- The operator reported `make restore from=20260509T171923Z.dump.full.prod.sql` failed because the script only checked the exact current-directory path. The file was actually under `knex/dumps/`, so bare filename resolution was added.
- Testing that same dump by basename against a disposable migrated database then failed on `notes.notes_2026032210`, proving the older dump included a runtime-created notes table that migrations do not create.
- The notes migration confirms there is no `notes.notes` parent table. `notes.note_add()` creates standalone runtime bucket tables with `CREATE UNLOGGED TABLE IF NOT EXISTS notes.notes_<bucket>`.
- After stripping the old notes bucket, the same old dump reached `lingocafe.events` and failed because the migration-created target database did not have an April 2026 event partition. Event partition creation is now generated/injected from dump contents.
