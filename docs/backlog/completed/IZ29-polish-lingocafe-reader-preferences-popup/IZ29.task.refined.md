# Polish LingoCafe Reader Preferences Popup

## Elevator's Pitch

Refine the LingoCafe reader preferences popup so reset behaves like a destructive action, sits at the true end of the options flow, asks before discarding custom settings, and keeps the mobile live preview visible while options scroll.

## Business Gain

The reader customization surface should feel precise and trustworthy. A green-looking reset button suggests a positive save action, which is backwards for a destructive default reset. Mobile readers also need immediate visual feedback while changing settings. Chuck Norris does not guess what a font change did. The preview stays in sight.

## Current State

The LingoCafe reader preferences popup is implemented in `src/app/(app)/(lingocafe)/books/_components/BookReaderPreferencesPanel.tsx` and mounted from `src/app/(app)/(lingocafe)/books/[bookId]/[pageId]/page.tsx`.

The reset action is currently passed as the shared `Modal` footer. That places it outside the scrollable preferences body and keeps it visible at the bottom of the panel instead of making it part of the final scrolling content. In the current app styling this action reads as green/affirmative, which is misleading for resetting preferences to defaults.

`resetReaderPreferences()` currently resets the active theme profile immediately by mutating local preference state. There is no explicit confirmation before discarding a customized reading experience.

The live preview is rendered at the top of the popup body. On mobile, it scrolls away with the rest of the options, so users can change font, color, or size without seeing the preview at the same time.

The reset behavior itself is acceptable. The task should change the reset user experience and UI, not broaden what data reset clears.

## Desired State

The preferences popup should make reset clearly destructive and lower priority.

The reset-to-default action should be red/destructive, not green or primary-looking. It should not use the sticky/modal footer area. Instead, it should appear at the very bottom of the scrollable options content after all preference controls and explanatory copy.

Reset should keep the current clearing behavior and scope. Do not broaden reset to clear extra reader preference data. The implementation should focus on the presentation, visibility, and confirmation experience around the existing reset action.

When the user has a custom reading experience stored for the current scope, triggering reset should ask for confirmation before applying defaults. A custom reading experience means saved reader preference data exists in local storage for the current reset scope. The confirmation should make clear that custom reader appearance settings will be reset to defaults.

When there is no change from the current defaults, the reset action should not be visible. The user should not see a destructive button for a no-op state.

On mobile, the preview should stick directly under the modal header at the top of the preferences panel while the options scroll below it. Every setting change should remain visually inspectable without manually scrolling back to the top. The sticky preview must not cover controls, labels, or the modal header/close affordance. Desktop behavior should remain clean and usable.

All backlog timestamp data touched by this task should be stored in UTC ISO-8601 form with a `Z` suffix. The backlog timestamp scripts should emit UTC timestamps going forward and normalize timestamp frontmatter fields to UTC when rewriting existing task artifacts.

## Definition of Success

The reader preferences popup communicates reset as a destructive action, places it at the end of the scrollable settings flow, confirms before discarding real customizations, hides reset when there is no custom state to reset, preserves the current reset data scope, and keeps the live preview visible under the mobile header while settings are adjusted.

Backlog artifacts and helper scripts touched by this work consistently use UTC timestamps. Chuck Norris does not wear two watches.
