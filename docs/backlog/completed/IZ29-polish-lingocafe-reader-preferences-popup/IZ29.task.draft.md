# Polish LingoCafe Reader Preferences Popup

## Elevator's Pitch

Refine the LingoCafe reader preferences popup so reset behaves like a destructive action, sits at the true end of the options flow, asks before discarding custom settings, and keeps the mobile live preview visible while options scroll.

## Business Gain

The reader customization surface should feel precise and trustworthy. A green-looking reset button suggests a positive save action, which is backwards for a destructive default reset. Mobile readers also need immediate visual feedback while changing settings. Chuck Norris does not guess what a font change did. The preview stays in sight.

## Current State

The LingoCafe reader preferences popup is implemented in `src/app/(app)/(lingocafe)/books/_components/BookReaderPreferencesPanel.tsx` and mounted from `src/app/(app)/(lingocafe)/books/[bookId]/[pageId]/page.tsx`.

The reset action is currently passed as the shared `Modal` footer. That places it outside the scrollable preferences body and keeps it visible at the bottom of the panel instead of making it part of the final scrolling content. In the current app styling this action reads as green/affirmative, which is misleading for resetting preferences to defaults.

`resetReaderPreferences()` currently resets the active theme profile immediately by mutating local preference state. There is no explicit confirmation before discarding a customized reading experience.

The live preview is rendered at the top of the popup body. On mobile, it scrolls away with the rest of the options, so users can change font, color, or size without seeing the preview at the same time.

## Desired State

The preferences popup should make reset clearly destructive and lower priority.

The reset-to-default action should be red/destructive, not green or primary-looking. It should not use the sticky/modal footer area. Instead, it should appear at the very bottom of the scrollable options content after all preference controls and explanatory copy.

When the user has a custom reading experience stored, triggering reset should ask for confirmation before applying defaults. The confirmation should make clear that the action discards custom reader appearance settings. If the reader is already at defaults, the UI should avoid a noisy confirmation path, either by disabling/hiding reset or by making the no-op harmless.

On mobile, the preview should stay sticky while the options scroll beneath or beside it, so every setting change remains visually inspectable without manually scrolling back to the top. Desktop behavior should remain clean and usable.

## Definition of Success

The reader preferences popup communicates reset as a destructive action, places it at the end of the scrollable settings flow, confirms before discarding real customizations, and keeps the live preview visible on mobile while settings are adjusted.

## Additional Context

The user request came in two points:

1. In the font settings and reading customization popup for LingoCafe, the reset-to-default button at the bottom is green and misleading. It should be red, should not be sticky, should be at the very bottom of the scrolling options, and should ask for confirmation when there is a custom reading experience.
2. On mobile, the preview should be sticky so all options scroll while the preview stays fixed/visible during changes.

## Assumptions

- "Mingo Cafe" in the operator note refers to the existing LingoCafe app and reader preferences popup.
- "custom reading experience" means reader preferences that differ from the defaults or persisted reader preference data in local storage.
- The confirmation is required before applying reset when there are meaningful custom values to discard.
- The reset should target the same preference scope that the current reset targets unless refinement decides otherwise.

## Constraints

- Stay scoped to the LingoCafe reader preferences/customization popup.
- Do not redesign the whole reader surface.
- Do not change the reader route shape or reader API contract.
- Preserve existing preference storage semantics unless implementing the confirmation reveals a specific reset-scope bug.
- Use shared UI primitives such as `Modal`, `Dialog`, or existing shadcn components where they fit.
- Avoid browser-native confirmation UI if an app-native confirmation can be implemented cleanly.
- Run `npm run qa` after implementation.

## Acceptance Criteria

- [ ] The reset-to-default action in the LingoCafe reader preferences popup is visually destructive/red.
- [ ] The reset action no longer appears in the shared modal footer or any always-visible sticky bottom area.
- [ ] The reset action appears after all scrollable preference options, at the bottom of the panel content.
- [ ] Resetting asks for confirmation when there is a custom reading experience to discard.
- [ ] The confirmation copy clearly states that custom reader appearance settings will be reset to defaults.
- [ ] Canceling the confirmation leaves all reader preferences unchanged.
- [ ] Confirming the reset applies the expected default reader preferences for the active scope.
- [ ] The UI handles the already-default/no-custom state without a misleading destructive confirmation flow.
- [ ] On mobile, the live preview remains visible while the user scrolls through and changes options.
- [ ] Mobile options remain scrollable without overlapping the sticky preview.
- [ ] Changing font size, font family, background, or text color updates the sticky mobile preview immediately.
- [ ] Desktop layout remains usable and does not regress because of the mobile sticky preview changes.
- [ ] The preferences popup remains accessible through the existing reader controls.
- [ ] The shared `Modal` behavior for open, close, escape, focus, and panel sizing remains intact.
- [ ] `npm run qa` passes after implementation.

## Dos

- Do keep the reset action visually distinct from save/confirm/primary actions.
- Do place reset in the natural reading order after the settings controls.
- Do confirm only when there is something meaningful to discard.
- Do verify mobile behavior with enough options scrolled to prove the preview stays visible.
- Do keep the implementation local to the reader preferences flow unless a shared primitive needs a small extension.

## Don'ts

- Don't leave reset styled as a green, primary, or affirmative action.
- Don't keep reset in a sticky footer.
- Don't make the reset confirmation fire for every click if the current state is already default.
- Don't introduce a custom overlay if the existing modal/dialog primitives cover the confirmation use case.
- Don't let the sticky preview cover controls, labels, or modal header/close affordances on mobile.
- Don't change unrelated reader navigation, table of contents, progress tracking, or book content rendering.

## Open Questions

- Should reset clear only the active theme profile, or should it also clear shared font-size storage when the user expects a full default reading experience?
- Should the reset action be disabled when there are no custom preferences, or visible as a no-op with neutral helper text?
- Should the confirmation use a nested shared `Modal`, shadcn alert dialog, or an existing app confirmation pattern if one is available?
