# Configure LingoCafe Icons and PWA Assets

## Elevator's Pitch

LingoCafe should use its own favicon, app icon, logo, and PWA icons from the assets now stored under `public/lingocafe` and `public/lingocafe/favicon`.

## Business Gain

The app looks like LingoCafe in browser tabs, installed home-screen shortcuts, and PWA metadata instead of wearing the default 42go jacket. Chuck Norris does not let a production app borrow another app's boots.

## Current State

The operator added website favicons to `public/lingocafe/favicon` and the regular icon and logo to `public/lingocafe`.

Known files include:

- `public/lingocafe/lingocafe.png` — 1024x1024 PNG, best available source for generated PWA icon sizes.
- `public/lingocafe/logo.png` — 1024x1024 PNG with alpha, candidate for app logo display if existing config consumers support it.
- `public/lingocafe/favicon/favicon.ico`
- `public/lingocafe/favicon/favicon-16x16.png`
- `public/lingocafe/favicon/favicon-32x32.png`
- `public/lingocafe/favicon/favicon-180x180.png`
- `public/lingocafe/favicon/favicon-192x192.png`
- `public/lingocafe/favicon/favicon-256x256.png`
- `public/lingocafe/favicon/mstile-150x150.png`

`WP59` added a LingoCafe `public.pwa` block with `startUrl: "/books"`, so the PWA configuration surface already exists for this app.

`src/config/lingocafe/config.ts` currently defines `public.pwa.icons.appleTouch180`, but it points to `/images/icons/default-180.png`.

The shared PWA type in `src/42go/pwa/types.ts` supports `appleTouch180`, `manifest192`, `manifest512`, and `maskable512`.

`src/app/manifest.ts` builds the web app manifest from `config.public.pwa.icons.manifest192`, `manifest512`, and `maskable512`.

`src/42go/pwa/HeadTags.tsx` emits the Apple touch icon from `config.public.pwa.icons.appleTouch180`.

The current source inspection did not find an obvious 42go config field dedicated to favicons.

`src/AppConfig.ts` has a top-level `logo?: string | ComponentType` field, and both public and app layout `AppTitle` components use `toolbarConfig?.icon || config.logo`. This means LingoCafe can likely wire the existing `public/lingocafe/logo.png` through `logo: "/lingocafe/logo.png"` without a new system feature, subject to visual verification.

The current source inspection also found `src/proxy.ts` excludes `/favicon.ico` and `/manifest.webmanifest` from request proxy matching.

Official Next.js App Router guidance says:

- `public` is the correct root folder for ordinary static assets that should be referenced from `/...` URLs.
- For static metadata files such as favicons, App Router recommends special metadata files inside `app`.
- `favicon.ico` can only live at the top level of the app route segment and is therefore a poor fit for one static favicon per dynamically resolved app.
- `icon` and `apple-icon` can be colocated in route segments or generated with code, but the current 42go system already emits Apple touch icon tags and the manifest from `config.public.pwa`.

## Desired State

The implementation should record the confirmed Next.js guidance in the task outcome or project docs: `public/lingocafe` is acceptable for app-specific static assets referenced by config, while App Router metadata file conventions are the preferred static metadata mechanism for single-app/static favicon cases.

LingoCafe then uses its own PWA icon assets through `config.public.pwa.icons` wherever the current 42go system supports them.

If the 42go AppConfig standard does not support configurable favicons, the task should raise that as a system-level issue and create or propose a follow-up task for adding app-configurable favicon support for every app. The expected system-level direction is a config-driven solution, not a static global `src/app/favicon.ico` replacement that only works for one app.

The follow-up system-level task has been drafted as `RO05`.

Within this LingoCafe-specific task, generate any missing PWA icon size that can be safely derived from the best available source image. The preferred source is `public/lingocafe/lingocafe.png` because it is a 1024x1024 icon source. In particular, generate a 512x512 PNG if it is needed for `manifest512` or `maskable512`.

Also wire the existing LingoCafe logo and general website favicon where the current system already supports doing so without inventing new framework behavior. If favicon support is missing, leave that portion to `RO05` rather than hardcoding a global favicon workaround.

## Definition of Success

LingoCafe uses LingoCafe-specific PWA icons through the existing configuration surface, missing icon sizes are generated from the best available LingoCafe source where practical, the app logo is configured if current config consumers support it, the asset-placement decision is documented against current Next.js guidance, and favicon handling is either implemented through existing shared config support or explicitly delegated to `RO05`.

## Assumptions

The assets added by the operator are intended to be committed static assets.

The currently added image sizes are the intended first pass unless refinement identifies missing required PWA sizes, such as a dedicated 512x512 manifest icon.

Because the current asset list includes 180, 192, and 256 pixel favicon PNGs but no explicit 512 pixel PNG, full manifest icon coverage should generate a 512x512 PNG from `public/lingocafe/lingocafe.png` if the available local tooling can do it cleanly.

The favicon should be app-specific for LingoCafe, not a global replacement that affects every app in the monorepo.

Existing `public.pwa` support should be reused before adding new configuration shape.

## Constraints

Use the existing multi-app AppConfig architecture.

Do not hardcode LingoCafe favicon behavior in a way that prevents other apps from configuring their own icons later.

Do not move assets unless the implementation chooses a project convention that is clearly better than the operator's current `public/lingocafe` placement.

Do not solve unrelated branding, landing page, or book cover work in this task.

Do not design the final shared 42go icon schema inside LT47; that belongs to `RO05`.

Run `npm run qa` after implementation.
