# Fix Auth Users Name Unique Constraint

## Elevator's Pitch

Separate credential usernames from OAuth display names so username uniqueness remains valid without blocking OAuth users who share the same real name.

## Business Gain

Users should be able to sign in with OAuth accounts that have the same real name but different email addresses. Password-based login should still have a unique username boundary when an app supports credentials.

## Current State

Google OAuth sign-in can fail when inserting into `auth.users` with:

`duplicate key value violates unique constraint "users_app_id_name_unique"`

The observed failing key is `(app_id, name)=(lingocafe, Marco Pegoraro)`. The current auth migration defines both `unique(["app_id", "name"])` and `unique(["app_id", "email"])` on `auth.users`.

The `name` column currently carries two meanings:

- For credentials login, it behaves like a username and is expected to be unique inside an app.
- For OAuth login, it is populated from provider profile data and usually means the user's real display name.

OAuth callback logic already looks up existing OAuth users by `app_id + email` before inserting a new `auth.users` row.

Credentials login currently accepts either `name` or `email` as the username field, using a case-insensitive lookup against `name` or `email`.

## Desired State

`auth.users` should distinguish between credential username and display name. The credential username should remain unique per app when present. The display name should be allowed to duplicate.

Fresh database rebuilds and existing migrated databases should end up with the same intended constraint model: unique `app_id + email`, unique `app_id + username` when a username exists, and no uniqueness constraint on real/display names.

For this project phase, local database history may be rewritten in place. The original auth migration should be edited as a clean destructive do-over for local rebuilds. A root-level `migrate.sql` file should provide the manual production patch that brings the existing deployed database into the same final schema shape.

## Definition of Success

OAuth sign-in succeeds for different email addresses that share the same real/display name within the same `app_id`.

The database still prevents duplicate user emails within the same app.

Password users still have unique app-local usernames.

Existing databases can be migrated without dropping unrelated auth data.

The deployed Quicklist database can be patched manually without losing existing data.

## Additional Context

The error was observed during Google OAuth sign-in for the `lingocafe` app on 2026-05-02. Middleware logs are expected and are not the defect.

The likely root cause is semantic overload of the `name` column plus the `users_app_id_name_unique` constraint created by `knex/migrations/20240320_auth.js`.

The historical auth task `ABA` intentionally implemented credentials login by username or email. That remains valid if username becomes its own field instead of sharing storage with OAuth display names.

There is an uncommitted candidate implementation in the working tree. It should be treated as draft execution context only until this task is fully refined and planned.

The operator explicitly accepts editing the existing auth migration in place because the local database will be destroyed and rebuilt. Production is non-critical but contains a few Quicklist datapoints that should be preserved through a manual SQL patch.

## Assumptions

`name` should be treated as a display name, not a username or login identifier.

`email` is the correct stable uniqueness boundary for app-local auth users.

Password-based users need a separate username field if username login remains supported.

## Constraints

Use Knex migrations for database schema changes.

For this task, do not add a new Knex migration for the username split. Edit the existing auth migration in place for the local clean rebuild path, and create a standalone root `migrate.sql` for the deployed database patch.

Do not weaken the existing `app_id + email` uniqueness rule.

Avoid reseeding or deleting auth data as the primary fix.

Keep password-based username login, but move it onto a dedicated username field.

Split demo user seed data into clearer per-app seed files: default test users, Quicklist test users, and Lingocafe test users.

If existing password users have duplicate names during the manual production patch, fail hard instead of generating usernames automatically.

## Don'ts

- Don't make display names globally or app-locally unique.
- Don't remove email uniqueness.
- Don't solve this by changing user names during OAuth insert.
- Don't delete user data to bypass the constraint.
- Don't leave credentials login silently dependent on whichever duplicate name the database returns first.
- Don't force OAuth users to invent a username during provider sign-in unless a later product flow explicitly needs it.
- Don't add a new Knex migration for this task unless the operator reverses the destructive local rebuild decision.
