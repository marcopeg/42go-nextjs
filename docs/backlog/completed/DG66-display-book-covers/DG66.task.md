# Display Book Covers

## Elevator's Pitch

Display book cover images on the authenticated LingoCafe `/books` route by resolving each book's cover from its book id or stored relative cover path.

## Business Gain

Book covers make the catalog easier to scan and make LingoCafe feel like a real reader instead of a bare database table. Chuck Norris does not browse placeholder cards. The covers move when he looks at them.

## Current State

LingoCafe has a `/books` route and ongoing work to replace the fake list with real catalog data.

The `lingocafe.books` table already has a nullable `cover` column, but the current seed data sets `cover: null` for every demo book.

The current `/books` UI does not render covers. It still has placeholder book data while the language-filtered catalog task is in progress.

## Desired State

The `/books` route displays a cover image for each book returned by the LingoCafe catalog API.

Cover images should be resolved consistently from each book's id. The first storage convention is `public/images/lingocafe/<book-id>.jpg`, exposed to the browser as `/images/lingocafe/<book-id>.jpg`.

The implementation may also keep using `lingocafe.books.cover` as an explicit override when that value is present, but the default behavior should work from `book.id` alone.

The cover files are static, committed repository assets for now. Runtime uploads are out of scope for this task.

## Definition of Success

The LingoCafe book catalog shows optimized cover images for seeded and future books using committed static assets under `public/images/lingocafe/`, with a generic placeholder image when a book-specific cover is missing.

## Additional Context

The operator asked: "display books covers. the /books route should display the books covers by resolving the relative image by book id. we need to refine where the images should be placed so to follow optimal nextjs patterns for image optimization and caching."

This should build on the language-filtered `/books` work instead of replacing it.

Use `public/images/lingocafe/<book-id>.jpg` as the default cover path. If `lingocafe.books.cover` is present, it can override the inferred path, but API responses must expose browser-safe public URLs only.

The implementation should add a generic placeholder image asset and use it when a book has no resolvable cover.

## Assumptions

Book cover assets are local committed application assets, not remote third-party URLs, unless later requirements say otherwise.

Next.js image optimization should be used where it fits the chosen asset location.

The first useful cover set should come from seeded/demo books and committed image files.

## Constraints

Follow the existing app-page convention for `/books`: client component, `AppLayout`, and browser-side fetching from the API.

Do not hardcode cover URLs only in the page component if the source of truth should live with book catalog data.

Do not introduce a storage convention that fights Next.js static asset serving or image optimization.

Keep the cover path stable across local development and production Docker builds.

Use the public asset convention `public/images/lingocafe/<book-id>.jpg` for book-specific covers.

Build and commit a generic placeholder cover image for missing covers.

Do not solve runtime upload storage in this task. Public assets are acceptable because covers are static and committed for now.

## Acceptance Criteria

- [ ] The `/books` route renders a cover image for every book that has a resolvable cover.
- [ ] The default cover URL is inferred as `/images/lingocafe/<book-id>.jpg`.
- [ ] If `lingocafe.books.cover` is present, it is treated as an explicit public URL override.
- [ ] Book-specific cover files live under `public/images/lingocafe/` and are committed static assets.
- [ ] The LingoCafe seed data or catalog setup provides cover references or book ids that resolve for the demo books.
- [ ] Missing covers render a generic committed placeholder image instead of a broken image.
- [ ] The placeholder image is created as part of the implementation.
- [ ] The client uses `next/image` for cover rendering.
- [ ] The API response shape includes whatever cover data the client needs without exposing filesystem paths.
- [ ] The solution works in local development and production standalone/Docker builds.
- [ ] `npm run qa` passes after implementation.

## Dos

## Don'ts

- Don't scatter cover paths across unrelated UI code.
- Don't store absolute local filesystem paths in API responses.
- Don't depend on remote image domains unless that is explicitly chosen and configured.
- Don't break the current `/books` authenticated app-page flow.
- Don't make the browser guess private server paths.
- Don't solve runtime uploads or persistent object storage in this task.

## Open Questions

Q: Should cover images live under `public/` with stable relative URLs, or be imported from source-controlled app assets?
A: Cover images should live under `public/images/lingocafe/` as committed static assets for now.

Q: Should the canonical `books.cover` value be a relative public URL, or should the app infer the cover path directly from `book.id`?
A: Infer the default cover path from `book.id` as `/images/lingocafe/<book-id>.jpg`. If `books.cover` is present, treat it as an explicit public URL override.

Q: What fallback visual should be shown when a book has no cover?
A: Use a generic image placeholder and build it as part of this task.

Q: Should generated/demo covers be committed as static assets or produced by a separate content pipeline?
A: Commit static cover assets at this point. A separate runtime/content pipeline is out of scope.
