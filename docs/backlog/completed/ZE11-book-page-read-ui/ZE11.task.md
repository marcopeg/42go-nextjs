# Book Page Read UI

## Elevator's Pitch

Implement the LingoCafe route and UI for reading one page from a book, using a URL that can survive full page reloads and restore the reader's vertical position.

## Business Gain

Readers need the "Read now" and "Continue reading" actions to land on a real reading surface. A saved position should reopen where the reader left off. Chuck Norris does not scroll back to his place. The page scrolls to him.

## Current State

`IL68` completed the book details route and base URL shape for book-specific navigation.

`PR58` completed the book details primary action. It creates read links in the shape `/books/<bookId>/<pageId>` and appends `progress_bps` as the resume query parameter when saved progress exists. It explicitly does not create the read page UI.

The requested gap is the page reading route and presentation layer.

## Desired State

The app supports `/books/[bookId]/[pageId]` for reading a specific page from a specific book. The route extends the `/books/[bookId]` detail URL from `IL68` by adding the page ID, so full page reloads reopen the same book page.

When the URL includes the progress indicator from `PR58`, the read UI should restore the reader's initial vertical scroll position using the `progress_bps` value. The value is a basis-point percentage from `0` to `10000`.

Opening a page should record a `page-open` event and create or update `lingocafe.books_progress` for the authenticated user with the current `book_id`, current `page_id`, and `progress_bps = 0`. This can be populated server-side by the page content API when it returns the requested page, so the app remembers the last opened page even if the user leaves before scrolling.

Scrolling the page should record `page-scroll` events from the client and update `lingocafe.books_progress` for the authenticated user. Scroll events should be collected over a 10-second window and only the last scroll position in that window should be sent, so continuous scrolling does not flood the event stream. The same last observed `progress_bps` value should be used to update saved progress for the current `book_id` and `page_id`.

The page body should render a small markdown subset from `books_pages.content`: paragraphs, italic, bold, and titles.

At the end of the page, the reader should show previous and next page buttons when those neighboring pages are available.

On desktop, the read page should feel similar to the book detail surface and include a back link to the book detail.

On mobile, the read page should render as a full-screen popup above the book details popup. Going back from the read page should slide this top popup away and reveal the book details underneath.

## Definition of Success

Users can open a stable read-page URL for a book page, refresh it without losing the selected page, and optionally resume at the vertical position encoded in the URL. Desktop reads as an internal page with a back link to book details. Mobile layers the read popup above book details so back navigation reveals the details popup underneath.

The read flow also records `page-open` and low-volume `page-scroll` activity into the LingoCafe event stream through the existing event model, with scroll reporting reduced to the last position observed within each 10-second collection window. Opening a page saves that page with `progress_bps = 0`, and later scroll reporting updates the user's saved `books_progress` row so "Continue reading" resumes from the latest opened page or latest reported scroll position.

## Assumptions

The read URL should be `/books/[bookId]/[pageId]`.

The `progress_bps` URL value should be treated as the requested initial scroll position, clamped to the valid `0` to `10000` range.

The page content should come from the LingoCafe book page data, not from the book details payload alone.

The read page content should render the existing `books_pages.content` value as a limited markdown document supporting paragraphs, italic, bold, and titles.
