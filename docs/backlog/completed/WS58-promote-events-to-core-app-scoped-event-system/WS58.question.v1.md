## Questions

### 1. PostgreSQL physical layout: which direction should win unless implementation proves it impossible?

The draft lists three viable shapes:

- one core parent table `events.events` partitioned by `created_at`, with mandatory `app_id` and app/time indexes
- list partition by `app_id`, then range partition by `created_at`
- separate parent table per app, each time-partitioned

My default recommendation is the first one: one core parent table, mandatory `app_id`, monthly time partitions, and indexes for `(app_id, created_at, id)` plus event-name queries. It keeps migrations and writers simple. App isolation is logical, not physical. Physical app partitioning can come later if volume punches us in the face.

Answer:

one core parent table `events.events` partitioned by `created_at`, with mandatory `app_id` and app/time indexes
(we go with your suggestion)


### 2. Should WS58 migrate old `lingocafe.events` rows into the new core table, or only switch future writes?

Choose the first implementation scope:

- future writes only, leaving historical `lingocafe.events` where they are
- one-time migration/copy from `lingocafe.events` into `events.events`
- compatibility export that reads both old and new tables without moving old rows

Future writes only is the smallest blast radius. Compatibility export is better for analytics continuity. One-time migration is heavier and needs careful duplicate handling.

Answer:

no need to migrate the data.
but you should remove any lingocafe events code support, and using the current events logging data point using the new core apis.

### 3. What should happen to app-specific dimensions like `book_id` and `page_id`?

The old LingoCafe table has `book_id` and `page_id` columns. A core table can either:

- keep only generic columns and put app-specific dimensions in `data`
- include optional generic dimension columns such as `entity_type`, `entity_id`, `parent_entity_id`
- keep convenience columns like `book_id` and `page_id` even though they are LingoCafe-shaped

My default recommendation is generic core columns plus `data`, with no `book_id` or `page_id` in core. Chuck Norris does not let one app sneak its furniture into the platform foundation.

Answer:

yes, those will be custom datapoints under "data".
the whole system is design to collect a hot buffer that will be downloaded. 
we are not going to use it for live analytics.
we don't even need much indexing more than the data download cursor support.

- no strict convention beyond length and character validation

This decision affects auth events, client events, docs, and future analytics.
