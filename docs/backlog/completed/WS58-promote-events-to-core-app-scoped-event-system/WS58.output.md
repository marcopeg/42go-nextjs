# Output — Promote Events to Core App-Scoped Event System

## Schema Decision

Use one core PostgreSQL parent table:

```text
events.events
```

The table is app-scoped through a mandatory `app_id` column and partitioned by `created_at` into monthly range partitions.

V1 does not use list partitioning by app and does not create one parent table per app.

## Why

PostgreSQL is the hot event buffer, not the analytics warehouse. The most valuable write path is simple and boring:

- one writer
- one table contract
- one partition helper
- one cursor model
- one export pipeline

App separation is logical through `app_id`. The export process can filter or partition files by app after download.

## Rejected For V1

### List partition by app, then time

Rejected for now because it raises migration and partition-management cost before the project has enough event volume to justify it.

### One parent table per app

Rejected because it fragments the core API and export tooling. It also makes every new app a schema-management event.

## Cursor Contract

Export progress uses:

```text
(created_at, id)
```

`created_at` is ingestion time and partition time.

`event_at` is behavioral time. It is useful for analysis, but it is not trusted as the export cursor.

This already handles multiple rows sharing the same timestamp because `id` is the tie-breaker.

## Event Time And Export Strategy

Recommended next hardening step: add a server-assigned `ingest_seq bigint` and use it as a compact deterministic tie-breaker in export ordering, for example `(created_at, ingest_seq)`.

Do not use a sequence by itself as the only correctness proof. PostgreSQL sequence allocation is not commit ordering, so a robust exporter should still use an ingestion-time safety horizon or overlap window before advancing durable state.

Keep PostgreSQL partitioning on ingestion time (`created_at`) for hot-cache reliability. Do not partition the hot table by client-provided `event_at`; late, backdated, future, or clock-skewed client events would make inserts depend on arbitrary old/future partitions.

Use `event_at` for analytics layout after download. The export/archive layer can write Parquet under paths such as:

```text
events/parquet/app_id=lingocafe/event_date=2026-05-11/batch-...
```

Late events can still be appended into their behavioral `event_date` partition, and periodic compaction can merge small files. Analysis should sort by `(event_at, created_at, id)` or `(event_at, ingest_seq)` when exact order matters.

## Core Row Shape

The intended generic row shape is:

- `created_at`
- `id`
- `app_id`
- `user_id`
- `event_at`
- `name`
- `data`
- `meta`

App-specific fields such as LingoCafe `book_id` and `page_id` live in `data`.

## Production Cleanup

WS58 does not migrate old `lingocafe.events` rows.

WS58 does not drop production tables.

Fresh local rebuilds should stop creating `lingocafe.events`, but production cleanup of the old table is manual.

## Export

The export skill now uses:

```bash
python3 .agents/skills/42go-events-export/scripts/export_events.py
```

`EVENTS_DATABASE_URL` can point exports at a dedicated source database. When unset, the exporter falls back to `DATABASE_URL`.

Default local archive:

```text
.local/42go-events-analytics
```

The old `LC_EVENTS_DATABASE_URL` is superseded.
