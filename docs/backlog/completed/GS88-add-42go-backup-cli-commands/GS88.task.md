# Add 42go Backup and Restore CLI Commands

## Elevator's Pitch

Move the database backup and restore operational surface into the 42go CLI. Backup should run as `42go backup --full` or `42go backup --light`, restore should run as `42go restore --from <dump>`, and the Makefile targets should remain only as wrappers around those CLI commands.

## Business Gain

Operators and developers get one consistent command surface for backups and restores across direct CLI usage, automation, and Makefile workflows. The Makefile remains convenient, but the reusable operational entry point moves into the 42go CLI where other repository commands already live.

## Current State

`make backup` currently runs `node .agents/skills/42go-backup/scripts/backup.mjs backup --mode "$(mode)"`. `make restore` currently runs `node .agents/skills/42go-backup/scripts/backup.mjs restore --from "$(from)"`. The backup and restore logic lives in the `42go-backup` skill script, and the skill documentation describes Makefile usage as the public command surface.

The repository also has a local pipx-installable `42go` CLI used by other workflows, including events export. There is no `42go backup --full`, `42go backup --light`, or `42go restore --from <dump>` command yet.

## Desired State

The 42go CLI owns the backup and restore implementation. Backup supports explicit flag-based mode selection:

- `42go backup --full`
- `42go backup --light`

If neither `--full` nor `--light` is provided, the CLI shows an inline multiple-choice prompt so the operator chooses which mode to run. The implementation should reject conflicting flags such as `--full --light`.

Restore supports:

- `42go restore --from <dump-path-or-name>`

If `--from` is not provided, the CLI lists available backups from newest to oldest and prompts the operator to choose one.

`make backup mode=full`, `make backup mode=light`, and `make restore from=<dump>` continue to work, but they become wrappers around the CLI commands instead of directly invoking the backup script. The Makefile assumes `42go` is available on PATH and fails hard if it is not. No local fallback logic is needed.

## Definition of Success

Backup and restore are available through the CLI, with Makefile targets acting only as wrappers. The backup/restore rules remain compatible with the current dump format and table-selection behavior, but the implementation lives in or is called from the CLI as the source of truth.

## Assumptions

- The CLI implementation should become the source of truth for backup and restore behavior, either by moving the existing script logic into the CLI package or by making the CLI call shared implementation code that owns the behavior.
- The existing dump output location and environment variable behavior should remain unchanged unless implementation discovers the current CLI conventions require a documented adjustment.
- Interactive prompts can use Typer-compatible prompting patterns already available through the CLI dependencies.

## Constraints

- Do not remove `make backup`.
- Do not remove `make restore`.
- Do not change backup mode names away from `full` and `light`.
- Do not fork backup logic into a second implementation with different table exclusion rules.
- Do not add Makefile fallback logic for missing `42go`; missing CLI should fail hard.
- Do not run `npm dev`.
- Run `npm run qa` after code changes during execution.

- `42go backup --full --light` fails loudly with a useful message.
- The CLI help output documents the backup command and both mode flags.

- The Makefile assumes `42go` is available and fails hard if it is missing.
