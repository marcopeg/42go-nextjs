# Design Reusable Overlay Surface Component and Agent Skill

## Elevator's Pitch

Create a generic core visual component for the 42Go / Forti2Go library that can power mobile full-screen pop-outs, desktop side panels, and centered modal-style overlays through one reusable API. The task also includes creating an agent skill that documents how to use the component, how to choose its modes and properties, and how future agents should apply it instead of inventing one-off dialogs, sheets, or popups.

## Business Gain

The platform gets one reusable overlay surface instead of many ad-hoc UX patterns. That means faster implementation of guided flows, more consistent mobile and desktop behavior, lower UI drift across apps, and a documented path for both humans and agents to reuse the same primitive. Chuck Norris does not let every feature build its own modal in the woods.

## Current State

The codebase already has a reusable panel abstraction from [AEC](../../completed/AEC-abstract-panel-component-aec/AEC.task.md), implemented under `src/42go/components/panel/`. That work is about static content panels and includes SRP primitives such as `Panel`, `PanelHeader`, `PanelBody`, `PanelFooter`, and `SimplePanel`. It does not own overlay behavior, portals, backdrop handling, Escape handling, focus management, body scroll locking, or responsive sheet/modal presentation.

The existing `Panel` component overlaps only in static composition language: title, description, actions, body, footer, spacing, and visual shell. It is not a modal/dialog primitive. GM77 should check whether any visual primitives can be reused without confusing responsibilities, but the overlay lifecycle must belong to the new component family.

There is no confirmed core primitive yet for a component that can:

- cover the full mobile viewport
- fade or slide from configurable anchors
- render optional header, body, and footer areas
- behave as a side panel on desktop
- behave as a centered modal on desktop
- provide overlay behavior and click-to-close defaults
- provide keyboard, focus, and scroll-lock behavior through one reusable contract
- give agents a standard documented way to reuse it

Existing code and planned UX work already point at this missing primitive:

- `src/app/(app)/(lingocafe)/books/_components/BookReaderPreferencesPanel.tsx` currently owns bespoke backdrop, Escape-key close, fixed shell, responsive sizing, and right-side panel behavior.
- `src/42go/components/docs/MobileSidebar.tsx` has its own mobile bottom-sheet style overlay.
- `src/42go/layouts/app/AppLayout.tsx` has separate mobile sidebar overlay behavior.
- [ABX](../ABX-improve-user-menu-abx/ABX.task.draft.md) needs a mobile full-screen slide-in user menu.
- [TF15](../TF15-move-reader-customization-to-core-overlay-surface/TF15.task.draft.md), [ZJ19](../ZJ19-after-gm77-add-mobile-reader-table-of-contents-popup/ZJ19.task.draft.md), and [BL01](../BL01-books-top-bar-language-level-switcher-via-gm77-panel/BL01.task.draft.md) are direct follow-up consumers of this primitive.

The project currently has Radix Dropdown Menu and Popover wrappers, but no committed Dialog or Sheet wrapper has been confirmed in the codebase. A code search also did not find an existing `@radix-ui/react-dialog` dependency.

The current authenticated app route tree has routes directly under `src/app/(app)/` and a LingoCafe route group under `src/app/(app)/(lingocafe)/`, but no `src/app/(app)/(default)/` route group yet.

## Desired State

Introduce one core reusable overlay surface component family with a clear name and a clear API.

The final public component name should be `Modal`. The implementation may still be built on Radix Dialog semantics under the hood, but callers should import and use `Modal`. The companion skill should be named `42go-ui-modal`.

The component should support both mobile and desktop with different visual behaviors:

- On mobile, the simplest and default behavior is a full-screen covering surface.
- Opening and closing animation should be inferred from `presentation` plus `anchor`, with no public `animation` prop for the first version.
- Slide directions should be configurable through a single directional concept such as `anchor` or equivalent:
  - `right`
  - `top`
  - `bottom`
  - `left`
- On mobile, the component should support optional `header`, `body`, and `footer` blocks.
- The `body` is the main content area and should scroll automatically when content overflows.
- The `header` should support:
  - title
  - subtitle
  - left-aligned two-line content by default
  - optional centered title mode
  - right-aligned actions
  - a default close action/button

The primary caller API should use prop slots plus `children` for the main body. A simple target shape is:

```tsx
<Modal title="foobar" actions={actions}>
  this is the main body
</Modal>
```

- The `footer` should support:
  - right-aligned actions
  - left-side helper slot/content

On desktop, the same primitive should also support:

- side-panel behavior that slides in/out from an anchor
- overlay with default click-to-close behavior
- configurable sizing through presets, such as `sm`, `md`, `lg`, `xl`, and `full`
- a centered modal mode above the overlay

On mobile, the component should always present full-screen by default and should not require caller props just to get the mobile full-screen behavior. On desktop, the component should support exactly two top-level presentation families:

- centered screen modal
- anchored side panel

The desktop display concept should use the simple top-level prop API from refinement round 2:

```tsx
<Modal
  open={open}
  onOpenChange={setOpen}
  title="Preferences"
  presentation="panel"
  anchor="right"
  size="md"
>
  Body content
</Modal>
```

The desktop display concept should use one `presentation` attribute with multiple values:

- `panel`
- `modal`

In `panel` mode, the component behaves like an anchored sheet or side panel.

In `modal` mode, the component behaves like a centered classic popup with configurable size constraints.

Animation behavior should be inferred:

- `presentation="modal"` uses centered modal animation, such as fade/scale.
- `presentation="panel"` uses slide animation based on `anchor`.
- There should be no public `animation` prop for now.

Sizing should use presets only in the first version. Raw min/max sizing props are not required for GM77 unless implementation proves a preset cannot cover the reader preferences migration or the demo surfaces.

The final task should also create a dedicated agent skill for this component, `42go-ui-modal`, so any future agent working on this repository can read one canonical guide describing:

- when to use the component
- when not to use it
- naming conventions
- all relevant properties
- mobile vs desktop behavior rules
- header/body/footer composition rules
- animation and anchor usage
- common recipes and examples

## Definition of Success

There is one agreed reusable core overlay primitive in the library, with naming that is defensible and not too narrow. The component supports the requested mobile and desktop behaviors without splitting into unrelated one-off components. The public API is documented clearly enough that future tasks can reuse it consistently. The companion agent skill exists and becomes the default guide whenever an agent needs a modal, popup, sheet, side panel, or full-screen overlay surface.

The task is not complete if it only creates another feature-specific popup, if it leaves keyboard/focus/backdrop behavior to every caller, or if the agent skill is too vague to prevent future one-off overlay work.

The concrete proof of reusability inside GM77 should be migration of the LingoCafe reader appearance preferences popup in `src/app/(app)/(lingocafe)/books/_components/BookReaderPreferencesPanel.tsx`, replacing its custom modal/popup surface mechanics where possible. The migration should replace both current mobile and desktop reader preference surfaces as much as practical, simplifying product code and moving general overlay logic into the core `Modal` component.

## Additional Context

The task should explicitly help decide the best product and code name for the component.

The naming problem matters because the component is broader than a classic modal:

- `modal` may be too narrow because side panels and mobile full-screen sheets are also in scope
- `popup` may be too informal and visually vague
- `sheet` may fit the sliding panel behavior well but may be too narrow for centered dialog behavior
- `panel` already exists as a static content primitive in `src/42go/components/panel/`
- `overlay` describes the layering mechanism but not necessarily the surface itself

The chosen direction is to expose a `Modal` component with desktop presentation options for centered modal vs side panel, while keeping mobile full-screen by default. Radix Dialog can still be used internally, but the public library primitive should stay named `Modal`.

The operator also wants demo/styleguide pages for comparing the existing static `Panel` with the new `Modal`. Create a new authenticated route group at `src/app/(app)/(default)/` and place `demo-panel` and `demo-modal` there. These pages should be linked in the main menu of the demo/default application, which serves as the boilerplate demo app.

This work should be treated as a core-library concern, not as a feature-specific one-off implementation. It should also define the skill name and skill trigger clearly enough that future agent tasks such as ABX, TF15, ZJ19, and BL01 can invoke it without guessing.

## Assumptions

- The component belongs in the shared 42Go / Forti2Go library area, not in an app-specific feature folder.
- Existing panel work from [AEC](../../completed/AEC-abstract-panel-component-aec/AEC.task.md) should inform the composition model for header/body/footer, but this new component is a different primitive with overlay behavior.
- A single component family with composable props or subcomponents is preferred over separate unrelated implementations for mobile full-screen popout, desktop panel, and desktop modal.
- The future implementation should use Radix as much as possible for dialog accessibility and lifecycle behavior.
- The implementation may need to add a Radix Dialog / shadcn dialog-style foundation because no existing Dialog wrapper or `@radix-ui/react-dialog` dependency has been confirmed.
- The companion agent skill should live with the repository skills and document the final component API after implementation.
- Follow-up tasks such as TF15, ZJ19, and BL01 should consume the primitive instead of redefining overlay mechanics locally.
- Demo/styleguide pages should be created so humans and agents can compare `Panel` and `Modal` behavior visually.
- Modal sizing should start with presets only, not raw sizing props.
- Modal animation should be inferred from `presentation` and `anchor`, not configured through a public `animation` prop.

## Constraints

- Preserve the multi-app nature of the library and avoid app-specific assumptions.
- Keep the component reusable across both mobile and desktop.
- Keep header and footer optional.
- Keep the body scroll behavior built in, not left to every caller to rediscover.
- Support both overlay plus anchored panel behavior and overlay plus centered modal behavior.
- Include accessible close behavior, keyboard behavior, focus behavior, and body scroll behavior in the shared primitive or in the chosen underlying primitive.
- Do not add raw min/max sizing props in the first version unless implementation proves presets are not enough.
- Do not add a public `animation` prop in the first version.
- The task includes the agent skill as a first-class deliverable, not as optional documentation.
- Do not narrow the solution too early to one implementation detail or one library name before the naming decision is reviewed.
- Do not mutate follow-up product tasks such as TF15, ZJ19, and BL01 as part of this refinement.
- The GM77 implementation scope does include migrating the LingoCafe reader appearance preferences popup as the first proof consumer, including both mobile and desktop behavior where practical.

- Do keep the API unified across mobile and desktop.
- Do define the behavior model in terms of reusable concepts such as mode, anchor, overlay, header, body, footer, actions, and helpers.
- Do make the mobile behavior simple by default.
- Do document how agents should choose between the component modes.
- Do review existing work such as [AEC](../../completed/AEC-abstract-panel-component-aec/AEC.task.md) and dependent candidates such as [ABX](../ABX-improve-user-menu-abx/ABX.task.draft.md), [TF15](../TF15-move-reader-customization-to-core-overlay-surface/TF15.task.draft.md), [ZJ19](../ZJ19-after-gm77-add-mobile-reader-table-of-contents-popup/ZJ19.task.draft.md), and [BL01](../BL01-books-top-bar-language-level-switcher-via-gm77-panel/BL01.task.draft.md).

- Do make `children` the main body content in the simple caller API.
- Do create demo/styleguide pages for both `Panel` and `Modal` so the design difference is visible.
- Do use the simple `presentation`, `anchor`, and `size` prop API rather than a nested `desktop` object.
- Do infer animation from `presentation` and `anchor`.
- Do start with size presets only.
- Do create the authenticated default route group if it does not already exist.

## Don'ts

- Don't create separate one-off components that overlap without a shared abstraction.
- Don't make the name so narrow that it only fits one of the required behaviors.
- Don't force every caller to manually build scrolling, close buttons, keyboard handling, or header/footer layout.
- Don't optimize only for desktop and treat mobile as an afterthought.
- Don't treat the agent skill as secondary documentation to maybe write later.
- Don't migrate every dependent LingoCafe flow inside GM77; only the reader appearance preferences popup is currently in scope as proof.
- Don't expose a public `animation` prop in the first version.
- Don't add raw sizing props before presets prove insufficient.

## Open Questions

- Whether `Modal` should reuse existing static `Panel` primitives internally or only match their visual language can be finalized during implementation after the demo/styleguide pages make the difference visible. This does not block planning.
