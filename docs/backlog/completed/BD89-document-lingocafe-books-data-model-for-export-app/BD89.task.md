# Document LingoCafe Books Data Model for Export App

## Elevator's Pitch

Create `docs/lingocafe/BOOKS_DATA_MODEL.md` and thoroughly document the inner workings of the LingoCafe books data model so the content export app can generate a reliable `content-export.sql` file for replacing book content from the LingoCafe main repository.

## Business Gain

The content app needs a precise contract for exporting books into the reader database. This documentation prevents the export pipeline from using brute-force deletes, guessing ID behavior, or leaving stale reader state behind. Chuck Norris does not truncate a catalog. He removes exactly the wrong rows.

## Current State

LingoCafe has a PostgreSQL schema under the `lingocafe` schema, including `books`, `books_pages`, `books_progress`, `profiles`, and `events`.

The core data model was implemented by `OG12`, and later reader tasks use it for the catalog, book details, read page, progress, and reader events.

There is not yet a dedicated `docs/lingocafe/BOOKS_DATA_MODEL.md` document that explains the books model as an export contract for another app.

## Desired State

`docs/lingocafe/BOOKS_DATA_MODEL.md` exists and explains the books-related tables, IDs, constraints, foreign keys, replacement semantics, stale page cleanup rules, and progress cleanup behavior in enough detail for the content app to generate `content-export.sql`.

The document should make clear that exports must surgically operate on their own content by `book_id` and `page_id`, not delete all LingoCafe content.

The export contract should be scoped to books explicitly present in the export batch. If a book exists in the target database but is no longer present in the source repository, this documentation must not instruct the export to delete it.

## Definition of Success

The content export app can use the documentation as a practical source of truth for generating SQL that upserts new books, updates existing books, upserts pages for exported books, removes stale pages only inside exported books, and cleans up progress rows that reference deleted pages.

## Additional Context

The operator asked to generate `docs/lingocafe/BOOKS_DATA_MODEL.md` and thoroughly document all there is to know about the inner workings of the books data model.

This documentation will be used by the content app to generate `content-export.sql` with all SQL instructions needed to create or replace books from the LingoCafe main repository.

The export app needs the relevant information to generate inserts into the appropriate tables for book and page entries.

The export app needs to understand how IDs work so it can replace existing content with updated versions. For example, it should be able to replace existing pages, remove stale pages that are no longer present in the content, and clean up the progress indicator table from references to pages that become stale in the new export.

The export must not delete all content. It must surgically work on its own content based on book and page IDs.

The operator clarified that books present in the target database but absent from the source export must not be deleted. The export should manage only new books and changes to existing exported books.

The operator clarified that the document should expose the contract and table structure carefully, and should include SQL examples because concrete SQL examples are expected to produce a better result for the future content export app.

The operator clarified that book IDs and page IDs are owned by the LingoCafe content repository. If a specific book is reflowed with a different set of pages, the export should delete target pages for that book that are no longer listed in the exported page set, then insert new pages or update existing pages. If a deleted page was linked to a `books_progress` row, that progress row should be removed as stale state.

The operator clarified that the migration source is the source of truth for schema behavior. The live database can be inspected as supporting evidence, but `knex/migrations/20260427150000_lingocafe_schema.js` is authoritative.

The current migration defines `books_progress_page_fk` with `ON DELETE CASCADE` from `lingocafe.books_progress (book_id, page_id)` to `lingocafe.books_pages (book_id, id)`. Therefore the document should explain that deleting stale pages automatically removes progress rows for those stale pages. If future schema inspection ever finds that cascade missing, then the export would need explicit progress cleanup before deleting stale pages.

## Assumptions

The export app will emit SQL, not call application APIs.

The exported content is authoritative for each `book_id` it includes.

Book IDs are stable identifiers coming from the LingoCafe content source repository.

Page IDs are stable within a book when the logical page remains the same.

All LingoCafe book and page IDs are considered owned by the LingoCafe content repository for the purpose of this documentation, but export execution is still scoped to the books included in the current export batch.

A removed page ID means the old page is stale for that book and should be deleted from `lingocafe.books_pages`.

Existing progress for pages that still exist should be preserved when possible.

Progress pointing at deleted pages must be removed through the existing `ON DELETE CASCADE` behavior defined in the migration.

## Constraints

The documentation must describe the current implemented schema, not an imaginary future schema.

The export contract must preserve unrelated LingoCafe content that is not part of the current export batch.

The export contract must not require deleting all rows from `lingocafe.books`, `lingocafe.books_pages`, or `lingocafe.books_progress`.

The export contract must not delete target books merely because they are absent from the current source export.

The export contract must treat each exported book as authoritative for that book's page set: pages currently in the target database for that `book_id` but absent from the exported page ID list are stale and should be removed.

The export contract must account for the composite page identity: `books_pages` uses `(book_id, id)` as its primary key.

The export contract must account for `books_progress` referencing `books_pages` through `(book_id, page_id)` and using `ON DELETE CASCADE`.

The documentation should be detailed enough to guide SQL generation for idempotent create-or-replace exports.

The documentation should focus on the contract and structure, and should include concrete SQL examples or templates that demonstrate how the content export app should generate `content-export.sql`.

The migration source must be treated as authoritative for schema details. Database inspection can be used as a supporting check, but the documentation should not contradict the migration source.

- Do document actual table names with schema prefixes.
- Do describe primary keys, unique constraints, foreign keys, and cascade behavior.
- Do explain replacement behavior in terms of `book_id` and page ID sets.
- Do include a safe, transaction-oriented export sequence.
- Do scope export behavior to the books included in the current export batch.
- Do call out which data belongs to content export and which data belongs to user activity.
- Do preserve unrelated content outside the exported book IDs.
- Do make stale page cleanup explicit.
- Do explain that reflowed books may legitimately delete stale pages and insert new page IDs.
- Do explain that stale progress is removed by the existing cascade when its page disappears.
- Do use the migration source as the source of truth for schema details.
- Do include concrete SQL examples that show the intended export behavior.

## Don'ts

- Don't tell the export app to wipe all LingoCafe books before importing.
- Don't tell the export app to delete books only because they are absent from the source export.
- Don't treat user progress as content owned by the export app.
- Don't suggest changing IDs on every export if the logical book or page did not change.
- Don't ignore the composite `(book_id, page_id)` relationship used by progress.
- Don't document speculative tables that do not exist yet as required for the export.
- Don't require manual cleanup that the SQL generator cannot automate.
- Don't contradict the migration source based on assumptions or live database drift.
