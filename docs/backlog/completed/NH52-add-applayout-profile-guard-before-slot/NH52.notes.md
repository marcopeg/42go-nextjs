# Execution Notes — Add AppLayout Profile Guard Before Slot
**Task**: ./NH52.task.md
**Plan**: ./NH52.plan.md

## Decisions

- Accepted the plan and transitioned `NH52` to `wip` before source edits.
- Kept the existing `TProfileLayoutGuard*` names and extended them to support `slot: "before" | "layout"` instead of introducing a parallel type tree.
- Implemented the before guard as blocking by default. A before-slot guard must explicitly release the AppLayout shell through `releaseBeforeGuard()`.
- Kept the LingoCafe onboarding UI and persistence path intact; only the lifecycle changed so successful completion can close the modal over the newly rendered shell.
- Changed LingoCafe guard configuration from `slot: "layout"` to `slot: "before"`.
- Added a server-injected `window.__APP_PROFILE_COMPLETION__` state for before guards so completed profiles can skip blocking on full page load.
- Added a user-scoped LingoCafe profile completion cache so completed profiles skip the before guard on client-side route changes.
- Replaced the first cache with a current-user-scoped profile completion state. Unknown user state now blocks by default, so login cannot inherit another tab/session's completed flag.
- Added a LingoCafe login-page reset hook to clear stale onboarding state before credentials/OAuth login enters the app layout.
- Removed script-based app/profile bootstrap entirely. App id and profile completion now flow through the client provider/context path, so client login transitions do not render `<script>` tags.
- Replaced `next-themes` with a script-free local theme provider because `next-themes` injects its own bootstrap script and Next 16 flags that during the client-side login transition.
- Updated the shared Sonner toaster to read theme state from the local provider.
- Cleared `.next` and restarted the dev server after the theme provider change so Turbopack stopped serving stale chunks.
- Made `X-42Go-AppID` a reserved proxy-owned header. Incoming requests may still match apps through configured header rules, but public clients can no longer choose the internal app id header directly.
- Removed the `window.__APP_ID__` client fallback. Client app config now flows through `AppConfigProvider`; normal menu/layout/policy consumers are already under that provider.
- Kept LingoCafe profile completion browser state separate from app id selection. It is only a UX cache for the before guard and is not a server authorization boundary.

## Problems Encountered

- First `npm run lint` failed on React 19 `set-state-in-effect` in `AppLayout` and `ProfileLayoutGuard`. Removed synchronous effect resets and keyed loaded guard state by guard/slot instead.
- Browser review after the first implementation showed an unacceptable transitional state for completed users on page load and every route change, because AppLayout blocked first and the guard checked profile completion afterward. Fixed by making the before guard expose a synchronous `shouldBlock` check backed by server-injected and session-cached profile completion.
- Live login-path verification caught a second gap: credentials login does SPA navigation, so the app layout can mount before a server refresh if the browser still has stale profile state. Fixed by scoping completion to `window.__APP_SESSION_USER_ID__`, clearing LingoCafe state on `/login`, and failing closed when the current user is unknown.
- The first server profile bootstrap used raw/`next/script` script tags in `app/layout.tsx`, which still triggered Next's dev overlay on client login navigations. Removed the script bootstrap and hydrate the state inside `Providers` instead.
- A follow-up login test still showed the same script warning even after removing the app bootstrap. The remaining script came from `next-themes`, so theme initialization now happens through a local external-store provider with no React-rendered script tag.
- The first custom theme provider returned a new object from `getServerSnapshot`, which made React warn that server snapshots should be cached. Fixed by using a stable server snapshot constant.
- A completed-profile login then exposed a render-phase update in the replacement theme provider. Initialization notified the external-store subscribers while a new provider was still rendering, so initialization now updates the snapshot silently and only user/system/storage changes notify listeners.
- Follow-up architecture review found that trusting client-provided `X-42Go-AppID` allowed app selection to bypass host/header matcher intent. Fixed by stripping that header before matching and by sharing the same reserved-header filtering with auth provider resolution.
