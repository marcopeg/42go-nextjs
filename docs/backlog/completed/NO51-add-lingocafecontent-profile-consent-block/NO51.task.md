# Add LingoCafe Profile Legal and Consent Blocks

## Elevator's Pitch

Add LingoCafe-specific profile blocks for required legal acknowledgements and optional consent choices. Users must accept Terms and Conditions and acknowledge the Privacy Policy before creating or saving a profile, while marketing emails and Early Birds participation stay optional and separately tracked.

## Business Gain

LingoCafe can separate mandatory service/legal acknowledgements from optional consent choices without burying them in account setup.

The required choices are:

- Terms and Conditions acceptance.
- Privacy Policy acknowledgement.

The optional choices are:

- Marketing emails about LingoCafe content updates and offers.
- Early Birds program enrollment, where the operator periodically sends questionnaires such as Google Forms.

Users get clear control. The database gets facts, not vibes. Chuck Norris signs consent forms with a roundhouse kick and still records the timestamp.

## Current State

LingoCafe already has an AppConfig-driven profile page through `app.profile.items`.

The existing LingoCafe custom profile block is `LingocafePreferences`, which manages native language, target language, and target level from `/profile`.

The `/books` route shows an onboarding component when the app finds the LingoCafe profile is incomplete.

The LingoCafe profile persistence flow stores app-specific data in `lingocafe.profiles`, including a JSON `data` field exposed as `profile.data` in API responses.

The current consent fields for LingoCafe content updates and Early Birds participation are not exposed in the profile UI or the `/books` onboarding flow.

The app does not yet have a dedicated required Terms and Privacy acknowledgement flow tied to profile creation and save.

## Desired State

Create a required LingoCafe profile block for Terms and Privacy inside the LingoCafe app area. A clear implementation name such as `LingoCafeTermsPrivacy` is acceptable.

The Terms and Privacy block renders two required checkboxes:

- `I accept the Terms and Conditions`
- `I acknowledge the Privacy Policy`

The Terms checkbox must link to the LingoCafe Terms and Conditions page. The Privacy checkbox must link to the LingoCafe Privacy Policy page.

Both required checkboxes must be checked before `/profile` save can succeed and before `/books` onboarding can create or complete a profile.

Create a separate custom LingoCafe profile block for optional marketing and opportunity choices. The original requested name was `LingoCafeContent`; a clearer UI title such as `Marketing & Opportunities` is acceptable as long as the component remains LingoCafe-specific and is wired through `app.profile.items`.

The optional Marketing and Opportunities block renders two checkboxes:

- `I consent to receive content updates and offers about LingoCafe services`
- `I want to participate in the Early Birds program and receive all the features for free`

The optional checkbox values are stored in the profile `data` field as readable consent objects:

- `consent_mkt.value`
- `consent_mkt.date`
- `consent_mkt.version`
- `consent_mkt.source`
- `consent_mkt.method`
- `consent_alpha.value`
- `consent_alpha.date`
- `consent_alpha.version`
- `consent_alpha.source`
- `consent_alpha.method`

The required Terms and Privacy acknowledgement values are stored in the profile `data` field with the same object pattern:

- `consent_terms.value`
- `consent_terms.date`
- `consent_terms.version`
- `consent_terms.source`
- `consent_terms.method`
- `consent_privacy.value`
- `consent_privacy.date`
- `consent_privacy.version`
- `consent_privacy.source`
- `consent_privacy.method`

The `value` subkey stores the boolean value.

The `date` subkey stores the date when the corresponding value last changed.

The `version` subkey identifies the legal document version, consent statement version, or stable content hash shown to the user at the time.

The `source` subkey identifies where the user submitted the choice, such as `profile` or `books-onboarding`.

The `method` subkey identifies the interaction pattern, initially `checkbox-submit`.

The same required and optional checkbox choices should appear in the `/books` onboarding component any time the LingoCafe profile is incomplete, so first-time and partially configured users can provide language preferences and legal/consent choices in one flow.

On first profile creation, both optional consent booleans must be stored explicitly. If an optional checkbox is left unchecked, the app still stores `false` plus the current change date.

Consent and acknowledgement evidence should be recorded in `profile.data` when the user submits/saves the form, not on every transient checkbox click. A click that is not saved should not become a legal record.

## Definition of Success

LingoCafe users cannot create or save a profile until Terms and Conditions and Privacy Policy acknowledgements are present. Users can also set or update the two optional content consent choices from `/profile`, and users with incomplete profiles see the same choices during `/books` onboarding. Saved profile data preserves readable consent objects, explicit `false` values, full ISO timestamps, and enough evidence metadata to prove what the user accepted or consented to at the time without overcollecting personal data.

## Additional Context

The operator asked: "focus on the lingocafe profile information. Create a new custom profile block called LingoCafeContent (place it inside the lingocafe app area)."

The operator later refined the request: add a specific `Terms and Privacy` block with two required checkboxes, and a separate `Marketing & Opportunities` block with optional marketing and alpha/Early Birds checkboxes.

This should build on the existing configurable profile page system and keep LingoCafe-specific UI under the LingoCafe app area, not inside shared `42go` platform blocks.

The current profile block documentation says custom blocks should use `type: "component"` and can participate in the global `Save preferences` action through `useProfileBlockHandle()`.

The existing `/api/lingocafe/profile` flow is the likely persistence path because it already owns LingoCafe profile saves and profile-update events.

Implementation planning should verify the current `profile.data` response and persistence shape before deciding whether to extend the existing profile API payload or add a focused adapter for legal acknowledgements and consent fields.

The marketing checkbox is for GDPR compliance when sending marketing emails to the user.

The Early Birds checkbox is for enrolling users in a program where the operator periodically sends questionnaires, likely through Google Forms.

Research notes from official guidance:

- GDPR consent records should prove who consented, when they consented, how they consented, and what they were told at the time.
- Strong evidence should include the exact statement or version/hash of the consent text, linked policy versions, and the source surface such as `/profile` or `/books` onboarding.
- IP address and user agent may help demonstrate context, but they are personal data and should be minimized, server-derived, and documented.
- Screen resolution, operating system fingerprinting, and broad device details are not necessary for basic consent proof and should not be collected unless a later legal/security review gives a concrete reason.
- For this task, store consent/acknowledgement evidence only in `profile.data`, not as append-only `lingocafe.events`, unless implementation uncovers a concrete product or legal reason to change that.

## Assumptions

The Terms and Privacy block and the Marketing and Opportunities block should be registered in `src/config/lingocafe/config.ts` under `app.profile.items`.

The blocks should initialize from existing `profile.data` values when they are present.

If a stored optional consent value is missing, the UI should treat it as unchecked.

Optional consent date fields should change only when the corresponding boolean value changes.

The stored date format must be a full ISO 8601 timestamp because the operator wants GDPR traceability.

When a profile is created and either optional consent checkbox is left unchecked, the app should still store the boolean `false` and the current ISO timestamp for that consent key.

Required Terms and Privacy acknowledgements should store boolean values, full ISO timestamps, and evidence metadata sufficient to prove which version of the terms/privacy text was accepted or acknowledged.

Evidence metadata should use readable `consent_{key}` objects for all four keys: `terms`, `privacy`, `mkt`, and `alpha`, each with `value`, `date`, `version`, `source`, and `method` subkeys.

## Constraints

Keep LingoCafe-specific profile UI inside the LingoCafe app area.

Use the exact optional profile `data` objects requested by the operator for marketing and Early Birds values/dates.

Do not store the consent choices in top-level `lingocafe.profiles` columns.

Do not trust client-supplied user IDs.

Preserve server-side validation for profile updates.

Keep authenticated app pages client-only and use browser-side fetches with `credentials: "same-origin"`.

Do not break the existing language preference profile flow or `/books` onboarding behavior.

Terms and Privacy page links are required. If those pages do not exist yet, planning should either add them as part of this implementation or explicitly depend on a separate task.

Do not collect browser fingerprinting fields such as screen resolution by default. Keep audit evidence proportional to consent proof.

Do not treat Privacy Policy acknowledgement as consent for unrelated optional processing.

Do not collect IP address or User-Agent for this task by default. The authenticated `lingocafe.profiles.user_id` already establishes who submitted the saved profile, while source, method, timestamp, and version cover the core audit trail with less personal data.

## Open Questions

Q: Should the consent date values be full ISO 8601 timestamps or date-only strings?
A: Full ISO 8601 timestamps for GDPR traceability.

Q: Should declining or unchecking a consent value also update the corresponding date?
A: Yes. The date represents when the value last changed.

Q: Should the Early Birds copy use the stored `alpha` naming or expose `Early Birds` only in the UI?
A: Store under the requested `consent_alpha` object and show `Early Birds` in the UI text.

Q: Should the optional checkboxes be required during `/books` onboarding?
A: No. Store explicit `false` plus date if they are unchecked on profile creation.

Q: On first profile creation, should unchecked optional boxes be stored as `false` plus a date or omitted until changed later?
A: Store `false` plus the current full ISO timestamp.

Q: Should the `/books` onboarding checkboxes appear only when the profile is missing or whenever it is incomplete?
A: Show them any time the profile is incomplete.

Q: What legal purpose does each optional consent line represent?
A: The first checkbox is for GDPR consent to send marketing emails about LingoCafe. The second checkbox is for Early Birds program enrollment and periodic questionnaires, such as Google Forms.

Q: Should Terms and Privacy be part of the same optional content consent block?
A: No. Add a separate required Terms and Privacy block with two required checkboxes: Terms and Conditions, and Privacy Policy.

Q: Are Terms and Privacy required to save/create a profile?
A: Yes. Without both required checkboxes, the user cannot create or save the profile.

Q: Should checkbox audit records be captured on every click?
A: No. Capture the legally meaningful state when the user submits or saves. Transient unsaved clicks should not become legal records.

Q: Which datapoints should be collected for basic consent evidence?
A: Collect value, date, version, source, and method under readable `consent_{key}` objects. The authenticated profile row already identifies who saved the values. Do not collect IP address, User-Agent, screen resolution, operating system, or other fingerprinting-style data by default.

Q: Should consent/acknowledgement history be stored only in `profile.data`, or also as append-only `lingocafe.events` records?
A: Store it only in `profile.data` unless implementation uncovers concrete evidence that another store is required.

Q: What exact naming pattern should the evidence datapoints use?
A: Use `consent_{key}` objects for all four keys: `terms`, `privacy`, `mkt`, and `alpha`.

Q: Which datapoints should be stored without overcollecting?
A: Store `value`, `date`, `version`, `source`, and `method`. Do not store IP address, User-Agent, screen resolution, operating system, or device fingerprinting fields by default.
