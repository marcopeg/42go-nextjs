# Execution Notes — Add Core Theme Profile Block and Enable It in LingoCafe
**Task**: ./FM72.task.md
**Plan**: ./FM72.plan.md

## Decisions

- Start by extending the shared profile block runtime with block-level dirty state so local-only theme edits can participate in the same save UX without entering the persisted profile payload.
- Feed aggregate block dirtiness into `useProfileController` as `externalDirty` so the existing leave-warning and top-bar dirty UX can be reused instead of duplicated.
- Implement the theme block with a draft selection plus a committed override so the UI stays stable between save success and the eventual `next-themes` update without relying on state-setting effects.

## Problems Encountered

- ESLint rejected the first theme-block draft because it synchronized local state from an effect with direct `setState` calls.
- The theme block was revised to remove the sync effect and model the selection as draft state plus a temporary committed override.

## Deviations From Plan

- `src/app/(app)/profile/page.tsx` did not require changes because the existing `onDirtyChange` wiring already updates the save button label once aggregate dirty state reaches the controller.
- `src/42go/config/ThemeProvider.tsx` did not require changes because the existing `useTheme()` API was sufficient for reading and applying the selected theme.
- No `/api/profile` payload-shaping change was needed because the theme block stays outside the shared profile store.

## Additional Requests

- Reworked the theme selector from a native select into a three-option segmented tab-style control matching the provided inspiration screenshot more closely.
- Updated the LingoCafe theme profile block copy to use the shorter "Theme" title and generic appearance helper text.
- Moved the theme helper text back into the panel header description slot so it is coherent with the other profile panels.
- Changed the selected tab from solid fill to an outlined active style and reduced the control height and icon spacing for a more compact profile-panel fit.
- Made the selected tab outline use the visible primary accent color with only a faint active fill.
- Moved the transparent border class into the inactive branch and used `var(--primary)` directly for the active border so the selected outline stays visible.

## Known Limitations

- `npm run qa` passed after the final theme tab styling change.
- The operator visually reviewed the `/profile` theme panel through screenshots and requested the final active style: a compact tab with light fill and a solid `var(--primary)` border.
- Dedicated browser verification of the dirty warning, post-save theme application, reload persistence, and profile payload shape was not performed in this session.
