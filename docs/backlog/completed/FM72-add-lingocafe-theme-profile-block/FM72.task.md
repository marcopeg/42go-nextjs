# Add Core Theme Profile Block and Enable It in LingoCafe

## Elevator's Pitch

Add a reusable core profile block for theme preference so a signed-in user can choose `light`, `dark`, or `system` as a local preference. Then enable that block in LingoCafe after the language preferences block so it can be tested there first. The choice must participate in the existing profile save flow and apply to the current UI only after a successful save, but it must remain local-storage-based rather than database-backed.

## Business Gain

Users get a clear place to manage theme preference inside their account profile without polluting profile persistence with device-local display state, and the platform gains a reusable profile block that other apps can opt into later instead of cloning one-off theme UI.

## Current State

LingoCafe already renders a custom profile block for language preferences and saves profile data through the shared `/api/profile` flow. The app already uses `next-themes` through the shared `ThemeProvider`, but there is no core profile block for theme preference and no LingoCafe profile entry for theme selection. Theme persistence today is local-storage-driven, and this preference should stay outside the profile data that is saved to the database.

## Desired State

The shared profile block system exposes a reusable theme block that can be configured from `app.profile.items`. LingoCafe uses that block after the language preferences block. The block edits a local theme selection, participates in the shared profile page save cycle, counts as normal unsaved state for user experience purposes, and applies the selected theme locally after the save succeeds so the interface updates without requiring a reload. The preference is stored through the existing local-storage-based theme system and is not written into `auth.users.profile`.

## Definition of Success

The user can open `/profile` in LingoCafe, choose `light`, `dark`, or `system`, click the shared save action, and see the theme update in the current session after the save completes. Reloading the app keeps the same preference through the existing local-storage persistence behavior, nothing new is written into the profile database payload, and the block is available as a reusable core block for future app configs.

## Additional Context

Relevant implementation points already exist:

- `src/config/lingocafe/config.ts` defines the LingoCafe profile block list and is where the new theme block should be inserted after `LingocafePreferences`.
- `src/42go/components/ProfileBlock/ProfilePageRenderer.tsx` already runs block validation, performs the central save, and then calls block-level `onSaveSuccess`.
- `src/42go/components/ProfileBlock/ProfileBlockRuntime.tsx` and `src/42go/components/ProfileBlock/types.ts` already support `useProfileBlockHandle({ onSaveSuccess })`.
- `src/42go/config/ThemeProvider.tsx` exposes `useTheme()` backed by `next-themes`, which is the likely place to apply the saved theme after success.
- The user explicitly wants this to be a core profile block, not a LingoCafe-only custom component, while LingoCafe is the first app that should enable it.
- The user explicitly does not want this preference persisted in the profile data stored in the database.

## Assumptions

- Valid values are exactly `light`, `dark`, and `system`.
- The theme should not change immediately on option click; it should change only after the profile save succeeds.
- The reusable block can live with the shared profile block system because the behavior is generic across apps.
- The theme preference remains optional and must not become a required profile validation field.

## Constraints

- Keep the generic theme block in the shared `42go` profile block system and keep any app-specific copy or styling concerns out of the block implementation.
- Use the existing profile page save orchestration instead of creating a separate theme-save endpoint or one-off local persistence flow.
- The implementation must respect the current app default theme fallback when no saved preference exists.
- Do not extend the LingoCafe profile schema just to support this block if the preference stays out of database-backed profile data.

## Acceptance Criteria

- [ ] The shared profile block system has a reusable theme preference block.
- [ ] The block appears after the language preferences block in `src/config/lingocafe/config.ts`.
- [ ] The block uses the shared profile store and shared save action, not a custom direct-save button.
- [ ] Changing the theme selection marks the profile page as dirty and participates in the standard unsaved-changes experience before save.
- [ ] After a successful save, the current session applies the saved theme through the existing theme system.
- [ ] Reloading the page preserves the same theme through the existing persistence behavior.
- [ ] Failed validation or failed persistence does not apply a new theme locally.
- [ ] The block can be referenced from `app.profile.items` without LingoCafe-specific component wiring.
- [ ] The theme preference is not written into `auth.users.profile` and does not require schema validation.
- [ ] The first rollout enables the block only in LingoCafe.

## Dos

- Do inspect how `next-themes` currently persists and restores the selected theme before wiring the block to local storage and save success behavior.
- Do use the existing `useProfile()` state and `useProfileBlockHandle()` lifecycle for validation and post-save application.
- Do keep the shared block generic and only use LingoCafe config to place it in the profile page.
- Do keep the option optional and outside AJV profile validation.
- Do make local-only theme edits behave like normal profile edits from the user's point of view.

## Don'ts

- Don't implement this as a LingoCafe-only component if the behavior is meant to be core and reusable.
- Don't bypass the shared `/api/profile` save pipeline.
- Don't apply the theme locally before the save succeeds.
- Don't write the theme preference into the database-backed profile payload.
- Don't create a separate special-case save UX just because the persistence target is local storage.

## Open Questions

- Should the block read its initial value directly from `next-themes` state, from local storage, or from a small shared adapter that hides those details from the block?
