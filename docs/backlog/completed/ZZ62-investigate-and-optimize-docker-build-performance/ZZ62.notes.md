# Execution Notes — Investigate and Optimize Docker Build Performance
**Task**: ./ZZ62.task.md
**Plan**: ./ZZ62.plan.md

## Decisions

- Accepted plan moved `ZZ62` into WIP.
- Local machine architecture is `arm64`.
- Docker client is Darwin arm64; Docker server is Linux arm64.
- `docker buildx ls` shows the active builder supports `linux/arm64`, `linux/amd64`, and `linux/amd64/v2`.
- Current `make prod.build` runs plain `docker build` with `--no-cache`; on this setup it targets the arm64 Docker server default.
- Current `make prod.build.light` runs plain cached `docker build`; on this setup it also targets the arm64 Docker server default.
- Current `make publish VERSION=0.0.20` runs `docker buildx build --platform linux/amd64,linux/arm64 --push`, publishing both server and ARM images.
- Baseline no-cache build took 76.07s before adding `--load`; dependency install stages were about 26s (`npm ci --omit=dev`) and 31s (`npm ci`).
- Baseline fully cached build took 1.28s but emitted a warning that the build result only remained in the build cache because the active builder is `docker-container` and no output was specified.
- A source-context cache probe took 24.04s; dependency install stages stayed cached while `COPY . .` and the Next build reran.
- Added `--load` to local `prod.build` and `prod.build.light` so `42go-next:latest` is actually loaded locally when using the docker-container builder.
- Added BuildKit npm cache mounts for dependency install stages. The first version used a single locked cache mount and serialized the two `npm ci` stages, so it was replaced with separate `npm-prod` and `npm-build` cache IDs.
- After optimization, a cached build with source/metadata changes and `--load` took 33.79s.
- After optimization, a no-change cached build with `--load` took 7.67s and loaded `42go-next:latest` as `linux/arm64`.
- After optimization, final `make prod.build` no-cache build with `--load` took 122.62s.
- Publish targets now use: `publish` for cached VPS-targeted `linux/amd64`, `publish.nocache` for no-cache VPS-targeted `linux/amd64`, and `publish.universal` for no-cache `linux/amd64,linux/arm64`.
- `npm run qa` passed.

## Problems Encountered

- Initial Docker daemon access from the sandbox failed with permission denied for the Colima socket. Retried with escalated Docker access and Docker commands worked.
- A shared locked npm cache mount made the first optimized build slower by serializing dependency installs; split cache IDs fixed the issue.

## Deviations From Plan

- Added `--load` to local build targets because the active Docker builder otherwise did not materialize `42go-next:latest` locally. This was discovered during execution and supports the intended behavior of the existing build targets.

## Additional Requests

- The operator requested `publish.nocache` as the explicit no-cache publish target name.

## Known Limitations

- Publish commands are being dry-run unless registry credentials and push intent are explicitly available.
