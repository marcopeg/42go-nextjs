# Investigate and Optimize Docker Build Performance

## Elevator's Pitch

The local production image build currently takes around five to ten minutes on a MacBook M2. This task measures where the time is spent, then implements safe local-first Docker build speedups such as cache-friendly Makefile targets, Dockerfile layer improvements, and a clearer publish command contract.

## Business Gain

Faster Docker builds reduce developer wait time and make production-image testing practical during local development. The future GitHub tag-to-image workflow will also benefit from a measured cache strategy instead of entering the ring blindfolded.

## Current State

Docker image builds are perceived as too slow, roughly five to ten minutes per run on a MacBook M2. Root causes are not yet documented step-by-step, and no validated optimization strategy is recorded.

The current `make prod.build` target runs `docker build --progress=plain --no-cache -f Dockerfile -t 42go-next:latest .`, so it deliberately disables Docker layer reuse. `make prod.build.light` runs the same build without `--no-cache`. The `publish` target uses `docker buildx build --platform linux/amd64,linux/arm64 ... --push`, which is a multi-architecture publish path and may be slower than needed for a normal Ubuntu VPS.

The Dockerfile currently uses a four-stage Next.js standalone build (`deps`, `build-deps`, `builder`, `runner`) with separate production and build dependency install stages. `.dockerignore` already excludes common heavy or sensitive inputs such as `.git`, `node_modules`, `.next`, docs, local env files, logs, and agent/editor state.

Dependency layer caching is considered safe when it is driven by `package.json` plus `package-lock.json` and `npm ci`. The Dockerfile already copies those manifests before running dependency installation, so those layers should be reusable until the dependency manifest or lockfile changes. The task should verify whether current commands accidentally bypass that cache, especially through `--no-cache`, and whether BuildKit cache mounts for the npm download cache can speed installs without weakening dependency reproducibility.

A `package-lock.json` generated on macOS is expected to be compatible with Linux Docker builds because the lockfile records package versions, resolution URLs, integrity hashes, and dependency metadata rather than installed native binaries. `npm ci` still runs inside the Linux container and installs/builds dependencies for that container platform. The task should still verify native and optional dependencies under the target image, especially because `node:22-alpine` uses musl libc rather than glibc.

## Desired State

A clear build-time profile exists for local production-image builds, with the slowest stages identified and ranked. Safe local-first optimizations are implemented and validated. Future CI/GitHub tag publishing guidance is documented separately enough to become its own task.

## Definition of Success

- Build stages and timings are measured and documented.
- Top bottlenecks are identified with evidence.
- Local cached build commands are improved or added where safe.
- The publish workflow is reshaped so the default target is cached and VPS-focused, while explicit no-cache and universal targets remain available.
- Implemented changes produce measurable build-time improvement over baseline.
- Future GitHub tag-to-image publishing is documented as follow-up scope, not implemented here.

## Additional Context

The current local execution environment is a MacBook with Apple M2 processor. The intended production target is a normal Contabo Ubuntu VPS. Unless the VPS architecture is confirmed as ARM, assume `linux/amd64` is the default deployment target because most commodity Ubuntu VPS instances are amd64.

Architecture distinction:

- `linux/arm64`: native CPU architecture for Apple Silicon Macs such as M1/M2/M3 and some ARM servers.
- `linux/amd64`: normal x86_64 server architecture used by most VPS providers, including the likely Contabo target.
- `make prod.build` currently uses plain `docker build` without `--platform`, so Docker builds for the current builder default. On an Apple M2 this is usually `linux/arm64` unless Docker is configured otherwise.
- `make publish` currently uses `docker buildx build --platform linux/amd64,linux/arm64 --push`, so it builds and publishes a multi-architecture manifest that can run on both common server types.
- Multi-architecture publishing is flexible but slower, especially when an M2 Mac has to emulate `linux/amd64`.

Expected direction: keep local build commands fast and cache-friendly, make the default publish path target the actual VPS, and keep multi-architecture publishing available only when cross-platform distribution is needed.

Preferred publish command shape:

- `make publish`: build and push the VPS-targeted image, using cache as much as possible.
- `make publish.nocache`: build and push the VPS-targeted image with no cache for release confidence or cache suspicion.
- `make publish.universal`: build and push the multi-architecture image (`linux/amd64,linux/arm64`) with no cache, matching the current heavyweight publish behavior.

This naming makes the common path fast and explicit while keeping clean and universal builds available when needed.

The operator wants practical local speedups implemented, not only a written investigation. The preferred shape is not `publish.light`; instead `make publish` should become the cached VPS-targeted path, while `make publish.nocache` and `make publish.universal` make slower choices explicit.

GitHub-based publishing is desired later: a Git tag should eventually trigger an image build and push to Docker Hub or another image registry. There is no CI yet, so this task should not implement GitHub Actions unless explicitly expanded later.

## Assumptions

- Docker build time can be reduced through caching and build-step optimization.
- The no-cache `prod.build` target is intentionally slow and useful as a clean baseline, not as the everyday local rebuild path.
- `prod.build.light` is likely the safer local rebuild default when source or dependency cache reuse is acceptable.
- Dependency install layer caching is safe when cache keys are effectively the dependency manifests, especially `package-lock.json`, and installation remains deterministic through `npm ci`.
- A macOS-generated `package-lock.json` is acceptable input for Linux Docker builds, but native or optional dependency behavior must be validated inside the target container.
- A normal Contabo Ubuntu VPS probably only needs a `linux/amd64` image unless the actual VPS architecture says otherwise.
- Local `docker build` on an M2 probably produces a `linux/arm64` image, which is useful for local testing but not sufficient for a likely `linux/amd64` VPS deployment.
- The default `publish` target should represent the normal production destination, not universal distribution.

## Constraints

- Keep the existing functional behavior of the built image.
- Do not reduce security posture just to gain speed.
- Keep `make prod.build` or an equivalent no-cache path available for clean baseline and release confidence.
- Do not implement GitHub Actions/tag publishing in this task unless the scope is explicitly changed.
- Any optimization that affects publishing must document architecture assumptions and cache-safety tradeoffs.

## Acceptance Criteria

- A baseline build-time report is documented.
- Slowest Docker build steps are explicitly listed with timing data.
- Candidate optimizations are documented with expected impact.
- Cached versus no-cache local builds are compared.
- Dependency install cache behavior is verified by changing app source without changing `package.json` or `package-lock.json` and confirming dependency install layers are reused.
- Safe Makefile and/or Dockerfile optimizations are implemented where they measurably help.
- Publish command naming is updated or documented so `publish` is the cached VPS path, `publish.nocache` is the no-cache VPS path, and `publish.universal` is the current multi-architecture no-cache path.
- GitHub tag-triggered registry publishing is captured as future work with enough notes to draft a follow-up task.
- Final recommendation includes next implementation steps.

## Dos

- Measure first, then optimize.
- Compare no-cache vs cached builds.
- Keep dependency caching keyed to `package.json` and `package-lock.json`; do not cache arbitrary `node_modules` state outside Docker's normal layer or BuildKit cache semantics.
- Treat `make prod.build` as the clean baseline and `make prod.build.light` as the first cached comparison.
- Evaluate whether publishing should default to single-arch for the Contabo VPS path.
- Prefer clear Makefile target names over hidden flags: the operator should know whether the build is cached, no-cache, VPS-targeted, or universal from the target name.
- Separate local-machine effects from future CI-runner effects.
- Keep findings evidence-based.

## Don'ts

- Do not rely on guesswork without timing data.
- Do not optimize by removing required build or security checks.
- Do not propose changes that only work on one machine setup unless clearly marked.
- Do not remove the clean no-cache build path.
- Do not add GitHub Actions as part of this task unless the operator explicitly expands scope.

## Open Questions

- Which exact Dockerfile stages are dominating the build time?
- How much of the delay comes from dependency install, app build, or image export?
- What is the actual architecture of the Contabo VPS: `linux/amd64` or ARM?
- Should `publish` and `publish.nocache` push both `latest` and `$(VERSION)`, or should one of them avoid touching `latest` during local validation?
- Which registry should the future GitHub tag workflow target: Docker Hub, GHCR, or another registry?
