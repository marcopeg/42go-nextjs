# Add CapRover Deploy Targets

## Elevator's Pitch

Add Makefile deploy targets that build the Docker image, publish it to Docker Hub, then trigger a CapRover deployment on the VPS.

## Business Gain

Deployment becomes a complete release action instead of a manual two-step. The operator gets one command that builds, ships, and tells the server to deploy, like a roundhouse kick with a receipt.

## Current State

`make publish` currently builds and pushes `$(IMAGE):latest` and `$(IMAGE):$(VERSION)` to Docker Hub using Docker buildx. `make publish.nocache` does the same VPS-targeted push without Docker cache. The VPS deployment is still a separate manual step.

Research found that CapRover has a CLI intended for automation and scripting. The CapRover CLI README says `caprover deploy` supports multiple deployment methods, including image name deployment, and can receive required values from command options, config files, or environment variables. The official site also describes CLI deployment as a supported deployment path.

The current CapRover CLI docs list `-i, --imageName <image>` as the Docker image deployment option. The image must already exist on the server or be accessible through a public or private registry that CapRover can access. This supports the desired flow: local buildx push to Docker Hub first, then CapRover deploy by image name.

CapRover also exposes a generic `caprover api` command, but its README warns that the API command is for careful use when the caller knows what they are doing. Prefer the documented deploy command first unless refinement proves the API path is the cleaner supported automation route.

## Desired State

The Makefile exposes deploy targets for the normal production release sequence:

- `make deploy`: run the cached VPS-targeted Docker build and Docker Hub publish, then trigger CapRover deploy.
- `make deploy.nocache`: run the no-cache VPS-targeted Docker build and Docker Hub publish, then trigger CapRover deploy.

`make publish` and `make publish.nocache` can remain the lower-level registry push primitives. The deploy commands should fail loudly if Docker publish succeeds but the CapRover deploy trigger fails.

## Definition of Success

- The exact CapRover CLI deploy mechanism for a Docker image is confirmed locally or against documentation.
- Required CapRover inputs are documented, including machine name, app name, server URL or login config, and authentication/token handling.
- `make deploy` builds and pushes the Docker image through the cached publish path, then triggers a deploy on the VPS.
- `make deploy.nocache` builds and pushes the Docker image through the no-cache publish path, then triggers a deploy on the VPS.
- The deploy trigger uses the intended image tag, preferably `$(IMAGE):$(VERSION)` rather than relying only on mutable `latest`.
- The workflow remains scriptable and does not require interactive prompts during `make deploy` or `make deploy.nocache`.
- Failure handling is clear and leaves the operator knowing whether the build, push, or deployment step failed.

## Additional Context

Useful research references:

- CapRover homepage: https://caprover.com/ states that CapRover includes CLI support for automation and scripting.
- CapRover CLI README: https://github.com/caprover/caprover-cli documents `caprover deploy` and says deployment supports captain-definition file, Dockerfile, tar file, and image name methods.
- CapRover CLI commands docs: https://caprover.com/docs/cli-commands.html list `-i, --imageName <image>` for deploying a Docker image.
- CapRover CLI README also says command data can come from environment variables, config files, command options, or prompts.
- CapRover deploy examples show non-interactive flags such as `caprover deploy -n machine-name -a app-name -b branchName`. For this task, the expected shape is `caprover deploy -n <machine> -a <app> -i <image>`, with `--appToken <token>` or existing local CLI login used for authentication.

Likely implementation direction:

- Add CapRover-related Makefile variables such as `CAPROVER_MACHINE`, `CAPROVER_URL`, `CAPROVER_APP`, `CAPROVER_IMAGE`, and `CAPROVER_APP_TOKEN`.
- Default `CAPROVER_APP` to `a42go-multi`, while still allowing override from the command line or `.env`.
- Make `CAPROVER_IMAGE` default to `$(IMAGE):$(VERSION)`.
- Allow local `.env` values to feed the Makefile for secrets and deployment settings, without committing those secrets. The preferred implementation is to conditionally include `.env` in the Makefile so `CAPROVER_APP_TOKEN` and other deployment variables can be sourced automatically.
- Prefer `--appToken $(CAPROVER_APP_TOKEN)` when the token is set. If the implementation also supports an existing local CLI login by machine name, keep that as a fallback rather than the primary deployment path.
- Add a separately callable CapRover trigger target such as `deploy.caprover` so deployment can be validated or retried without rebuilding and repushing.
- Add `deploy: publish deploy.caprover`.
- Add `deploy.nocache: publish.nocache deploy.caprover`.

## Assumptions

- The VPS is managed by CapRover.
- Docker Hub is the target registry for the production image.
- The CapRover app is already created and configured on the VPS as `a42go-multi`.
- CapRover can pull the pushed Docker Hub image from the VPS. If the Docker Hub repository is private, registry credentials must already be configured in CapRover or added as part of this work.
- The currently published Docker image tags are compatible with the CapRover deployment flow.
- A CapRover app token can be provided through local `.env` as `CAPROVER_APP_TOKEN`.
- `make publish.universal` remains publish-only because it is a distribution target, not the normal VPS release path.

## Constraints

- Do not make `make deploy`, `make deploy.nocache`, or their lower-level targets interactive.
- Do not store CapRover passwords, app tokens, or Docker Hub credentials in the repository.
- Preserve the existing `publish`, `publish.nocache`, and `publish.universal` behavior as registry publish targets unless implementation proves a small internal restructuring is needed.
- Keep the existing VPS-focused `publish` platform behavior from the Docker build optimization work.
- Avoid undocumented CapRover API calls unless the CLI cannot support image deployment cleanly.

- [x] Current CapRover CLI documentation is checked to confirm the image deploy option: `caprover deploy -i/--imageName <image>`.
- [x] `caprover deploy --help` is checked locally during implementation to confirm the installed CLI version matches the documented syntax.
- [x] Required environment variables or Makefile variables are documented.
- [x] `CAPROVER_APP` defaults to `a42go-multi` and can be overridden.
- [x] `CAPROVER_APP_TOKEN` can be sourced from local `.env` or the shell environment without being committed.
- [x] `make deploy` runs the cached VPS-targeted Docker Hub publish path and then triggers CapRover deploy for the VPS app.
- [x] `make deploy.nocache` runs the no-cache VPS-targeted Docker Hub publish path and then triggers CapRover deploy for the VPS app.
- [x] The deploy step references `$(IMAGE):$(VERSION)`, the same immutable version tag that was just pushed.
- [x] `make publish.universal` remains publish-only.
- [x] A dry-run or command inspection path exists for validating the generated Docker and CapRover commands without deploying.
- [x] The production deployment documentation is updated to explain the new `make deploy` and `make deploy.nocache` behavior.
- [x] Failure cases are documented for missing CapRover login/config, missing app name, private registry pull errors, and deploy trigger failure.

## Dos

- Verify the installed CapRover CLI syntax with `caprover deploy --help`.
- Prefer the official CapRover CLI deploy path.
- Use environment variables or local ignored `.env` config for secrets.
- Keep `$(IMAGE)` and `$(VERSION)` as the source of truth for the deployed image tag.
- Make the release command deterministic and scriptable.
- Document setup steps for first-time CapRover login/configuration.
- Keep the deployment trigger separately callable so a failed CapRover trigger can be retried without rebuilding.

## Don'ts

- Do not commit CapRover credentials.
- Do not rely on interactive prompts inside `make deploy` or `make deploy.nocache`.
- Do not silently deploy `latest` if `$(VERSION)` was pushed and should be the release artifact.
- Do not replace the Docker Hub publish step with server-side source builds unless the operator changes direction.
- Do not introduce GitHub Actions in this task unless explicitly requested.

## Open Questions

- What is the CapRover URL to use with `--appToken`?
- Is the Docker Hub repository public or private? If private, confirm that CapRover has registry pull access before relying on image-name deploy.
