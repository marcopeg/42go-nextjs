# Refinement Questions v1 — Harden Magic Link Email Validation

Answer inline under each question, then return to chat and say `next round`.

## Questions

### 1. Scope of server enforcement

Should this task validate only the app-owned `/api/auth/email/request` and `/api/auth/email/verify-code` routes, or must it also harden the direct NextAuth email sign-in path that calls `EmailProvider.normalizeIdentifier` and `sendVerificationRequest`?

Answer:

For now, only the paths. Don't hack into next auth. 

### 2. Account creation policy

JD93 allows verified email-only login to create a new app-scoped user. Should EM99 preserve implicit account creation for valid emails, or should magic-link sending be limited to existing contacts/users for some apps?

Answer:

It shouldn't impact the current behavior. This is just a filter so that we don't let the magic link flow begin if the email is invalid or not accepted. So right now, the magic link will generate accounts, and that's totally fine—I mean, after verification of course—but this is just a filter. 

### 3. Plus aliases

Should any `+` in the email local-part be rejected for every domain, or only for domains/providers known to support plus-address aliases?

Answer:

This is the rule for Gmail that I am aware of. I want you to investigate and figure out all the possibilities that you can find online for, like, major account providers, of course, but I want you to run an extensive research to avoid duplicate Lingocaffe accounts for the same email provider account. 

### 4. Gmail dot aliases

Gmail ignores dots in the local-part, so `marco@gmail.com` and `mar.co@gmail.com` may resolve to the same mailbox. Should this task reject dotted Gmail local-parts as aliases, or only reject plus-addressing?

Answer:

Yes, again, I only gave you some examples. I want you to extensively research and make sure that you accept only real accounts and you reject accounts that are just aliases to the best of your capabilities. 

### 5. Other alias patterns

Some providers support aliases through `-`, `_`, custom domain catch-alls, or provider-specific subaddressing. Should v1 attempt to block only plus-addressing plus known disposable domains, or should it include a stricter provider-specific alias policy even if false positives increase?

Answer:

Again, you need to research and build the best possible filtering function that you can conceivably build. 

### 6. Disposable and risky domain source

Should the implementation vendor a curated static disposable-domain denylist into the repo, add a maintained package dependency, or keep a smaller manually curated list with documentation for updates?

Answer:

Yes, do that and document this feature in the public documentation and in the developer documentation. Document that we need to review this list periodically. So, there should be a periodic list to update this part of the feature. 

### 7. Privacy relay and forwarding providers

Should providers such as Apple private relay, DuckDuckGo Email Protection, Firefox Relay, Proton aliases, and other forwarding/privacy-address services be blocked, or are they acceptable real accounts?

Answer:

Consider those real accounts. 

### 8. Configuration surface

Should the strict validation policy be global for all 42go email-auth providers, or should apps be able to override parts of it through `AppConfig.auth.providers[].config`, such as allowing plus aliases or choosing a blocklist mode?

Answer:

For now, just build everything inside this static function. No configuration is needed for this. 

### 9. User-facing error behavior

Should the client show specific messages such as "Plus aliases are not allowed" and "Temporary email domains are not allowed", or should all invalid inputs map to one generic message like "Enter a valid personal email address"?

Answer:

For now, a generic message is best. 

### 10. Structured server response

Can the server return structured validation reason codes for tests and UI rendering while still using generic copy for users, or should server responses avoid reason codes entirely?

Answer:

No, return a generic error. I prefer to be protected than to provide an explanation. 

### 11. Validation timing in the UI

Should validation run only on submit/Enter/button click, or should the UI also validate on blur or while typing after the field has been touched?

Answer:

Only at the enter or click .

### 12. Test coverage level

Is a validator unit-test suite plus route-level tests with sender/throttle spies enough, or should this task also include a browser-level test proving Enter and button click do not call the email request API for invalid addresses?

Answer:

I think for this, as long as a static function, unit tests are enough.
