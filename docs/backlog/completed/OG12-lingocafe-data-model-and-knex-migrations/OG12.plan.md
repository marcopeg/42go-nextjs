# Plan — LingoCafe Data Model and Knex Migrations

## Goal

Implement the first LingoCafe PostgreSQL schema and migration set so the app can support onboarding-aware profile creation, book catalog browsing, reading-position resume, export-safe page replacement, and a raw append-only event stream for future analytics.

## Milestones

- [x] Step 1 — Create the LingoCafe schema migration

  - Create: `knex/migrations/<timestamp>_lingocafe_schema.js`
  - Modify: `docs/backlog/drafts/OG12-lingocafe-data-model-and-knex-migrations/OG12.task.md` only if the final executable contract exposes a missing column, key, or naming mismatch discovered during migration authoring
  - Delete: n/a
  - Touch points: `CREATE SCHEMA`, `knex.schema.withSchema("lingocafe")`, `auth.users` foreign keys, composite key on `books_pages`, composite FK from `books_progress` to `books_pages(book_id, id)`, `published_at`, `progress_bps`, nullable `content`
  - Validation: Implemented in `knex/migrations/20260427150000_lingocafe_schema.js`; syntax-checked with `node --check`, then applied successfully with `make migrate`.
  - Notes: The final executable contract keeps the core schema explicit and limited to the agreed MVP tables.

- [x] Step 2 — Encode export-safe ordering and resume semantics
  - Achieve: Make the migration enforce the data guarantees that the reader app depends on: one stable reading-progress row per user/book, one ordered page sequence per book, and cascade behavior that invalidates progress when an exported page disappears.
  - Create: n/a
  - Modify: `knex/migrations/<timestamp>_lingocafe_schema.js`
  - Delete: n/a
  - Touch points: `unique(user_id, book_id)`, `unique(book_id, position)`, `onDelete("CASCADE")` for `books_progress` relations, nullable vs non-null page pointers, timestamp defaults
  - Validation: The migration now enforces `unique(user_id, book_id)`, `unique(book_id, position)`, `progress_bps` range checks, and cascading page invalidation through the `(book_id, page_id)` foreign key.
  - Notes: Structural page re-exports now invalidate stored resume state automatically, matching the product rule captured in the task.

  - Create: n/a
  - Modify: `knex/migrations/<timestamp>_lingocafe_schema.js`
  - Delete: n/a
  - Touch points: `id` PK, `user_id`, `created_at`, `event_at`, `name`, `book_id`, `page_id`, `data`, `meta`, JSONB defaults if chosen, partitioned parent-table DDL
