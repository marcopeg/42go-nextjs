# Point LingoCafe Book Covers to External Static Assets

## Elevator's Pitch

LingoCafe book covers should resolve from a configurable external static assets host using each book's project identifier. The default URL should be `https://assets.lingocafe.app/{project}/reader`.

## Business Gain

Book cover assets can be served from the same external static asset location as the generated book project output. That keeps the app lean and lets the content pipeline own cover delivery. Chuck Norris does not ship stale cover files. The CDN refreshes itself.

## Current State

`DG66` established local cover resolution through committed public assets under `/images/lingocafe/`, with fallback behavior when a book-specific cover is missing. This task completely replaces that local book-specific image approach for LingoCafe covers.

The current LingoCafe database model includes `project` on `lingocafe.books`, and the current reader API payload exposes `cover` and `coverFallback` fields for UI rendering. Shared cover rendering is used across the catalog, book detail, and reader surfaces through `BookCover`.

The current `lingocafe.books` table does not include a `cover` column. Older DG66 backlog notes mention `books.cover`, but that is stale context and must not be treated as current implementation input.

The operator now wants book covers to aim at the external static assets URL format `https://assets.lingocafe.app/{project}/reader`.

The external assets base path should be configurable through an environment variable such as `LC_ASSETS_BASE_PATH="https://assets.lingocafe.app"`. If the variable is not set, the default base path should be `https://assets.lingocafe.app`.

## Desired State

LingoCafe book cover resolution uses each book's `project` value and a configurable base path to build the external cover URL `{LC_ASSETS_BASE_PATH}/{project}/reader`.

`LC_ASSETS_BASE_PATH` defaults to `https://assets.lingocafe.app`, so the out-of-the-box URL remains `https://assets.lingocafe.app/{project}/reader`.

The generated external URL becomes the target for book covers wherever the book cover is shown. The old local `/images/lingocafe/<book-id>.jpg` convention should not remain in the fallback chain. Missing project values or failed external image loads should use the generic placeholder fallback.

## Definition of Success

Book covers shown in the LingoCafe catalog, book detail page, and reader surfaces request their image from `{LC_ASSETS_BASE_PATH}/{project}/reader` for each book project, use `https://assets.lingocafe.app` as the default base path, and fall back directly to the generic placeholder when the external image is unavailable or the project value is missing.

## Additional Context

The operator asked for LingoCafe book covers to aim at the external static assets URL and later clarified the final image path should be `{LC_ASSETS_BASE_PATH}/{project}/reader`.

The operator later added that `https://assets.lingocafe.app` should be a configurable environment variable like `LC_ASSETS_BASE_PATH="https://assets.lingocafe.app"`, with `https://assets.lingocafe.app` as the default value.

During refinement, the operator clarified that:

- `books.cover` is not part of the current LingoCafe data model and should not be included in this task.
- The old local book-specific image approach is fully replaced by this development.
- Failed external images and missing project values should use the generic placeholder fallback.
- The new external image path should use `{LC_ASSETS_BASE_PATH}/{project}/reader`.
- The CDN handles content negotiation for the extensionless `reader` resource.

This should build on the existing book cover pipeline rather than replacing the book UI surfaces.

## Assumptions

The `{project}` placeholder refers to the existing `lingocafe.books.project` value already exposed in the reader book payloads.

The external assets host is intended to be a public browser-accessible static host.

`LC_ASSETS_BASE_PATH` is the intended environment variable name unless implementation discovers an existing LingoCafe configuration naming pattern that should be reused.

The current generic placeholder cover can remain as the fallback.

If a book has no usable `project` value, the API or cover resolution path should return no generated external cover URL and let the UI use the generic placeholder.

## Constraints

Preserve the existing catalog, detail, and reader route behavior.

Preserve the `BookCover` fallback-on-error behavior.

Do not remove the ability to handle missing covers.

Do not keep the old local `/images/lingocafe/<book-id>.jpg` book-specific image convention as a fallback for this behavior.

If Next.js image optimization is still used for remote covers, configure the external image host deliberately.

Do not scatter the external URL template across unrelated UI components.

Do not require `LC_ASSETS_BASE_PATH` to be set for local development; the default should be applied when it is absent.

Run `npm run qa` after implementation.

- [ ] `LC_ASSETS_BASE_PATH` is supported as the configurable external assets base path.
- [ ] `LC_ASSETS_BASE_PATH` defaults to `https://assets.lingocafe.app` when the environment variable is not set.
- [ ] The `{project}` placeholder is populated from the book's existing `project` field.
- [ ] The implementation does not introduce or depend on a `lingocafe.books.cover` database column.
- [ ] The catalog `/books` surface uses the external cover URL.
- [ ] The book detail `/books/[bookId]` surface uses the external cover URL.
- [ ] The reader `/books/[bookId]/[pageId]` surface uses the external cover URL where it renders the book cover.
- [ ] Missing `project` values fall back to the generic placeholder.
- [ ] Failed external images fall back directly to the generic placeholder.
- [ ] The old local `/images/lingocafe/<book-id>.jpg` book-specific image convention is not used as a fallback path for the new cover behavior.
- [ ] The URL resolution logic is centralized in the existing book data or cover resolution path.
- [ ] Next.js remote image configuration supports `assets.lingocafe.app` if required by the implementation.
- [ ] Existing book navigation and reading behavior are not regressed.
- [ ] `npm run qa` passes after implementation.

## Dos

- Do use the existing `project` field as the source for the external asset path.
- Do use `LC_ASSETS_BASE_PATH` as the base path for generated external cover URLs.
- Do default `LC_ASSETS_BASE_PATH` to `https://assets.lingocafe.app`.
- Do keep cover URL construction deterministic and easy to test.
- Do preserve a clear generic placeholder fallback for failed remote image loads and missing project values.
- Do keep the cover behavior consistent across catalog, detail, and reader surfaces.
- Do update image host configuration if `next/image` requires it.

## Don'ts

- Don't hardcode per-book cover URLs by title or book id.
- Don't hardcode `https://assets.lingocafe.app` in a way that prevents environment-specific overrides.
- Don't introduce a new `lingocafe.books.cover` column for this work.
- Don't make the UI components each invent their own external cover URL.
- Don't remove existing cover fallback behavior.
- Don't keep the old local book-specific image path as a fallback stage.
- Don't change the book routes just to change cover asset resolution.
- Don't solve unrelated asset hosting or upload pipeline work in this task.
