# Build pipx-installable 42Go Events CLI

## Elevator's Pitch

Create a pipx-installable Python CLI for this repository, exposed as `42go`, and start its event tooling with `42go events pull` and `42go events query stats`.

## Business Gain

The operator can install the repository as a local command-line tool, pull production events during the day, and query local Parquet statistics without depending on Makefile internals or running heavy analytics against the production database.

## Current State

- `make events` runs `.agents/skills/42go-events-export/scripts/export_events.py`, preferring `.local/42go-events/.venv/bin/python` when the local venv exists.
- The exporter reads `events.events` incrementally from `EVENTS_DATABASE_URL` using `(created_at, id)` as cursor, writes monthly CSV and Parquet pairs under `.local/42go-events/events/`, and advances `state.json` only after touched Parquet files pass a DuckDB smoke read.
- The local archive currently contains paired files for `events_202605` and `events_202606`; CSV and Parquet row counts match at 545 and 436 rows respectively, for 981 unique events total.
- The Parquet schema mirrors the event table columns: `created_at`, `id`, `app_id`, `user_id`, `event_at`, `name`, `data`, and `meta`; `data` and `meta` are JSON strings in local files.
- The database migration `knex/migrations/20260511143000_core_events_schema.js` defines `events.events` as a monthly range-partitioned PostgreSQL table by `created_at` with indexes on `(app_id, created_at, id)` and `(app_id, name, created_at DESC)`.
- The existing `analyze_events.py wau` command proves DuckDB can read local Parquet files and compute weekly active users, but it is only a smoke query and not yet a reusable analytics system.
- The local venv already has `duckdb`, `pyarrow`, `psycopg`, and `psycopg-binary` installed from `.agents/skills/42go-events-export/requirements.txt`.
- No existing `.agents/skills/42go-cli` skill was found during refinement; creating or updating that skill is part of this task.
- Sibling/local CLI examples exist and should guide the shape:
  - `/Users/marcopeg/dv/ImageMagik/pyproject.toml` exposes a Typer CLI through `[project.scripts]`.
  - `/Users/marcopeg/dv/LingoCafe/books/pyproject.toml` exposes `gut = "gutenberg.cli:cli"` with a `src/` package and Typer command tree.
  - `/Users/marcopeg/dv/LingoCafe/cli/pyproject.toml` exposes `lg = "lingocafe_cli.cli:cli"` with Typer and repository-root helpers.

## Desired State

- A Python package exists under `cli/` with a `pyproject.toml`, `src/` package layout, and `[project.scripts]` entry point exposing a `42go` executable from the `42go-cli` distribution package.
- The package can be installed locally with pipx, for example from this checkout or by pulling this repo, following the same style as the existing ImageMagik and LingoCafe CLI projects.
- The CLI uses a Typer-style command tree:
  - `42go events pull`
  - `42go events query stats`
- `42go events pull` incorporates the current `make events` behavior by running the existing export logic that downloads new `events.events` rows into monthly CSV and Parquet files.
- `42go events query stats` reads the local Parquet archive and prints human-readable high-level live stats, including covered months, total events, distinct ids, app count, user count, event-name count, date range, and per-month counts.
- The event tooling has reusable Python helpers for archive path resolution, Parquet globbing, DuckDB access, CSV/Parquet parity checks, and JSON payload querying where useful later.
- The CLI is fully navigable by an agent from built-in help: invoking bare `42go` displays help, and every command/subcommand supports the standard `--help` option with useful command descriptions and parameter documentation.
- A `42go-cli` skill exists under `.agents/skills/` and is kept in sync with the implemented command tree. The skill contains a flat command index with brief explanations, and complex commands can have dedicated sub-documents with richer agent-facing documentation.
- The implementation chooses Parquet plus DuckDB as the primary analytics path, with CSV retained for human inspection and parity checks.
- The design leaves a clear later path for more `42go events query ...` commands and future aggregate rebuild/publish commands.

## Definition of Success

- A future implementer can install the repo CLI with pipx and run `42go --help`.
- Running bare `42go` prints the root help instead of failing silently or requiring a command.
- `42go events --help`, `42go events pull --help`, and `42go events query stats --help` work.
- `42go events pull` can replace the practical use of `make events` for exporting fresh production events into `.local/42go-events/events/`.
- `42go events query stats` proves the CLI can read the Parquet files and compute live statistics without querying PostgreSQL.
- The CLI help tree is complete enough that an agent can discover available commands and parameters by walking `42go --help` and nested `--help` output.
- `.agents/skills/42go-cli/SKILL.md` exists and documents the current command index.
- The recommended approach supports repeated same-day runs without duplicating raw events or corrupting exporter state.
- The first query command creates the foundation for later metrics such as weekly active users, daily active users, event counts by name, reader sessions, translation usage, and book engagement.
- The raw `.local/42go-events/events/` archive remains read-only from the analytics perspective except for the existing exporter.

## Additional Context

- Raw export should stay separate from analytics. The exporter is already responsible for downloading and normalizing production rows into monthly CSV and Parquet files.
- Use `created_at` for export cursor and local partition names. Use `event_at` for behavioral analytics.
- Current local data summary from Parquet: 981 rows, 981 distinct ids, 1 app (`lingocafe`), 20 users, 17 event names.
- Top event names currently include `page.scroll`, `page.translate`, `page.open`, `read.settings.changed`, `book.info`, `user.login`, and auth/profile/consent events.
- The long-term goal is to compute rolling and historical product metrics locally from appended event archives and eventually expose those metrics through JSON files or production-facing storage.
- Name the console command `42go` unless implementation discovers that Python packaging or shell compatibility makes that unsafe. Numeric console script names are allowed in practice, but this should still be verified with pipx.
- The first query command is intentionally small. It is a probe with teeth: it validates installability, command structure, Parquet access, and output shape.
- Clarification v1 resolved that the package belongs in `cli/`, `make events` should become a wrapper around `42go events pull`, the current export/query scripts should be cleaned up into the CLI shape, and `query stats` should output human-readable text.
- Clarification v2 resolved that the Python distribution name should be `42go-cli`, the console command should remain `42go`, old event scripts should move fully into CLI-owned logic, `query stats` should include per-month counts, the CLI must be help-navigable for agents, and a `42go-cli` skill must be created or kept updated.

## Assumptions

- The CLI package should live under `cli/`, following the nearby `/Users/marcopeg/dv/LingoCafe/cli` and `/Users/marcopeg/dv/LingoCafe/books` patterns.
- The package distribution name should be `42go-cli`, while the console script remains `42go`.
- The package should move event export/query logic from `.agents/skills/42go-events-export/scripts/export_events.py` and `.agents/skills/42go-events-export/scripts/analyze_events.py` into CLI-owned modules cleanly rather than duplicating it blindly.
- The current Makefile `events` target should become a compatibility wrapper around `42go events pull` after the CLI exists.
- Typer is the preferred CLI framework because the sibling projects already use it.
- The Typer app should use no-args-is-help behavior at the root and subcommand levels where useful.
- DuckDB is the best default query engine for local Parquet analytics because it can query file globs directly, handle JSON extraction, and avoid loading everything into memory.
- PyArrow remains useful for Parquet schema inspection and parity checks, but metric SQL should usually run through DuckDB.
- Aggregated outputs should not be committed unless refinement explicitly chooses repo-tracked sample fixtures or documentation examples.
- The first `42go-cli` skill version can live at `.agents/skills/42go-cli/SKILL.md`; richer command documentation can live in a local subfolder such as `.agents/skills/42go-cli/references/` or `.agents/skills/42go-cli/commands/` when command complexity justifies it.

## Constraints

- Do not run heavy analytics against the production database.
- Do not delete, truncate, retain, or clean up source production events as part of analytics.
- Do not mutate raw CSV or Parquet archive files from the analytics pipeline.
- Do not commit `.local/42go-events` data or credentials.
- Preserve `make events` compatibility by making it delegate to `42go events pull`.
- The CLI must be installable with pipx from the repository.
- The CLI must be discoverable through standard help behavior; do not require agents to know hidden command names.
- The `42go-cli` skill must be updated in the same task when commands or help text are added.
- Keep production credentials in environment variables or `.env` loading behavior. Do not bake secrets into CLI config.
- Keep raw exporter state (`events/state.json`, `manifest.jsonl`, and `inflight.json`) owned by the pull/export logic.
