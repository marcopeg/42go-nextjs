# Improve Backlog Draft Skill Scaffolding

## Elevator's Pitch

Improve the repo-local `backlog-draft` skill by moving deterministic draft scaffolding out of the markdown skill body and into Python. The target workflow is: the agent interprets the operator's rough input, performs any useful lookup, prepares structured task section content, calls a scaffold script, and receives the new TaskID plus clickable draft file path.

## Business Gain

Draft creation becomes more reliable and cheaper for agents to execute. Python owns repeatable mechanics such as IDs, folders, frontmatter, body sections, and backlog index placement. The skill stays lean and focuses on judgment: extracting the best possible draft from rough operator input. Chuck Norris does not babysit boilerplate. Boilerplate reports for duty.

## Current State

The local `backlog-draft` skill at `.agents/skills/backlog-draft/SKILL.md` currently carries a large amount of scaffold responsibility in its body.

It defines:

- when the skill should be invoked
- backlog storage paths
- TaskID rules
- filename rules
- frontmatter rules
- `BACKLOG.md` link conventions
- state consistency rules
- the full task body template
- post-creation steps
- chaining behavior into refinement

The support script at `.agents/skills/backlog-draft/scripts/generate_task_id.py` currently generates unused AA11-style IDs by scanning the backlog root. It does not scaffold folders, write frontmatter, create body sections, update `docs/backlog/BACKLOG.md`, or return a structured draft file path.

Shared backlog helpers already exist under `.agents/skills/backlog-core/scripts/`. In particular, `backlog_tool.py sync-indexes` can rebuild the active index from task folders, and `backlog_lib.py` already contains reusable logic for slugification, task file naming, task discovery, active task selection, and index rendering.

This means each agent using `backlog-draft` must recreate the same file structure and markdown template from skill text. That increases skill size, context load, and the chance of drift when the backlog format changes.

The current skill body also starts with invocation guidance. The operator's preference is that this skill is not optimized for human reading; skill-picking guidance should live in frontmatter `description` where the harness can use it. Human-oriented help, if needed, should live outside the LLM-loaded skill body, such as a README or non-loaded comments.

## Desired State

Implement a cleaner division of responsibilities for `backlog-draft`.

The repo-local Python scaffold flow should own deterministic work:

- generate the TaskID
- create the draft task folder using `<taskid>-<task-slug>`
- write `<taskid>.task.draft.md`
- write required frontmatter with `taskId`, `status`, `createdAt`, and `updatedAt`
- preserve optional supported frontmatter such as `group` when supplied
- write every standard body section
- use `information is missing` for missing section content
- update `docs/backlog/BACKLOG.md` under `## Drafts` if practical
- enforce or verify that the new task appears in exactly one active backlog section
- return the TaskID and draft path in a reliable agent-friendly format

The preferred agent workflow is to perform interpretation before calling the scaffold:

1. read the operator input
2. inspect relevant backlog files or sidecar notes when useful
3. derive the title, slug-relevant title text, group if clear, and section content
4. call the scaffold script with structured content for the task sections
5. verify the returned TaskID, task folder, draft path, and index state
6. report the TaskID and draft path
7. hand off to refinement only when requested

From an agentic platform point of view, preparing structured section content before scaffolding is likely more efficient than creating a placeholder file and heavily editing it afterward. It reduces rename churn, avoids fragile post-write path manipulation, and lets Python keep the filesystem and index operations atomic. The implementation should still choose a practical input channel for multi-section content, such as JSON via stdin or an input JSON file, instead of trying to pass large markdown bodies through shell arguments.

The scaffold output should be machine-readable and still easy to use in a final response. JSON is the preferred default unless implementation constraints argue otherwise. It should include at least:

- `taskId`
- `taskFolder`
- `draftPath`

It may also include:

- `backlogPath`
- `backlogEntry`
- `status`
- `createdAt`
- `updatedAt`

The `SKILL.md` body should become almost entirely operational. Invocation and trigger language should move mostly into the frontmatter `description`, because that is the likely harness-facing selection surface. If human-level help is needed, keep it outside the skill body in a dedicated README or other documentation that does not burden normal LLM context.

Scope is local only: update `.agents/skills/backlog-draft` in this repository. Do not update `/Users/marcopeg/.agents/skills/backlog-draft` in this task. The operator will copy changes over manually when satisfied.

## Definition of Success

`backlog-draft` has a smaller and clearer repo-local `SKILL.md`, and deterministic scaffold work lives in Python instead of markdown instructions. Creating a draft still produces the same or better backlog output: valid frontmatter, complete standard sections, correct folder/file naming, correct TaskID handling, correct backlog index placement, and correct reporting of TaskID plus draft path.

The task is successful only if the new workflow is easier for future agents to follow and harder to apply inconsistently.

## Additional Context

The current script name, `generate_task_id.py`, may no longer fit if the script becomes responsible for the full scaffold. The implementation should decide whether to extend it, rename it, or add a new script such as `scaffold_draft.py`.

The current `backlog-core` helper code may be reusable. The implementation should avoid duplicating index rendering and slug/path conventions if the shared helper functions can support the draft scaffold cleanly.

The project instructions mention `.agents/backlog`, but the live repository currently contains and uses `docs/backlog`. This mismatch should not move the backlog storage location as part of HW55. The final `backlog-draft` behavior should continue targeting the live `docs/backlog` system unless a separate migration task changes that storage model.
