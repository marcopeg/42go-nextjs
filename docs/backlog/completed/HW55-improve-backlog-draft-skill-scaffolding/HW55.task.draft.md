# Improve Backlog Draft Skill Scaffolding

## Elevator's Pitch

Improve the `backlog-draft` skill by investigating whether task scaffolding should move out of the markdown skill instructions and into the support Python script. The target flow is: run one scaffold command, receive the generated TaskID and draft file path, then let the skill focus on editing the generated draft using the operator input and its drafting logic.

## Business Gain

The draft workflow becomes smaller, stricter, and less repetitive. Agents spend less context remembering filesystem details and more effort writing a useful draft. Chuck Norris does not hand-type boilerplate. Boilerplate fills itself out and waits for orders.

## Current State

The local `backlog-draft` skill at `.agents/skills/backlog-draft/SKILL.md` currently carries a large amount of scaffold responsibility in its body.

It defines:

- when the skill should be invoked
- backlog storage paths
- TaskID rules
- filename rules
- frontmatter rules
- `BACKLOG.md` link conventions
- state consistency rules
- the full task body template
- post-creation steps
- chaining behavior into refinement

The support script at `.agents/skills/backlog-draft/scripts/generate_task_id.py` currently generates unused AA11-style IDs by scanning the backlog root. It does not scaffold folders, write frontmatter, create body sections, or return a structured draft file path.

This means every agent using the skill must recreate the same file structure and markdown template from skill text. That increases skill size, context load, and the chance of drift when the backlog format changes.

There is also a concern that the early body section of `SKILL.md` explains when to use the skill even though skill selection should be maximized through the skill frontmatter `description`. If the harness chooses skills primarily from the description, trigger wording may belong there instead of consuming body space.

## Desired State

Investigate and implement a cleaner division of responsibilities for `backlog-draft`.

The support Python script should be evaluated as the owner of deterministic scaffolding:

- generate the TaskID
- derive or accept a task slug/title input where appropriate
- create the draft task folder
- write the draft file with required frontmatter
- write every standard body section
- fill missing scaffold content with `information is missing`
- return the TaskID and draft file path in a machine-readable or easy-to-parse format

The skill should become thinner and focus on the value that requires agent judgment:

- run the scaffold command
- read the resulting file path and TaskID
- edit the generated file using the operator's input
- preserve the current drafting rules around uncertainty, missing information, assumptions, constraints, and open questions
- update or verify the backlog index according to the chosen script boundary
- hand off to refinement only when requested

The task should also review the skill description and first body section. The goal is to decide whether the "when to use this skill" material should be moved mostly or entirely into the frontmatter `description` so the harness can pick the skill more reliably while the body starts with operational workflow.

## Definition of Success

`backlog-draft` has a smaller and clearer `SKILL.md`, and deterministic scaffold work lives in Python instead of markdown instructions. Creating a draft still produces the same or better backlog output: valid frontmatter, complete standard sections, correct folder/file naming, correct TaskID handling, and correct reporting of TaskID plus draft path.

The task is successful only if the new workflow is easier for future agents to follow and harder to apply inconsistently.

## Additional Context

The current script name, `generate_task_id.py`, may no longer fit if the script becomes responsible for the full scaffold. The implementation should decide whether to extend it, rename it, or add a new script such as `scaffold_draft.py`.

The project instructions mention `.agents/backlog`, but the live repository currently contains and uses `docs/backlog`. This mismatch should be reviewed as part of the skill cleanup only if it affects the `backlog-draft` instructions. Do not move the backlog storage location as part of this task unless explicitly chosen during refinement.

The current skill's storage model says `docs/backlog`; the actual backlog tree also exists at `docs/backlog`.

## Assumptions

- The scaffold operation is deterministic enough to belong in Python.
- The agent should still write the best possible draft prose after the scaffold exists.
- The current task template sections should remain unless refinement decides to change the backlog format intentionally.
- The frontmatter `description` is likely important for skill selection by the harness.
- Moving invocation guidance into `description` may improve picking and reduce duplicated explanation in the body.
- Backlog index handling may either remain in the skill or move into the scaffold script, but the final boundary must be explicit.

## Constraints

- Preserve the existing backlog file format unless a deliberate format change is accepted.
- Preserve AA11-style TaskID generation.
- Preserve required frontmatter fields: `taskId`, `status`, `createdAt`, and `updatedAt`.
- Preserve draft status semantics.
- Preserve the standard body sections unless refinement changes the template.
- Do not create refined task files from `backlog-draft`.
- Do not let the script invent rich draft content from thin input.
- Keep generated placeholder text predictable.
- Keep the skill compatible with direct rough input, dictation, and partial ideas.

## Dos

- Do move deterministic file, frontmatter, path, and template mechanics into Python if the investigation confirms the boundary.
- Do keep the agent responsible for interpreting operator intent and improving the draft prose.
- Do make the script output easy to parse.
- Do keep the final skill short and operational.
- Do use the skill `description` for trigger language if that is how the harness chooses skills.
- Do preserve the existing task template unless there is an intentional refinement decision.
- Do document the final boundary between script-owned and skill-owned behavior.

## Don'ts

- Don't make the Python script responsible for creative task interpretation.
- Don't hide important behavioral rules so deeply in Python that agents cannot understand the workflow.
- Don't leave duplicate scaffold rules in both the script and the skill unless there is a clear reason.
- Don't break existing draft-and-refine chaining.
- Don't move backlog storage locations as a side effect of this cleanup.
- Don't rely on ad hoc string parsing if a structured script output is practical.

## Open Questions

- Should the script update `docs/backlog/BACKLOG.md`, or should the skill keep responsibility for index updates after editing the draft?
- Should the scaffold command accept a title and slug, or should it create a temporary slug that the skill may rename after deriving the final title?
- Should script output be JSON, plain key-value lines, or another format?
- Should the existing `generate_task_id.py` be replaced by a broader scaffold command, or kept as a small reusable helper?
- How much invocation text can safely move into the skill description without making the description too long?
- Does the harness actually use only the frontmatter `description` for skill picking, or are body headings also considered?
- Should this task update only the repo-local skill, or also the global fallback skill if it exists separately?
