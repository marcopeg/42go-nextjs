# Plan — Fix Reader Scroll Position Progress

## Goal

Make LingoCafe reader progress use the actual desktop and mobile scroll containers, persist low-volume scroll progress, and restore the saved position without letting `page-open` reset an already saved same-page position.

## Milestones

### Milestone 1 — Fix the Server Progress Contract

- [x] Step 1 — Add a read-progress lookup helper
  - Achieve: Let the page API fetch the authenticated user's existing `books_progress` row for the requested book without duplicating query logic in the route.
  - Create: n/a
  - Modify: `src/app/api/(lingocafe)/lingocafe/_lib/reader.ts`
  - Delete: n/a
  - Touch points: `BookProgressRow`, a new helper such as `loadBookProgress(userId, bookId)`, existing `getDB()` access pattern.
  - Validation: TypeScript accepts the helper return type as either the current book progress row or `null`.
  - Notes: The helper should only read progress for the authenticated user. Chuck Norris does not accept user IDs from browsers.

- [x] Step 2 — Make page-open progress persistence conditional
  - Achieve: Keep recording `page-open` every time, but update `books_progress` only when no record exists for the book or when the opened page differs from the saved page.
  - Create: n/a
  - Modify: `src/app/api/(lingocafe)/lingocafe/_lib/reader.ts`, `src/app/api/(lingocafe)/lingocafe/books/[bookId]/pages/[pageId]/route.ts`
  - Delete: n/a
  - Touch points: `saveBookProgress`, a new helper such as `saveBookOpenProgress`, `trackReaderEvent`, `getBookPage`.
  - Validation: Code path can distinguish these cases: no progress row creates `{ page_id: pageId, progress_bps: 0 }`; same-page row does not update; different-page row updates to `{ page_id: pageId, progress_bps: 0 }`; `page-open` event still writes in every case.
  - Notes: Prefer one helper that returns the effective progress after the conditional write so the route response can use it.

- [x] Step 3 — Return effective progress with page data
  - Achieve: Let the read page restore progress without depending only on the book detail `Continue reading` URL parameter.
  - Create: n/a
  - Modify: `src/app/api/(lingocafe)/lingocafe/_lib/reader.ts`, `src/app/api/(lingocafe)/lingocafe/books/[bookId]/pages/[pageId]/route.ts`, `src/app/(app)/(lingocafe)/books/_components/book-types.ts`
  - Delete: n/a
  - Touch points: `BookPageResponse`, `ReaderBookPage`, `mapBookPageDetail` or route-level response wrapping, progress shape such as `{ pageId, progressBps }`.
  - Validation: API response includes the effective progress for the requested page when available, and the client normalizer rejects malformed progress data safely.
  - Notes: Use this precedence during implementation: a valid URL `progress_bps` is an explicit override; otherwise restore persisted progress only when it belongs to the currently opened page. Direct navigation without `progress_bps` should restore same-page saved progress.
