# Store Event Exports in Monthly CSV and Parquet Files

## Elevator's Pitch

Change the `42go-events-export` local archive from one CSV and Parquet file per export run into monthly files that mirror the database's monthly `events.events_YYYYMM` partition structure.

## Business Gain

Monthly event files make local analytics easier to query, incrementally refresh, and reason about over time. They align the local archive with the database partition model instead of creating many small timestamped batch files.

## Current State

The `42go-events-export` skill downloads new `events.events` rows ordered by `(created_at, id)` and writes paired files named `batch-YYYYMMDDTHHMMSSZ.csv` and `batch-YYYYMMDDTHHMMSSZ.parquet`. The database stores events in the partitioned `events.events` table using monthly partitions named like `events_YYYYMM` over `created_at`.

## Desired State

Exported rows are grouped by their database partition month, using `created_at` as the month key. The archive root becomes `.local/42go-events`. Each run updates stable monthly CSV and Parquet files named like the database partitions, for example `events/csv/events_YYYYMM.csv` and `events/parquet/events_YYYYMM.parquet`.

If a monthly file already exists, the exporter merges existing rows with new rows, de-duplicates by event `id`, and rewrites the whole monthly file atomically. The raw archive mirrors the online `events.events` table by time. It does not introduce app-scoped raw files or directories; app-specific data architecture belongs in a later ETL layer fed by these raw monthly files.

## Definition of Success

The local `.local/42go-events` CSV and Parquet archive reflects the same monthly time structure and naming convention as the database event partitions. Repeated exports update existing monthly files when needed, create new monthly files for new months, delete old timestamped local batch files, and continue to advance the export cursor only after all affected files are valid.

## Additional Context

- Relevant code: `.agents/skills/42go-events-export/scripts/export_events.py`, `.agents/skills/42go-events-export/scripts/analyze_events.py`, `.agents/skills/42go-events-export/SKILL.md`, `.agents/skills/42go-events-export/README.md`.
- The operator referred to `.park` files; the existing implementation uses Parquet files with `.parquet` extension.
- The monthly database structure is defined in `knex/migrations/20260511143000_core_events_schema.js` through `events.events` partitioning by `created_at` and partitions named `events_YYYYMM`.
- The migration names partitions with `format('events_%s', to_char(v_start, 'YYYYMM'))`; local CSV and Parquet filenames should mimic that convention.

## Assumptions

- Monthly grouping should use `created_at`, because the export cursor and database partition range are based on `created_at`.
- `event_at` remains available for behavioral analytics but should not define the local file partition.
- CSV remains for human inspection and Parquet remains for DuckDB/Python analytics.
- The existing cursor model can continue to use `(created_at, id)`.
- The raw exporter should use one global timeline over `events.events`. App-specific filtering, app-specific raw folders, and curated analytics datasets are out of scope for this task.

## Constraints

- Do not delete, truncate, retain, or clean up source PostgreSQL events.
- Do not run heavy analytics against the source database.
- Do not advance `events/state.json` until every affected CSV and Parquet monthly file has been written and Parquet smoke checks pass.
- Do not introduce duplicate rows when appending to an existing month.
- Preserve `app_id` and all existing exported columns.
- Only delete local legacy timestamped `batch-*.csv` and `batch-*.parquet` files from the analytics archive. Never delete source database events.

## Acceptance Criteria

- The export script groups fetched rows by `created_at` month.
- The default archive root changes from `.local/42go-events-analytics` to `.local/42go-events`.
- The archive layout stores stable monthly CSV files named `events_YYYYMM.csv` instead of timestamped CSV batch files.
- The archive layout stores stable monthly Parquet files named `events_YYYYMM.parquet` instead of timestamped Parquet batch files.
- When a monthly file exists, new rows for that month are merged with existing rows and the whole monthly file is rewritten atomically.
- The append/merge path prevents duplicate event IDs in the monthly files.
- A single export run can update more than one month if fetched rows cross a month boundary.
- `events/state.json` advances only after every affected monthly CSV and Parquet file is valid.
- `manifest.jsonl` or equivalent metadata records which months/files were updated by an export run.
- The skill docs and README describe the monthly layout and the append semantics.
- `analyze_events.py` reads the new monthly Parquet layout.
- Local legacy timestamped `batch-*.csv` and `batch-*.parquet` files are deleted from the archive as part of the layout migration.
- The raw export remains time-partitioned only; it does not create app-scoped raw files or directories.
- Validation covers a dry run or fixture-style run that updates an existing month and creates a new month.

## Dos

- Use the database's `created_at` month as the local archive partition key.
- Use `.local/42go-events` as the default archive root.
- Use local file names that mirror the database partition naming convention: `events_YYYYMM`.
- Keep source database access read-only.
- Prefer atomic writes for rewritten monthly files.
- Keep CSV and Parquet content derived from the same normalized row set.

## Don'ts

- Do not keep creating one Parquet file per export activity.
- Do not keep creating one CSV file per export activity.
- Do not use `event_at` as the file month unless refinement explicitly changes that decision.
- Do not advance the cursor after only one file format succeeds.
- Do not introduce app-scoped raw archive directories or per-app raw export files.
- Do not commit `.local` analytics files.

## Resolved Decisions

- Monthly CSV and Parquet filenames should match the database partition naming strategy: `events_YYYYMM`.
- Monthly Parquet output should stay as one file per month and be rewritten atomically on append/merge.
- Old timestamped local `batch-*` CSV and Parquet files should be deleted.
- The raw archive should mirror the online table by time only. Further app-specific or curated data architecture should be handled by independent ETL from the raw monthly files.
